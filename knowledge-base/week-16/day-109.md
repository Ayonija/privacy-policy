## DSA (2 hours)

### Pattern: Hybrid DP — Sorted State Space + Binary Search

**Core idea:**
Complex DP problems often combine pattern recognition (what to track in state) with algorithmic tools (binary search to find valid transitions efficiently). Today's problems all reduce to: "sort the state space so DP transitions are monotone, then use binary search to find the optimal predecessor."

**Trigger condition:**
- "Longest chain where each step adds/removes one character" → LCS-like, sort by length
- "Min replacements to make array strictly increasing using values from arr2" → DP states = (position, last_value); binary search for smallest valid replacement
- "Max nodes reachable by jumping with height constraint" → sort by height, DP from short to tall

**Complexity:**
String Chain: O(W × L²) where W = words, L = word length | Strictly Increasing: O(n × m × log m) | Jump Game V: O(n × d)

### Full Solution Walkthrough — Longest String Chain (LC 1048) — Revision

**Problem:** Find the longest chain of words where each word is a "predecessor" of the next (adding exactly one letter anywhere gives the next word).

**Approach:** Sort by word length. For each word, try removing each character to find the predecessor. Use a hash map `dp[word]` = length of longest chain ending at word.

```java
import java.util.*;

class Solution {
    public static int longestStrChain(String[] words) {
        Arrays.sort(words, (a, b) -> a.length() - b.length()); // process shorter words first
        Map<String, Integer> dp = new HashMap<>();

        int result = 1;
        for (String word : words) {
            dp.put(word, 1); // minimum: just this word alone
            for (int i = 0; i < word.length(); i++) {
                String pred = word.substring(0, i) + word.substring(i + 1); // remove character at position i
                if (dp.containsKey(pred)) {
                    dp.put(word, Math.max(dp.get(word), dp.get(pred) + 1));
                }
            }
            result = Math.max(result, dp.get(word));
        }

        return result;
    }
}
```

**Why sort by length?** When processing word of length L, all predecessors (length L-1) are already computed. DP is well-defined.

**Complexity:** O(W × L) total work: W words, each tries L predecessor deletions, each deletion is O(L) string operation. With dict lookup O(L) for hashing → O(W × L²).

**Trace on `["a","ab","abc","bc","abd","abcd"]`:**
- Sort: a, ab, bc, abc, abd, abcd
- dp[a]=1, dp[ab]=2(ab→a), dp[bc]=1(no pred), dp[abc]=3(abc→ab), dp[abd]=3(abd→ab), dp[abcd]=max(abcd→abc=4, abcd→abd=4) → 4

### Full Solution Walkthrough — Jump Game V (LC 1340) — Hard

**Problem:** Given `arr` and `d`. From index i, you can jump to j if: `|i-j| ≤ d` AND `arr[k] < arr[i]` for all k between i and j (exclusive). Find max number of indices you can visit.

**Key insight:** Process indices in order of increasing height. When processing index i, all indices j reachable from i have arr[j] < arr[i] and are already computed.

```java
import java.util.*;

class Solution {
    public static int maxJumps(int[] arr, int d) {
        int n = arr.length;
        int[] dp = new int[n]; // at minimum, can visit just index i
        Arrays.fill(dp, 1);

        // Sort indices by arr value (ascending)
        Integer[] indices = new Integer[n];
        for (int i = 0; i < n; i++) indices[i] = i;
        Arrays.sort(indices, (a, b) -> arr[a] - arr[b]);

        for (int i : indices) {
            // Jump right: j = i+1, i+2, ..., i+d
            for (int j = i + 1; j < Math.min(i + d + 1, n); j++) {
                if (arr[j] >= arr[i]) break; // can't jump over taller or equal element
                dp[i] = Math.max(dp[i], dp[j] + 1);
            }
            // Jump left: j = i-1, i-2, ..., i-d
            for (int j = i - 1; j > Math.max(i - d - 1, -1); j--) {
                if (arr[j] >= arr[i]) break; // can't jump over taller or equal element
                dp[i] = Math.max(dp[i], dp[j] + 1);
            }
        }

        int ans = 0;
        for (int x : dp) ans = Math.max(ans, x);
        return ans;
    }
}
```

**Why break on arr[j] >= arr[i]?** The jump rule requires ALL elements between i and j to be strictly less than arr[i]. If arr[j] ≥ arr[i], we can't jump past it — and we can't jump to any further index because the path is blocked.

**Why process in ascending height order?** When computing dp[i], any j reachable from i has arr[j] < arr[i] (strictly), so dp[j] is already finalized when we reach arr[i] in sorted order. No circular dependency.

### Full Solution Walkthrough — Make Array Strictly Increasing (LC 1187) — Hard

**Problem:** Replace minimum elements of arr1 to make it strictly increasing. Replacements must come from arr2 (can use any value from arr2 at any position).

**State:** `dp` = dictionary mapping `last_value → min_operations` to make the current prefix strictly increasing ending with that value.

**Transition for each arr1[i]:**
- **Keep arr1[i]:** if arr1[i] > prev_value → valid; ops unchanged
- **Replace with arr2[j]:** use the smallest arr2[j] > prev_value (greedy: smaller last value = more flexibility for future); ops += 1

```java
import java.util.*;

class Solution {
    public static int makeArrayIncreasing(int[] arr1, int[] arr2) {
        // Sort and deduplicate arr2
        TreeSet<Integer> set = new TreeSet<>();
        for (int x : arr2) set.add(x);
        int[] sorted2 = new int[set.size()];
        int idx = 0;
        for (int x : set) sorted2[idx++] = x;
        int m = sorted2.length;

        // dp maps: last_placed_value -> min_operations_so_far
        Map<Integer, Integer> dp = new HashMap<>();
        dp.put(-1, 0); // start: no element placed yet, prev value = -1 (sentinel)

        for (int val : arr1) {
            Map<Integer, Integer> newDp = new HashMap<>();
            for (Map.Entry<Integer, Integer> e : dp.entrySet()) {
                int prev = e.getKey(), ops = e.getValue();
                // Option 1: keep arr1[i] (no replacement)
                if (val > prev) {
                    if (!newDp.containsKey(val) || newDp.get(val) > ops)
                        newDp.put(val, ops);
                }
                // Option 2: replace with smallest sorted2[j] > prev
                int pos = Arrays.binarySearch(sorted2, prev + 1);
                if (pos < 0) pos = -(pos + 1);
                if (pos < m) {
                    int replaceVal = sorted2[pos];
                    int newOps = ops + 1;
                    if (!newDp.containsKey(replaceVal) || newDp.get(replaceVal) > newOps)
                        newDp.put(replaceVal, newOps);
                }
            }
            dp = newDp;
            if (dp.isEmpty()) return -1; // impossible
        }

        int ans = Integer.MAX_VALUE;
        for (int v : dp.values()) ans = Math.min(ans, v);
        return ans;
    }
}
```

**Why only smallest arr2[j] > prev?** Greedy: among all valid replacements for the current position, the smallest one leaves the most room for future positions (smaller last value = more values are valid next time). Using a larger replacement never helps.

**Complexity:** O(n × m) states in the worst case (n positions, m possible values), each with O(log m) binary search. Total O(n × m × log m).

**Trace on `arr1=[1,5,3,6,7], arr2=[1,3,2,4]`:**
- arr2 sorted: [1,2,3,4]
- Start: dp={-1:0}
- val=1: keep (1>-1): dp={1:0}; replace(smallest arr2>-1=1): dp={1: min(0,1)}={1:0}
- val=5: keep(5>1): dp={5:0}; replace(arr2>1=2): dp={5:0, 2:1}
- val=3: keep only if 3>prev. From 5: 3<5 invalid; from 2: 3>2 keep {3:1}; replace from 5: arr2>5=None; from 2: arr2>2=3, cost 2 → dp={3:1} (min)
- ... (continue; answer = 1)

## System Design (30 min)

### Topic: Rate Limiting — Redis Lua Implementation and Global vs. Per-User Limits

**Atomic rate limiting with Lua:**
Redis executes Lua scripts atomically — no interleaved commands from other clients. This prevents race conditions in check-then-set patterns.

```lua
-- Token bucket via Lua (called via EVALSHA)
local key_tokens = KEYS[1]       -- e.g., "ratelimit:tokens:user123"
local key_time = KEYS[2]         -- e.g., "ratelimit:time:user123"
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])   -- tokens per second
local now = tonumber(ARGV[3])    -- current Unix timestamp (ms)

local tokens = tonumber(redis.call('get', key_tokens)) or capacity
local last_time = tonumber(redis.call('get', key_time)) or now

-- Refill tokens since last request
local elapsed = (now - last_time) / 1000.0
tokens = math.min(capacity, tokens + elapsed * rate)

if tokens >= 1.0 then
    tokens = tokens - 1
    redis.call('set', key_tokens, tokens)
    redis.call('set', key_time, now)
    redis.call('expire', key_tokens, 3600)
    redis.call('expire', key_time, 3600)
    return 1  -- allowed
else
    return 0  -- denied
end
```

**Limit hierarchy:**
| Level | Key pattern | Purpose |
|-------|------------|---------|
| Global | `ratelimit:global` | Protect entire service from overload |
| Per-endpoint | `ratelimit:endpoint:{path}` | Expensive endpoints get tighter limits |
| Per-user | `ratelimit:user:{user_id}` | Fair usage per customer |
| Per-IP | `ratelimit:ip:{ip}` | Prevent anonymous abuse |

**Headers for clients:**
```
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1715340000
Retry-After: 30  (on 429 response)
```

**Interview talking point:**
*"Implement rate limiting in layers: global → endpoint → user → IP. This way, a DDoS from one IP doesn't exhaust the global limit, and one heavy user doesn't affect others. Use Redis Lua for atomicity. Always return rate limit headers so clients can implement smart backoff — sending `Retry-After` prevents clients from hammering you after a 429."*
