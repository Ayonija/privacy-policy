## DSA (2 hours)

### Pattern: LIS with Custom Ordering and Segment Tree Acceleration

**Core idea:**
LIS generalizes to any strict partial order, not just numeric increase. Key insight: if you sort by the ordering relation first, LIS DP on the sorted array gives the correct answer. For difference-bounded LIS (consecutive diff ≤ k), a segment tree accelerates from O(n²) to O(n log max).

**Trigger condition:**
- "Largest subset where every pair divides each other" → sort + LIS with divisibility check
- "Stack 3D boxes where each fits inside the next" → sort dimensions per box + sort boxes + LIS check all 3 dims
- "LIS where consecutive elements differ by at most k" → segment tree range max query

**Complexity:**
Divisible Subset: O(n²) | Cuboids: O(n² log n) or O(n²) | LIS II with segment tree: O(n log V) where V = max value

### Full Solution Walkthrough — Largest Divisible Subset (LC 368) — Revision

**Problem:** Find the largest subset where every pair (a, b) satisfies a|b or b|a.

**Key observation:** After sorting, transitivity of divisibility means: if a|b and b|c, then a|c. So we only need to check `nums[i] % nums[j] == 0` for consecutive elements in the subset.

```python
def largestDivisibleSubset(nums: list[int]) -> list[int]:
    nums.sort()
    n = len(nums)
    dp = [1] * n       # LIS length ending at i
    parent = [-1] * n  # for reconstruction

    for i in range(1, n):
        for j in range(i):
            if nums[i] % nums[j] == 0 and dp[j] + 1 > dp[i]:
                dp[i] = dp[j] + 1
                parent[i] = j

    # Find the index with maximum dp value
    max_len = max(dp)
    idx = dp.index(max_len)

    # Reconstruct the subset
    result = []
    while idx != -1:
        result.append(nums[idx])
        idx = parent[idx]

    return result[::-1]  # reverse to get ascending order
```

**Why sort first?** If nums[i] % nums[j] == 0 and j > i (nums[j] > nums[i] after sort), that's impossible since a larger number can't divide a smaller one (except 0). Sorting ensures the divisibility relation is one-directional.

**Trace on `[1, 2, 3, 4, 5, 6]`:**
- dp: [1, 2, 2, 3, 2, 3] — subset [1,2,4] (length 3) and [1,2,6] (length 3)
- Multiple valid answers, first found is returned

### Full Solution Walkthrough — Maximum Height by Stacking Cuboids (LC 1691) — Hard

**Problem:** Given cuboids with dimensions, find the maximum height of a stack. A cuboid A can be placed on B if A.w ≤ B.w AND A.d ≤ B.d AND A.h ≤ B.h (all three dimensions, after rotating optimally).

**Key insight:** Optimally rotate each cuboid so its dimensions are non-decreasing (w ≤ d ≤ h). Then sort all cuboids. Claim (provable by exchange argument): the optimal stack always has cuboids in sorted order. So run LIS on sorted cuboids checking all 3 dimensions.

```python
def maxHeight(cuboids: list[list[int]]) -> int:
    # Sort each cuboid's 3 dimensions (optimal rotation: smallest dim last is height)
    # Actually sort so w≤d≤h to normalize
    cuboids = [sorted(c) for c in cuboids]
    # Sort all cuboids lexicographically
    cuboids.sort()

    n = len(cuboids)
    # dp[i] = max stack height with cuboid i on top
    dp = [c[2] for c in cuboids]  # at minimum, just cuboid i's height alone

    for i in range(n):
        for j in range(i):
            # Cuboid j fits below cuboid i if all dims of j ≤ corresponding dims of i
            if (cuboids[j][0] <= cuboids[i][0] and
                    cuboids[j][1] <= cuboids[i][1] and
                    cuboids[j][2] <= cuboids[i][2]):
                dp[i] = max(dp[i], dp[j] + cuboids[i][2])

    return max(dp)
```

**Why does sorting dimensions work?** If cuboid A must fit inside B, we want to maximize the stack height. Sorting dimensions of each cuboid (w ≤ d ≤ h) and putting the largest dimension as "height" provably gives the maximum height for any single cuboid placed on top (proof by exchange argument — swapping the role of dimensions to maximize h always helps or doesn't hurt).

**Why does sorting all cuboids work?** If A can be placed on B, then after sorting dimensions, all 3 dimensions of A ≤ all 3 of B. This means the sorted order of B comes before A. So checking whether B comes before A in sorted order and all dims ≤ is both necessary and sufficient.

**Complexity:** O(n log n) sort + O(n²) DP.

### Full Solution Walkthrough — LIS II (LC 2407) — Hard

**Problem:** Find the length of the longest strictly increasing subsequence where the difference between consecutive elements is at most k.

**O(n²) DP (baseline):**
```python
def lengthOfLIS_slow(nums: list[int], k: int) -> int:
    n = len(nums)
    dp = [1] * n
    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i] <= nums[j] + k:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)
```

**O(n log V) with Segment Tree:**
Maintain segment tree where `tree[v]` = max LIS length achievable ending with value exactly v.
For each number nums[i]: query max in range [nums[i]-k, nums[i]-1], then update position nums[i].

```python
def lengthOfLIS(nums: list[int], k: int) -> int:
    max_val = max(nums)
    # Segment tree for range max query, point update
    size = max_val + 1
    tree = [0] * (2 * size)

    def update(pos: int, val: int) -> None:
        pos += size  # leaf node
        tree[pos] = max(tree[pos], val)
        while pos > 1:
            pos //= 2
            tree[pos] = max(tree[2 * pos], tree[2 * pos + 1])

    def query(l: int, r: int) -> int:
        # Max in [l, r] inclusive
        l += size
        r += size + 1
        result = 0
        while l < r:
            if l & 1:
                result = max(result, tree[l])
                l += 1
            if r & 1:
                r -= 1
                result = max(result, tree[r])
            l //= 2
            r //= 2
        return result

    ans = 0
    for num in nums:
        lo = max(0, num - k)  # diff constraint: prev value in [num-k, num-1]
        best = query(lo, num - 1)
        curr = best + 1
        update(num, curr)
        ans = max(ans, curr)

    return ans
```

**Complexity:** O(n log V) where V = max value. Each of n numbers does O(log V) query + O(log V) update.

**When to use segment tree vs. patience sort:**
- Standard LIS (diff = any): patience sort O(n log n)
- LIS with diff ≤ k: segment tree O(n log V)
- LIS with custom comparison (divisibility, envelopes): patience sort after sort

## System Design (30 min)

### Topic: GraphQL — Federation, Schema Stitching, and Persisted Queries

**Schema Federation (Apollo):**
Multiple GraphQL services each expose their own schema. A "gateway" composes them into a unified supergraph. Clients query the gateway without knowing which service owns which type.

```graphql
# User Service — defines User
type User @key(fields: "id") {
  id: ID!
  name: String!
}

# Order Service — extends User with orders
extend type User @key(fields: "id") {
  id: ID! @external
  orders: [Order!]!
}
```

**DataLoader patterns for federation:**
- Each service receives batch requests for entities by key (the `@key` field)
- Gateway calls `User Service: {ids: [1,2,3]}` → returns 3 users → passes to Order resolver

**Persisted Queries (APQ):**
- Problem: GraphQL queries can be large and are not cacheable (all POSTs)
- APQ: client sends hash of query on first request; server caches query by hash; subsequent requests send only hash
- Benefits: smaller payloads (hash vs. full query text), CDN caching of GET requests with hash in URL

**Security considerations:**
- **Depth limiting:** prevent deeply nested queries that cause N^depth DB hits
- **Query complexity:** assign cost to each field; reject if total cost > threshold
- **Rate limiting by complexity:** don't rate-limit by request count alone — a single expensive query can be worse than 100 simple ones

**Interview talking point:**
*"For a large system with multiple teams, Apollo Federation lets each team own their GraphQL schema independently. The gateway stitches them automatically. The key performance tool is persisted queries — they reduce payload size and enable CDN caching, solving the biggest GraphQL caching challenge. For security, implement query depth limiting (max 5–7 levels) and complexity scoring to prevent abuse."*
