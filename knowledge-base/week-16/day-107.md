## DSA (2 hours)

### Pattern: LIS with Counting and 2D Applications

**Core idea:**
Standard LIS gives length. Counting LIS requires a parallel array: `cnt[i]` = number of LIS of length `dp[i]` ending at index i. Mountain Array removal uses LIS from both ends at every potential peak. Delete Columns reduces to finding the max columns that form a row-wise non-decreasing sequence, which is LIS on column compatibility.

**Trigger condition:**
- "How many LIS are there?" → parallel cnt[] array
- "Make array mountain: strictly up then strictly down" → LIS from left + LIS from right at each index
- "Max columns to keep so each row is non-decreasing" → LIS on columns (column i before column j if all rows satisfy strs[row][i] ≤ strs[row][j])

**Complexity:**
Number of LIS: O(n²) | Mountain Array: O(n log n) | Delete Columns: O(m² × n)

### Full Solution Walkthrough — Number of LIS (LC 673) — Revision

**Problem:** Count the total number of longest increasing subsequences in nums.

**Parallel arrays:**
- `dp[i]` = length of LIS ending at index i
- `cnt[i]` = number of LIS of that length ending at index i

**Recurrence:**
```
For j < i where nums[j] < nums[i]:
    if dp[j] + 1 > dp[i]:   # found a longer LIS ending at i
        dp[i] = dp[j] + 1
        cnt[i] = cnt[j]      # reset count
    elif dp[j] + 1 == dp[i]: # tied length — add those ways
        cnt[i] += cnt[j]
```

```python
def findNumberOfLIS(nums: list[int]) -> int:
    n = len(nums)
    dp = [1] * n    # LIS length ending at i
    cnt = [1] * n   # count of such LIS

    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i]:
                if dp[j] + 1 > dp[i]:
                    dp[i] = dp[j] + 1
                    cnt[i] = cnt[j]
                elif dp[j] + 1 == dp[i]:
                    cnt[i] += cnt[j]

    max_len = max(dp)
    return sum(cnt[i] for i in range(n) if dp[i] == max_len)
```

**Trace on `[1, 3, 5, 4, 7]`:**
```
i=0: dp=[1], cnt=[1]
i=1: nums[0]<nums[1] → dp[1]=2, cnt[1]=1
i=2: nums[1]<nums[2] → dp[2]=3, cnt[2]=1
i=3: nums[0]<nums[3]: dp[3]=2,cnt[3]=1; nums[1]<nums[3]: dp=3,cnt=1
i=4: from i=2(dp=3→dp[4]=4,cnt=1); from i=3(dp=3→tied,cnt+=1) → cnt[4]=2
max_len=4, sum = cnt[4] = 2  (sequences: 1,3,5,7 and 1,3,4,7)
```

### Full Solution Walkthrough — Minimum Removals to Make Mountain Array (LC 1671) — Hard

**Problem:** Make array a "mountain" by removing minimum elements. Mountain: strictly increases to peak, then strictly decreases. Peak must not be at ends.

**Approach:** Find the peak index k that maximizes `lis_left[k] + lis_right[k] - 1` (subtract 1 to avoid counting the peak twice). Both sides must have at least 2 elements (lis_left[k] > 1 and lis_right[k] > 1 — peak can't be at an edge).

```python
from bisect import bisect_left

def minimumMountainRemovals(nums: list[int]) -> int:
    n = len(nums)

    def compute_lis_lengths(arr: list[int]) -> list[int]:
        tails = []
        result = []
        for num in arr:
            pos = bisect_left(tails, num)
            if pos == len(tails):
                tails.append(num)
            else:
                tails[pos] = num
            result.append(pos + 1)  # LIS length at this position
        return result

    # LIS length ending at each index (left to right)
    left = compute_lis_lengths(nums)
    # LIS length starting at each index = LIS of reversed array (right to left)
    right = compute_lis_lengths(nums[::-1])[::-1]

    max_mountain = 0
    for k in range(1, n - 1):  # peak can't be at ends
        if left[k] > 1 and right[k] > 1:  # must have elements on both sides
            max_mountain = max(max_mountain, left[k] + right[k] - 1)

    return n - max_mountain
```

**Trace on `[2, 1, 1, 5, 6, 2, 3, 1]`:**
- left  = [1, 1, 1, 2, 3, 2, 3, 2]  (LIS ending at each index)
- right = [2, 1, 1, 3, 2, 1, 2, 1]  (LIS starting at each index)
- Valid peaks (both > 1): index 3 → 2+3-1=4; index 4 → 3+2-1=4
- max_mountain = 4; answer = 8 - 4 = 4 ✓

**Why reverse and flip?** LIS starting at index k (going right to left) = LIS of the reversed array ending at position n-1-k, then re-indexed back.

### Full Solution Walkthrough — Delete Columns to Make Sorted III (LC 960) — Hard

**Problem:** n strings of length m. Delete minimum columns so the remaining columns form lexicographically non-decreasing strings for every row (i.e., the concatenated remaining columns give sorted strings for all rows).

**LIS formulation:** Find the maximum number of columns to KEEP such that the kept columns form a "compatible" sequence. Column j is compatible after column i if for all rows r: `strs[r][i] ≤ strs[r][j]`.

```python
def minDeletionSize(strs: list[str]) -> int:
    n = len(strs)      # number of rows (strings)
    m = len(strs[0])   # number of columns

    # dp[j] = max columns we can keep with column j as the last kept column
    dp = [1] * m

    for j in range(m):
        for i in range(j):
            # Can column i come immediately before column j?
            if all(strs[row][i] <= strs[row][j] for row in range(n)):
                dp[j] = max(dp[j], dp[i] + 1)

    return m - max(dp)  # columns deleted = total - max kept
```

**Complexity:** O(m² × n) — m² pairs of columns, each checked across n rows.

**Why LIS?** We're finding the longest "increasing" sequence of columns, where "increasing" means lexicographically non-decreasing across all rows — exactly like LIS but with a custom comparison function.

## System Design (30 min)

### Topic: API Versioning Strategies and REST Best Practices

**Why version APIs?**
Breaking changes (remove fields, rename, change types) would break existing clients. Versioning lets you evolve the API while supporting old clients for a defined deprecation period.

**Versioning strategies:**

| Strategy | URL example | Pros | Cons |
|----------|------------|------|------|
| URL path | `/api/v1/users` | Explicit, cacheable, easily tested | Bookmarks break; harder to route |
| Query param | `/users?version=2` | Simple, URL stays clean | Non-standard; ignored by caches |
| Header | `Accept: application/vnd.api+json;version=1` | Clean URLs | Not testable in browser; hard to document |
| Subdomain | `v1.api.example.com` | Fully separate; independent scale | DNS overhead; hard to share auth |

**Best practices:**
1. **URL versioning** for public APIs — explicit, discoverable, CDN-cacheable
2. Support at least 2 versions simultaneously; deprecate with `Sunset` header: `Sunset: Sat, 31 Dec 2025 23:59:59 GMT`
3. **Non-breaking changes** (adding fields, endpoints) never require a version bump
4. **Breaking changes** (remove field, change type, redefine semantics) always require new version
5. Never remove a version without 6+ months notice to external partners

**REST error response structure:**
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "email field is required",
    "details": [{"field": "email", "issue": "missing"}],
    "requestId": "req-abc-123",
    "timestamp": "2025-05-10T10:00:00Z"
  }
}
```

**Interview talking point:**
*"For public APIs, I choose URL versioning because it's explicit, widely understood, and CDN-cacheable (the URL uniquely identifies the resource version). Internal APIs can use headers. I always include a requestId in error responses — it's the single most useful field for debugging production issues, allowing correlation across distributed traces."*
