## DSA (2 hours)

### Pattern: Tree DP — Post-order + Rerooting

Each node's DP state is computed from its children's states in a post-order DFS. For problems that ask for a value computed from every node as a root (not just node 0), a second top-down pass "reroots" the answer in O(n).

**Trigger condition:**
- Optimal substructure on a tree (choices at each node depend on subtree values)
- "What if every node were the root?" — rerooting eliminates O(n²) naive approach
- Node state depends on parent AND children (two-pass required)

**Complexity:**
- Time: O(n) for both post-order and rerooting passes
- Space: O(n) recursion stack + O(n) for auxiliary arrays

### Full Solution Walkthrough — House Robber III (LC 337) — Revision

**Problem:** Rob houses on a binary tree; cannot rob both a node and its parent. Maximize total stolen.

**Key invariant:** At each node, the optimal answer depends on whether you rob this node. Return both choices.

```java
class Solution {
    static class TreeNode { int val; TreeNode left, right; TreeNode(int val) { this.val = val; } }

    public int rob(TreeNode root) {
        int[] result = dp(root);
        return Math.max(result[0], result[1]);
    }

    // Returns [rob_this_node, skip_this_node]
    private int[] dp(TreeNode node) {
        if (node == null) return new int[]{0, 0};

        int[] left = dp(node.left);
        int[] right = dp(node.right);

        // Rob this node: cannot rob either child
        int robThis = node.val + left[1] + right[1];

        // Skip this node: take the best outcome from each child independently
        int skipThis = Math.max(left[0], left[1]) + Math.max(right[0], right[1]);

        return new int[]{robThis, skipThis};
    }
}
```

**Trace on example tree [3, 4, 5, 1, 3, None, 1]:**
- dp(1): (1, 0) — leaf
- dp(3): (3, 0) — leaf
- dp(4): rob=4+0+0=4, skip=max(1,0)+max(3,0)=4 → (4, 4)
- dp(1): (1, 0) — leaf (right child of 5)
- dp(5): rob=5+0+0=5, skip=max(0,0)+max(1,0)=1 → (5, 1)
- dp(3 root): rob=3+4+1=8, skip=max(4,4)+max(5,1)=4+5=9 → (8, 9) → answer=9

**Why return a tuple instead of a single value:** A single value dp(node) would require knowing the parent's choice before computing the child — a chicken-and-egg problem. Returning both choices breaks the dependency.

### Full Solution Walkthrough — Binary Tree Cameras (LC 968) — Hard

**Problem:** Place minimum cameras on binary tree nodes. Each camera covers the node itself, its parent, and its children.

**Three states:**

```java
class Solution {
    static class TreeNode { int val; TreeNode left, right; TreeNode(int val) { this.val = val; } }

    static final int UNCOVERED = 0, HAS_CAMERA = 1, COVERED = 2;
    int cameras = 0;

    public int minCameraCover(TreeNode root) {
        cameras = 0;
        // If root itself comes back UNCOVERED, no parent exists to cover it
        if (dp(root) == UNCOVERED) cameras++;
        return cameras;
    }

    // Returns: 0=UNCOVERED, 1=HAS_CAMERA, 2=COVERED
    private int dp(TreeNode node) {
        if (node == null) {
            // Null nodes are considered COVERED (don't need a camera)
            return COVERED;
        }

        int left = dp(node.left);
        int right = dp(node.right);

        // Rule 1: If any child is uncovered, we MUST place a camera here
        if (left == UNCOVERED || right == UNCOVERED) {
            cameras++;
            return HAS_CAMERA;
        }

        // Rule 2: If any child has a camera, this node is covered by that camera
        if (left == HAS_CAMERA || right == HAS_CAMERA) {
            return COVERED;
        }

        // Rule 3: Both children are covered but have no camera — this node is uncovered.
        // We pass responsibility to the parent (parent will decide to place a camera).
        return UNCOVERED;
    }
}
```

**Why null nodes return COVERED:** If a leaf's null child were UNCOVERED, every leaf would be forced to place a camera. Instead, null children are "safe" — coverage is only required for real nodes.

**State transition table:**

| Left | Right | Action |
|------|-------|--------|
| UNCOVERED | any | Place camera here → HAS_CAMERA |
| any | UNCOVERED | Place camera here → HAS_CAMERA |
| HAS_CAMERA | any | Covered by child → COVERED |
| any | HAS_CAMERA | Covered by child → COVERED |
| COVERED | COVERED | No camera nearby → UNCOVERED |

### Full Solution Walkthrough — Sum of Distances in Tree (LC 834) — Hard

**Problem:** For each node in an unweighted tree, find the sum of distances to all other nodes. Return an array.

**Rerooting key formula:** When we move the "root" from node `p` to its child `c`:
- `count[c]` nodes (the subtree of `c`) each get 1 step closer → subtract `count[c]`
- `n - count[c]` nodes (everything outside `c`'s subtree) each get 1 step farther → add `n - count[c]`
- Therefore: `ans[c] = ans[p] - count[c] + (n - count[c])`

```java
import java.util.*;

class Solution {
    public int[] sumOfDistancesInTree(int n, int[][] edges) {
        Map<Integer, List<Integer>> graph = new HashMap<>();
        for (int i = 0; i < n; i++) graph.put(i, new ArrayList<>());
        for (int[] e : edges) {
            graph.get(e[0]).add(e[1]);
            graph.get(e[1]).add(e[0]);
        }

        int[] count = new int[n]; // count[i] = size of subtree rooted at i (when rooted at 0)
        int[] ans = new int[n];   // ans[i] = sum of distances from i to all nodes in its subtree
        Arrays.fill(count, 1);

        // Post-order: compute count and ans[0] contribution
        dfs1(0, -1, graph, count, ans);
        // Top-down: reroot from parent to child using the formula
        dfs2(0, -1, n, graph, count, ans);
        return ans;
    }

    private void dfs1(int node, int parent, Map<Integer, List<Integer>> graph, int[] count, int[] ans) {
        for (int child : graph.get(node)) {
            if (child != parent) {
                dfs1(child, node, graph, count, ans);
                count[node] += count[child];
                // Adding child's subtree: each of count[child] nodes is 1 step farther from node
                ans[node] += ans[child] + count[child];
            }
        }
    }

    private void dfs2(int node, int parent, int n, Map<Integer, List<Integer>> graph, int[] count, int[] ans) {
        for (int child : graph.get(node)) {
            if (child != parent) {
                ans[child] = ans[node] - count[child] + (n - count[child]);
                dfs2(child, node, n, graph, count, ans);
            }
        }
    }
}
```

**Trace for n=6, edges forming a path 0-1-2-3-4-5:**
- After dfs1: count=[6,5,4,3,2,1], ans[0]=0+1+2+3+4+5=15
- dfs2: ans[1] = 15 - 5 + (6-5) = 15-5+1 = 11
- ans[2] = 11 - 4 + (6-4) = 11-4+2 = 9
- ans[3] = 9-3+(6-3) = 9-3+3 = 9 ... correct (by symmetry, 3 is center)

## System Design (30 min)

### Topic: Consistent Hashing — Ring Structure and Virtual Nodes

**Core idea:** Map both servers and keys onto a hash ring [0, 2^32). Each key is owned by the first server clockwise from its hash position.

**Why not simple modular hashing?**
- With `server = hash(key) % n`, adding/removing a server reassigns ~all keys → massive cache misses
- Consistent hashing reassigns only ~K/n keys on average when one server changes

**Ring implementation:**

```java
import java.security.MessageDigest;
import java.util.SortedMap;
import java.util.TreeMap;

class ConsistentHashRing {
    private final int virtualNodes;
    private final SortedMap<Long, String> ring = new TreeMap<>();

    public ConsistentHashRing(int virtualNodes) {
        this.virtualNodes = virtualNodes;
    }

    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] digest = md.digest(key.getBytes());
            long h = 0;
            for (int i = 0; i < 4; i++) h = (h << 8) | (digest[i] & 0xFFL);
            return h & 0xFFFFFFFFL; // keep in [0, 2^32)
        } catch (Exception e) { return key.hashCode() & 0xFFFFFFFFL; }
    }

    public void addServer(String serverId) {
        for (int i = 0; i < virtualNodes; i++) {
            long pos = hash(serverId + "#" + i);
            ring.put(pos, serverId);
        }
    }

    public void removeServer(String serverId) {
        for (int i = 0; i < virtualNodes; i++) {
            long pos = hash(serverId + "#" + i);
            ring.remove(pos);
        }
    }

    public String getServer(String key) {
        long pos = hash(key);
        SortedMap<Long, String> tail = ring.tailMap(pos);
        long nodePos = tail.isEmpty() ? ring.firstKey() : tail.firstKey();
        return ring.get(nodePos);
    }
}
```

**Virtual nodes — why they matter:**
- Without virtual nodes: servers may own large or tiny arcs of the ring depending on their hash → load imbalance
- With V=150 virtual nodes per server: each server's load deviates by <5% standard deviation from ideal
- Each physical node owns V arc segments, spread across the ring

**Key properties:**
- `getServer`: O(log(n × V)) via binary search on sorted positions
- `addServer`: O(V log(nV)) — insert V new positions
- On add/remove: only the successor node's keys are affected (on average K/n keys)

**Interview talking point:**
*"Consistent hashing ensures that when we add a server to a 10-node cluster, only 1/11 of keys need to be remapped instead of ~90%. With 150 virtual nodes per server, load variance stays under 5%. I'd use a sorted map with binary search for the ring and MD5 or xxHash for position hashing."*

### Tree DP — 2-State (House Robber III)

**Situation:** "I was given House Robber III — maximize robbery on a binary tree with no adjacent nodes robbed. A single-value dp(node) function doesn't work because the answer at any node depends on whether the parent chose to rob — creating a circular dependency."

**Task:** "My goal was to break the dependency by returning a pair from dp(node): (rob_this, skip_this), so the parent can choose optimally without requiring dp to know the parent's choice."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "This is Tree DP with a 2-state tuple return. The key insight is that any node's children are independent of each other — only the parent-child edge creates a constraint. Return both choices and let the parent decide."
2. *Initialize:* "Base case: dp(null) = (0, 0) — null nodes contribute nothing whether robbed or not. The pair propagates bottom-up."
3. *Core loop logic:* "At each node: rob_this = node.val + skip_left + skip_right. skip_this = max(rob_left, skip_left) + max(rob_right, skip_right). The left and right subtrees combine independently."
4. *Convergence guarantee:* "Post-order DFS — each node is processed exactly once after both children. O(n) time, O(h) space for recursion where h = tree height."
5. *Duplicate handling / edge case proactivity:* "The root has no parent. After the root call, take max(rob_root, skip_root) — the root is free to be either robbed or not."

**Result:** "This gives O(n) time (single DFS pass) vs. O(2^n) for brute force trying all subsets of nodes. For n=10,000 nodes, that's 10,000 operations vs. 10^3010."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer tuple-return DP here |
|-------------|---------------------------|---------------------------------|
| Memoized map from node to single int | Works if you track parent state | More complex — must thread parent's choice through call stack |
| BFS level-order | Only for perfect binary trees | Can't handle arbitrary tree structure |

### Tree DP — 3-State (Binary Tree Cameras)

**Situation:** "Binary Tree Cameras: place minimum cameras; each covers itself, parent, and children. A 2-state approach (has_camera / no_camera) fails because a no-camera node could be covered by its child OR its parent — ambiguous, so placement decisions become non-local."

**Task:** "Use 3 states — UNCOVERED (0), HAS_CAMERA (1), COVERED (2) — to disambiguate exactly who is responsible for covering each node, allowing a single bottom-up pass with greedy-optimal decisions."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "Tree DP with 3-state return. The additional state (UNCOVERED vs. COVERED without a camera) lets the parent know whether it MUST place a camera. This turns a global optimization into a local greedy decision."
2. *Initialize:* "Base case: null returns COVERED — null nodes don't need cameras. If they returned UNCOVERED, every leaf would be forced to place a camera unnecessarily."
3. *Core loop logic:* "Rule 1 (highest priority): if any child returns UNCOVERED, MUST place camera here → cameras++, return HAS_CAMERA. Rule 2: if any child has a camera, this node is covered → return COVERED. Rule 3 (default): both children are covered but have no camera → return UNCOVERED (defer to parent)."
4. *Convergence guarantee:* "Post-order. O(n) time. The 3-state machine is a greedy: never place a camera unless forced, because waiting for the parent is always at least as good as placing now."
5. *Duplicate handling / edge case proactivity:* "After the root call, if root returns UNCOVERED, no parent exists to cover it — add 1 more camera. This special-case is easy to forget."

**Result:** "Achieves the optimal (minimum) camera count in O(n) — the greedy is provably optimal because a camera at any node covers 3 nodes; deferring placement maximizes each camera's coverage."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer 3-state greedy DP here |
|-------------|---------------------------|-----------------------------------|
| Integer programming / brute force | Theoretical verification | Exponential — infeasible for large trees |
| 2-state (has_camera / no_camera) | Simpler problems | Ambiguous coverage source; requires backtracking to resolve |

### Rerooting DP (Sum of Distances in Tree)

**Situation:** "Sum of Distances in Tree: for each node, find the sum of distances to all other nodes. Naive: run BFS from every node → O(n²). For n=30,000, that's 900M operations — TLEs."

**Task:** "Use rerooting DP: one post-order pass computes the answer for node 0, then one top-down pass propagates the answer to all other nodes in O(1) each, giving O(n) total."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "Rerooting DP — when a problem asks for the same computation from every node as root, compute for one root and propagate via a relational formula instead of recomputing from scratch."
2. *Initialize:* "Pass 1 (post-order): count[node] = subtree size, ans[node] = sum of distances from node to all nodes in its subtree. Initialize count[i] = 1 for all i."
3. *Core loop logic:* "Pass 1: ans[node] += ans[child] + count[child] (adding child's subtree, each node is 1 farther than from child). Pass 2: ans[child] = ans[parent] - count[child] + (n - count[child]). Moving root from parent to child: count[child] nodes get 1 step closer (subtract), n-count[child] nodes get 1 step farther (add)."
4. *Convergence guarantee:* "Two DFS passes, each O(n). Space O(n) for count and ans arrays."
5. *Duplicate handling / edge case proactivity:* "The parent-tracking variable prevents revisiting the parent as a child. For n=1, ans=[0] trivially — handle by returning early."

**Result:** "O(n²) → O(n). For n=30,000 nodes: 900M → 30,000 operations. Easily fits within 1-second time limit."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer rerooting here |
|-------------|---------------------------|---------------------------|
| Naive BFS from every node | n ≤ 500 | O(n²) — feasible for small trees only |
| Centroid decomposition | Weighted edges, non-constant subtree queries | More complex; rerooting is simpler for uniform edge weights |

## Behavioral (30 min)

**Target LP:** Dive Deep (Amazon) / Be Curious (Meta)

**STAR Story: "Diagnosing a Hidden O(n²) Tree Traversal in Our Analytics Pipeline"**

**Situation (20%):** Our team's analytics pipeline computed per-node aggregates over a large organizational hierarchy (a tree of ~50,000 nodes representing departments and sub-teams). The pipeline had been slowing down for months; at peak it took 22 minutes to complete. Everyone assumed the database was the bottleneck since we had slow queries visible in monitoring. I wasn't satisfied with that assumption.

**Task:** I volunteered to own a deep performance investigation, with a goal of cutting the pipeline runtime to under 3 minutes without changing the underlying data model.

**Action (60-70%):**
1. I started by reading the actual code rather than trusting the dashboard. I added fine-grained timing around each stage using Python's `time.perf_counter`. The DB stage took 40 seconds. The tree computation stage took 21 minutes. The team had been looking at the wrong bottleneck entirely.
2. I traced the tree computation: for each of 50,000 nodes, the code called a `get_subtree_distance_sum(node)` function that re-traversed the entire subtree from scratch. This was O(n²) — exactly the pattern in Sum of Distances in Tree.
3. I mapped the problem to the rerooting DP pattern: one post-order pass to compute subtree sizes and the answer for the root, then one top-down pass to propagate answers to all nodes using the formula `ans[child] = ans[parent] - count[child] + (n - count[child])`. This reduces O(n²) to O(n).
4. I implemented the two-pass solution in Python, added unit tests on small synthetic trees to verify correctness (including the edge case of a single-node tree and a path graph), and then ran it against the full 50,000-node dataset in staging.
5. I documented the algorithmic change clearly in the PR description, including a complexity proof and a before/after timing table, so reviewers could approve with confidence rather than trust blindly.

**Result (10-20%):** The pipeline runtime dropped from 22 minutes to 47 seconds — a 28× speedup. The DB stage (40 seconds) was now the actual bottleneck, and we addressed it separately with query batching (bringing total runtime to 18 seconds). The deep-dive mindset — not trusting the first dashboard I saw — was what made the discovery possible. I shared the pattern with the team in a post-mortem and we now profile by stage before any perf investigation.

**Interview tip:** Dive Deep stories score highest when you demonstrate you went further than your role required and found the non-obvious root cause.
