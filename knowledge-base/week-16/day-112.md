## DSA (2 hours)

### Pattern: Bitmask DP — Bitmask as State

Represent a subset of n items (n ≤ 20) as a single integer. Each bit i is 1 if item i is included. Iterate over all 2^n masks; at each mask, try adding one more item or one more subset.

**Trigger condition:**
- n ≤ 20 (or very small) — exponential state space is feasible
- "Which items have been used/visited?" is the key state
- Permutations or assignments where order matters among selected items
- Partition problems where you must decide exactly who goes in which group

**Complexity:**
- Time: O(2^n × n) for single-item transitions; O(3^n) for subset enumeration over all masks
- Space: O(2^n) for the DP table

### Full Solution Walkthrough — Beautiful Arrangement (LC 526) — Revision

**Problem:** Count permutations of 1..n where perm[i] % i == 0 or i % perm[i] == 0 for every position i (1-indexed).

**Key insight:** Fill positions 1 through n in order. At step `pos` (= popcount of current mask), we are placing a number at position `pos`. The mask records which numbers have been used.

```python
def countArrangement(n):
    dp = [0] * (1 << n)
    dp[0] = 1   # empty arrangement: 1 way

    for mask in range(1, 1 << n):
        pos = bin(mask).count('1')   # 1-indexed position being filled
        for i in range(n):
            if mask & (1 << i):      # number i+1 is used in this mask
                num = i + 1
                # Is placing num at position pos valid?
                if num % pos == 0 or pos % num == 0:
                    # Remove num from mask → previous state
                    dp[mask] += dp[mask ^ (1 << i)]

    return dp[(1 << n) - 1]
```

**Trace for n=2:**
- dp[0] = 1
- mask=01 (bit 0 set → num=1): pos=1. 1%1==0 ✓. dp[01] += dp[00] = 1. dp[01]=1
- mask=10 (bit 1 set → num=2): pos=1. 2%1==0 ✓. dp[10] += dp[00] = 1. dp[10]=1
- mask=11: pos=2
  - i=0 (num=1, bit 0 set): 1%2≠0, 2%1==0 ✓. dp[11] += dp[10] = 1
  - i=1 (num=2, bit 1 set): 2%2==0 ✓. dp[11] += dp[01] = 1
  - dp[11] = 2 → answer = 2 (arrangements: [1,2] and [2,1] both valid)

**Why popcount(mask) = position:** We fill positions 1, 2, ..., n in order. When we've placed exactly k numbers (popcount(mask) = k), we're about to fill position k. This is the standard "iterate masks in order of popcount" trick.

### Full Solution Walkthrough — Partition to K Equal Sum Subsets (LC 698) — Hard

**Problem:** Given nums and k, can you partition nums into k non-empty subsets with equal sum?

**Key insight:** dp[mask] = the current partial sum in the "active" (not yet completed) subset. When the partial sum hits `target`, it resets to 0 (mod target trick). If dp[full_mask] == 0, all k subsets were completed.

```python
def canPartitionKSubsets(nums, k):
    total = sum(nums)
    if total % k != 0:
        return False
    target = total // k
    nums.sort(reverse=True)   # pruning: try largest numbers first
    if nums[0] > target:
        return False

    n = len(nums)
    dp = [-1] * (1 << n)
    dp[0] = 0   # with no numbers selected, partial sum = 0

    for mask in range(1 << n):
        if dp[mask] == -1:
            continue   # unreachable state
        for i in range(n):
            if mask & (1 << i):
                continue   # num[i] already used
            new_mask = mask | (1 << i)
            new_sum = dp[mask] + nums[i]
            if new_sum <= target:
                dp[new_mask] = new_sum % target
                # When new_sum == target, it resets to 0 (one subset complete)

    return dp[(1 << n) - 1] == 0
```

**The mod target trick explained:**
- dp[mask] tracks only the partial sum in the CURRENT (incomplete) subset
- When we add a number and the sum reaches exactly `target`, one full subset is done: reset partial sum to 0 via `% target`
- We do not need to track how many complete subsets exist — if dp[full] == 0, all numbers were used and the last subset was complete

**Why sort descending:** Large numbers that exceed the remaining capacity are detected earlier, pruning more branches. In the worst case (k=1) this is still O(2^n × n), but in practice it's much faster.

### Full Solution Walkthrough — Minimum Number of Work Sessions (LC 1986) — Hard

**Problem:** Given tasks with durations and a sessionTime limit, what is the minimum number of sessions to complete all tasks? Each session is at most sessionTime total.

**Pattern:** Two layers of bitmask DP.
1. Precompute which subsets fit in one session.
2. Use subset enumeration to find minimum sessions to cover all tasks.

```python
def minSessions(tasks, sessionTime):
    n = len(tasks)
    full = (1 << n) - 1

    # Step 1: Precompute fit[mask] = True if all tasks in mask fit in one session
    fit = [False] * (1 << n)
    fit[0] = True
    for mask in range(1, 1 << n):
        s = sum(tasks[i] for i in range(n) if mask & (1 << i))
        fit[mask] = (s <= sessionTime)

    # Step 2: dp[mask] = min sessions to complete exactly the tasks in mask
    dp = [float('inf')] * (1 << n)
    dp[0] = 0

    for mask in range(1, 1 << n):
        # Enumerate all subsets of mask that fit in one session
        # These represent the last session we schedule
        sub = mask
        while sub > 0:
            if fit[sub] and dp[mask ^ sub] + 1 < dp[mask]:
                dp[mask] = dp[mask ^ sub] + 1
            sub = (sub - 1) & mask   # iterate all non-empty subsets of mask

    return dp[full]
```

**Subset enumeration idiom — critical:**
```
sub = mask
while sub > 0:
    process(sub)        # sub is a subset of mask
    sub = (sub - 1) & mask   # clear the lowest set bit of sub, keep within mask
```
This visits ALL non-empty subsets of `mask`. Total work across all masks = O(3^n) (each element is either: not in mask, in mask but not sub, in both).

**Complexity:** O(3^n) — feasible for n ≤ 14 (LeetCode constraint is n ≤ 14).

## System Design (30 min)

### Topic: LRU Cache — O(1) Get and Put

**Requirements:** Fixed-capacity cache. `get(key)` returns value or -1. `put(key, value)` adds/updates; if at capacity, evict the Least Recently Used entry.

**Data structures:** HashMap (key → node for O(1) lookup) + Doubly Linked List (maintains usage order: head = most recent, tail = least recent).

```python
class Node:
    def __init__(self, key=0, val=0):
        self.key = key
        self.val = val
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.cache = {}           # key → Node
        # Dummy sentinels to avoid edge cases
        self.head = Node()        # most-recent end
        self.tail = Node()        # least-recent end (evict from here)
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node):
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def _move_to_head(self, node):
        self._remove(node)
        self._add_to_head(node)

    def _remove_tail(self):
        # Returns the LRU node (just before dummy tail)
        node = self.tail.prev
        self._remove(node)
        return node

    def get(self, key):
        if key not in self.cache:
            return -1
        self._move_to_head(self.cache[key])
        return self.cache[key].val

    def put(self, key, value):
        if key in self.cache:
            self.cache[key].val = value
            self._move_to_head(self.cache[key])
        else:
            node = Node(key, value)
            self.cache[key] = node
            self._add_to_head(node)
            if len(self.cache) > self.cap:
                lru = self._remove_tail()
                del self.cache[lru.key]
```

**Why dummy sentinels:** Avoid checking for null head/tail during every insert and remove. Both `_add_to_head` and `_remove_tail` work identically whether the list is empty or full.

**All operations are O(1):**
- `get`: hashmap lookup O(1) + move node O(1)
- `put`: hashmap insert/update O(1) + add to head O(1) + optional remove tail O(1)

**Interview talking point:**
*"LRU needs O(1) for both lookup and ordering. A hashmap alone gives O(1) lookup but O(n) for finding LRU. A doubly-linked list gives O(1) head/tail ops but O(n) lookup. Combining them gives O(1) for everything. The key is storing a pointer to the list node in the hashmap — that's the bridge between the two structures."*
