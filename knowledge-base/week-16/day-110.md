### Full Solution Walkthrough — Maximal Square (LC 221) — Revision

**Problem:** Find the largest square containing only 1s in a binary matrix. Return its area.

**Invariant:** `dp[i][j]` = side length of the largest square whose bottom-right corner is at (i, j).

```python
def maximalSquare(matrix: list[list[str]]) -> int:
    if not matrix:
        return 0
    m, n = len(matrix), len(matrix[0])
    dp = [[0] * n for _ in range(m)]
    max_side = 0

    for i in range(m):
        for j in range(n):
            if matrix[i][j] == '1':
                if i == 0 or j == 0:
                    dp[i][j] = 1
                else:
                    dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1
                max_side = max(max_side, dp[i][j])

    return max_side ** 2
```

**Why min of 3 neighbors?** dp[i][j] is bounded by: how tall the left column is, how wide the top row is, and what diagonal supports (dp[i-1][j-1]). The minimum of these three limits the square side.

### Full Solution Walkthrough — Minimum White Tiles After Covering With Carpets (LC 2209) — Hard

**Problem:** Binary string floor (1=white, 0=black). Place numCarpets carpets each of length carpetLen. Minimize uncovered white tiles.

**State:** `dp[j][i]` = min uncovered white tiles in floor[0..i-1] using j carpets.

**Transitions for each position i:**
1. Don't place carpet ending here: `dp[j][i] = dp[j][i-1] + (floor[i-1]=='1')`
2. Place carpet covering [i-carpetLen, i-1]: `dp[j][i] = dp[j-1][max(0, i-carpetLen)]`

```python
def minimumWhiteTiles(floor: str, numCarpets: int, carpetLen: int) -> int:
    n = len(floor)
    INF = float('inf')

    # dp[j][i] = min white tiles in floor[0..i-1] using j carpets
    dp = [[INF] * (n + 1) for _ in range(numCarpets + 1)]
    dp[0][0] = 0
    for i in range(1, n + 1):
        dp[0][i] = dp[0][i-1] + (floor[i-1] == '1')  # no carpets: count all whites

    for j in range(1, numCarpets + 1):
        for i in range(1, n + 1):
            dont_place = dp[j][i-1] + (floor[i-1] == '1')
            start = max(0, i - carpetLen)
            do_place = dp[j-1][start]  # carpet covers [start, i-1], cost = 0 for those tiles
            dp[j][i] = min(dont_place, do_place)

    return dp[numCarpets][n]
```

**Complexity:** O(numCarpets × n). No prefix sum needed since carpet covers exactly carpetLen tiles.

### Full Solution Walkthrough — Freedom Trail (LC 514) — Hard

**Problem:** A ring of characters. Spell a key string by rotating to each character and pressing a button. Rotations can be clockwise or counterclockwise (take the shorter path). Minimize total steps.

**State:** `dp(key_idx, ring_pos)` = min steps to spell key[key_idx..] starting with ring at ring_pos.

**Recurrence:**
For each character in ring matching key[key_idx], try all positions:
- Rotation cost = min(|ring_pos - pos|, n - |ring_pos - pos|) — shorter of the two directions
- Button press cost = 1
- Future cost = dp(key_idx+1, pos)

```python
def findRotateSteps(ring: str, key: str) -> int:
    from functools import lru_cache
    from collections import defaultdict

    n = len(ring)
    m = len(key)

    # Precompute positions of each character in ring
    char_pos = defaultdict(list)
    for i, c in enumerate(ring):
        char_pos[c].append(i)

    @lru_cache(maxsize=None)
    def dp(key_idx: int, ring_pos: int) -> int:
        if key_idx == m:
            return 0
        target = key[key_idx]
        result = float('inf')
        for pos in char_pos[target]:
            diff = abs(ring_pos - pos)
            dist = min(diff, n - diff)      # shorter rotation
            cost = dist + 1 + dp(key_idx + 1, pos)  # rotate + press + future
            result = min(result, cost)
        return result

    return dp(0, 0)
```

**Complexity:** O(m × n) states, O(n) transitions each → O(m × n²). With precomputed positions, the n positions per character at worst.

## System Design (30 min)

### Topic: Full API Design Mock — REST + GraphQL + Rate Limiting Integration

**The prompt:** *"Design an API for a social media platform (Twitter-like) that serves both mobile and web clients."*

**1. Requirements (3 min)**
- Functional: post tweets, follow users, get timeline (home + user), like/retweet
- Non-functional: < 200ms p99 latency, 100M users, 10M writes/day, 1B reads/day (read-heavy 100:1)

**2. API Design Decision (5 min)**

| Layer | Choice | Reason |
|-------|--------|--------|
| External (mobile) | GraphQL | Mobile needs flexible queries (different screens need different fields); eliminates over-fetching |
| External (web) | REST + GraphQL | Web can use REST for simple pages; GraphQL for complex feeds |
| Internal (service-to-service) | gRPC | Binary protocol, typed contracts, lower latency |

**3. REST + GraphQL Endpoints (10 min)**

REST endpoints (simple, cacheable):
```
GET  /users/{id}               → user profile (cached at CDN, TTL=5min)
GET  /users/{id}/tweets        → paginated tweets (cursor-based)
POST /tweets                   → create tweet (not cacheable)
POST /follow/{target_id}       → follow (idempotent with Idempotency-Key header)
```

GraphQL (complex queries):
```graphql
type Query {
  timeline(limit: Int!, after: String): TimelineConnection!
  user(id: ID!): User
}
type Tweet {
  id: ID!
  content: String!
  author: User!    # resolved via DataLoader (N+1 prevention)
  likeCount: Int!
  retweetCount: Int!
}
```

**4. Rate Limiting Strategy (5 min)**
| Tier | Limit | Window |
|------|-------|--------|
| Unauthenticated | 300 req | 15 min |
| Free user | 1,500 req | 15 min |
| Pro user | 15,000 req | 15 min |
| Write endpoints (POST) | 50 req | 15 min |

Implementation: Redis token bucket per tier. Return `X-RateLimit-Remaining` and `Retry-After` headers.

**5. Failure handling (5 min)**
- API Gateway: validates JWT, routes requests, applies rate limits — single failure point mitigated with active-active multi-region
- GraphQL timeout: enforce max query complexity score; reject complex queries before execution
- Rate limit bypass: DDoS → CDN absorbs; WAF blocks malicious IPs

**Interview talking point:**
*"The key design tension: REST for simple reads (cacheable at CDN) vs. GraphQL for complex queries (flexible but not trivially cacheable). I use REST for the most traffic-heavy, simple endpoints (user profiles, tweet pages) and GraphQL for authenticated feed queries where the client controls what data it needs. Rate limiting must be per-tier and per-endpoint — a bulk write endpoint needs a tighter limit than a read endpoint."*
