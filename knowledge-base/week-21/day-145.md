## DSA (1.5 hours)

### Pattern: Palindrome DP (Interval-Based)
- Core idea: `dp[i][j]` = minimum insertions to make `s[i..j]` a palindrome. If `s[i]==s[j]`, `dp[i][j] = dp[i+1][j-1]` (outer characters already match). Otherwise, `dp[i][j] = 1 + min(dp[i+1][j], dp[i][j-1])` (insert one character to match either end).
- Trigger condition: "Make a string a palindrome with minimum operations" or "find palindromic subsequence length." Note: minimum insertions = `n - LPS(s)` where LPS = Longest Palindromic Subsequence.
- Time complexity: O(n²) | Space complexity: O(n²), row-optimizable to O(n)

### Pattern: Day-Partitioned Job Schedule DP
- Core idea: `dp[i][j]` = min difficulty to schedule jobs `[i..n-1]` across `d` days. On day `d`, you must do at least one job, and the difficulty of that day = max difficulty of jobs done that day. Partition remaining jobs optimally for the next day.
- Trigger condition: You must divide a sequence of tasks across K time periods, each period has a cost based on its hardest/most difficult task, minimize total cost.
- Time complexity: O(n²×d) | Space complexity: O(n×d), optimizable to O(n)

    //   Iterate by length: for len 2..n; i = 0..n-len; j = i+len-1
    // Key Option B: answer = n - LPS(s) using LC 516 code

    // LC 1335 — Minimum Difficulty of a Job Schedule (cold solve — delete this stub and rewrite from scratch)
    // TODO: int minDifficulty(int[] jobDifficulty, int d)
    // Key: if n < d return -1 (can't do 1 job per day)
    //   dp[i][j] = min difficulty scheduling jobs[j..n-1] in i remaining days
    //   For i days and starting at job j: try doing jobs j..k today (difficulty = max(j..k))
    //   then dp[i][j] = min over k of (maxDifficulty[j..k] + dp[i-1][k+1])
    //   Fill: dp[1][j] = max(jobs[j..n-1]) for base case

    // LC 516 — Longest Palindromic Subsequence (revision — write from memory)
    public int longestPalindromeSubseq(String s) {
        int n = s.length();
        int[][] dp = new int[n][n];
        for (int i = 0; i < n; i++) dp[i][i] = 1;
        for (int len = 2; len <= n; len++) {
            for (int i = 0; i <= n - len; i++) {
                int j = i + len - 1;
                dp[i][j] = s.charAt(i) == s.charAt(j)
                    ? dp[i+1][j-1] + 2
                    : Math.max(dp[i+1][j], dp[i][j-1]);
            }
        }
        return dp[0][n-1];
    }
}

#### Trending Algorithm

**Naive approach:** Count views in last 24 hours — too slow, too memory-heavy.

**Sliding Window Counter (Redis):**
```
For each video view event:
  ZINCRBY trending:hour:<current_hour> 1 video_id
  EXPIRE trending:hour:<current_hour> 7200  // 2 hour TTL

To get trending:
  Aggregate last N hour buckets:
    ZUNIONSTORE trending:combined 6 trending:hour:1 ... trending:hour:6
    ZREVRANGE trending:combined 0 49  // top 50 videos
```

**Why not a simple counter?** A video posted 3 days ago with 10M views shouldn't outrank a video posted 1 hour ago with 100K views. Sliding window captures velocity, not just volume.

#### Real-Time Analytics Architecture

```
User events (views, likes, comments, shares)
    │
    ▼ (Kafka, 1M events/sec peak at YouTube)
┌────────────────────────────────────────────┐
│  STREAM PROCESSOR (Apache Flink)           │
│  - Windowed aggregation (1min, 5min, 1hr)  │
│  - Top-K computation (Count-Min Sketch)    │
│  - Anomaly detection (bot traffic)         │
└────────────────┬───────────────────────────┘
                 │                   │
          ┌──────▼──────┐    ┌───────▼────────┐
          │  Redis       │    │  ClickHouse /  │
          │  (trending,  │    │  BigQuery      │
          │   counters)  │    │  (analytics DB)│
          └─────────────┘    └────────────────┘
```

**Count-Min Sketch:** Probabilistic data structure for approximate frequency counting. Uses O(w×d) space (w=width, d=depth) instead of O(n) for n distinct items. Suitable for counting views among billions of events with < 1% error at 1% of the memory.

#### View Count Consistency Problem
**Challenge:** YouTube has billions of views; incrementing a counter in MySQL for every view causes thundering herd on popular videos.

**Solution:**
1. **Write to Kafka** on each view event (fire-and-forget, < 1ms).
2. **Kafka consumer** aggregates in batch: every 30 seconds, sum all events and do a single atomic ADD in Redis.
3. **Periodic flush** from Redis to Cassandra/MySQL (durable store) every 5 minutes.
4. **Display count** reads from Redis (fast, eventually consistent). Exact count available in analytics DB with delay.

**Key trade-off:** Display count may be 30–60 seconds behind real count. Acceptable (users don't need real-time exact counts).

**Interview talking point:** "If asked how YouTube handles trending in real-time for billions of events, answer: Sliding window counters in Redis (ZINCRBY on time-bucketed sorted sets). Apache Flink processes the Kafka event stream to compute windowed view counts. Count-Min Sketch enables approximate top-K computation in O(1) space per event rather than maintaining exact counts for every video. Trending is refreshed every 60 seconds."

| How do you derive minimum insertions to make a palindrome from LPS? | `minInsertions(s) = s.length() - longestPalindromeSubsequence(s)`. Every character NOT in the LPS needs a matching character inserted. |
