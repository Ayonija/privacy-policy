# Day 139 — Decorator Pattern (Structural)
**Week 21 | Phase: LLD Mastery | Month 4**

## Focus
The Decorator pattern adds behavior to an object at runtime by wrapping it in another object that shares the same interface. It solves the subclass-explosion problem when features can be combined in any order or combination. The layering is the key mental model — each wrapper adds exactly one responsibility.

---

## Core Concept

### Analogy
A base pizza is ₹100. Add cheese (₹20), then olives (₹15), then paneer (₹30). Each topping wraps the previous pizza, adds its own cost, and exposes the same `cost()` and `description()` interface. You can stack toppings in any order. No subclass `CheesePaneerPizza` needed.

### What It Does
- Wraps an object to add behavior at runtime without changing the original class
- Decorated object and decorators share the same interface — client code is unchanged
- Each decorator delegates to the wrapped object and adds its own behavior before/after

### When to Reach For It (Spot Triggers)
- "Add features dynamically at runtime"
- "Many optional add-ons in any combination" (beverage toppings, insurance riders, UI styling)
- "Avoid subclass explosion when features can be mixed freely"
- "Extend behavior without modifying the source class (Open/Closed Principle)"

---

## Interface Contract Rule
Every decorator MUST implement the same interface as the object it wraps. This is what allows transparent stacking — the client holds a reference to `Coffee` regardless of how many decorators are layered.

---

## Pseudocode

```
// Component interface
interface Coffee {
    cost(): int
    description(): String
}

// Concrete component
class SimpleCoffee implements Coffee {
    cost() { return 50 }
    description() { return "Simple Coffee" }
}

// Abstract decorator — holds a reference to Coffee
abstract class CoffeeDecorator implements Coffee {
    protected wrapped: Coffee
    constructor(coffee: Coffee) { this.wrapped = coffee }
}

// Concrete decorator 1
class MilkDecorator extends CoffeeDecorator {
    cost() { return wrapped.cost() + 15 }
    description() { return wrapped.description() + ", Milk" }
}

// Concrete decorator 2
class SugarDecorator extends CoffeeDecorator {
    cost() { return wrapped.cost() + 10 }
    description() { return wrapped.description() + ", Sugar" }
}

// Usage — order of wrapping = order of feature layering
coffee = new SimpleCoffee()          // cost = 50
coffee = new MilkDecorator(coffee)   // cost = 65
coffee = new SugarDecorator(coffee)  // cost = 75
// coffee.description() → "Simple Coffee, Milk, Sugar"
```

---

## Order Matters
`BufferedReader(new FileReader("file.txt"))` — FileReader reads raw chars; BufferedReader adds buffering on top. Reversing them would fail because `BufferedReader` expects a `Reader`, not a `BufferedReader`. The outermost decorator's behavior is applied last on the way in and first on the way out.

---

## Real-World Uses

| Context | Decorator Layer Added |
|---|---|
| Java I/O: `BufferedInputStream(FileInputStream)` | Buffering on top of raw byte stream |
| Java I/O: `GZIPOutputStream(FileOutputStream)` | Compression on top of file writing |
| Coffee/pizza pricing systems | Topping cost + description |
| HTTP middleware (Express.js, Django) | Auth, logging, CORS — each wraps request handler |
| UI component styling (HTML/CSS-style) | Border, scroll, color added per component |
| Logging middleware | Log request/response around existing handler |

---

## Don't Confuse

| Pattern | Intent | Behavior |
|---|---|---|
| Decorator | ADD new behavior / features at runtime | Method behavior changes with each layer |
| Proxy | CONTROL access — same behavior, adds indirection | Core behavior unchanged; adds caching, auth, lazy load |

**One-line test:** Am I adding capability? → Decorator. Am I adding a gatekeeper? → Proxy.

## STAR Interview Framework

**Situation:** "Design a beverage ordering system for a coffee shop. Customers choose a base drink (espresso, simple coffee, dark roast) and can add milk, sugar, whip cream in any combination. The system must compute the total cost and generate an accurate description for the receipt."

**Task:** "Design a flexible pricing system that handles any combination of add-ons without creating a separate class for every possible combination."

**Action:**
1. Define `Beverage` interface with `cost(): int` and `description(): String`
2. Implement concrete beverage classes: `SimpleCoffee`, `Espresso`, `DarkRoast`
3. Create abstract `BeverageDecorator implements Beverage` holding a `wrapped: Beverage`
4. Implement concrete decorators: `MilkDecorator`, `SugarDecorator`, `WhipDecorator` — each adds cost delta and appends to description
5. Client constructs by wrapping: `new WhipDecorator(new MilkDecorator(new Espresso()))`
6. Avoid creating `EspressoWithMilkAndWhip` subclass — combination count grows exponentially without Decorator

**Result:** System supports M toppings across N base drinks with only M + N classes instead of N * 2^M. Adding a new topping requires one new decorator class — zero changes to existing code (Open/Closed). Pricing engine generates correct cost and description through delegation chain.

## Flashcards

| Q | A |
|---|---|
| What problem does Decorator solve that inheritance cannot? | Avoids subclass explosion when features can be combined in any order or combination; inheritance would require a class per combination |
| What must every decorator implement? | The same interface (or extend the same abstract class) as the component it wraps — this makes decoration transparent to the client |
| Decorator vs Proxy — one-line distinction | Decorator adds new behavior; Proxy controls access while keeping behavior the same |
| In Java I/O, give one example of Decorator in use | `BufferedReader(new FileReader("file.txt"))` — FileReader is the component; BufferedReader decorates it with buffering |
| Does order of decoration matter? | Yes — outermost decorator's behavior applies last. `new A(new B(base))` means B wraps base first, A wraps that result |
| Spot trigger for Decorator | "Many optional add-ons in any combination", "extend behavior without modifying source", "avoid subclass explosion" |
