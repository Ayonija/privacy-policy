# Day 140 — Facade Pattern (Structural)
**Week 21 | Phase: LLD Mastery | Month 4**

## Focus
The Facade pattern provides a single simplified entry point into a complex subsystem of many classes. It hides internal orchestration from the caller without removing access to the subsystem directly when needed. Master it for checkout flows, SDK design, compiler front-ends, and any "one method that does everything" interview question.

---

## Core Concept

### Analogy
You walk into a restaurant and say "one thali, please." The waiter handles the rest — tells the chef what to cook, checks with the store-room for ingredients, coordinates the order. You did not talk to the chef or the store-room directly. The waiter is the Facade. The kitchen subsystem still exists and could handle direct orders if needed.

### What It Does
- Gives callers one clean entry point into a complex set of classes
- Hides the sequence and orchestration of subsystem calls
- Subsystem classes are still accessible directly — Facade is not a lock, it is a shortcut

### When to Reach For It (Spot Triggers)
- "Too many classes to call in the right order"
- "Simplify a messy subsystem — give it one clean API"
- "Callers should not need to know how the subsystem works"
- "Layered architecture — UI layer should not call database, queue, and payment directly"

## Pseudocode

```
// Subsystem classes (complex, each with its own interface)
class PaymentService {
    charge(cart: Cart): Receipt { ... }
}

class InventoryService {
    reserve(items: List<Item>): void { ... }
}

class ShippingService {
    schedule(address: Address, items: List<Item>): TrackingId { ... }
}

// Facade — one entry point that orchestrates all three
class OrderFacade {
    private payment: PaymentService
    private inventory: InventoryService
    private shipping: ShippingService

    placeOrder(cart: Cart, address: Address): OrderConfirmation {
        receipt = payment.charge(cart)          // step 1
        inventory.reserve(cart.items)           // step 2
        trackingId = shipping.schedule(address, cart.items)  // step 3
        return new OrderConfirmation(receipt, trackingId)
    }
}

// Client code — only calls one method
facade = new OrderFacade(payment, inventory, shipping)
confirmation = facade.placeOrder(cart, address)
// Client never knows about payment/inventory/shipping coordination
```

Key points:
- `PaymentService`, `InventoryService`, `ShippingService` are unchanged
- `OrderFacade` is a new convenience layer, not a replacement
- Adding a new subsystem step (e.g., fraud check) is one change in `placeOrder()` — zero changes to clients

---

## Subsystem Still Accessible
Facade does not prevent direct subsystem access. Power users or internal services can still call `PaymentService.charge()` directly when they need fine-grained control. This distinguishes Facade from encapsulation — it simplifies, does not restrict.

## Real-World Uses

| Context | What Facade Hides |
|---|---|
| E-commerce checkout | Payment + inventory + shipping + notifications |
| Compiler front-end | Lexer + parser + AST builder + semantic analyzer |
| Spring JdbcTemplate | Connection management + statement prep + result mapping + close |
| AWS SDK high-level clients | HTTP calls + auth signing + retry + serialization |
| Home theater system | TV power + HDMI switch + lights + sound system |
| `slf4j` logging facade | Hides whether Log4j, Logback, or JUL is underneath |

---

## Don't Confuse

| Pattern | Problem It Solves | Key Question |
|---|---|---|
| Facade | One subsystem is complex — simplify its surface | "Is one subsystem too hard to use directly?" |
| Adapter | Two existing things are incompatible | "Do two things not fit each other's interface?" |

**One-line test:** Hiding complexity of one thing → Facade. Bridging incompatibility between two things → Adapter.

## STAR Interview Framework

**Situation:** "Design a home theater system. Watching a movie requires: turning on the TV, switching TV input to HDMI-2, dimming the lights to 30%, turning on the sound system, setting volume to 15, and starting the Blu-ray player. The user should not have to do all of this manually every time."

**Task:** "Give the user one method to start a movie experience, hiding all subsystem orchestration."

**Action:**
1. Identify subsystems: `TV`, `Lights`, `SoundSystem`, `BluRayPlayer` — each has its own interface
2. Create `HomeTheaterFacade` with a `watchMovie(movie: String)` method
3. Inside `watchMovie()`: call `lights.dim(30)`, `tv.on()`, `tv.setInput("HDMI-2")`, `sound.on()`, `sound.setVolume(15)`, `bluray.play(movie)` — in the right order
4. Create `endMovie()` facade method that reverses the sequence
5. Subsystems remain independently accessible for custom configurations

**Result:** User calls `theater.watchMovie("Interstellar")` — one line instead of six coordinated calls. Adding a new device (e.g., projector screen auto-lower) is one change in `watchMovie()`, zero changes to the user's code. Junior developers can use the system correctly without knowing subsystem internals.

## Flashcards

| Q | A |
|---|---|
| What problem does Facade solve? | A subsystem has too many classes to coordinate correctly; Facade provides one clean entry point that handles the orchestration |
| Does Facade prevent direct subsystem access? | No — subsystem classes remain accessible; Facade is a convenience layer, not a lock |
| Facade vs Adapter — one-line distinction | Facade simplifies one subsystem. Adapter bridges two incompatible interfaces. |
| Spot trigger for Facade | "Too many classes to call in order", "simplify messy subsystem", "one clean API for callers" |
| Name a real Java/Spring Facade | `JdbcTemplate` (hides connection + statement + result set management) or `slf4j` (hides concrete logging implementation) |
