# Day 138 ‚Äî Adapter Pattern (Structural)
**Week 21 | Phase: LLD Mastery | Month 4**

## Focus
The Adapter pattern bridges two incompatible interfaces so they can work together without modifying either side. It wraps an existing class and translates its interface into the one the client expects. Master it for any question involving third-party integration, legacy API glue, or SDK wrappers.

---

## Core Concept

### Analogy
A foreign plug (US 2-pin) and an Indian socket (3-pin) are incompatible. You use a travel adapter ‚Äî it accepts the plug on one side and fits the socket on the other. Neither the plug nor the socket changes. The adapter is the translator.

### What It Does
- Wraps one interface to look like another
- Lets two incompatible classes collaborate without changing either
- Client only ever sees the target interface; the adaptee is hidden

### When to Reach For It (Spot Triggers)
- "Integrate a third-party / legacy library that has a different interface"
- "Two interfaces don't match but both are out of your control"
- "Wrap an old API to match the new contract"
- "Use an existing class whose interface isn't what you need"

---

## Two Variants

| Variant | Mechanism | When to Use |
|---|---|---|
| Object Adapter | Composition ‚Äî holds a reference to adaptee | Preferred; works even when adaptee is final or from external lib |
| Class Adapter | Inheritance ‚Äî extends adaptee + implements target | Only in languages with multiple inheritance; tightly couples you to adaptee |

**Rule:** Prefer Object Adapter (composition). Class Adapter requires extending the adaptee which breaks encapsulation and is impossible with final/sealed classes.

## Pseudocode

```
// Target interface ‚Äî what client expects
interface JsonDataFetcher {
    fetchAsJson(query: String): JsonObject
}

// Adaptee ‚Äî existing class with incompatible interface
class XmlLibrary {
    fetchAsXml(query: String): XmlDocument
}

// Object Adapter ‚Äî wraps XmlLibrary, implements JsonDataFetcher
class XmlToJsonAdapter implements JsonDataFetcher {
    private adaptee: XmlLibrary

    constructor(adaptee: XmlLibrary) {
        this.adaptee = adaptee
    }

    fetchAsJson(query: String): JsonObject {
        xmlDoc = adaptee.fetchAsXml(query)
        return convertXmlToJson(xmlDoc)   // translation logic lives here
    }
}

// Client code ‚Äî only sees JsonDataFetcher
client = new ApiClient(new XmlToJsonAdapter(new XmlLibrary()))
client.getData("invoices")   // client never knows XML exists
```

Key points:
- `XmlLibrary` never changes
- `XmlToJsonAdapter` is the only class that knows about both interfaces
- Swap `XmlLibrary` for any other source by writing a new adapter ‚Äî client untouched

## Real-World Uses

| Context | What Is Being Adapted |
|---|---|
| Payment gateway wrappers (Stripe, Razorpay) | Third-party SDK ‚Üí internal `PaymentProcessor` interface |
| Legacy billing system integration | Old SOAP/XML API ‚Üí new REST/JSON service layer |
| Java `Arrays.asList()` | Array ‚Üí List interface |
| Java `InputStreamReader` | `InputStream` ‚Üí `Reader` (character stream) |
| Spring `HandlerAdapter` | Various controller types ‚Üí one dispatcher interface |
| Third-party auth SDKs | OAuth provider SDK ‚Üí internal `AuthProvider` interface |

---

## Don't Confuse

| Pattern | Intent | Key Question |
|---|---|---|
| Adapter | Make two existing incompatible interfaces work together | "Do I have two things that don't fit each other?" |
| Facade | Simplify one complex subsystem with a single clean entry point | "Is one subsystem too complex to call directly?" |

**One-line test:** If you are bridging incompatibility ‚Üí Adapter. If you are hiding complexity ‚Üí Facade.

## STAR Interview Framework

**Situation:** "You need to integrate a legacy XML billing system into a new JSON-based microservices platform. The billing system is a black box ‚Äî you can't modify it. Your payment service expects all billing providers to implement a `BillingProvider` interface that returns `JsonInvoice` objects."

**Task:** "Design a solution that lets the new platform use the legacy system without coupling the platform code to XML or the legacy API shape."

**Action:**
1. Define `BillingProvider` interface with `getInvoice(orderId): JsonInvoice`
2. Identify `LegacyBillingSystem` as the adaptee ‚Äî it exposes `fetchInvoiceXml(orderId): XmlDocument`
3. Create `LegacyBillingAdapter implements BillingProvider` ‚Äî constructor takes a `LegacyBillingSystem` instance (composition)
4. Implement `getInvoice()` by calling `fetchInvoiceXml()` then parsing and mapping XML ‚Üí `JsonInvoice`
5. Register `LegacyBillingAdapter` wherever `BillingProvider` is injected ‚Äî zero changes to payment service

**Result:** Legacy system integrated with zero changes to either the legacy code or the payment service. Adding a second billing provider requires only a new adapter class. XML parsing logic is isolated to one class ‚Äî easy to test and swap.

## Flashcards

| Q | A |
|---|---|
| What problem does the Adapter pattern solve? | Two existing interfaces are incompatible; Adapter wraps one to match the other so client can use it without modification |
| Object Adapter vs Class Adapter ‚Äî which is preferred and why? | Object Adapter (composition) ‚Äî works with final classes, looser coupling, doesn't inherit adaptee's implementation |
| In the Adapter pattern, which side changes: client, adaptee, or adapter? | Only the Adapter changes / is created; client and adaptee remain untouched |
| Spot trigger for Adapter in an interview | "Integrate third-party/legacy library", "two interfaces don't match", "wrap old API" |
| Adapter vs Facade ‚Äî one-line distinction | Adapter: bridges incompatibility between two things. Facade: simplifies complexity of one thing. |
