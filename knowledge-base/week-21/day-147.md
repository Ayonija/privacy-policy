## DSA (1.5 hours)

### Pattern: BFS/DFS Backtracking on Strings with Validity Pruning
- Core idea: Generate all strings at each edit distance level (BFS) or enumerate all possible removal combinations (DFS). The key optimization for Remove Invalid Parentheses: compute the minimum number of `(` to remove and `)` to remove upfront. Only explore branches that respect these bounds.
- Trigger condition: "Remove minimum characters to make string valid." The minimum removal count bounds the search depth, making BFS by edit distance natural (stop at first level that yields valid results).
- Time complexity: O(2^n) worst case, but BFS stops at first valid level | Space complexity: O(n × 2^n) for BFS queue

### Pattern: Expression Backtracking with Running Values
- Core idea: Walk through a digit string, inserting `+`, `-`, `*` between digits. Track `current_expression`, `running_total`, and `last_operand` to handle multiplication correctly (you need to undo the last multiplication when inserting `*`).
- Trigger condition: "Insert operators between digits to evaluate to a target value." The `last_operand` trick is essential: for `*`, the new total = `running_total - last_operand + last_operand * new_num`.
- Time complexity: O(4^n × n) — 4 choices per digit gap (no-op: multi-digit number, +, -, *) × O(n) to build string | Space complexity: O(n) recursion depth

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Remove Invalid Parentheses | 301 | Hard (Cold Solve) | BFS Backtracking + Validity Check | BFS level by level (each level = one more removal); stop at first level with valid strings; use Set to avoid duplicates |
| 2 | Expression Add Operators | 282 | Hard (Cold Solve) | Expression Backtracking | Track `last_operand` for correct multiplication handling: `new_total = total - last + last * curr` |
| 3 | Generate Parentheses | 22 | Medium (Revision) | Recursive Backtracking | REVISION: Add `(` if `open < n`; add `)` if `close < open`; base case `open == close == n` |

```java
class Solution {
    // LC 301 — Remove Invalid Parentheses (cold solve — delete this stub and rewrite from scratch)
    // TODO: List<String> removeInvalidParentheses(String s)
    // Key: BFS approach — level = number of characters removed
    //      Start with {s} in queue. For each string, check if valid (count open/close balance).
    //      If any valid found at this level, collect all valid, stop BFS.
    //      If none valid, generate all strings with one character removed → add to next level set.
    //      Use Set<String> to avoid re-visiting.
    //      Validity check: iterate, track balance; balance goes negative → invalid; end balance != 0 → invalid.

    // LC 282 — Expression Add Operators (cold solve — delete this stub and rewrite from scratch)
    // TODO: List<String> addOperators(String num, int target)
    // Key: backtrack(index, currentExpr, runningTotal, lastOperand)
    //      At each index, try all sub-numbers num[index..j] (watch for leading zeros: "05" invalid)
    //      If first number (no operator yet): recurse with (num[index..j], num[index..j], 0)
    //      Else try +, -, *:
    //        +: recurse(j+1, expr+"+"+str, total + currNum, currNum)
    //        -: recurse(j+1, expr+"-"+str, total - currNum, -currNum)
    //        *: recurse(j+1, expr+"*"+str, total - last + last*currNum, last*currNum)
    //      Base: index == num.length() && runningTotal == target → add expr to result

    // LC 22 — Generate Parentheses (revision — write from memory)
    public List<String> generateParenthesis(int n) {
        List<String> result = new ArrayList<>();
        backtrack(result, new StringBuilder(), 0, 0, n);
        return result;
    }
    private void backtrack(List<String> result, StringBuilder sb, int open, int close, int n) {
        if (open == n && close == n) { result.add(sb.toString()); return; }
        if (open < n) { sb.append('('); backtrack(result, sb, open+1, close, n); sb.deleteCharAt(sb.length()-1); }
        if (close < open) { sb.append(')'); backtrack(result, sb, open, close+1, n); sb.deleteCharAt(sb.length()-1); }
    }
}
```

## System Design (1 hour)

### Topic: YouTube / Netflix — Auth, Uploads & Notifications

#### Authentication & Authorization

**Auth Flow:**
```
1. User logs in → POST /auth/login → returns JWT access token (15min TTL) + refresh token (30 days)
2. Client stores refresh token in HttpOnly cookie (XSS-safe)
3. Every request includes Authorization: Bearer <access_token>
4. API Gateway validates JWT signature (public key) locally — no DB call
5. If access token expired: POST /auth/refresh with refresh token → new access token
6. Refresh tokens stored in Redis: key = refresh_token_hash, value = user_id, TTL = 30 days
7. Logout: delete refresh token from Redis (instant invalidation)
```

**Why JWT for access tokens?**
- Stateless: API Gateway validates signature without DB lookup → low latency, horizontally scalable.
- Downside: Cannot revoke a non-expired JWT without a blocklist.
- Mitigation: Short TTL (15 min). Only refresh tokens need revocation (stored in Redis).

**OAuth2 / Social Login (Google, Apple):**
```
User clicks "Sign in with Google"
    │
    ▼
Redirect to Google OAuth endpoint
    │ (user authenticates with Google)
    ▼
Google redirects back with authorization code
    │
    ▼
YouTube backend: exchange code for Google ID token
    │
    ▼
Extract user profile (email, name, picture) from ID token
    │
    ├── User exists? → issue YouTube JWT
    └── New user? → create account → issue YouTube JWT
```

**S3 presigned URL benefits:** Client uploads directly to S3 — upload bytes never pass through your backend servers. Reduces bandwidth cost and backend load by 100%.

#### Notification System

**Types of notifications:**
- New video from subscribed channel
- Comment reply
- Like milestone (100K likes)
- Live stream starting

**Architecture:**
```
Event Sources (Kafka topics)
  video.published
  comment.replied
  like.milestone
  live.started
        │
        ▼
┌───────────────────────────────────────────┐
│  NOTIFICATION SERVICE                     │
│  - Consumes Kafka topics                  │
│  - Fetches subscriber list (Cassandra)    │
│  - Checks user notification preferences  │
│  - Fan-out: send to delivery services    │
└─────────┬──────────────┬─────────────────┘
          │              │
  ┌───────▼──────┐  ┌────▼────────────┐
  │  Push Service│  │  Email Service  │
  │  (FCM/APNs) │  │  (SendGrid/SES) │
  └──────────────┘  └─────────────────┘
          │
  ┌───────▼──────────────────┐
  │  Device Tokens (Redis)   │
  │  user_id → [token1, ...] │
  └──────────────────────────┘
```

**Fan-out challenge for popular channels:**
- PewDiePie: 110M subscribers. Uploading a video triggers 110M notification fan-outs.
- Solution: Batch processing + priority queues. Fan-out happens asynchronously in background. Target: all notifications delivered within 5 minutes of upload (not real-time).
- Rate limiting: max 3 notifications per user per day per channel to avoid spam.

**In-App Notification Feed:**
```
Table: notifications (Cassandra)
Partition key: user_id
Clustering: (created_at DESC, notification_id)

GET /notifications?user_id=X&limit=20&page_token=Y
```

**Push Notification Delivery:**
- FCM (Firebase Cloud Messaging) for Android + Web.
- APNs (Apple Push Notification Service) for iOS.
- Both are fire-and-forget from YouTube's side; delivery is best-effort.
- Store notification in DB first (durable), then send push (ephemeral). Push failure → user still sees it in notification feed.

**Interview talking point:** "If asked how YouTube delivers notifications to 110M subscribers within minutes of a popular creator uploading, answer: Async Kafka consumer fan-out. The video.published event triggers a Notification Service that reads the subscriber list in batches from Cassandra and enqueues FCM/APNs push requests. Fan-out is not synchronous — it runs as a background job over 1–5 minutes. Notifications are also written to a per-user notification feed in Cassandra so they appear in-app even if the push was not delivered."

## Flashcards

| Q | A |
|---|---|
| Why does BFS find minimum removals naturally in Remove Invalid Parentheses? | BFS explores all strings at edit distance 1, then 2, etc. The first level that produces valid strings is the minimum removal count — no deeper search needed. |
| What is the `last_operand` trick in Expression Add Operators and when is it used? | When inserting `*`: `new_total = running_total - last_operand + last_operand * curr_num`. Undoes the last addition/subtraction and replaces it with multiplication. Used because multiplication has higher precedence. |
| Why are JWT access tokens short-lived (15 min) while refresh tokens are long-lived (30 days)? | JWTs cannot be revoked without a blocklist. Short TTL limits the damage window if a token is stolen. Refresh tokens are stored in Redis and can be deleted instantly (true revocation). |
| What is a presigned S3 URL and why does it reduce backend load for uploads? | A time-limited URL signed with AWS credentials allowing the bearer to PUT an object to S3. The client uploads directly to S3 — upload bytes never pass through your backend servers. |
| How does YouTube deliver notifications to 110M subscribers of a popular channel? | Async Kafka consumer fan-out in background batches. Target: all notifications within 1–5 minutes. Notification also written to per-user Cassandra feed so in-app view works even if push fails. |
