## DSA (1.5 hours)

### Pattern: Constraint Satisfaction Backtracking
- Core idea: Place a candidate in a cell/position, check if the current state is still valid (constraint checking), recurse for the next position, then undo the placement (backtrack). The key to efficiency is pruning early: don't recurse into states that already violate constraints.
- Trigger condition: Place N non-attacking queens, fill a Sudoku grid, color a map. Constraints must be checked before recursing. If any placement completely determines others (like Sudoku rows/cols/boxes), maintain sets for O(1) constraint checking.
- Time complexity: O(N!) worst case for N-Queens, O(9^81) worst case for Sudoku (but massive pruning makes it fast in practice) | Space complexity: O(N) recursion depth

### Pattern: Subset Enumeration (Power Set)
- Core idea: For each element, decide to include or exclude it. Use index-based recursion to avoid duplicates. Add current subset to result at each recursive call (not just leaves). Handle duplicates by sorting and skipping repeated elements at the same recursion level.
- Trigger condition: "Return all subsets", "enumerate all combinations." If input has duplicates, sort first then skip same-value elements at the same index.
- Time complexity: O(2^n × n) | Space complexity: O(n) recursion depth + O(2^n × n) result space

    }
    private void backtrack(int[] nums, int start, List<Integer> current, List<List<Integer>> result) {
        result.add(new ArrayList<>(current));
        for (int i = start; i < nums.length; i++) {
            current.add(nums[i]);
            backtrack(nums, i + 1, current, result);
            current.remove(current.size() - 1);
        }
    }

## System Design (1 hour)

### Topic: YouTube / Netflix — Comments, Likes & Social Graph

#### Comments System Design

**Functional Requirements:**
- Users can post, edit, delete comments on a video.
- Users can reply to comments (one level of nesting for YouTube; Netflix has no comments).
- Users can like/dislike individual comments.
- Comments are paginated (load more on scroll).
- Comments are sorted by: Top (most liked) or Newest.

**Scale:**
- YouTube: ~2B videos, ~10 comments/video average = 20B comments total.
- YouTube receives ~500K new comments per minute during peak.
- Read QPS for comments on a popular video: 50,000–100,000 reads/sec at peak.

**Data Model (Cassandra):**
```
Table: comments
Partition key: video_id
Clustering key: (created_at DESC, comment_id)

Columns:
  video_id     UUID
  comment_id   UUID (TIMEUUID for natural sort)
  parent_id    UUID (null for top-level, comment_id for replies)
  user_id      UUID
  content      TEXT
  like_count   COUNTER (Cassandra counter column)
  created_at   TIMESTAMP
  updated_at   TIMESTAMP
```

**Why Cassandra?**
- Video ID as partition key → all comments for a video on same node.
- TIMEUUID clustering → chronological pagination without offset (use page token = last seen comment_id).
- Cassandra counters for like counts → conflict-free atomic increments.
- Trades strong consistency for write availability (acceptable: comment counts are approximate).

**Pagination Strategy:**
```
GET /v1/videos/{video_id}/comments?page_token=<comment_id>&limit=20&sort=newest

Response:
{
  "comments": [...],
  "next_page_token": "<last_comment_id_in_response>",
  "has_more": true
}

Next request uses next_page_token as WHERE comment_id < :token (keyset pagination)
```
Why keyset pagination over OFFSET? Offset-based pagination degrades to O(offset) on large datasets; keyset is O(log n) and stable under concurrent inserts.

#### Likes / Dislikes System

**Challenge:** Popular videos get millions of like/dislike clicks per hour. Cannot write to a single database row per click.

```
User clicks Like
    │
    ▼
Like Service
    │
    ├──► Kafka event: {video_id, user_id, action: LIKE, timestamp}
    │
    └──► Redis: SADD liked_videos:{user_id} video_id  (user's liked set)
              INCR like_count:{video_id}              (approximate fast counter)

Kafka Consumer (batch, every 30s):
    ├──► Cassandra: BATCH UPDATE like counts (durable store)
    └──► MySQL: UPDATE user_video_interactions SET liked=true WHERE user_id AND video_id
                (for "did I already like this?" lookup)
```

**Idempotency:** Before liking, check Redis set `liked_videos:{user_id}` — if video_id already present, this is a duplicate click (ignore or toggle to unlike).

#### Social Graph (Subscriptions / Follows)

**YouTube Subscriptions:**
- User A subscribes to Channel C → A sees C's new videos in feed.
- Data model: simple edge table `(subscriber_id, channel_id, created_at)`.
- At YouTube scale (2B users, avg 50 subscriptions each) = 100B edges.

**Storage for Social Graph:**
```
Option 1: MySQL
  Table: subscriptions (subscriber_id, channel_id, created_at)
  Reads: SELECT channel_id FROM subscriptions WHERE subscriber_id = ?
  Problem: 100B rows → slow even with index for users with thousands of subscribers

Option 2: Graph Database (Neo4j / Neptune)
  Edges natively, efficient for multi-hop queries
  Problem: Hard to scale to 100B+ edges

Option 3: Adjacency list in Cassandra (YouTube's actual approach)
  Table: subscriptions
  Partition: subscriber_id → list of channel_ids
  Fast per-user fan-out; not optimized for "all subscribers of a channel" queries
```

**Feed Generation (Fan-out on write vs. read):**
```
Fan-out on write (push model):
  When Channel C uploads a video:
  → For each of C's N subscribers, write video_id to their feed table
  Pro: fast read (feed is pre-computed)
  Con: expensive for channels with 100M+ subscribers (100M writes per upload)

Fan-out on read (pull model):
  When User A opens YouTube:
  → Fetch user's subscriptions → fetch latest video from each → merge
  Pro: cheap writes
  Con: slow reads if user follows 500+ channels

Hybrid (YouTube's approach):
  - Push model for regular users (< 1M subscribers)
  - Pull model for "celebrity" channels (> 1M subscribers) — too expensive to push
  - Cache merged feed in Redis for active users
```

**Interview talking point:** "If asked how YouTube handles comment pagination at scale, answer: Cassandra with video_id as partition key and TIMEUUID as clustering key. Keyset pagination using the last seen comment_id as a page token — this is O(log n) per page fetch vs. O(offset) for OFFSET-based pagination, and it's stable under concurrent inserts. Like counts use Cassandra counter columns with a Kafka-backed batch flush for durability."

## Flashcards

| Q | A |
|---|---|
| What three data structures track constraints in N-Queens backtracking? | `Set<Integer>` for occupied columns, diagonals (row-col), and anti-diagonals (row+col). All O(1) lookup. |
| How do you index the 3×3 box in Sudoku Solver? | `boxIndex = (row / 3) * 3 + (col / 3)`. Gives values 0–8 for the nine 3×3 boxes. |
| Why is keyset pagination better than OFFSET for YouTube comments? | OFFSET requires scanning and discarding rows to reach the offset — O(offset) cost. Keyset uses the last seen ID as a WHERE clause, which is O(log n) via index. Stable under concurrent inserts. |
| How does YouTube handle like counting on popular videos without a write bottleneck? | Write clicks to Kafka (async). Redis INCR for fast approximate count. Kafka consumer batch-writes to Cassandra every 30s for durability. |
| What is the hybrid fan-out strategy YouTube uses for subscription feeds? | Push (fan-out on write) for channels with < 1M subscribers. Pull (fan-out on read) for celebrity channels with > 1M subscribers to avoid 100M+ writes per upload. |
