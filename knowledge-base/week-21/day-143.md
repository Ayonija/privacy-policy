## Machine Coding: Elevator System

### Problem Statement
Design an elevator system for a 20-floor building with 4 elevators. Support:
- Passengers requesting an elevator from any floor
- Elevator dispatching to the nearest/most optimal elevator
- Open/close door, move up/down, idle
- Concurrent requests from multiple floors

---

### Patterns Used

| Pattern | Role in Elevator System |
|---|---|
| State | Elevator state machine: IDLE / MOVING_UP / MOVING_DOWN / DOOR_OPEN |
| Strategy | Scheduling algorithm: FCFS, SCAN (elevator algorithm), nearest-first |
| Observer | Floor request buttons notify `ElevatorController` of pending requests |
| Facade | `ElevatorController` as single entry point — client calls `requestElevator(floor, dir)` |

---

### Entities

```
Elevator
  - id: int
  - currentFloor: int
  - state: ElevatorState (IDLE / MOVING_UP / MOVING_DOWN / DOOR_OPEN)
  - pendingRequests: PriorityQueue<Request>
  - capacity: int
  - currentLoad: int

Floor
  - floorNumber: int
  - upButton: Button
  - downButton: Button

Request
  - sourceFloor: int
  - destinationFloor: int
  - direction: Direction (UP / DOWN)
  - timestamp: long

ElevatorController (Facade)
  - elevators: List<Elevator>
  - scheduler: SchedulingStrategy
  - requestElevator(floor, direction): void
  - assignRequest(request, elevator): void

Door
  - state: OPEN / CLOSED
  - open(): void
  - close(): void

Button (inside elevator)
  - floor: int
  - press(): void  // adds destination request to elevator queue
```

---

### State Machine

```
State transitions:

IDLE ──────── receive request ──────► MOVING_UP or MOVING_DOWN
MOVING_UP ── reach target floor ───► DOOR_OPEN
MOVING_DOWN ─ reach target floor ──► DOOR_OPEN
DOOR_OPEN ── timer expires + close ► IDLE (if no pending) or MOVING_UP/DOWN (if more requests)

State machine pseudocode:
class ElevatorStateMachine {
    state: ElevatorState

    onRequestReceived() {
        if (state == IDLE) {
            state = (targetFloor > currentFloor) ? MOVING_UP : MOVING_DOWN
        }
        // if already moving, add to queue — direction logic in scheduler
    }

    onFloorReached(floor: int) {
        state = DOOR_OPEN
        door.open()
        wait(DOOR_OPEN_DURATION)
        door.close()
        state = pendingRequests.isEmpty() ? IDLE : nextMoveState()
    }
}
```

---

### Scheduling Strategy (Strategy Pattern)

```
interface SchedulingStrategy {
    selectNextFloor(elevator: Elevator, pendingRequests: List<Request>): int
}

class FCFSStrategy implements SchedulingStrategy {
    selectNextFloor(elevator, requests) {
        return requests.get(0).destinationFloor   // first come, first served
    }
}

class SCANStrategy implements SchedulingStrategy {
    // Elevator sweeps in current direction until no requests in that direction,
    // then reverses — minimizes total travel (elevator algorithm / disk scheduling equivalent)
    selectNextFloor(elevator, requests) {
        if (elevator.state == MOVING_UP) {
            floorsAbove = requests where floor >= currentFloor, sorted ascending
            if (!floorsAbove.isEmpty()) return floorsAbove.first()
            else return requests sorted descending, first()
        }
        // symmetric for MOVING_DOWN
    }
}
```

### Key Methods

```
// ElevatorController (Facade entry point)
requestElevator(floor: int, direction: Direction): void {
    request = new Request(floor, direction)
    bestElevator = selectBestElevator(request)  // nearest idle or same-direction
    bestElevator.addRequest(request)
}

// Elevator
addRequest(request: Request): void {
    synchronized(pendingRequests) {   // thread-safe
        pendingRequests.add(request)
        if (state == IDLE) startMoving()
    }
}

moveToFloor(targetFloor: int): void {
    while (currentFloor != targetFloor) {
        currentFloor += (targetFloor > currentFloor) ? 1 : -1
        Thread.sleep(FLOOR_TRAVEL_TIME)
        checkForPickups(currentFloor)   // pick up passengers on the way
    }
    onFloorReached(targetFloor)
}

openDoor(): void {
    state = DOOR_OPEN
    door.open()
    Thread.sleep(DOOR_OPEN_DURATION)
    door.close()
    state = IDLE
    processNextRequest()
}
```

---

### Thread Safety Note
Concurrent floor requests arrive from multiple floors simultaneously. The `pendingRequests` queue must be thread-safe. Use `synchronized` blocks or `java.util.concurrent.PriorityBlockingQueue`. Elevator state transitions must be atomic — use a `ReentrantLock` or `synchronized` on state changes.

```
// Thread-safe request addition
synchronized void addRequest(Request r) {
    pendingRequests.offer(r)
    if (state == ElevatorState.IDLE) {
        state = computeNextState()
        thread.notify()   // wake up elevator thread
    }
}
```

### Edge Cases

| Edge Case | Handling |
|---|---|
| Overloaded elevator (exceeds capacity) | Reject `addRequest()`, notify controller to find next available elevator |
| Emergency stop | New state `EMERGENCY` — reject all new requests, broadcast alert via Observer |
| Power failure | Persist request queue to durable store; on recovery, reload and resume |
| All elevators busy | Queue request in `ElevatorController`; assign when first elevator becomes IDLE |
| Passenger presses floor already served | Deduplicate requests in `pendingRequests` |

## How to Attack Any Machine Coding Problem: 4-Step Method

**Step 1 — Nouns:** Identify entities (Elevator, Floor, Button, Door, Request, Controller). Each noun is a class candidate.

**Step 2 — Verbs:** Identify behaviors (requestElevator, move, open, close, addRequest). Each verb is a method candidate.

**Step 3 — What Varies:** Ask "what part of this changes?" → scheduling algorithm varies (Strategy), elevator behavior by state varies (State), multiple subscribers want to know about requests (Observer). Apply pattern where variation exists.

**Step 4 — Start Simple:** Begin with one elevator, one floor, one request. Get the state machine working. Add multi-elevator coordination, then concurrency, only when interviewer pushes for it.

## STAR Interview Framework

**Situation:** "Design an elevator system for a 20-floor building with 4 elevators. Thousands of passengers use it daily; multiple requests arrive concurrently from different floors."

**Task:** "Design a system that correctly dispatches elevators, manages state transitions, and handles concurrent requests safely."

**Action:**
1. Identify entities (Elevator, Floor, Request, ElevatorController, Door, Button) and their relationships
2. Model Elevator state machine (IDLE → MOVING_UP/DOWN → DOOR_OPEN → IDLE)
3. Extract scheduling as a Strategy — start with SCAN, mention FCFS and nearest-first as alternatives
4. Use Observer pattern: Floor buttons are subjects; ElevatorController observes and dispatches
5. Wrap with Facade: external callers only call `ElevatorController.requestElevator(floor, direction)`
6. Thread-safe queue for concurrent requests; discuss lock granularity (per-elevator vs global)
7. Handle edge cases: overload, emergency stop, power failure

**Result:** Extensible design — new scheduling algorithm = new Strategy class, zero other changes. Adding floors = update Floor count, zero structural changes. Thread-safety guaranteed by per-elevator synchronized queue. Interviewer can extend the problem in any direction (new elevator type, priority VIP floors) without breaking the design.
