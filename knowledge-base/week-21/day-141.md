# Day 141 — Proxy Pattern (Structural)
**Week 21 | Phase: LLD Mastery | Month 4**

## Focus
The Proxy pattern places a stand-in object in front of a real object to control access, add lazy loading, caching, or logging — without the client knowing. The proxy and the real object implement the same interface, making the substitution transparent. Master the three proxy types: Virtual, Protection, and Remote.

---

## Core Concept

### Analogy
A club bouncer stands in front of the entrance. The bouncer checks IDs before letting you in (Protection Proxy). If the club is still being set up, the bouncer makes you wait until it's ready (Virtual Proxy / lazy load). The bouncer remembers frequent guests so they get through faster (Caching Proxy). You never interact with the club directly — only through the bouncer.

### What It Does
- Puts a stand-in in front of the real object
- Proxy implements the same interface as the real object — client is unaware of the substitution
- Adds control, deferral, caching, or logging without changing the real object

---

## Three Proxy Types

| Type | Purpose | Real-World Example |
|---|---|---|
| Virtual Proxy | Lazy-load expensive object — create it only when first needed | Load high-res image only on first `display()` call |
| Protection Proxy | Access control — check permissions before delegating | Only admin users can call certain methods |
| Remote Proxy | Represents object in another process/machine | Java RMI stub; gRPC client stub |

Bonus:
- **Caching Proxy** — cache results of expensive operations (common interview mention)
- **Logging Proxy** — intercept calls to log method name, args, timing

## Pseudocode — Virtual Proxy (Lazy Load)

```
// Subject interface
interface Image {
    display(): void
}

// Real object — expensive to create
class HeavyImage implements Image {
    private filename: String

    constructor(filename: String) {
        this.filename = filename
        this.loadFromDisk()    // expensive I/O — takes 2 seconds
    }

    display() { println("Displaying " + filename) }
    loadFromDisk() { println("Loading " + filename + " from disk...") }
}

// Virtual Proxy — delays creation until first use
class ImageProxy implements Image {
    private filename: String
    private real: HeavyImage = null   // not loaded yet

    constructor(filename: String) {
        this.filename = filename      // no I/O here
    }

    display() {
        if (real == null) {
            real = new HeavyImage(filename)   // lazy load on first call
        }
        real.display()
    }
}

// Client code — same interface, unaware of proxy
images = [new ImageProxy("photo1.jpg"), new ImageProxy("photo2.jpg")]
// No disk I/O yet. Images appear in gallery.
images[0].display()   // triggers load only for photo1; photo2 still not loaded
```

## Pseudocode — Protection Proxy

```
class DatabaseProxy implements Database {
    private real: RealDatabase
    private currentUser: User

    query(sql: String): ResultSet {
        if (sql.contains("DROP") && !currentUser.isAdmin()) {
            throw AccessDeniedException("Insufficient permissions")
        }
        return real.query(sql)
    }

    write(sql: String): void {
        if (!currentUser.canWrite()) {
            throw AccessDeniedException("Read-only user")
        }
        return real.write(sql)
    }
}
```

## Caching Proxy Pattern (Important for System Design)

```
class CachingDatabaseProxy implements Database {
    private real: RealDatabase
    private cache: Map<String, ResultSet> = {}

    query(sql: String): ResultSet {
        if (cache.has(sql)) {
            return cache.get(sql)     // cache hit — skip real DB call
        }
        result = real.query(sql)
        cache.put(sql, result)
        return result
    }
}
```

## Real-World Uses

| Context | Proxy Type |
|---|---|
| Lazy-loaded images in browser / mobile | Virtual |
| Spring `@Transactional` — wraps method with transaction management | Proxy (AOP-generated) |
| Spring `@Cacheable` — wraps method to cache return values | Caching |
| Java RMI / gRPC client stubs | Remote |
| OAuth middleware — validate token before handler | Protection |
| Rate limiting (allow N calls/minute) | Protection / Caching |
| Hibernate entity lazy loading | Virtual |

---

## Don't Confuse

| Pattern | Intent | Behavior of Wrapped Object |
|---|---|---|
| Proxy | CONTROL access — lazy load, cache, auth, log | Core behavior unchanged; proxy adds indirection |
| Decorator | ADD new behavior/features at runtime | Behavior augmented — new cost, new description, new output |

**One-line test:** Is the core behavior the same but gated/deferred/cached? → Proxy. Am I genuinely adding new capability? → Decorator.

## STAR Interview Framework

**Situation:** "Design a database access layer for a read-heavy analytics service. Read queries are expensive (50–200ms each) and many queries repeat within short time windows. Write queries must be rate-limited to protect the database from bulk import storms."

**Task:** "Add caching for reads and rate limiting for writes transparently — existing service code should not change."

**Action:**
1. Define `Database` interface with `query(sql): ResultSet` and `write(sql): void`
2. Implement `RealDatabase` with actual DB connection
3. Build `CachingRateLimitedProxy implements Database` — composition over `RealDatabase`
4. In `query()`: check in-memory LRU cache; on hit return immediately; on miss delegate to real, store result with TTL
5. In `write()`: check rate limit counter (token bucket); reject if over limit; otherwise delegate to real and invalidate affected cache keys
6. Inject proxy instead of real database in service constructor — zero changes to service code

**Result:** Read query latency drops 80% for repeated queries. Write rate capped at 100/minute protects DB during batch imports. Service code is completely unmodified — all access control logic lives in the proxy layer. Adding a logging proxy on top requires zero service changes (stack proxies if needed, though that edges toward Decorator territory).

## Flashcards

| Q | A |
|---|---|
| What is a Virtual Proxy? | Creates the real object lazily — only on first actual use — to defer expensive initialization |
| What is a Protection Proxy? | Checks permissions before delegating to the real object; client gets AccessDenied if unauthorized |
| Proxy vs Decorator — one-line distinction | Proxy controls access (behavior unchanged); Decorator adds new behavior (behavior augmented) |
| How does the client know it's talking to a proxy vs the real object? | It doesn't — both proxy and real object implement the same interface |
| Spot trigger for Proxy | "Control access/permissions", "lazy-load expensive object", "add caching or logging transparently" |
