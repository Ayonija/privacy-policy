# Day 142 — Confusion Pairs: Structural Patterns
**Week 21 | Phase: LLD Mastery | Month 4**

## Focus
The most-failed part of structural pattern interviews is confusing pairs that look similar on the surface. This day locks in the precise distinctions between Adapter/Facade and Decorator/Proxy, then adds three bonus GoF structural patterns (Composite, Flyweight, Bridge) that frequently appear in senior-level LLD rounds.

---

## Confusion Pair 1: Adapter vs Facade

| Dimension | Adapter | Facade |
|---|---|---|
| Problem | Two existing, incompatible interfaces need to work together | One complex subsystem is hard to use directly |
| Direction | Wraps one interface to match another | Wraps multiple classes under one simplified API |
| What changes | A new adapter class is added | A new facade class is added |
| Subsystem modified? | No — adaptee unchanged | No — subsystems unchanged |
| Number of things involved | Two (target interface + adaptee) | Many (one facade over entire subsystem) |
| Analogy | Travel plug adapter | Restaurant waiter |

**Decision test:** "Is the problem incompatibility (Adapter) or complexity (Facade)?"

**Trap question:** "I have a PaymentGateway with 15 methods but I only want to expose `charge()` and `refund()`." → **Facade** (simplifying complexity). "I have a PaymentGateway with interface X but my system expects interface Y." → **Adapter** (incompatibility).

## Confusion Pair 2: Decorator vs Proxy

| Dimension | Decorator | Proxy |
|---|---|---|
| Problem | Add new behavior/features at runtime | Control access: lazy load, cache, auth, log |
| Core behavior | Changes — new cost, new output, new capability | Unchanged — proxy defers/gates the same behavior |
| Intent | Enrich the object | Intercept/control access to the object |
| Stacking | Designed to be stacked (coffee + milk + sugar) | Typically one proxy per real object |
| Analogy | Pizza toppings layered on base | Club bouncer in front of entrance |

**Decision test:** "Am I adding capability (Decorator) or controlling access (Proxy)?"

**Trap question:** "Add logging to every method call." → **Proxy** (control/intercept, no new functionality). "Add compression to a data writer." → **Decorator** (new behavior — data is now compressed, not just gated).

## Bonus Pattern: Composite

**Problem:** You need to treat individual items and collections of items identically.

**Analogy:** Folders and files — a folder can contain files or other folders, but `open()` and `getSize()` work the same on both.

**Structure:**
```
interface Component {
    display(): void
    getSize(): int
}

class File implements Component {        // Leaf
    display() { print filename }
    getSize() { return fileSize }
}

class Folder implements Component {      // Composite
    private children: List<Component>
    add(c: Component) { children.add(c) }
    display() { for each child: child.display() }
    getSize() { return sum of child.getSize() }
}
```

**Key three classes:** Component (interface), Leaf, Composite (holds children list)

**Use when:** file systems, UI component trees (widget containing sub-widgets), org hierarchies (manager contains employees), arithmetic expression trees.

**Spot trigger:** "Treat leaf and container uniformly", "recursive tree structure", "uniform interface over part-whole hierarchy"

## Bonus Pattern: Flyweight

**Problem:** Creating millions of similar objects wastes memory because most of their state is identical.

**Analogy:** Chess game with 16 pawns. Each pawn has the same image, color, movement rules (intrinsic state). Only position on the board differs per pawn instance (extrinsic state). Share one `PawnType` object; pass position as a parameter at draw time.

**Key concept:**
- **Intrinsic state** — shared, immutable, stored inside Flyweight object (e.g., glyph shape, color, texture)
- **Extrinsic state** — unique per instance, passed in at use time (e.g., position, context)

```
class PawnFlyweight {
    private color: String     // intrinsic — shared
    private image: Bitmap     // intrinsic — shared
    draw(x: int, y: int) {   // x, y are extrinsic — passed in
        renderAt(image, x, y)
    }
}

class FlyweightFactory {
    private cache: Map<String, PawnFlyweight> = {}
    get(color: String): PawnFlyweight {
        if (!cache.has(color)) cache.put(color, new PawnFlyweight(color))
        return cache.get(color)
    }
}
```

**Use when:** game entities (particles, trees, soldiers), font glyph rendering, large-scale UI icon sets.

**Spot trigger:** "Millions of similar objects", "most state is shared", "memory-intensive system with repeated data"

## Bonus Pattern: Bridge

**Problem:** When both abstraction and implementation have multiple variants, inheritance explodes. Bridge decouples them so each can vary independently.

**Analogy:** A remote control (abstraction) and a TV brand (implementation) are separate. You can have Sony TV + Basic Remote, Sony TV + Smart Remote, LG TV + Basic Remote — without a class per combination.

**Structure:**
```
// Implementation hierarchy
interface Renderer {
    render(shape: String): void
}
class VectorRenderer implements Renderer { ... }
class RasterRenderer implements Renderer { ... }

// Abstraction hierarchy — holds reference to Renderer (bridge)
abstract class Shape {
    protected renderer: Renderer
    constructor(r: Renderer) { this.renderer = r }
    abstract draw(): void
}
class Circle extends Shape {
    draw() { renderer.render("circle") }
}
```

**Use when:** abstraction and implementation both need to change independently, cross-platform UI rendering, database drivers.

**Spot trigger:** "Avoid permanent binding between abstraction and implementation", "multiple orthogonal dimensions of variation"

---

## Structural Pattern Decision Tree

```
Problem involves:
├── Two things with incompatible interfaces → Adapter
├── One complex subsystem, too many classes → Facade
├── Adding new behavior/features at runtime → Decorator
├── Controlling access (lazy, cache, auth, log) → Proxy
├── Treating leaf and container uniformly (tree) → Composite
├── Millions of similar objects, shared state → Flyweight
└── Two orthogonal hierarchies varying independently → Bridge
```

## STAR Interview Framework

**Situation:** "You're interviewing at a company designing a document editor. The interviewer asks: 'When would you use Proxy vs Decorator? Give real-world examples.'"

**Task:** "Give a precise, confident distinction with concrete examples — not a vague 'both wrap objects' answer."

**Action:**
1. State the core distinction: "Proxy controls access — behavior is unchanged but you add indirection. Decorator adds new behavior — the wrapped object's output or capability genuinely changes."
2. Proxy example: "In our document editor, loading high-resolution page previews for a 500-page PDF is expensive. A Virtual Proxy defers loading until the user scrolls to that page. The behavior — rendering the page — is identical; we just delay it."
3. Decorator example: "When a user marks a paragraph as bold and italic, we wrap the `TextComponent` with a `BoldDecorator`, then an `ItalicDecorator`. Each adds rendering behavior. The same `render()` interface is maintained but the output now includes bold+italic styling."
4. Distinguish stacking: "Decorators are designed to stack (bold + italic + underline). Proxies are typically singular — one layer of indirection between client and real object."

**Result:** Interviewer sees you understand the structural similarity (same interface wrapping) but precise divergence in intent. Follow-up readiness: "What about a logging Proxy that logs method calls — isn't that new behavior?" → "Logging is cross-cutting infrastructure, not domain behavior. The method's contract is unchanged — same output, same side effects. That's why AOP and Proxy are used for logging, not Decorator."

## Flashcards

| Q | A |
|---|---|
| Adapter vs Facade — one-line decision test | Incompatibility between two interfaces → Adapter. Complexity of one subsystem → Facade. |
| Decorator vs Proxy — one-line decision test | Adding capability → Decorator. Controlling access / deferring / caching → Proxy. |
| In Composite, what are the three key classes? | Component (interface), Leaf (no children), Composite (holds List of Component children) |
| In Flyweight, what is intrinsic vs extrinsic state? | Intrinsic: shared, immutable, stored in flyweight (e.g., glyph shape). Extrinsic: per-instance, passed at use time (e.g., position) |
| When does Bridge beat inheritance? | When abstraction and implementation both have multiple variants — Bridge lets each vary independently; inheritance would require a class per combination |
| Spot trigger for Flyweight | "Millions of similar objects", "memory-intensive system with repeated state", "game particles / font glyphs" |
| Spot trigger for Composite | "Treat leaf and container uniformly", "recursive tree structure", "part-whole hierarchy" |
| Spot trigger for Bridge | "Two orthogonal dimensions of variation", "avoid binding abstraction to implementation", "cross-platform rendering" |
