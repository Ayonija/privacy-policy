## Class Relationships

### Association — "uses-a" (loose, no ownership)
- One class uses another, but neither owns the other
- Both can exist independently and have their own lifecycle
- Relationship can be one-way or bidirectional
- Example: `Student` uses `Library` — a student can exist without a library; a library exists without any particular student

```
// Association: Student has a reference to Library, but doesn't own it
class Student {
    private library: Library   // reference, not owned

    borrowBook(bookId: String) {
        library.checkOut(bookId, this)
    }
}
```

UML: **solid line** between classes (no diamond, no arrow unless directed)

---

### Aggregation — "has-a" (weak ownership)
- Container holds parts, but parts exist independently of the container
- Parts can be shared between multiple containers
- Destroying the container does not destroy the parts
- Example: `Department` has `Professors` — professors existed before joining the department and can move to another department or exist without one

```
class Department {
    private professors: List<Professor>   // professors exist independently

    addProfessor(p: Professor) { professors.add(p) }
    removeProfessor(p: Professor) { professors.remove(p) }
    // When department is dissolved, Professor objects still live
}
```

UML: **hollow diamond** on the container side (Department ◇—— Professor)

---

### Composition — "has-a" (strong ownership)
- Container owns parts; parts cannot exist without the container
- Parts are created by the container and destroyed when container is destroyed
- Example: `House` has `Rooms` — a room has no meaning outside a house; when house is demolished, rooms cease to exist

```
class House {
    private rooms: List<Room>

    constructor(numberOfRooms: int) {
        for i in 0..numberOfRooms:
            rooms.add(new Room())   // House creates Room
    }
    // When House is garbage-collected, Room objects are also collected
    // No external reference to Room exists
}
```

UML: **filled (solid) diamond** on the container side (House ◆—— Room)

### All Five UML Arrows

| Relationship | Arrow Type | Example |
|---|---|---|
| Association | Solid line (no diamond) | Student — Library |
| Aggregation | Hollow diamond on owner end | Department ◇—— Professor |
| Composition | Filled diamond on owner end | House ◆—— Room |
| Inheritance (extends) | Solid line + hollow triangle arrowhead (pointing to parent) | Dog ——▷ Animal |
| Implementation (implements) | Dashed line + hollow triangle arrowhead (pointing to interface) | Dog - - -▷ Animal (interface) |

**Memory trick:**
- Diamond = ownership. Hollow = weak (parts survive). Filled = strong (parts die with owner).
- Dashed = interface/abstract (you're implementing a contract, not inheriting implementation).

---

### When to Use Each in LLD Interviews

| Choose | When |
|---|---|
| Association | Two classes reference each other but neither owns the other; relationship is behavioral |
| Aggregation | Container holds a collection but the elements preexist and can be re-used elsewhere |
| Composition | One entity is a fundamental part of another; separate existence makes no sense |

**Interview tell:** If the interviewer asks "can X exist without Y?" — Yes → Aggregation or Association. No → Composition.

Examples:
- `Order` and `OrderItem` → **Composition** (item has no meaning without the order)
- `Library` and `Books` → **Aggregation** (books can exist in multiple libraries or on their own)
- `Driver` and `Car` → **Association** (driver and car exist independently; driver uses a car)

## LLD Interview Strategy — Full 5-Step Method

### Step 1: Clarify Requirements (5 minutes)
Ask before designing. Interviewers reward structured questions.

Functional:
- What are the core use cases? (e.g., "User can borrow a book, return a book, search catalog")
- What are the actors? (User, Admin, System?)
- Any specific workflows to design? (payment flow, notification flow?)

Non-functional:
- Scale? (1K users vs 10M users changes caching/database choices)
- Consistency requirements? (eventual vs strong)
- Any real-time requirements? (alerts, notifications?)

**Do not design until you have answers to at least 3 functional and 1 non-functional question.**

---

### Step 2: Identify Entities → Classes (nouns)
Read the problem and underline every noun. Each is a class candidate.

"Design a parking lot" → ParkingLot, Floor, ParkingSpot, Vehicle, Ticket, PaymentGateway, Gate, Display

Filter out nouns that are attributes (not entities): "color of car" → `Car.color`, not a class.

Draw class names in a rough box layout before adding any methods or relationships.

---

### Step 3: Identify Behaviors → Methods (verbs)
For each class, ask "what does this thing DO?"

"Design a parking lot" → ParkingLot.findAvailableSpot(), Ticket.calculateFee(), Gate.open(), PaymentGateway.charge()

Assign verbs to the noun that owns the responsibility. Avoid "god class" that does everything.

---

### Step 4: Find What VARIES → Apply Pattern
Ask: "What part of this system is likely to change?"

| What varies | Apply |
|---|---|
| Algorithm or policy (pricing, routing, sorting) | Strategy |
| Object behavior based on its state | State |
| Feature combinations (add-ons, options) | Decorator |
| Type of object to create | Factory / Abstract Factory |
| Notification to multiple receivers | Observer |
| Access control or lazy loading | Proxy |
| Simplifying a complex API surface | Facade |
| Incompatible interfaces | Adapter |

Only apply a pattern where you can articulate what specifically varies. Do not pattern-spray.

---

### Step 5: Start Simple, Add Complexity on Push
Start with the minimal working design — one parking spot type, one payment method, no concurrency.

When interviewer says "what if we have multiple vehicle types?" → add inheritance / Strategy.
When interviewer says "what if this is distributed?" → add thread safety, queues, etc.
When interviewer says "how would this scale?" → discuss caching, database indexing, horizontal scaling.

**Interviewers test depth by pushing. Save the complexity discussion for when they ask.**

## Common Mistakes in LLD Interviews

| Mistake | Correct Approach |
|---|---|
| Forcing patterns everywhere ("I used 7 patterns") | Apply patterns only where variation exists; explain why |
| God class (one class with 20 methods) | Distribute responsibilities — single-responsibility principle |
| Singleton overuse for shared objects | Prefer dependency injection; Singleton makes testing hard |
| Coding to concrete classes | Use interfaces and inject dependencies; enables testing and substitution |
| Jumping to code without design | Draw class names + relationships first; code only when design is confirmed |
| Ignoring non-functional requirements | State that you're designing for the given scale upfront |
| Not naming patterns explicitly | Say "I'd use Strategy here" — interviewers reward vocabulary |

## Interviewer Trigger → Pattern Mapping

| Interviewer says... | Reach for... |
|---|---|
| "What if we want to add new payment methods later?" | Strategy |
| "The system has different behaviors depending on status" | State |
| "Add optional features to the order dynamically" | Decorator |
| "Notify multiple services when an event occurs" | Observer |
| "Create objects without specifying exact class" | Factory / Abstract Factory |
| "There should only be one instance" | Singleton (use sparingly) |
| "Hide the complexity of this subsystem" | Facade |
| "Integrate this third-party library" | Adapter |
| "Delay loading until needed" | Proxy (Virtual) |
| "The tree structure should be uniform" | Composite |
| "We have millions of similar objects" | Flyweight |

## STAR Interview Framework

**Situation:** "The interviewer says: 'Walk me through how you approach an LLD design problem from scratch. How do you decide which patterns to use?'"

**Task:** "Demonstrate a structured, repeatable methodology — not guesswork or pattern-matching by feel."

**Action:**
1. "I start by clarifying — I ask for 2-3 functional requirements and one non-functional constraint before drawing anything. This prevents designing the wrong system."
2. "Then I identify entities by underlining nouns. Each noun is a class candidate. I immediately ask 'is this an attribute or an entity?' to avoid bloating the design."
3. "For behaviors, I go class by class and assign verbs. I watch for god-class signs — if one class has 8+ methods, I split it."
4. "The most important step: I ask 'what varies here?' For each variation point, I name the pattern that handles that kind of variation. I don't apply patterns blindly — I apply them where I can articulate the need."
5. "I start simple and explicitly tell the interviewer: 'This is my base design. I can add [caching / polymorphism / concurrency] if we want to go deeper.' This invites a collaborative extension conversation."

**Result:** Every design I deliver has a clear rationale for each pattern used. When interviewers push — 'what if we scale to 10M users?' — I have a pre-planned extension point because I identified what varies in Step 4. The methodology itself becomes a signal of seniority.
