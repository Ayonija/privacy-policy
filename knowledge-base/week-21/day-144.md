## DSA (1.5 hours)

### Pattern: State Machine DP on Graph / Circular Structure
- Core idea: Define `dp[i][j]` as the minimum cost to spell `ring[0..i-1]` characters with the ring's current position at index `j`. Transitions involve rotating the ring clockwise or counterclockwise to reach the character needed for the next step.
- Trigger condition: You have a current "position" in a circular or graph structure, and actions have costs based on the transition between states. The problem asks for the optimal sequence of transitions.
- Time complexity: O(m×n²) where m = key length, n = ring length | Space complexity: O(m×n), optimizable to O(n)

### Pattern: Interval Scheduling DP with Binary Search
- Core idea: Sort jobs by end time. `dp[i]` = max profit using jobs from `[0..i]`. For each job `i`, binary search for the latest non-overlapping job `p(i)`. Recurrence: `dp[i] = max(dp[i-1], profit[i] + dp[p(i)])`.
- Trigger condition: Jobs have start time, end time, and value. Maximize total value without scheduling overlap. Classic weighted interval scheduling.
- Time complexity: O(n log n) | Space complexity: O(n)

    public int rob(int[] nums) {
        int prev2 = 0, prev1 = 0;
        for (int num : nums) {
            int cur = Math.max(prev1, prev2 + num);
            prev2 = prev1;
            prev1 = cur;
        }
        return prev1;
    }

## System Design (1 hour)

### Topic: YouTube / Netflix — Recommendation Engine

#### Why Recommendations Matter
- YouTube: 70% of watch time comes from recommended videos (up from 60% in 2016).
- Netflix: 75–80% of content watched is driven by the recommendation engine.
- Getting recommendations right is the single biggest lever for increasing engagement and retention.

#### Two Core Approaches

**1. Collaborative Filtering (CF)**
- Premise: "Users similar to you liked this video."
- Build a user-item matrix: rows = users, columns = videos, cells = rating/watch fraction.
- Matrix factorization (ALS, SVD): decompose into user-embedding matrix U (users × K) and item-embedding matrix V (items × K). K = 32–256 latent factors.
- Recommendation: for user u, compute `U[u] · V[v]` for all videos v; rank by score.
- Problem: cold start — new users or new videos have no history.

**2. Content-Based Filtering**
- Premise: "You liked videos with these features; here are similar ones."
- Extract video features: title embeddings (BERT), tags, category, duration, audio/visual features (CNN over frames).
- Build video-feature matrix; compute cosine similarity between videos.
- Works for new videos (no cold start problem).

**3. Two-Tower Neural Network (Modern Approach)**
```

──────────────                ──────────────
user_id embedding             video_id embedding
watch_history embedding       title embedding (BERT)
location, time-of-day         tags, category

        ▼                             ▼

        │                             │
        └───────────┬─────────────────┘
                    ▼

                    ▼

```

#### Recommendation Pipeline

```

│                   OFFLINE TRAINING (daily/weekly)          │
│                                                            │
│  Watch logs ──► Feature Engineering ──► Model Training     │
│  (Hadoop/Spark)   (Spark/Flink)         (TensorFlow)       │
│                                                            │
│  Output: Updated user embeddings + item embeddings         │
│          Written to feature store (Redis / Feast)          │
└────────────────────────────────────────────────────────────┘
                          │ embeddings
                          ▼
┌────────────────────────────────────────────────────────────┐
│                   CANDIDATE GENERATION (online, ~100ms)    │
│                                                            │
│  User request ──► Retrieve user embedding from Redis       │
│                ──► ANN search (Faiss / ScaNN) over         │
│                    item embeddings → top 500 candidates    │
└────────────────────────────────────────────────────────────┘
                          │ 500 candidates
                          ▼
┌────────────────────────────────────────────────────────────┐
│                   RANKING (online, ~20ms)                  │

                          │ top 20–50 results
                          ▼
                   Homepage / Sidebar feed
```

#### Scale Numbers
| Component | Scale |
|-----------|-------|
| User-video interactions per day | ~5B events (YouTube) |
| Embedding dimension K | 64–256 |
| Candidate generation latency | < 100ms |
| Ranking latency | < 20ms |
| Model retraining frequency | Daily (batch), Hourly (incremental) |
| Feature store reads | Millions per second (Redis) |

#### Database Choices
| Data | Storage | Reason |
|------|---------|--------|
| User embeddings | Redis (in-memory) | Low-latency read at serving time |
| Item embeddings | Faiss index (in-memory or SSD) | ANN similarity search |
| Watch history | DynamoDB | Scalable key-value, per-user partition |
| Model weights | S3 | Versioned, large file |
| Training data | S3 + Apache Parquet | Batch ML training |

**Interview talking point:** "If asked how YouTube serves personalized recommendations at 2B users × < 100ms latency, answer: Two-stage pipeline — candidate generation retrieves top 500 videos in ~50ms using approximate nearest neighbor (ANN) search over pre-computed embeddings stored in memory. A lightweight ranking model then scores 500 → 20 in ~20ms. Embeddings are updated offline daily, not computed on-the-fly per request."
