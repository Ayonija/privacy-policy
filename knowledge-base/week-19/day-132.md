# Day 132 — Trees Hard: BST Recovery & Camera Placement
**Week 19 | Phase 3 — Mock Interviews | Month 5**

## Focus
Two complementary hard tree problems — restoring a BST where two nodes were swapped using inorder violation detection (LC 99) and optimally placing cameras to cover every node with the minimum count using greedy post-order DP (LC 968) — both test deep structural understanding of binary trees.

---

## DSA (1.5 hours)

### Pattern: Inorder Traversal Violation Detection (Recover BST)
- For LC 99, inorder traversal of a valid BST produces a strictly sorted sequence. Two swapped nodes create one or two inversions. Track `prev` (the last visited node). First inversion: `first = prev, second = curr`. If a second inversion occurs later, update `second = curr`. After traversal, swap `first.val` and `second.val`.
- Trigger condition: "BST where exactly two nodes are swapped — find them in O(n) without extra space." Inorder traversal exposes the violation.
- Time complexity: O(n) | Space complexity: O(h) recursive stack; O(1) with Morris traversal

### Pattern: Greedy Post-Order DP (Minimum Camera Cover)
- For LC 968, assign each node one of three states returned by the helper: 0 = not covered (needs camera from parent), 1 = covered (no camera here), 2 = has camera. Post-order logic: if any child returns 0 → place camera here (return 2); if any child returns 2 → covered without camera (return 1); if all children return 1 → return 0 (signal parent to cover us). If root returns 0, add one more camera.
- Trigger condition: "Minimum cameras to cover all nodes in a binary tree — each camera covers the node, its parent, and its children."
- Time complexity: O(n) | Space complexity: O(h)

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Recover Binary Search Tree | 99 | Hard | Inorder Traversal + Two-Pointer Violation | Two swapped nodes create 1–2 inversions in inorder; `first = prev` at first inversion, `second = curr` at each inversion |
| 2 | Binary Tree Cameras | 968 | Hard | Greedy Post-Order DP | Place camera at a node only when a child signals it's uncovered (state 0); leaf nodes always signal state 0 to force camera placement at their parent |
| 3 | Lowest Common Ancestor of BST | 235 | Easy (Revision) | BST Property Traversal | REVISION: if both p, q < node → go left; if both > node → go right; else current node is LCA |

### Code Skeleton

```java
class Solution {

    // LC 99 — Recover Binary Search Tree
    private TreeNode first = null, second = null, prev = null;

    public void recoverTree(TreeNode root) {
        inorder(root);
        // Swap the values of the two misplaced nodes
        int temp = first.val;
        first.val = second.val;
        second.val = temp;
    }

    private void inorder(TreeNode node) {
        if (node == null) return;
        inorder(node.left);

        // Detect inversion: prev should always be less than current in a valid BST
        if (prev != null && prev.val > node.val) {
            if (first == null) first = prev;  // first violation: prev is the larger displaced node
            second = node;                     // second is updated at every inversion
        }
        prev = node;

        inorder(node.right);
    }


    // LC 968 — Binary Tree Cameras
    private int cameras = 0;

    public int minCameraCover(TreeNode root) {
        // If root itself is uncovered after traversal, place one more camera
        if (dfs(root) == 0) cameras++;
        return cameras;
    }

    // Returns: 0 = not covered, 1 = covered (no camera), 2 = has camera
    private int dfs(TreeNode node) {
        if (node == null) return 1;  // null nodes are considered "covered"

        int left = dfs(node.left);
        int right = dfs(node.right);

        if (left == 0 || right == 0) {
            // At least one child is uncovered → must place camera here
            cameras++;
            return 2;
        }
        if (left == 2 || right == 2) {
            // At least one child has a camera → this node is covered
            return 1;
        }
        // Both children are covered but have no cameras → signal parent to cover us
        return 0;
    }
}

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Ride Matching & Geospatial Storage

**Core components:**
- **Driver Location Service:** Drivers send GPS pings every 4s. Each ping is an HTTP/WebSocket event → Kafka topic `driver-location-updates` → Location Worker updates Redis Geo index: `GEOADD drivers:{city} lng lat driver_id`
- **Geospatial Index:** Redis `GEOSEARCH` or legacy `GEORADIUS` queries drivers within a radius. Data is stored as sorted sets with geohash scores — O(log n + m) per query
- **Matching Service:** On new ride request: (1) query Redis for drivers within 2km radius, sorted by distance; (2) filter by availability (driver status = AVAILABLE in Redis hash); (3) attempt to lock driver using `SET driver:{id}:status MATCHED NX EX 30` (atomic, prevents double-matching); (4) if lock fails, try next driver; (5) publish match event to Kafka
- **Supply/Demand Zones:** City divided into H3 hexagonal cells at resolution 8 (~0.5 km²). Aggregate driver count per cell in Redis. Used by surge pricing and dispatch to find supply-rich zones quickly

**Key trade-offs:**
- **Radius query vs zone-based:** radius is precise but scans variable counts; zone-based is O(1) lookup but has cell-boundary gaps — Uber uses zone-based for surge, radius for precise matching
- **Locking driver with Redis NX vs DB transaction:** Redis NX lock is fast (~1ms) but can fail on Redis restart; DB transaction is durable but adds 10-50ms latency — Uber uses Redis NX with short TTL (30s) as optimistic lock and DB as truth of record
- **Driver state machine:** drivers cycle OFFLINE → AVAILABLE → MATCHED → IN_TRIP → AVAILABLE; state must be consistent — store in Redis for latency, async sync to PostgreSQL for durability

**Interview talking point:** "If asked how the matching service prevents two riders from being assigned the same driver simultaneously, answer: The matching service uses Redis SET NX (set if not exists) as an atomic lock: `SET driver:{id}:status MATCHED NX EX 30`. If two matching workers race, only one wins the atomic set; the other sees the lock exists and moves to the next candidate driver. The TTL ensures the lock expires if the transaction never completes."

## Flashcards

| Q | A |
|---|---|
| In LC 99, why does a single inversion in inorder traversal mean the two swapped nodes are adjacent? | If the two displaced nodes are adjacent in inorder order, they create exactly one pair where `prev.val > curr.val`; `first = prev` and `second = curr` at that single inversion |
| In LC 99, why do you update `second = curr` at every inversion rather than only the first? | The true second node (the smaller displaced one) may be far from the first; there can be two inversions; the second inversion's `curr` is the correct second node |
| In LC 968, why do null nodes return state 1 (covered) rather than 0? | Null nodes don't exist and don't need cameras; treating them as "covered" prevents false signals that would cause cameras to be placed unnecessarily at leaf nodes' parents |
| In LC 968, what is the greedy argument for placing the camera at the parent of an uncovered leaf rather than at the leaf itself? | A camera at the parent covers the leaf, the parent, and the grandparent — one camera covers more nodes than a camera at the leaf, which only covers the leaf and its parent |
| In Uber's matching, why use Redis NX lock with TTL rather than a database transaction for driver assignment? | Redis NX is atomic and sub-millisecond; a DB transaction under 200K concurrent requests/minute would bottleneck on row-level locking; TTL ensures stale locks self-expire |
