## DSA (1.5 hours)

### Pattern: BFS with Column/Row Coordinate Tracking
- For LC 987, assign each node a (col, row) coordinate during BFS: root is (0,0); left child is (col-1, row+1); right child is (col+1, row+1). Group nodes by column using a TreeMap (sorted by col). Within each column, sort by (row ascending, val ascending). Collect values column by column.
- Trigger condition: "Return nodes of a binary tree grouped by vertical column, sorted top-to-bottom and left-to-right within ties."
- Time complexity: O(n log n) for sorting | Space complexity: O(n)

### Pattern: Two-Pass Tree DP (Re-rooting Technique)
- For LC 834, first DFS from root: compute `count[v]` = subtree size rooted at v, and `ans[root]` = sum of distances from root to all other nodes. Second DFS: for each child c of parent p, `ans[c] = ans[p] - count[c] + (n - count[c])`. Moving the "root" from p to c gains `n - count[c]` distance reductions and adds `count[c]` distance increments.
- Trigger condition: "Compute a quantity at every node where results depend on the full tree structure — re-rooting avoids O(n²) by propagating parent's answer to child."
- Time complexity: O(n) | Space complexity: O(n)

### Code Skeleton

```java
class Solution {

    // LC 987 — Vertical Order Traversal of a Binary Tree
    public List<List<Integer>> verticalTraversal(TreeNode root) {
        // TreeMap<col, TreeMap<row, PriorityQueue<val>>> for automatic sorting
        TreeMap<Integer, TreeMap<Integer, PriorityQueue<Integer>>> map = new TreeMap<>();

        // Use a queue of (node, col, row)
        Deque<Object[]> bfsQueue = new ArrayDeque<>();
        bfsQueue.offer(new Object[]{root, 0, 0});

        while (!bfsQueue.isEmpty()) {
            Object[] curr = bfsQueue.pollFirst();
            TreeNode node = (TreeNode) curr[0];
            int col = (int) curr[1];
            int row = (int) curr[2];

            map.computeIfAbsent(col, k -> new TreeMap<>())
               .computeIfAbsent(row, k -> new PriorityQueue<>())
               .offer(node.val);

            if (node.left != null) bfsQueue.offer(new Object[]{node.left, col - 1, row + 1});
            if (node.right != null) bfsQueue.offer(new Object[]{node.right, col + 1, row + 1});
        }

        List<List<Integer>> result = new ArrayList<>();
        for (TreeMap<Integer, PriorityQueue<Integer>> colMap : map.values()) {
            List<Integer> colList = new ArrayList<>();
            for (PriorityQueue<Integer> pq : colMap.values()) {
                while (!pq.isEmpty()) colList.add(pq.poll());
            }
            result.add(colList);
        }
        return result;
    }


    // LC 834 — Sum of Distances in Tree
    private int[] count, ans;
    private List<List<Integer>> graph;

    public int[] sumOfDistancesInTree(int n, int[][] edges) {
        count = new int[n];  // subtree sizes
        ans = new int[n];
        graph = new ArrayList<>();
        for (int i = 0; i < n; i++) graph.add(new ArrayList<>());
        Arrays.fill(count, 1);  // each node counts itself

        for (int[] e : edges) {
            graph.get(e[0]).add(e[1]);
            graph.get(e[1]).add(e[0]);
        }

        dfs1(0, -1);   // compute count[] and ans[0]
        dfs2(0, -1);   // propagate ans to all nodes
        return ans;
    }

    private void dfs1(int node, int parent) {
        for (int neighbor : graph.get(node)) {
            if (neighbor == parent) continue;
            dfs1(neighbor, node);
            count[node] += count[neighbor];
            ans[node] += ans[neighbor] + count[neighbor];  // distance to subtree root + size of subtree
        }
    }

    private void dfs2(int node, int parent) {
        for (int neighbor : graph.get(node)) {
            if (neighbor == parent) continue;
            int n = count.length;
            // Moving root from node to neighbor: lose count[neighbor] close ones, gain n-count[neighbor] far ones
            ans[neighbor] = ans[node] - count[neighbor] + (n - count[neighbor]);
            dfs2(neighbor, node);
        }
    }
}

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Real-Time Location Tracking

**Core components:**
- **Driver app** sends GPS coordinates every 4 seconds via persistent WebSocket or HTTPS to the Location Gateway
- **Location Gateway** — WebSocket servers that maintain persistent connections with driver apps; receives 1.25M location events/sec at peak (5M drivers × 0.25/sec)
- **Kafka topic `driver-location`** — Location Gateway publishes each event; decouples ingestion from processing; provides replay for downstream consumers
- **Location Worker** — Kafka consumer; updates Redis Geo index (`GEOADD`) and driver's last-known position in a Redis hash (`driver:{id}:lat`, `driver:{id}:lng`, `driver:{id}:updated_at`); also updates driver's H3 cell membership for zone-level analytics
- **Location History DB** — async writer persists location trail to Cassandra (time-series, high write throughput) for trip replay and dispute resolution
- **Staleness detection** — a background job scans driver Redis keys; if `updated_at` is older than 30s, mark driver STALE and remove from active geospatial index

**Key trade-offs:**
- **WebSocket vs HTTP polling:** WebSocket is persistent — lower overhead per location event (no HTTP headers), real-time bidirectional; HTTP polling adds latency and header overhead at 5M drivers × 0.25 req/sec = significant bandwidth
- **Redis Geo vs PostGIS:** Redis Geo is in-memory and O(log n) for radius queries; PostGIS supports complex geospatial operations but is disk-based with higher latency — Uber uses Redis for real-time matching, PostGIS for analytics
- **Cassandra for history:** location history is append-only, time-series — Cassandra's wide-row model (partition key = `driver_id`, clustering key = `timestamp`) gives O(1) write and efficient range reads for a driver's recent path

**Interview talking point:** "If asked how Uber handles 1.25M location updates per second, answer: The Location Gateway is a fleet of WebSocket servers behind a load balancer. Each server maintains ~10K persistent driver connections. Events are published to Kafka (partitioned by driver_id for ordering). Location Workers consume in parallel, each handling a subset of driver_ids. Redis writes are pipelined — batching 100 updates per pipeline call reduces round-trips by 99x. The whole pipeline stays under 200ms end-to-end."

## Flashcards

| Q | A |
|---|---|
| In LC 987, what determines the order of nodes within the same (col, row) position? | Node value, ascending — use a min-heap (PriorityQueue) per (col, row) cell to resolve ties |
| In LC 834, what does `count[v]` represent after the first DFS? | The size of the subtree rooted at v (including v itself) |
| In LC 834, why does `ans[child] = ans[parent] - count[child] + (n - count[child])` work? | Moving root from parent to child: `count[child]` nodes get 1 closer (subtract `count[child]`); `n - count[child]` nodes get 1 farther (add `n - count[child]`) |
| In LC 987, why use TreeMap rather than HashMap for column grouping? | TreeMap keeps columns in sorted order automatically; HashMap would require sorting after collection |
| In Uber's location tracking, why partition Kafka by driver_id? | Ensures all location events for a given driver are processed by the same Kafka consumer, preserving ordering for staleness detection and position history |
