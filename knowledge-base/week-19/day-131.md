# Day 131 — Trees Hard: Path Sum DP & Tree Serialization
**Week 19 | Phase 3 — Mock Interviews | Month 5**

## Focus
Tackle two canonical hard tree problems — computing the maximum path sum through any node using post-order DP (LC 124) and serializing/deserializing a binary tree for storage and reconstruction (LC 297) — both staples in Google and Meta hard rounds.

---

## DSA (1.5 hours)

### Pattern: Post-Order Tree DP (Maximum Path Through Node)
- For LC 124, at each node compute the maximum "gain" from its left and right subtrees: `maxGain(node) = node.val + max(0, maxGain(left)) + max(0, maxGain(right))`. This local sum (which can use both sides) updates the global answer. But the value returned to the parent can only use one side: return `node.val + max(0, max(leftGain, rightGain))`.
- Trigger condition: "Maximum path sum in a tree where the path can start and end at any node." Post-order is mandatory — you need children's values before computing the parent's contribution.
- Time complexity: O(n) | Space complexity: O(h) call stack

### Pattern: Preorder Serialization with Null Markers
- For LC 297, serialize using preorder DFS and encode null pointers with a sentinel (e.g., "#"). Deserialization reconstructs by consuming tokens left-to-right — preorder naturally gives you the root first, then rebuild left subtree, then right.
- Trigger condition: "Encode a binary tree to a string and rebuild it exactly — order must be recoverable." Preorder with null markers is the simplest correct approach.
- Time complexity: O(n) | Space complexity: O(n)

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Binary Tree Maximum Path Sum | 124 | Hard | Post-Order Tree DP | At each node, update global max with left + node + right; return node + max(left, right) to parent (can only pick one branch going up) |
| 2 | Serialize and Deserialize Binary Tree | 297 | Hard | Preorder DFS + Null Sentinel | Use "#" for nulls; preorder gives root-first order that lets you reconstruct unambiguously |
| 3 | Binary Tree Level Order Traversal | 102 | Medium (Revision) | BFS | REVISION: standard BFS with level-size counter; `for (int i = 0; i < size; i++)` pattern |

### Code Skeleton

```java
class Solution {

    // LC 124 — Binary Tree Maximum Path Sum
    private int maxSum = Integer.MIN_VALUE;

    public int maxPathSum(TreeNode root) {
        maxGain(root);
        return maxSum;
    }

    private int maxGain(TreeNode node) {
        if (node == null) return 0;

        // Only take gain if positive (ignore negative subtrees)
        int leftGain = Math.max(0, maxGain(node.left));
        int rightGain = Math.max(0, maxGain(node.right));

        // Path through this node (uses both sides — can't go further up with both)
        int priceNewPath = node.val + leftGain + rightGain;
        maxSum = Math.max(maxSum, priceNewPath);

        // Return max single-branch gain to parent
        return node.val + Math.max(leftGain, rightGain);
    }


    // LC 297 — Serialize and Deserialize Binary Tree
    private static final String NULL_MARKER = "#";
    private static final String DELIMITER = ",";

    public String serialize(TreeNode root) {
        StringBuilder sb = new StringBuilder();
        serializeHelper(root, sb);
        return sb.toString();
    }

    private void serializeHelper(TreeNode node, StringBuilder sb) {
        if (node == null) {
            sb.append(NULL_MARKER).append(DELIMITER);
            return;
        }
        sb.append(node.val).append(DELIMITER);
        serializeHelper(node.left, sb);
        serializeHelper(node.right, sb);
    }

    public TreeNode deserialize(String data) {
        Deque<String> tokens = new ArrayDeque<>(Arrays.asList(data.split(DELIMITER)));
        return deserializeHelper(tokens);
    }

    private TreeNode deserializeHelper(Deque<String> tokens) {
        String token = tokens.pollFirst();
        if (NULL_MARKER.equals(token)) return null;
        TreeNode node = new TreeNode(Integer.parseInt(token));
        node.left = deserializeHelper(tokens);
        node.right = deserializeHelper(tokens);
        return node;
    }
}

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Overview & Scale

**Scale numbers (memorize for the interview):**
- 500M+ trips/year, ~15M trips/day
- 5M+ active drivers globally; 100M+ registered riders
- Peak: ~200K concurrent ride requests in a city cluster
- Match latency target: ≤ 5 seconds from request to driver assignment
- Location update frequency: every 4 seconds per active driver

**Core services at high level:**
- **Rider App** → API Gateway → **Ride Request Service**
- **Driver Location Service** — receives GPS pings, maintains real-time driver positions
- **Matching Service** — finds nearest available driver for a request
- **Trip Service** — manages trip lifecycle (requested → accepted → in_progress → completed)
- **Pricing Service** — computes fare (base rate + surge multiplier)
- **Notification Service** — push to driver and rider (APNs/FCM)
- **Payment Service** — charges rider post-trip

**Key architectural decisions:**
- Location data is write-heavy and time-critical → dedicated geospatial store (e.g., Redis geohash or Uber's H3 library) rather than general-purpose SQL
- Ride matching is compute-heavy and latency-sensitive → stateless matching workers reading from in-memory driver index
- Trip state must be durable and consistent → PostgreSQL with event sourcing or a state machine table

**Key trade-offs:**
- **Geohash vs S2 cells vs H3:** Geohash is simple but has edge problems at cell boundaries; S2 (Google) and H3 (Uber) use hierarchical hexagonal cells with uniform area, reducing boundary artifacts
- **Push vs pull for driver location:** Drivers push GPS every 4s; pulling would miss rapid movement; push + Kafka ingestion + Redis geospatial index gives real-time accuracy
- **Eventual vs strong consistency for driver availability:** Strong consistency prevents double-booking but adds latency; Uber uses optimistic locking with a driver status field (AVAILABLE/MATCHED) and re-tries on conflict

**Interview talking point:** "If asked how Uber finds the nearest driver in O(1) per city, answer: Driver locations are stored in a Redis Geo index (backed by a sorted set of geohash scores). `GEOSEARCH` with a radius query returns drivers within X km in O(log n + m) time. The city is partitioned into zones (geohash prefix buckets) so each query only scans the relevant zone's index, not all global drivers."

## Flashcards

| Q | A |
|---|---|
| In LC 124, why does `maxGain` return `node.val + max(leftGain, rightGain)` rather than `node.val + leftGain + rightGain`? | The path going upward can only continue in one direction from the current node; using both branches creates a fork which can't extend further to the parent |
| In LC 124, why do you take `max(0, maxGain(child))`? | If a subtree's max gain is negative, you skip it entirely — a negative branch reduces the total path sum |
| In LC 297 preorder serialization, what does the "#" sentinel represent? | A null child pointer — without it you can't distinguish "node with no left child" from "the number stored in the left child" |
| In LC 297, why is preorder (root → left → right) preferred over inorder for serialization? | Preorder gives you the root first, so you can create the node immediately during deserialization and recursively build left then right; inorder requires knowing boundaries |
| In Uber's architecture, why is driver location stored in Redis rather than PostgreSQL? | Drivers update every 4 seconds — at 5M active drivers, that's 1.25M writes/second; Redis Geo (sorted set) handles this at sub-millisecond latency; PostgreSQL would bottleneck on write throughput |
