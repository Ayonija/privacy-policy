## Focus

## DSA (1.5 hours)

### Pattern: Min-Max Bounded Sliding Window
- For LC 2444, a subarray is "fixed-bound" if it contains at least one `minVal` and one `maxVal` and no element outside `[minVal, maxVal]`. Track: last index of a minVal (`last_min`), last index of a maxVal (`last_max`), last index of an out-of-bounds element (`last_bad`). For each right, the count of valid subarrays ending at `right` is `max(0, min(last_min, last_max) - last_bad)`.
- Trigger condition: "Count subarrays where both a minimum and maximum constraint must be satisfied simultaneously, and elements outside a range invalidate the window."
- Time complexity: O(n) | Space complexity: O(1)

### Pattern: Two Pointer + Prefix for Good Subarrays
- For LC 1793, use binary search or two pointers to find the maximum score window around each element where the score = `min(nums[i..j]) * (j - i + 1)`. The key insight: for any optimal window, one endpoint is the element itself; the score decreases as you include elements smaller than the current minimum.
- Trigger condition: "Maximize a window score defined as `min(window) * window_length`; enforce a positional constraint (must include a fixed index)."
- Time complexity: O(n) with two pointers | Space complexity: O(1)

# LC 2444 — Count Subarrays With Fixed Bounds
def countSubarrays(nums: list, minK: int, maxK: int) -> int:
    last_min = last_max = last_bad = -1
    result = 0
    for i, n in enumerate(nums):
        if n < minK or n > maxK:
            last_bad = i
        if n == minK:
            last_min = i
        if n == maxK:
            last_max = i

    return result


# LC 1793 — Maximum Score of a Good Subarray
def maximumScore(nums: list, k: int) -> int:
    n = len(nums)
    left, right = k, k
    min_val = nums[k]
    result = nums[k]

    while left > 0 or right < n - 1:
        # Expand in the direction that sacrifices less (larger neighbor)
        if left == 0:
            right += 1
        elif right == n - 1:
            left -= 1
        elif nums[left - 1] >= nums[right + 1]:
            left -= 1
        else:
            right += 1

## System Design (1 hour)

### Topic: Twitter / X — Media Storage & CDN Pipeline

**Core components:**
- Media upload path: client → API gateway → Media Service → S3 (raw upload) → async transcoding pipeline (video: multiple resolutions via FFmpeg workers; images: compression + format conversion to WebP) → CDN origin
- Storage tiers: original uploaded files in S3 (standard tier); processed/transcoded variants in S3 (infrequent access); thumbnails and previews in S3 with CDN caching
- CDN (CloudFront / Akamai / Fastly): serves media at edge nodes globally; cache-control headers set by Media Service; cache invalidation on delete or re-upload
- Media metadata: stored in media table (`media_id, tweet_id, user_id, s3_key, content_type, width, height, duration_ms, status`); status tracks transcoding pipeline state (PENDING → PROCESSING → READY → FAILED)
- Video streaming: HLS (HTTP Live Streaming) with adaptive bitrate — multiple quality segments served from CDN; player selects quality based on network bandwidth

**Key trade-offs:**
- Synchronous vs asynchronous transcoding: sync blocks tweet creation until all variants are ready (bad UX for videos); async lets tweet appear immediately while processing continues in background — media status checked by client polling or webhook
- CDN cache TTL: long TTL (days/weeks) for media that never changes once uploaded; short TTL for profile images that change infrequently; immediate invalidation on deletion required for legal/content-moderation compliance
- Storage cost: video at scale is expensive — Twitter uses aggressive re-encoding (VP9/H.265) and deduplication (perceptual hash to avoid storing duplicate viral videos)

**Interview talking point:** "If asked how Twitter handles video uploads at scale, answer: Uploads go directly to S3 (presigned URL from client to avoid passing large files through the API tier). An SQS/Kafka event triggers async transcoding workers that produce HLS segments at multiple bitrates. Once ready, segment metadata is written to the media table and the CDN is primed. The tweet is visible immediately after the raw upload; the video player polls for READY status and plays as soon as segments are available."
