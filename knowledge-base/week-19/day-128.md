## DSA (1.5 hours)

### Pattern: Two-Pointer String Comparison (Lexicographic Suffix Selection)
- For LC 1163, the answer is always a suffix of `s`. Use two pointers `i = 0, j = 1` to compare candidate suffixes. Advance a third pointer `k` to compare characters at `s[i+k]` vs `s[j+k]`. If `s[i+k] < s[j+k]`, move `i` past the candidate; if `s[i+k] > s[j+k]`, move `j`. If equal, increment `k`. When pointers cross, the winner is the suffix starting at the remaining pointer.
- Trigger condition: "Find the lexicographically largest/smallest suffix without generating all O(n²) suffixes or sorting in O(n log n)."
- Time complexity: O(n) amortized | Space complexity: O(1)

### Pattern: Sliding Window on Sorted + Deduplicated Array
- For LC 2009, after sorting and deduplicating the array, we need the minimum replacements to make it contain exactly n consecutive integers. Binary search (or sliding window): for each left endpoint `nums[i]`, find the furthest right endpoint where `nums[j] <= nums[i] + n - 1`. The window `[i, j]` already contains `j - i + 1` valid distinct values, so we need to replace `n - (j - i + 1)` duplicates/out-of-range elements.
- Trigger condition: "Make an array of n elements contain n consecutive values with minimum replacements."
- Time complexity: O(n log n) for sort + O(n) sliding window | Space complexity: O(n)

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Last Substring in Lexicographical Order | 1163 | Hard | Two Pointer Suffix Comparison | Never need to go back — advance the losing pointer past the comparison point; answer is always a suffix |
| 2 | Minimum Number of Operations to Make Array Continuous | 2009 | Hard | Sort + Dedup + Sliding Window | Count elements already in valid window `[nums[i], nums[i]+n-1]`; replace = n - window_size |
| 3 | 3Sum Closest | 16 | Medium (Revision) | Sort + Two Pointer | REVISION: fix one element, two-pointer on rest; track closest sum; move pointer based on how sum compares to target |

### Code Skeleton

```python
# LC 1163 — Last Substring in Lexicographical Order
def lastSubstring(s: str) -> str:
    i, j, k = 0, 1, 0
    n = len(s)
    while j + k < n:
        if s[i + k] == s[j + k]:
            k += 1
        elif s[i + k] > s[j + k]:
            # j's suffix loses; advance j past the comparison point
            j = j + k + 1
            k = 0
            if j <= i:
                j = i + 1
        else:
            # i's suffix loses; advance i past the comparison point
            i = i + k + 1
            k = 0
            if i == j:
                i += 1  # or j += 1 to avoid same pointer
    return s[i:]


# LC 2009 — Minimum Operations to Make Array Continuous
def minOperations(nums: list) -> int:
    n = len(nums)
    nums = sorted(set(nums))  # sort and deduplicate
    result = n
    left = 0

    for right in range(len(nums)):
        # Slide left forward so nums[right] - nums[left] < n
        while nums[right] - nums[left] >= n:
            left += 1
        # Window [left, right] has (right - left + 1) elements already valid
        # Replacements needed = n - (right - left + 1)
        result = min(result, n - (right - left + 1))

    return result
```

## System Design (1 hour)

### Topic: Twitter / X — Caching Strategy

**Core components:**
- Cache layers: L1 = in-process cache (small, per-service, e.g., Caffeine in JVM); L2 = distributed cache (Redis cluster, shared across service instances); L3 = CDN (edge cache for static/semi-static content like profile images, stylesheets)
- What to cache: user profile data (TTL: minutes), precomputed timelines (TTL: until invalidated by new tweet), tweet metadata for trending/viral tweets (TTL: hours), follow graph for fan-out service (TTL: minutes with write-through invalidation)
- Timeline cache: Redis list per user storing up to 800 tweet IDs; populated by fan-out workers on tweet creation; TTL set to ~7 days (inactive users cold-start from DB)
- Tweet hydration cache: since timelines store only tweet_ids, the Timeline Service hydrates tweet content from a tweet cache (Redis hash: `tweet:{id}` → full tweet JSON); very high cache hit rate for viral tweets
- Cache invalidation: write-through for user profile updates; event-driven invalidation (Kafka event triggers cache delete) for follow/unfollow changes; TTL expiry for everything else

**Key trade-offs:**
- Cache-aside vs write-through: cache-aside is simpler (read miss → fetch from DB → populate cache) but risks thundering herd on cold start; write-through keeps cache consistent but adds write latency
- Thundering herd on cache miss: a viral tweet cache miss could trigger thousands of simultaneous DB reads; mitigate with request coalescing (single DB fetch per cache miss, queue remaining requests) or probabilistic early expiration
- Memory vs hit rate: caching the top 1% of tweets (by access frequency) captures ~99% of read traffic — use LFU or LRU eviction; monitor cache miss rate per service

**Interview talking point:** "If asked how Twitter handles a sudden viral tweet overwhelming the cache, answer: First, the cache hit rate is extremely high for viral tweets — they're read millions of times after the initial cache population. For a cold-start miss storm, Twitter uses request coalescing: only one goroutine/thread fetches from the DB for a given cache key; all other concurrent requests wait and share the result. Additionally, probabilistic early expiration refreshes cache entries slightly before TTL to smooth the miss distribution."

## Flashcards

| Q | A |
|---|---|
| In LC 1163, why is the answer always a suffix of `s`? | Any non-suffix substring has a suffix starting at the same character that is lexicographically ≥ it (the suffix extends to the end); so the max substring must extend to the end |
| In LC 1163, when you advance pointer `i` to `i + k + 1`, why skip `k` positions? | All suffixes starting at `i+1` through `i+k` are proven to be smaller than `s[j..]` at that comparison depth; no need to re-check them |
| What is the key preprocessing step in LC 2009 before sliding the window? | Sort and deduplicate `nums`; duplicates can be freely replaced and should not be counted as "valid" elements for a given window |
| In LC 2009, what does the window `[left, right]` on the deduplicated array represent? | All distinct values that already fit in the range `[nums[left], nums[left] + n - 1]`; size = `right - left + 1`; replacements needed = `n - (right - left + 1)` |
| What is cache request coalescing and when does Twitter use it? | Only one thread fetches from DB for a given cache miss; concurrent requests wait and share the result; prevents thundering herd when a viral tweet's cache entry expires |
