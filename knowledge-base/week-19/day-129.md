## DSA (1.5 hours)

### Pattern: Merge Sort with Cross-Partition Counting
- For LC 493, a "reverse pair" is `(i, j)` where `i < j` and `nums[i] > 2 * nums[j]`. A naive O(n²) check is too slow. During merge sort, after partitioning, count pairs where the left-half element > 2 × right-half element using a two-pointer across the two sorted halves (before merging). Then merge normally. Each merge step counts cross-partition pairs in O(n) time.
- Trigger condition: "Count pairs satisfying a cross-partition inequality where brute force is O(n²); think merge sort to reduce to O(n log n) by exploiting sorted order."
- Time complexity: O(n log n) | Space complexity: O(n) for merge buffer

### Pattern: Binary Search on Length + Rolling Hash for Duplicate Detection
- For LC 1044, binary search on the length `L` of the duplicate substring. For a given `L`, check if any substring of length `L` appears twice using Rabin-Karp rolling hash — compute hash for first window, then slide by updating hash in O(1). Store seen hashes in a set; collision → verify actual string equality to avoid false positives.
- Trigger condition: "Longest duplicate (or repeated) substring — length search space is sortable; O(n²) naïve check avoids with rolling hash in O(n) per binary search step."
- Time complexity: O(n log n) expected | Space complexity: O(n)

```python
# LC 493 — Reverse Pairs
def reversePairs(nums: list) -> int:
    def merge_count(arr):
        if len(arr) <= 1:
            return 0

        mid = len(arr) // 2
        left = arr[:mid]
        right = arr[mid:]
        count = merge_count(left) + merge_count(right)

        # Count cross-partition reverse pairs BEFORE merging
        # (both halves are sorted after recursive calls)
        j = 0
        for val in left:
            while j < len(right) and val > 2 * right[j]:
                j += 1
            count += j

        # Standard merge step
        i = k = 0
        while i < len(left) and k < len(right):
            if left[i] <= right[k]:

# LC 1044 — Longest Duplicate Substring (Rabin-Karp)
def longestDupSubstring(s: str) -> str:
    MOD = (1 << 61) - 1  # Mersenne prime for low collision probability
    BASE = 31

    def has_duplicate(length: int):
        # Compute rolling hash for first window
        h = 0
        power = 1
        for i in range(length):
            h = (h * BASE + ord(s[i]) - ord('a') + 1) % MOD
            if i > 0:
                power = power * BASE % MOD

        seen = {h: [0]}  # hash → [start indices]
        for start in range(1, len(s) - length + 1):
            # Slide window: remove leftmost char, add new rightmost char
            h = (h - (ord(s[start - 1]) - ord('a') + 1) * power) % MOD
            h = (h * BASE + ord(s[start + length - 1]) - ord('a') + 1) % MOD
            if h in seen:
                # Verify to handle hash collision
                candidate = s[start:start + length]
                for prev in seen[h]:
                    if s[prev:prev + length] == candidate:
                        return candidate
                seen[h].append(start)
            else:
                seen[h] = [start]
        return ""

    lo, hi = 0, len(s) - 1
    result = ""
    while lo <= hi:
        mid = (lo + hi) // 2
        dup = has_duplicate(mid)
        if dup:
            result = dup
            lo = mid + 1
        else:
            hi = mid - 1
    return result
```

## System Design (1 hour)

### Topic: Twitter / X — Rate Limiting & Anti-Abuse

**Core components:**
- Rate limiting goals: prevent API abuse (scraping, spam bots), protect backend services from traffic spikes, enforce fair usage across clients
- Rate limit types: per-user (e.g., 300 tweets/3h), per-app (OAuth token), per-IP (unauthenticated); different limits for read vs write endpoints
- Algorithms:
  - **Token bucket:** user gets N tokens; each request consumes 1 token; tokens replenish at fixed rate R/sec; allows burst up to N
  - **Sliding window counter:** Redis `ZADD`/`ZREMRANGEBYSCORE` with scored timestamps; accurate but memory-intensive
  - **Fixed window counter:** simpler — Redis `INCR` with TTL; fast but vulnerable to boundary bursts (2× traffic at window boundary)
- Implementation: rate limiter as a middleware layer before API servers; checks Redis counter; returns HTTP 429 Too Many Requests on limit exceeded with `Retry-After` header
- Bot detection: beyond rate limits — behavioral signals (posting frequency patterns, account age, follower/following ratio, device fingerprint); ML model scores new accounts at registration and continuously

**Key trade-offs:**
- Distributed rate limiting accuracy: with multiple API servers, per-server counters allow each server to permit N requests = N × num_servers total; centralized Redis counter is accurate but adds ~1–2ms latency per request
- Sliding window vs fixed window: sliding window is accurate (no boundary spike) but requires O(requests) memory per user; fixed window is O(1) memory but permits 2× burst at window boundary
- Allowlist vs strictlist: verified/paid accounts (Twitter Blue) get higher rate limits; this requires rate limit tier lookup on every request — cache tier in Redis

**Interview talking point:** "If asked to design a distributed rate limiter for Twitter's API, answer: Use a Redis cluster with a sliding window log per (user_id, endpoint). Each request does `ZADD key now now` then `ZREMRANGEBYSCORE key 0 (now - window)` then `ZCARD key` — if count exceeds limit, reject with 429. To reduce Redis latency, batch requests using Lua scripts (atomic check-and-increment). For extreme scale, use a token bucket in-process with periodic Redis sync, accepting slight over-counting in exchange for lower latency."

## Flashcards

| Q | A |
|---|---|
| In LC 493, why count reverse pairs before the merge (not after)? | After merging, the two halves are interleaved and sorted — you can no longer tell which element came from left vs right, so you lose the `i < j` ordering constraint |
| In LC 493's counting step, why does a two-pointer across the sorted halves work? | Both halves are already sorted after recursive calls; if `left[i] > 2 * right[j]` then `left[i] > 2 * right[0..j]` — monotonicity lets j advance without going back |
| What is the purpose of using a Mersenne prime as MOD in Rabin-Karp? | Mersenne primes (2^p - 1) allow fast modular reduction using bit operations, reducing collision probability compared to smaller moduli |
| In LC 1044, why verify string equality when a hash collision is detected? | Hash collisions produce false positives — two different strings may have the same hash; string comparison confirms a genuine duplicate |
| What is the "boundary burst" vulnerability in fixed-window rate limiting? | A user can make N requests at the end of window W and N requests at the start of window W+1 — 2N requests in a short period, yet both windows count separately |
