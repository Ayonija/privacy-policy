## DSA (2 hours)

### Pattern: 0/1 Knapsack

**Core idea:**
Each item can be chosen at most once. State `dp[w]` = max value achievable with capacity w. Iterate weights from high to low to prevent reusing the same item. For 2D knapsack, maintain `dp[m][n]` iterating both dimensions backwards.

**Trigger condition:**
- "Maximum value / count subject to capacity constraint"
- "Can we partition a set into two equal-sum subsets?"
- "Count schemes using exactly k resources achieving profit ≥ p"
- Items binary: taken or not (contrast with unbounded knapsack where forward iteration allows reuse)

**Complexity:**
O(n × W) time | O(W) space after 1D optimization

**The key rule — why iterate backwards:**
`dp[j] = max(dp[j], dp[j-w] + v)` — when updating `dp[j]`, we read `dp[j-w]`, which must still reflect the state *before* this item was considered. Forward iteration would overwrite `dp[j-w]` first, effectively allowing the same item twice (unbounded behavior).

### Full Solution Walkthrough — Partition Equal Subset Sum (LC 416) — Revision

**Problem:** Can you partition the array into two subsets with equal sums?

**Reduction:** If total is odd → False. Otherwise, find a subset summing to `total // 2`.

```python
def canPartition(nums: list[int]) -> bool:
    total = sum(nums)
    if total % 2 != 0:
        return False
    target = total // 2
    dp = [False] * (target + 1)
    dp[0] = True  # empty subset sums to 0

    for num in nums:
        for j in range(target, num - 1, -1):  # backwards → each item used once
            dp[j] |= dp[j - num]

    return dp[target]
```

**Trace on `[1, 5, 11, 5]` → target = 11:**
After 1: `dp[1]=T`
After 5: `dp[6]=T, dp[5]=T`
After 11: `dp[11]=T` ✓

### Full Solution Walkthrough — Ones and Zeroes (LC 474) — Hard

**Problem:** Given strings of 0s and 1s, find the max strings you can pick using at most m zeros and n ones total.

**2D knapsack:** Each string has two "weights": (zeros, ones). Capacity = (m, n). Value of each item = 1.

```python
def findMaxForm(strs: list[str], m: int, n: int) -> int:
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for s in strs:
        zeros = s.count('0')
        ones = s.count('1')
        # Iterate both dimensions backwards (0/1 knapsack rule)
        for i in range(m, zeros - 1, -1):
            for j in range(n, ones - 1, -1):
                dp[i][j] = max(dp[i][j], dp[i - zeros][j - ones] + 1)

    return dp[m][n]
```

**Why O(S × m × n)?** S strings, each requiring a full 2D sweep. No further optimization unless m or n is large.

### Full Solution Walkthrough — Profitable Schemes (LC 879) — Hard

**Problem:** n members, each crime uses some group members and generates some profit. Count schemes where ≤ n members are used AND total profit ≥ minProfit. Answer mod 10⁹+7.

**3D DP → 2D:** State `dp[members][profit]` = count of schemes using exactly `members` people with profit capped at `minProfit` (values past minProfit are equivalent). This collapses the third dimension.

```python
def profitableSchemes(n: int, minProfit: int, group: list[int], profit: list[int]) -> int:
    MOD = 10**9 + 7
    # dp[j][p] = number of schemes using j members with profit = p (capped at minProfit)
    dp = [[0] * (minProfit + 1) for _ in range(n + 1)]
    dp[0][0] = 1  # base: 0 members, 0 profit, 1 way (do nothing)

    for g, p in zip(group, profit):
        # Iterate backwards on both dimensions (0/1 knapsack)
        for members in range(n, g - 1, -1):
            for prof in range(minProfit, -1, -1):
                new_prof = min(prof + p, minProfit)  # cap at minProfit
                dp[members][new_prof] = (dp[members][new_prof] + dp[members - g][prof]) % MOD

    # Answer: sum all states with exactly minProfit profit, any member count
    return sum(dp[k][minProfit] for k in range(n + 1)) % MOD
```

**Why cap at minProfit?** Any profit ≥ minProfit counts as "profitable" — we don't distinguish between minProfit and 2×minProfit. Capping lets us use a fixed-size array.

**Trace logic:** After processing each crime, `dp[members][minProfit]` accumulates all valid profitable schemes. Final answer sums over all member counts.

## System Design (30 min)

### Topic: REST vs GraphQL — Fundamentals

**REST (Representational State Transfer):**
- Resource-oriented: each endpoint represents one resource (`/users/{id}`, `/posts/{id}/comments`)
- HTTP verbs carry semantics: GET (read, idempotent), POST (create), PUT (full replace, idempotent), PATCH (partial update), DELETE (remove, idempotent)
- Responses return a fixed data shape — client gets what the server decides to send
- Stateless: each request carries full context; no server-side session

**GraphQL:**
- Single endpoint: `POST /graphql` with a query in the body
- Client specifies exact fields it needs — no over-fetching (getting unused fields) or under-fetching (needing multiple round trips)
- Strongly typed schema: SDL (Schema Definition Language) is the contract
- Introspection: clients can query the schema itself

**Core trade-offs:**

| Dimension | REST | GraphQL |
|-----------|------|---------|
| Caching | HTTP cache (CDN, browser) works natively on URL | Harder — all POSTs; need persisted queries or APQ |
| Tooling | Universal — every language/tool supports HTTP | Requires GraphQL client (Apollo, urql) |
| Over/under-fetch | Both problems common | Solved — client requests exactly what it needs |
| Schema evolution | Versioning required (v1, v2) | Additive: deprecate fields, never break |
| Complexity | Low — easy to reason about | Higher — resolvers, N+1 problem, federation |

**Interview talking point:**
*"Choose REST for simple CRUD APIs with predictable, uniform data needs and strong caching requirements. Choose GraphQL when clients have heterogeneous data needs (mobile vs. web), when you're building a BFF (Backend for Frontend) that aggregates multiple services, or when you want schema-driven development. Many systems use REST for public APIs and GraphQL internally."*
