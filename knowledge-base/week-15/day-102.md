## DSA (2 hours)

### Pattern: Knapsack with Difference Minimization and Exact Count

**Core idea:**
Beyond max-value knapsack, many problems require:
- **Difference minimization:** maintain `dp[d]` = best value achievable with rod difference = d
- **Counting subsets:** `dp[j]` = number of ways to form sum j (not whether it's possible)
- **Complement counting:** total − invalid, where invalid = subsets below a threshold

**Trigger condition:**
- "Split into two groups to maximize the smaller group" → difference DP
- "Count ways to assign +/− signs to reach target" → counting knapsack
- "Count partitions where both sides ≥ k" → complement: total − 2 × (subsets with sum < k)

**Key insight — Target Sum reduction:**
Assign + to subset P and − to subset N. Then P − N = target and P + N = total. So P = (total + target) / 2. Count subsets summing to P using a counting knapsack.

### Full Solution Walkthrough — Target Sum (LC 494) — Revision

**Problem:** Assign + or − to each number to reach target. Count ways.

**Reduction:** Let P = sum of numbers with + sign, N = sum with − sign.
- P − N = target
- P + N = total
- → P = (total + target) / 2

Count subsets summing to P. Guard: if (total + target) is odd or |target| > total → 0.

```python
def findTargetSumWays(nums: list[int], target: int) -> int:
    total = sum(nums)
    if (total + target) % 2 != 0 or abs(target) > total:
        return 0
    t = (total + target) // 2

    dp = [0] * (t + 1)
    dp[0] = 1  # one way to form sum 0: choose nothing

    for num in nums:
        for j in range(t, num - 1, -1):  # backwards → 0/1 knapsack
            dp[j] += dp[j - num]

    return dp[t]
```

**Why counting vs. boolean?** `dp[j] += dp[j-num]` accumulates the number of distinct subsets, whereas `dp[j] |= dp[j-num]` only tracks reachability.

### Full Solution Walkthrough — Tallest Billboard (LC 956) — Hard

**Problem:** Given rod segments, build two equal-height steel supports for a billboard. Maximize their height.

**State:** `dp[d]` = maximum height of the shorter support when the height difference between the two supports is exactly d.
- `dp[0] = 0` initially (no segments used, both supports empty, equal height 0)
- All other `dp[d] = −∞` (unachievable)

**Transition for each segment of height h:**
- **Skip:** no change
- **Add to taller support:** difference grows by h → `dp_new[d+h] = max(dp_new[d+h], dp[d])`
- **Add to shorter support:**
  - If h ≤ d: shorter stays shorter → `dp_new[d-h] = max(dp_new[d-h], dp[d] + h)`
  - If h > d: shorter becomes the new taller → `dp_new[h-d] = max(dp_new[h-d], dp[d] + d)`
    - (The old shorter becomes the new shorter at height = old shorter + d = old taller)

**Answer:** `dp[0]` — when both supports are equal height, "shorter" = that height.

```python
def tallestBillboard(rods: list[int]) -> int:
    dp = {0: 0}  # diff -> max shorter rod height

    for h in rods:
        new_dp = dict(dp)
        for d, shorter in dp.items():
            # Add to taller side
            key = d + h
            new_dp[key] = max(new_dp.get(key, 0), shorter)
            # Add to shorter side
            if h <= d:
                key = d - h
                new_dp[key] = max(new_dp.get(key, 0), shorter + h)
            else:
                key = h - d
                new_dp[key] = max(new_dp.get(key, 0), shorter + d)

        dp = new_dp

    return dp.get(0, 0)
```

**Trace on `[1, 2, 3, 6]`:**
- After 1: `{0:0, 1:0}`
- After 2: `{0:0, 1:0, 2:0, 3:0, 1:2}` → `{0:0, 1:2, 2:0, 3:0}` (take max at each key)
- After 3: `{0:3, ...}` → `dp[0] = 3`
- After 6: `{0:6, ...}` → `dp[0] = 6` ✓ (answer: 6)

**Complexity:** O(n × Σrods) with dictionary; bounded by total sum ≤ 5000.

### Full Solution Walkthrough — Number of Great Partitions (LC 2518) — Hard

**Problem:** Count partitions of `nums` into two non-empty groups where both groups sum ≥ k. Answer mod 10⁹+7.

**Complement counting:**
- Total ordered partitions = 2^n (each element goes to group 1 or group 2)
- A partition is "bad" if group 1 sum < k OR group 2 sum < k
- By symmetry: bad = 2 × (subsets with sum < k) — if total ≥ 2k, both sides can't both be < k simultaneously, so these events are disjoint
- If total < 2k: every partition has at least one side < k → answer = 0

**Key math:** `answer = (2^n − 2 × cnt_bad) % MOD` where `cnt_bad = count of subsets with sum in [0, k-1]`.

```python
def countPartitions(nums: list[int], k: int) -> int:
    MOD = 10**9 + 7
    total = sum(nums)
    n = len(nums)

    if total < 2 * k:
        return 0

    # Count subsets with sum < k using 0/1 knapsack
    dp = [0] * k
    dp[0] = 1  # empty subset sums to 0

    for num in nums:
        for j in range(k - 1, num - 1, -1):
            dp[j] = (dp[j] + dp[j - num]) % MOD

    cnt_bad = sum(dp) % MOD
    total_ordered = pow(2, n, MOD)
    return (total_ordered - 2 * cnt_bad + MOD) % MOD
```

**Verify on `nums=[1,2,3,4], k=4`:** total=10 ≥ 8. Subsets with sum < 4: {}, {1}, {2}, {3}, {1,2} → 5. Answer = 16 − 10 = 6 ✓

**Why +MOD before %?** `(2 * cnt_bad)` could exceed `total_ordered`, making the subtraction negative in Python's modular arithmetic. Adding MOD ensures non-negative.

## System Design (30 min)

### Topic: REST API Design — CRUD, Idempotency, Pagination

**Resource naming conventions:**
- Use nouns, not verbs: `/users`, `/orders/{id}/items` (not `/getUser`)
- Plural for collections, singular for specific resources
- Nest max 2 levels deep: `/users/{id}/posts` (not `/users/{id}/posts/{id}/comments/{id}`)

**Idempotency in practice:**
- GET, PUT, DELETE are idempotent by spec — safe to retry on network failure
- POST is not idempotent — use an `Idempotency-Key` header (UUID from client) stored server-side for 24h to detect and deduplicate retries
- PATCH can be non-idempotent (e.g., `{ "increment": 5 }`) — design with care

**Pagination patterns:**
| Pattern | How | Best for | Downside |
|---------|-----|----------|---------|
| Offset (`?page=2&limit=20`) | SQL `LIMIT/OFFSET` | Simple, any order | Slow on large offsets; page drift on inserts |
| Cursor (`?after=cursor`) | Encode last-seen ID or timestamp | Time-ordered feeds, large datasets | Can't jump to page N |
| Keyset (`?after_id=123`) | `WHERE id > 123 LIMIT 20` | High-performance, stable | Requires unique sort key |

**Interview talking point:**
*"For APIs serving real-time feeds (like Twitter timeline), cursor-based pagination is mandatory. Offset pagination breaks when new items are inserted — the user sees duplicates or misses items on the next page. Encode the cursor as an opaque base64 string so clients can't reverse-engineer the internals."*
