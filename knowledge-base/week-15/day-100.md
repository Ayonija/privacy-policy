### Full Solution Walkthrough — Ugly Number II (LC 264) — Revision

**Ugly numbers:** Numbers whose only prime factors are 2, 3, 5. Generate them in sorted order.

**3-pointer DP (pattern: merge K sorted lists applied to DP):**
```python
def nthUglyNumber(n: int) -> int:
    ugly = [1] * n
    i2 = i3 = i5 = 0   # pointers into ugly[] for each factor

    for i in range(1, n):
        next2 = ugly[i2] * 2
        next3 = ugly[i3] * 3
        next5 = ugly[i5] * 5
        ugly[i] = min(next2, next3, next5)
        if ugly[i] == next2: i2 += 1
        if ugly[i] == next3: i3 += 1
        if ugly[i] == next5: i5 += 1   # all pointers advance if tied

    return ugly[n - 1]
```

**Why advance all tied pointers?** Avoid duplicates. If `min = 6 = 2×3 = 3×2`, both `i2` and `i3` produced it — advance both so 6 is not generated again.

### Full Solution Walkthrough — Number of Ways to Stay (LC 1269) — Hard

**Problem:** Array of length `arrLen`, start at index 0. Make exactly `steps` moves (left, right, stay). Count ways to end at index 0.

**State:** `dp[i]` = ways to be at index `i` after current step. At most `min(steps, arrLen-1)` positions reachable (at step k, can only be at positions 0 to k).

```python
def numWays(steps: int, arrLen: int) -> int:
    MOD = 10**9 + 7
    max_pos = min(steps // 2, arrLen - 1)  # can't go further right than steps//2
    dp = [0] * (max_pos + 1)
    dp[0] = 1   # start at position 0

    for _ in range(steps):
        ndp = [0] * (max_pos + 1)
        for i in range(max_pos + 1):
            if dp[i] == 0: continue
            # Stay
            ndp[i] = (ndp[i] + dp[i]) % MOD
            # Move left
            if i > 0:
                ndp[i-1] = (ndp[i-1] + dp[i]) % MOD
            # Move right
            if i < max_pos:
                ndp[i+1] = (ndp[i+1] + dp[i]) % MOD
        dp = ndp

    return dp[0]
```

**Key optimization:** `max_pos = min(steps // 2, arrLen - 1)` — you can't be more than `steps//2` positions away from 0 and still return in remaining steps. This reduces O(steps × arrLen) to O(steps × min(steps/2, arrLen)).

### Full Solution Walkthrough — K Inverse Pairs Array (LC 629) — Hard

**Problem:** Count arrays of `n` integers (1..n, no repeats) with exactly `k` inverse pairs.

**Recurrence:** Adding integer `n` at position `i` from the right of an array of `(n-1)` elements creates `i` new inverse pairs (all elements to its right are smaller than `n`). So `dp[n][k] = sum(dp[n-1][k-i])` for `i` in `0..min(n-1, k)`.

This is a prefix sum over the previous row.

```python
def kInversePairs(n: int, k: int) -> int:
    MOD = 10**9 + 7
    # dp[j] = number of arrays of current length with exactly j inverse pairs
    dp = [0] * (k + 1)
    dp[0] = 1   # base: length 1, 0 inverse pairs

    for length in range(2, n + 1):
        # For each length, dp_new[j] = sum(dp[j-i]) for i in 0..min(length-1, j)
        # = prefix_sum[j] - prefix_sum[j - length]  (sliding window sum)
        prefix = [0] * (k + 2)
        for j in range(k + 1):
            prefix[j + 1] = (prefix[j] + dp[j]) % MOD

        ndp = [0] * (k + 1)
        for j in range(k + 1):
            lo = max(0, j - (length - 1))
            ndp[j] = (prefix[j + 1] - prefix[lo] + MOD) % MOD
        dp = ndp

    return dp[k]
```

**Complexity:** O(n × k) with prefix sums. Without prefix sums: O(n × k × n) — TLE.

## System Design (30 min)

### Topic: Full Chat Service Design Review — Interview Simulation

**The prompt:** *"Design WhatsApp / a real-time messaging system for 500M users."*

**Walk through this structure in 30 minutes:**

**1. Requirements (3 min)**
- Functional: send/receive messages (1:1 and group), online presence, push for offline users, message history
- Non-functional: < 100ms delivery latency, 99.99% availability, messages never lost

**2. Capacity estimation (2 min)**
- 500M users, 100M daily active
- 40B messages/day → ~500K messages/second
- Average message 200 bytes → ~100MB/s writes

**3. Core architecture (10 min)**
```
Client → API Gateway → Chat Server (WebSocket) → Cassandra (messages)
                              ↕ Redis Pub/Sub (routing)
                       Notification Service → APNs/FCM
                       Presence Service → Redis (user status)
```

**4. Key design decisions (10 min)**
| Decision | Choice | Reason |
|----------|--------|--------|
| Protocol | WebSocket | Full-duplex; low latency; battery efficient vs. polling |
| Message store | Cassandra | High write throughput; partition by conversation_id + timestamp |
| Routing | Redis Pub/Sub | Ephemeral cross-server routing; microsecond latency |
| Offline delivery | APNs/FCM | Platform push; in-app inbox as fallback |
| Message ordering | Per-conversation sequence counter in Redis | Simpler than vector clocks for 1:1 |
| Dedup | Idempotency key (UUID) per message | At-least-once delivery + client-side dedup |

**5. Failure handling (5 min)**  
- Chat server crash: client reconnects to any server; Redis routing clears stale entry
- Cassandra node down: replica factor 3; any 2 nodes serve reads/writes
- Push failure: retry queue + in-app inbox fallback

**Interview talking point:**  
*"The most important architectural decision is separating the ephemeral routing layer (Redis) from the persistent message layer (Cassandra). Redis state is disposable — it can be rebuilt from reconnections. Cassandra state is durable — it must survive all failures. This separation is what allows each layer to scale independently."*
