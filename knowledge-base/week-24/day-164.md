# Day 164 — Classic System Design — WhatsApp Chat + Google Drive

## Focus
WhatsApp tests real-time messaging at extreme scale with guaranteed ordering and offline delivery. Google Drive tests chunked file upload, delta sync, and metadata consistency across devices. Both are heavy-hitter system design problems in senior interviews: WhatsApp for its 1.16M messages/sec challenge and Google Drive for its multi-device sync complexity.

## HLD (2 hours)

### WhatsApp / Real-Time Chat

**Requirements**

Functional:
- 1-1 messaging and group messaging (up to 1000 members)
- Message delivery status: sent, delivered, read receipts
- Last seen / online presence
- Push notifications for offline users
- End-to-end encryption

Non-functional:
- 2B users; peak 200M concurrent users online simultaneously
- 100B messages/day
- Delivery latency < 500ms for online users
- Message ordering guaranteed within a conversation
- 99.99% availability

**Scale Estimation**
- Message QPS: 100B / 86,400 approximately 1.16M messages/sec
- Storage: 100B × 200 bytes × 365 × 5 years approximately 36 PB
- Connections: 200M concurrent WebSocket connections at peak
- Connection servers needed: 200M connections / ~100K connections per server = 2,000 chat servers

**Connection Management**

Architecture:
- Each client maintains a persistent WebSocket connection to one Chat Server
- Chat Servers are stateless with respect to message storage; connection routing state is stored in Redis
- Redis: `userId → chatServerId` (which server holds this user's WebSocket connection)
- Chat Servers are horizontally scalable; stateless means any server can serve any user after reconnect

Message flow (online recipient):
```
Sender → Chat Server A
    → lookup recipient's chatServerId in Redis
    → if same server: push directly to recipient WebSocket
    → if different server: route via internal message queue to Chat Server B
                           → Chat Server B pushes to recipient WebSocket
```

Message flow (offline recipient):
```
Sender → Chat Server A
    → write message to Cassandra (durability first)
    → lookup recipient status in Redis presence store
    → if offline: send push notification via FCM (Android) or APNs (iOS)
    → recipient device receives push, reconnects WebSocket, requests missed messages
```

**Why Write to DB Before Delivery**

Even for online recipients: message is written to Cassandra before the WebSocket push. This ensures:
- Message is durable even if Chat Server crashes between write and push
- Recipient can always re-fetch from DB if WebSocket delivery fails
- Sender receives "sent" acknowledgment only after DB write confirms

**Message Ordering with Snowflake IDs**

Snowflake ID (64-bit):
```
[41 bits: timestamp ms] [5 bits: datacenter] [5 bits: worker] [12 bits: sequence]
```
- Monotonically increasing within a datacenter-worker pair
- Globally sortable by time without central coordination
- 4096 unique IDs per millisecond per worker (12-bit sequence)
- Used as Cassandra clustering key: naturally time-ordered within a conversation partition

Cassandra schema:
```
messages (
  chatId      UUID,         -- partition key (conversation or group ID)
  messageId   BIGINT,       -- clustering key (Snowflake, ASC)
  senderId    UUID,
  content     BLOB,         -- encrypted ciphertext
  sentAt      TIMESTAMP,
  type        TEXT          -- TEXT, IMAGE, VIDEO, FILE
)
```

**Group Messaging**

Small groups (<= 100 members):
- Fan-out at message queue: write one copy of the message reference per active group member's inbox
- Each inbox entry: `{chatId, messageId, recipientId}`

Large groups (> 100 members):
- Fan-out at read: store message once in the group's partition; members pull on group open
- Presence service tracks which members are currently active in the group
- Only push to online members; offline members pull on reconnect

**Message Status Tracking**

- Sent: Chat Server A acknowledges receipt from sender (after DB write)
- Delivered: recipient device acknowledges receipt to Chat Server B (app receives message)
- Read: recipient opens the conversation → sends read receipt back to sender
- Single checkmark (sent) / double checkmark (delivered) / blue double checkmark (read)

Status storage: Redis hash per message for ephemeral status; only final status (READ) persisted to Cassandra.

**End-to-End Encryption (Signal Protocol)**

- Each device has a long-term identity key pair (Ed25519 for signing, Curve25519 for key exchange)
- Per-session message encryption key derived via Double Ratchet Algorithm (combination of Diffie-Hellman + symmetric ratchet)
- Key exchange happens at first contact using pre-published one-time keys (X3DH protocol)
- Server stores only: public keys, ciphertext, and message metadata — never plaintext
- Forward secrecy: each message encrypted with a new key; past messages cannot be decrypted even if current session key is compromised

**Storage Summary**
- Messages: Cassandra (36PB distributed; time-series append-only; partitioned by chatId)
- User metadata: MySQL (user profiles, contact lists, group membership, device tokens)
- Online presence: Redis (`userId → {status: ONLINE/OFFLINE, lastSeen: timestamp}`); heartbeat renewal every 30s; TTL=60s for auto-offline

### Google Drive (File Storage & Sync)

**Requirements**

Functional:
- Upload and download files of any size
- Sync files across multiple devices automatically
- Share files/folders with other users (viewer/editor/owner roles)
- Version history (restore previous versions)
- Folder organization with nested structure

Non-functional:
- 1B users; 15GB free storage per user
- Strong consistency for metadata (file renames, moves must be immediately visible)
- Eventual consistency acceptable for file data (blob content)
- Sync propagated to all devices within seconds of a change

**Scale Estimation**
- Total theoretical storage: 1B users × 15GB = 15 EB; actual used ~10% = 1.5 EB
- File changes: 50M/day = approximately 580/sec writes
- Metadata QPS: 10x file QPS = 5,800/sec (browsing and listing is frequent)
- Chunk dedup savings: across 1B users many files are duplicates; content-addressed storage achieves 30-50% storage savings

**File Upload (Chunked)**

```
Client:
1. Split file into 4MB chunks
2. Compute SHA256 hash of each chunk
3. POST /chunks/check {hashes[]} → server returns list of hashes it already has
4. For hashes server doesn't have: POST /chunks/{hash}/upload → receive presigned S3 URL
5. PUT chunk to S3 directly via presigned URL
6. POST /files/complete {name, chunks: [{hash, order}]} → server records file metadata
```

Why 4MB chunks:
- Small enough that any single chunk upload fits in memory and can be retried individually
- Large enough that overhead per chunk is negligible relative to data
- Resume: on connection drop, re-send only chunks whose hash server didn't have

Content-addressed storage (deduplication):
- Chunks are stored by their SHA256 hash as the key: `s3://chunks/{sha256hash}`
- If two users upload the same file → same hashes → zero additional storage used
- Saves ~30-50% storage at scale across 1B users

**Delta Sync**

Problem: User edits a 1GB Keynote presentation, changing only slide 3. Without delta sync = re-upload 1GB. With delta sync = re-upload one 4MB chunk.

Process:
1. Client maintains a local index: `{filePath → [{chunkHash, order}]}`
2. On file change: recompute hashes of all chunks
3. Compare with stored index: identify changed chunk hashes
4. Upload only changed chunks (POST /chunks/{hash}/upload for each new hash)
5. POST /files/{fileId}/update {newChunkList} — server creates new version

**Metadata Service**

PostgreSQL (strong consistency for metadata required):
```sql
files    (id, name, parentFolderId, ownerId, size, mimeType, createdAt, updatedAt)
versions (id, fileId, chunkList JSON, versionNumber, createdAt, createdBy)
folders  (id, name, parentFolderId, ownerId)
permissions (fileId, userId, role: VIEWER/EDITOR/OWNER, inheritedFrom)
```

Why PostgreSQL (not Cassandra) for metadata:
- Need ACID transactions: rename + update parent folder must be atomic
- Need strong consistency: immediately see changes across all devices
- Need complex queries: folder tree traversal, permission inheritance
- Metadata is small (kilobytes per file); fits in a single PostgreSQL cluster with read replicas

**Sync Service (Multi-Device)**

Change detection:
- Client registers a WebSocket (or long-polling) connection to Sync Service
- Sync Service subscribes to Kafka topic: `file-changes:{userId}`
- Metadata Service publishes to Kafka on every file/folder create, update, delete, move

Change propagation:
1. User A renames file on Device 1 → Metadata Service writes to PostgreSQL → publishes to Kafka
2. Sync Service consumes Kafka message → pushes change event to all User A's connected devices via WebSocket
3. Device 2 receives event → downloads updated metadata → downloads changed chunks if content changed

Conflict resolution:
- Last-write-wins by default (based on updatedAt timestamp)
- Alternative (Dropbox approach): create a conflict copy: `filename (UserA's conflicted copy 2026-06-08).ext` — no data loss

**Sharing & Permissions**

ACL model:
- `permissions (fileId, principalId, principalType: USER/GROUP, role: VIEWER/EDITOR/OWNER)`
- Folder permissions cascade to all children unless explicitly overridden at child level
- On permission check: walk up folder tree to find nearest explicit permission

Link sharing:
- Generate cryptographically random token (32 bytes = 256 bits entropy)
- Store: `share_links (token, fileId, permissions, createdBy, expiryTs, passwordHash?)`
- Anyone with the link (+ optional password) gets access; token revocable at any time

**Version History**
- Every file update creates a new version record pointing to the new chunk list
- Free tier: 30 days of version history; paid tier: unlimited
- Restore: POST /files/{fileId}/versions/{versionId}/restore — creates a new version with the old chunk list (no data movement needed, just metadata pointer update)

## STAR Interview Framework

**Situation:** "We need to design Google Drive sync for 1B users. Users frequently make small edits to large files (presentations, code) and need all their devices updated within seconds. Storage cost is a major concern at 1.5 EB actual usage."

**Task:** "Design a file storage and sync system with efficient upload (no full re-upload on small edits), cross-device real-time sync, deduplication, and version history."

**Action:**
1. Chunked upload (4MB chunks) with SHA256 content-addressing: client checks which chunks server already has before uploading; deduplication across all users saves ~30-50% storage
2. Delta sync: client maintains local chunk hash index; on file change only uploads chunks whose hash changed; 1GB file with 4MB edit = 4MB upload
3. PostgreSQL for metadata with strong consistency; Kafka change feed pushes updates to all connected devices via WebSocket Sync Service
4. Version history as metadata-only operation: each version points to a chunk list; restore = update pointer, no data movement

**Result:** Average sync latency < 3 seconds. 40% storage savings from dedup and delta sync at scale. Version history implemented at near-zero marginal storage cost for unchanged chunks.

## Flashcards

| Q | A |
|---|---|
| How does Google Drive avoid uploading the same file chunk twice? | SHA256 hash per 4MB chunk; client sends all hashes to server first; server returns which hashes it already has; client only uploads missing chunks (content-addressed deduplication) |
| How does WhatsApp guarantee message ordering? | Snowflake IDs (41-bit timestamp + datacenter + worker + 12-bit sequence) as Cassandra clustering key; monotonically increasing; sortable without central coordination |
| What is delta sync in Google Drive? | Client computes SHA256 hashes of all file chunks; compares against stored index; only uploads chunks whose hash changed; dramatically reduces upload size for large files with small edits |
| How does WhatsApp handle offline message delivery? | Write message to Cassandra first; send push notification via FCM/APNs; offline client fetches undelivered messages from Cassandra on reconnect using last-seen Snowflake ID as cursor |
| How does Google Drive handle sync conflicts between two devices? | Last-write-wins by default (based on updatedAt timestamp); alternatively create a conflict copy with a timestamped filename so no data is lost |
| Why is WhatsApp end-to-end encrypted, and how does it affect the server? | Signal Protocol (Double Ratchet + X3DH) derives session keys entirely client-side; server stores only public keys and encrypted ciphertext; even a full DB breach reveals no readable message content |
