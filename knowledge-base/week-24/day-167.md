## DSA (1.5 hours — OA Contest Mode)

### Pattern: Two-Pass DFS (Precompute + Propagate)
- Core idea: In problems that ask "what is the tree height if node X is removed?", a naive approach re-roots the tree for each query in O(N) per query. Instead, DFS pass 1 computes `depth[v]` (root-to-node) and `height[v]` (node-to-deepest-leaf) for every node. DFS pass 2 propagates `maxHeightWithout` downward — for each child, the best height achievable by skipping that child equals the maximum of the parent's inherited `maxHeightWithout` and the sibling subtree heights.
- Trigger condition: Multiple removal queries on a tree asking for resulting height/diameter — precompute subtree stats and propagate answers in a second DFS.
- Time complexity: O(N + Q) | Space complexity: O(N)

### Contest Template (write from memory before solving):
```java
// Two-pass DFS: precompute depth/height, then propagate complement height
int[] depth = new int[n + 1];
int[] height = new int[n + 1];
int[] answer = new int[n + 1]; // answer[node] = tree height after removing node

// Pass 1: DFS to fill depth[] and height[]
private int dfs1(int node, int parent, int d, int[][] children) {
    depth[node] = d;
    int h = 0;
    for (int child : children[node]) {
        if (child == parent) continue;
        h = Math.max(h, 1 + dfs1(child, node, d + 1, children));
    }
    return height[node] = h;
}

// Pass 2: DFS to fill answer[]
// maxWithout = best height achievable if this subtree did not exist
private void dfs2(int node, int parent, int maxWithout, int[][] children) {
    answer[node] = maxWithout;          // removing node → best height from outside
    // for each child, its complement = max(maxWithout, best sibling height + offset)
    // sort children heights descending; child gets max(maxWithout, 2nd-best + offset)
    int h1 = -1, h2 = -1;              // top-2 child heights
    for (int child : children[node]) {
        if (child == parent) continue;
        int hc = 1 + height[child];
        if (hc >= h1) { h2 = h1; h1 = hc; }
        else if (hc > h2) h2 = hc;
    }
    for (int child : children[node]) {
        if (child == parent) continue;
        int sibling = (1 + height[child] == h1) ? h2 : h1;
        int childMaxWithout = Math.max(maxWithout, depth[node] + Math.max(0, sibling));
        dfs2(child, node, childMaxWithout, children);
    }
}
```

class Solution {

    // LC 2458 — Height of Binary Tree After Subtree Removal Queries
    // (cold solve — 30-min timer, narrate approach before opening IDE)
    // TODO: public int[] treeQueries(TreeNode root, int[] queries)
    // Key: DFS1 computes depth[node] and height[node] for all nodes.
    //      DFS2 propagates maxHeightWithout downward: for each child c,
    //      answer[c.val] = max(parentMaxWithout, depth[c] + height(sibling)).
    //      Use top-2 child heights to serve both children in O(1).
    private int[] depthArr, heightArr, ans;

    public int[] treeQueries(TreeNode root, int[] queries) {
        depthArr = new int[100001];
        heightArr = new int[100001];
        ans = new int[100001];
        dfs1(root, 0);
        dfs2(root, 0);
        int[] result = new int[queries.length];
        for (int i = 0; i < queries.length; i++) result[i] = ans[queries[i]];
        return result;
    }

    private int dfs1(TreeNode node, int d) {
        if (node == null) return -1;
        depthArr[node.val] = d;
        int h = Math.max(dfs1(node.left, d + 1), dfs1(node.right, d + 1)) + 1;
        heightArr[node.val] = h;
        return h;
    }

    private void dfs2(TreeNode node, int maxWithout) {
        if (node == null) return;
        ans[node.val] = maxWithout;
        int lh = node.left  == null ? -1 : heightArr[node.left.val]  + 1;
        int rh = node.right == null ? -1 : heightArr[node.right.val] + 1;
        // left child complement: best of (maxWithout, depth[node]+rh)
        dfs2(node.left,  Math.max(maxWithout, depthArr[node.val] + rh));
        // right child complement: best of (maxWithout, depth[node]+lh)
        dfs2(node.right, Math.max(maxWithout, depthArr[node.val] + lh));
    }

    // (OA simulation — set 20-min timer before reading)
    // TODO: public int minTime(int n, int[][] edges, List<Boolean> hasApple)
    // Key (unlock after 20 min if stuck): build undirected adj list; DFS from 0;

    public int minTime(int n, int[][] edges, List<Boolean> hasApple) {
        List<List<Integer>> adj = new ArrayList<>();
        for (int i = 0; i < n; i++) adj.add(new ArrayList<>());
        for (int[] e : edges) { adj.get(e[0]).add(e[1]); adj.get(e[1]).add(e[0]); }
        return dfsApple(0, -1, adj, hasApple);
    }

    private int dfsApple(int node, int parent, List<List<Integer>> adj, List<Boolean> hasApple) {
        int cost = 0;
        for (int child : adj.get(node)) {
            if (child == parent) continue;
            int childCost = dfsApple(child, node, adj, hasApple);
            if (childCost > 0 || hasApple.get(child)) cost += childCost + 2;
        }
        return cost;
    }
}

#### Feed Generation Strategies

```
Fan-out on WRITE (push model):
  - On post creation: write one row into Feeds table for every follower
  - Read feed: SELECT post_id FROM Feeds WHERE user_id=? ORDER BY created_at DESC LIMIT 20
  - Read cost: O(1) — single index scan
  - Write cost: O(followers) — 1M followers = 1M inserts per post
  - Good for: typical users with <100K followers
  - Problem: celebrity accounts create write thundering herd

Fan-out on READ (pull model):
  - No Feeds table writes on post creation
  - Read feed: SELECT p.* FROM Posts p
               JOIN Follows f ON p.user_id = f.followee_id
               WHERE f.follower_id = ? ORDER BY p.created_at DESC LIMIT 20
  - Read cost: O(following_count) — joins across followees' posts
  - Write cost: O(1)
  - Good for: celebrities with 10M+ followers
  - Problem: slow reads for users following many accounts

Hybrid (Twitter/Instagram approach):
  - Push for users with < 1,000,000 followers
  - Pull for celebrity accounts (> 1,000,000 followers)
  - At read time: merge pre-computed push feed with freshly pulled celebrity posts
  - Merge is cheap: both are sorted by created_at → O(C) merge where C = celebrity count followed
```

**Design pattern:** Denormalized counters (like_count, follower_count) trade write overhead (must atomically increment on like/follow) for fast reads (no COUNT(*) on every profile load). Use `UPDATE Users SET follower_count = follower_count + 1 WHERE user_id = ?` — atomic in SQL.

**Trade-off:** Fan-out on write optimizes read latency at the cost of write amplification. Fan-out on read eliminates write amplification but adds read latency. Hybrid picks the boundary (e.g., 1M followers) based on measured cost per model.

**Interview talking point:** "If asked how Twitter handles Elon Musk's 100M+ follower account in their feed system, answer: celebrity accounts use fan-out on read — when a follower loads their feed, the system fetches Elon's latest tweets directly and merges them with the pre-computed push feed of other accounts the user follows. This avoids writing one tweet to 100M feed records synchronously."

## Flashcards

| Q | A |
|---|---|
| In LC 2458, what two things does DFS pass 1 compute and what does DFS pass 2 propagate? | Pass 1 computes `depth[node]` (root-to-node distance) and `height[node]` (node-to-deepest-leaf). Pass 2 propagates `maxHeightWithout` downward — the best tree height achievable if this node's subtree were removed. |
| Why track top-2 child heights in the two-pass DFS instead of just the best? | Each child's complement needs the best height from its siblings. If you only stored the best, the child with the best height would get an incorrect complement (it cannot use its own height). Top-2 lets you serve both children correctly in O(1). |
| What index is essential for paginated feed queries and why? | `INDEX (user_id, created_at DESC)` on the Feeds table — it lets the DB seek directly to the user's latest posts without a full table scan, making page N as fast as page 1. |
| What prevents a user from liking the same post twice at the DB level? | `PRIMARY KEY (user_id, post_id)` on the Likes table — inserting a duplicate throws a unique constraint violation, which the application catches and returns "already liked". |
| In LC 1443 (Collect All Apples), when do you add 2 to the child's cost? | When `childCost > 0` (the child's subtree has at least one apple) OR `hasApple[child]` is true — in both cases you must make the round trip (2 edges) to that child. If neither is true, you can skip the child entirely. |
