### Consensus Algorithms

**Why consensus is needed:** Multiple replicas must agree on the same sequence of writes despite node crashes and network partitions. Without it, two replicas can diverge — split-brain — and return conflicting data.

**Raft**
- Leader-based: all writes go through the elected leader
- Leader election: candidates broadcast RequestVote; first to reach quorum (N/2+1) wins; term number prevents stale leaders from being accepted
- Log replication: leader appends entry → AppendEntries RPC to all followers → commits once majority acknowledges → applies to state machine
- Safety guarantee: a candidate must have at least as up-to-date a log as the voter to receive that vote
- Used in: etcd (Kubernetes backing store), CockroachDB, TiKV, Consul

**Paxos**
- Original consensus algorithm (Lamport, 1989)
- Two phases: Prepare (promise phase — proposer asks acceptors to promise not to accept lower-numbered proposals) → Accept (proposer sends accepted value)
- Multi-Paxos: elect a stable leader to skip Prepare for subsequent rounds
- ZooKeeper uses ZAB (ZooKeeper Atomic Broadcast) — a Paxos variant with primary-backup semantics
- Rarely implemented directly; use Raft or a library

**Quorum**
- Majority of nodes must agree: quorum = N/2 + 1
- 3 nodes → quorum = 2; 5 nodes → quorum = 3; 7 nodes → quorum = 4
- Read quorum + Write quorum > N ensures at least one node always has the latest write
- DynamoDB/Cassandra: configurable R + W > N for strong consistency (e.g., R=2, W=2, N=3)

### Distributed Locks (Redlock)

**Problem:** Two processes must not execute the same critical section simultaneously — e.g., charge a payment exactly once, send a notification once.

**Redlock Algorithm (Redis)**
1. Record current timestamp
2. Try acquiring lock on all N Redis nodes sequentially with same key and TTL (N=5 recommended)
3. Lock is acquired only if: majority (N/2+1) nodes responded OK AND total elapsed time < lock TTL / 2
4. Use the lock; release on all nodes upon completion or failure (even failed nodes)
5. If acquisition fails → immediately release any partial locks acquired

**Flaw:** Clock drift and GC pauses can cause two clients to simultaneously believe they hold the lock.

**Fencing tokens (mitigation)**
- Lock service issues a monotonically increasing integer token with every lock grant
- Every downstream write includes the token
- Storage layer rejects writes with a token lower than the maximum it has seen
- Protects against: delayed writes after lock expired due to GC pause or network partition

**ZooKeeper Distributed Locks (preferred for critical operations)**
- Create ephemeral sequential znode under `/locks/resource`
- Client with the smallest sequence number = current lock holder
- Others watch their immediate predecessor znode; receive notification when it deletes
- Ephemeral nodes auto-delete on client session loss — no manual TTL management needed
- Used for: Kafka broker election, HBase master election, Hadoop NameNode HA

### Leader Election

**ZooKeeper Pattern**
1. All candidates create ephemeral sequential znodes at `/election/node-`
2. Each candidate lists children; if its znode is the smallest → it is leader
3. Non-leaders watch the znode immediately preceding theirs (not all — avoids herd effect on large clusters)
4. If predecessor deletes → re-check: if now smallest = become leader; else watch new predecessor

**etcd Lease-Based (Raft-backed)**
- Candidate creates a key with a TTL-based lease; key present = leader is active
- Leader continuously renews lease heartbeat
- If leader crashes, TTL expires, key deleted, remaining candidates re-compete
- Used by: Kubernetes controller manager, scheduler leader election

**Real-world use cases**
- Kafka: each partition has one elected leader broker; followers replicate in-sync
- MySQL Group Replication: primary elected via consensus
- Elasticsearch: master node elected by majority of master-eligible nodes

### Two-Phase Commit (2PC)

**Phase 1 — Prepare:** Coordinator sends PREPARE to all participants; each writes to WAL and undo log, locks resources, then votes YES or NO.

**Phase 2 — Commit/Abort:** If all YES → coordinator sends COMMIT; if any NO → sends ABORT. Participants execute and release locks.

**Critical flaw: blocking protocol**
- If coordinator crashes after Phase 1 before sending Phase 2, participants hold resource locks indefinitely
- No participant knows the global outcome; cannot proceed without hearing from coordinator
- 3PC partially addresses this with an extra round trip but adds latency and still has edge cases under certain network partitions

**When 2PC is acceptable**
- Single datacenter, low-latency, tightly controlled environments
- Internal DB cross-table transactions (hidden inside a single RDBMS, not across services)
- Avoid across microservices — use Saga instead

### Saga Pattern

**Core idea:** A long-running business transaction is broken into a sequence of local transactions. Each has a corresponding compensating transaction used for rollback. No distributed locks; no blocking.

**Choreography Saga**
- No central controller; services emit domain events and react to each other's events
- Example: OrderService publishes `OrderCreated` → PaymentService listens → publishes `PaymentCharged` → InventoryService listens → publishes `InventoryReserved` → ShippingService listens
- Pros: fully decoupled, no single point of failure, services are independently deployable
- Cons: overall transaction state is distributed across events — hard to trace, hard to debug

**Orchestration Saga**
- Dedicated Saga Orchestrator (state machine service) directs each step via explicit commands
- Orchestrator calls PaymentService → on success calls InventoryService → on success calls ShippingService
- On failure at any step: orchestrator triggers compensating transactions in reverse order
- Pros: central visibility of transaction state, auditable, easy to debug
- Cons: orchestrator is a single point of failure (mitigate with HA); extra service to maintain

**Compensating Transactions**
- Each step must have a defined rollback action
- Example (Order→Payment→Inventory→Shipping):
  - Shipping fails → compensate Inventory (release reservation) → compensate Payment (issue refund) → cancel Order
- Compensation must be idempotent: safe to execute multiple times (on retry)
- Compensations are best-effort — if they fail, alert and handle manually (the "sorry email" pattern)

**Idempotency**
- Every operation produces the same result regardless of how many times it is called with the same input
- Implement with idempotency keys: client generates UUID per request; server stores result keyed by UUID in Redis with TTL; returns cached result on duplicate retry
- Critical for: payment processing, notification sending, order creation

### Disaster Recovery

**RPO (Recovery Point Objective):** Maximum acceptable data loss measured in time. RPO = 1 hour means at most 1 hour of data can be lost in a disaster (last backup can be 1 hour old).

**RTO (Recovery Time Objective):** Maximum acceptable downtime before the system is restored. RTO = 4 hours means the system must be operational within 4 hours of a failure.

**DR Strategies (ascending cost, descending RPO/RTO)**

| Strategy | RPO | RTO | Cost | Mechanism |
|---|---|---|---|---|
| Backup & Restore | Hours–days | Hours–days | Low | Periodic snapshots to S3; restore from scratch on DR event |
| Pilot Light | Minutes–hours | 30–60 min | Medium | Core database always running in DR region; app servers shut down; scale out on event |
| Warm Standby | Seconds–minutes | Minutes | Medium-high | Scaled-down replica of production running in DR region; scale up on DR event |
| Hot Standby (Active-Active) | Near zero | Near zero | High | Full-capacity DR region serving live traffic; instant failover via DNS/LB change |

**Key techniques**
- Database replication across regions: async for speed (small data loss window), sync for zero RPO at cost of write latency
- S3 cross-region replication: replicate object storage to DR region automatically
- Route 53 health checks + DNS failover: automatic traffic rerouting on primary endpoint failure
- Regularly test DR runbooks: untested DR plans fail under pressure (practice Game Days)

## STAR Interview Framework

**Situation:** "We're building a payment processing platform for a fintech startup. Multiple services — debit account, update ledger, send receipt — need to coordinate, but each owns its own database. We've had double-charge incidents caused by network retries during peak load."

**Task:** "Design a distributed lock service and transaction coordination mechanism to prevent duplicate payment processing while maintaining high availability across two datacenters."

**Action:**
1. Added idempotency keys at the API gateway layer — each payment request carries a client-generated UUID; server atomically checks-and-sets the key in Redis with 24h TTL; retries receive the cached response instantly
2. Replaced 2PC with an Orchestration Saga: debit → update ledger → send receipt; each step has a compensating transaction; Saga orchestrator persists its state to PostgreSQL (durable, survives crash)
3. For the high-value daily settlement job (must run exactly once across 12 replicas), used ZooKeeper ephemeral sequential znodes for leader election rather than Redlock — ZooKeeper's strong consistency is better suited for financial critical sections
4. Added fencing tokens to the settlement write path: storage layer rejects any write with a stale token, protecting against GC-paused-then-resumed workers writing with an expired lock

**Result:** Zero double-charge incidents in the six months post-deployment. Settlement job safely elects a single leader. Saga orchestrator's persistent state log enabled debugging of three failed transaction sequences within minutes versus hours previously.

## Flashcards

| Q | A |
|---|---|
| What is quorum in distributed systems? | Majority of nodes (N/2+1) that must agree for an operation to succeed; prevents split-brain by ensuring any two quorums overlap by at least one node |
| Raft vs Paxos — key difference? | Raft decomposes consensus into leader election + log replication (designed for understandability, single stable leader); Paxos is the original general algorithm, harder to implement correctly, used as basis for ZAB (ZooKeeper) |
| What problem does Redlock solve, and what is its limitation? | Distributed mutual exclusion across Redis instances; limitation: vulnerable to clock skew and GC pauses — two clients can simultaneously believe they hold the lock; mitigate with fencing tokens |
| Why is 2PC problematic in microservices? | Blocking protocol — if coordinator crashes after Phase 1, participants hold resource locks indefinitely with no way to proceed; replaced by Saga which uses local transactions + compensating rollbacks |
| Saga choreography vs orchestration — when to use each? | Choreography for simple flows with few services where decoupling matters most; orchestration when you need central visibility, complex rollback logic, or auditability of transaction state |
| What is a compensating transaction? | A transaction that logically reverses a previously committed local transaction; must be idempotent because it may be called multiple times during retry |
| RPO vs RTO — define both | RPO = max data loss acceptable measured in time (how old can the last backup be?); RTO = max downtime acceptable measured in time (how long can the system be down?) |
| Hot standby vs warm standby? | Hot = full-capacity replica in another region serving live traffic (RPO near zero, RTO near zero, highest cost); warm = scaled-down replica not serving live traffic (RPO seconds, RTO minutes, lower cost) |
