## HLD (2 hours)

### System Design Interview Framework (5 Steps)

**Step 1 — Requirements Clarification (5 minutes)**

Questions to ask:
- "What are the core features we need to support?" (Functional)
- "How many users / DAU / QPS are we designing for?"
- "What's the read-to-write ratio?"
- "What latency target should we hit? (p99)"
- "Do we need strong consistency or is eventual consistency acceptable?"
- "What's the acceptable data loss? Any SLA / SLO to hit?"
- "Are there any specific compliance or security requirements?"

Why this matters: wrong assumptions lead to wrong architecture. A system designed for 1K QPS looks nothing like one designed for 1M QPS. An interviewer is evaluating whether you ask the right clarifying questions before designing.

Key tells of a strong candidate at Step 1:
- Asks about read vs write ratio (drives caching and DB choice)
- Asks about consistency requirements (drives CP vs AP decision)
- Asks about global vs single-region (drives CDN, geo-distribution)
- Does not spend more than 5 minutes; moves forward confidently

---

**Step 2 — Scale Estimation (5 minutes)**

QPS calculation:
- DAU × average daily actions / 86,400 = average QPS
- Multiply by 2-3x for peak (rule of thumb)

Storage calculation:
- Record count × record size × retention period
- Separate for hot storage (fast, expensive) vs cold storage (S3, cheap)

Cache sizing:
- Apply 80/20 rule: 20% of data = 80% of read traffic
- Cache size = 20% of total records × record size

Bandwidth:
- Ingress = write QPS × average payload size
- Egress = read QPS × average response size

Useful conversion constants:
| Daily Operations | QPS |
|-----------------|-----|
| 1M / day | ~12/sec |
| 10M / day | ~116/sec |
| 100M / day | ~1,200/sec |
| 1B / day | ~11,600/sec |

System throughput limits (for sanity-checking your design):
| System | Throughput |
|--------|-----------|
| Redis | ~100K ops/sec per instance |
| MySQL writes | ~1K/sec sustained |
| MySQL reads | ~10K/sec |
| Cassandra writes | ~50K/sec per node |
| Kafka | ~1M messages/sec across all partitions |
| S3 PUT | ~3,500/sec per prefix |
| S3 GET | ~5,500/sec per prefix |

---

**Step 3 — High-Level Design (10 minutes)**

Draw boxes first, then explain data flow.

Standard boxes to consider: Client → Load Balancer → API Servers → Cache → Database → Message Queue → Workers → Object Storage → CDN → Search

Write path: trace a single write request from client to all systems it touches.
Read path: trace a single read request, showing where caching applies.

DB choice justification template:
- "I'll use [DB] because the access pattern is [key-value/time-series/relational/document], the write volume is [N/sec], and we need [strong/eventual] consistency."

Always state trade-offs:
- "I'm choosing eventual consistency here because a slight delay in seeing new tweets is acceptable; strong consistency would require synchronous replication which adds write latency."

---

**Step 4 — Deep Dives (20 minutes)**

Focus on the hardest or most interesting problem in the system. Common deep dive areas:

| Area | What to Cover |
|------|--------------|
| DB sharding | Shard key choice, hotspot avoidance, cross-shard queries |
| Caching | Write strategy, eviction policy, cache stampede prevention |
| Real-time delivery | WebSocket vs SSE, connection management, offline handling |
| Deduplication | Idempotency key implementation, Redis NX, Bloom filters |
| Failover | Circuit breaker, retry with jitter, bulkhead isolation |
| Fan-out | Push vs pull model, celebrity problem, hybrid approach |
| File uploads | Chunked upload, resumable, delta sync, dedup |
| Distributed locks | Redlock, fencing tokens, ZooKeeper znodes |

Have opinions: "I'd use X over Y because..." is stronger than "either could work".

Use specific numbers: "Redis can handle 100K ops/sec, which covers our 11,600 read QPS with 10x headroom."

---

**Step 5 — Wrap Up (5 minutes)**

- State the biggest bottleneck in the current design and how you'd address it
- Monitoring strategy: "I'd alert on SLO breach — p99 > 200ms or error rate > 0.1% — using Prometheus + Grafana + PagerDuty"
- 10x scale plan: "At 10x scale I'd add read replicas to the DB, shard the cache, and add a second region"
- What you'd do differently: shows self-awareness and senior thinking

### HLD Decision Cheat Sheet

| Problem | Solution |
|---------|----------|
| Read-heavy system | Redis cache + read replicas + CDN for static |
| Write-heavy system | Message queue + CQRS + DB sharding |
| Real-time bidirectional | WebSockets (chat, collaborative editing, gaming) |
| Real-time server push | SSE (simpler, unidirectional — live feeds, notifications) |
| Large file storage | S3 + multipart upload + CDN |
| Full-text search | Elasticsearch with inverted index |
| Geospatial queries | Redis Geo (GEOADD/GEORADIUS) or PostGIS |
| Distribute cache load | Consistent hashing with virtual nodes |
| Distributed transaction | Saga (choreography or orchestration, not 2PC) |
| Prevent duplicate processing | Idempotency key: hash(userId+action+correlationId) in Redis NX |
| Prevent cascade failure | Circuit breaker + retry with exponential backoff + jitter + bulkhead |
| Global low-latency reads | CDN + Anycast + geo-distributed DB with cross-region replication |
| Audit trail / event replay | Event sourcing + Kafka as append-only log |
| Heavy fan-out writes | Fan-out on read for celebrities; fan-out on write for regular users |
| Seat / resource contention | Redis SET NX EX (distributed lock) + optimistic locking in DB |
| Rate limiting | Token bucket (allows burst) or sliding window (most accurate) via Redis |
| Scheduling | DB + poller OR Redis sorted set as delay queue (score = scheduledAt epoch) |
| Dedup cache warmup | Bloom filter to check membership before DB query |

### Availability Nines

| SLA | Downtime per Year | Downtime per Month |
|-----|------------------|--------------------|
| 99% | 87.6 hours | 7.3 hours |
| 99.9% | 8.76 hours | 43.8 minutes |
| 99.99% | 52.6 minutes | 4.4 minutes |
| 99.999% | 5.26 minutes | 26.3 seconds |

To achieve 99.99%: eliminate single points of failure; multi-AZ deployment; automated failover; chaos testing; < 2 minute MTTR on common incidents.

### Full HLD Topic Recap

| Category | Key Concepts | Interview Talking Point |
|----------|-------------|------------------------|
| Fundamentals | CAP theorem, sync vs async, latency vs throughput | "Choose C vs A depending on partition tolerance requirement; always state which you sacrifice and why it's acceptable" |
| Scalability | Horizontal scaling, load balancing, stateless design | "Stateless services scale horizontally; all session and user state lives in Redis or DB, never in application memory" |
| Databases | SQL vs NoSQL, sharding, indexing, replication | "Start with PostgreSQL; switch to NoSQL when schema flexibility or horizontal write scale is required" |
| Caching | Cache-aside, write-through, LRU/LFU, Redis | "80/20 rule: cache the 20% of data that accounts for 80% of reads; Redis at 100K ops/sec covers most workloads" |
| Messaging | Kafka vs SQS, pub/sub, at-least-once, DLQ | "Use Kafka when multiple independent consumers need the same events and replay matters; SQS for point-to-point task queues" |
| APIs | REST vs gRPC vs GraphQL, rate limiting, pagination | "gRPC for internal service-to-service; REST for external; GraphQL when clients need flexible query shapes" |
| Search | Elasticsearch, inverted index, BM25, typeahead | "ES for full-text; Trie or ES prefix query for typeahead; always have a primary DB as source of truth alongside ES" |
| Storage | S3, chunked upload, CDN, presigned URLs | "S3 for all objects; presigned URLs for direct client upload to bypass app servers; CloudFront CDN for global serving" |
| Security | OAuth2, JWT, mTLS, TLS, signal protocol | "JWT for stateless auth (15-60 min expiry); HttpOnly cookie for refresh token; mTLS between internal services via service mesh" |
| Observability | RED metrics, distributed tracing, SLO alerting | "Alert on SLO breach (p99 > 500ms, error rate > 0.1%), not raw thresholds; correlation IDs from the first day" |
| Distributed | Raft, Saga, idempotency, Redlock, fencing tokens | "Saga replaces 2PC for microservices; idempotency keys on all write APIs; fencing tokens with Redlock for safety" |
| Reliability | Circuit breaker, retry + jitter, bulkhead, DLQ | "Circuit breaker prevents cascade failure; retry with jitter prevents thundering herd; bulkhead isolates blast radius" |
| Infra | K8s HPA, canary deploy, service mesh | "Canary limits blast radius to X% of users; service mesh handles mTLS and distributed tracing without code changes" |

## STAR Interview Framework

**Situation:** "In a senior system design interview, the interviewer presents a blank canvas: 'Design a system like WhatsApp / Uber / Twitter from scratch. You have 45 minutes.'"

**Task:** "Produce a complete, defensible, well-reasoned architecture that covers functional requirements, scale, core components, and at least one deep dive — all within 45 minutes."

**Action:**
1. Requirements (5 min): ask 4-5 targeted questions about scale, read/write ratio, consistency, and real-time requirements before drawing anything
2. Scale estimation (5 min): calculate QPS, storage, and cache sizing on the whiteboard; state assumptions clearly
3. High-level design (10 min): draw the core boxes, trace the write path, trace the read path, justify DB and cache choice
4. Deep dive (20 min): pick the hardest component (fan-out, ordering, locking, sync); go deep with specific technical choices and trade-offs
5. Wrap-up (5 min): state bottleneck, monitoring approach, 10x scale plan

**Result:** Structured, predictable interview performance. Interviewer evaluates thought process and trade-off reasoning, not just the final answer. Following the framework prevents the most common failure mode — jumping to implementation before understanding requirements.

## Flashcards

| Q | A |
|---|---|
| What are the 5 steps of the system design interview framework? | Requirements (5 min) → Scale Estimation (5 min) → High-Level Design (10 min) → Deep Dives (20 min) → Wrap Up (5 min) |
| Convert 100M requests/day to QPS | 100M / 86,400 approximately 1,200 requests/sec |
| What is the SLA downtime for 99.99% availability per year? | 52.6 minutes per year / 4.4 minutes per month |
| When would you choose Kafka over SQS? | When multiple independent consumer groups need the same events; Kafka retains messages for replay; SQS deletes on consumption — point-to-point only |
| What does the 80/20 caching rule mean in practice? | Cache the 20% of data that accounts for 80% of reads; at 100M records × 1KB = 100GB total, cache 20GB in Redis to serve most traffic |
| CAP theorem — what do you always give up in a distributed system during a partition? | You must sacrifice either Consistency (AP: Cassandra, DynamoDB) or Availability (CP: ZooKeeper, HBase); you cannot have both C and A during a network partition |
| Redis ops/sec vs MySQL writes/sec — practical implication for caching? | Redis ~100K ops/sec; MySQL writes ~1K/sec — Redis is ~100x faster for writes; explains why a thin Redis cache in front of MySQL can handle 100x more read throughput |
| When should you use WebSockets vs SSE? | WebSockets for bidirectional real-time (chat, collaborative editing, gaming); SSE for unidirectional server push (live feeds, live scores, notifications, location tracking to rider) |
| How do you handle thundering herd when a hot Redis cache key expires? | Three options: (1) probabilistic early expiration (XFetch — re-cache before TTL expires); (2) mutex / Redis SET NX EX on cache miss (first thread fetches, others wait); (3) background refresh job refreshes before TTL |
| What monitoring strategy should you implement for a new service from day one? | RED method (Rate, Errors, Duration per endpoint); set SLOs on p99 latency and error rate; alert on SLO breach, not raw thresholds; add distributed tracing with correlation IDs; DLQ depth alert for async services |
