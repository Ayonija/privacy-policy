## DSA (1.5 hours — OA Contest Mode)

### Pattern: Binary Search on Answer Value (Feasibility Check)
- Core idea: Instead of searching for an element in a sorted array, binary search on the answer space (a value range). Define a `feasible(mid)` function that checks whether mid satisfies the problem constraint. Find the minimum (or maximum) mid where `feasible(mid)` is true. The key design decisions are: (1) setting correct lo and hi bounds, (2) writing a correct and efficient feasibility check, and (3) choosing the right convergence direction.
- Trigger condition: Use binary search on value when the problem involves "minimum/maximum X such that condition Y holds" and the feasibility condition is monotone — if mid is feasible, all values above (or below) mid are also feasible.
- Time complexity: O(N log(max-min)) where N is the cost of one feasibility check | Space complexity: O(1)

### Contest Template (write from memory before solving):
```java
// Binary search on answer — minimization variant
int lo = /* minimum possible answer */, hi = /* maximum possible answer */;
while (lo < hi) {
    int mid = lo + (hi - lo) / 2;
    if (feasible(mid)) hi = mid;    // mid works; try smaller
    else              lo = mid + 1; // mid fails; need larger
}
return lo; // lo == hi == answer

// Binary search on answer — maximization variant
while (lo < hi) {
    int mid = lo + (hi - lo + 1) / 2; // round up to avoid infinite loop
    if (feasible(mid)) lo = mid;    // mid works; try larger
    else              hi = mid - 1; // mid fails; need smaller
}
return lo;

// Feasibility check pattern (example: capacity to ship in D days)
private boolean feasible(int[] weights, int capacity, int days) {
    int usedDays = 1, currentLoad = 0;
    for (int w : weights) {
        if (currentLoad + w > capacity) { usedDays++; currentLoad = 0; }
        currentLoad += w;
    }
    return usedDays <= days;
}
```

    public int minEatingSpeed(int[] piles, int h) {
        int lo = 1, hi = 0;
        for (int p : piles) hi = Math.max(hi, p);
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (feasibleKoko(piles, mid, h)) hi = mid;
            else lo = mid + 1;
        }
        return lo;
    }
    private boolean feasibleKoko(int[] piles, int k, int h) {
        int hours = 0;
        for (int p : piles) hours += (p + k - 1) / k; // ceil(p / k)
        return hours <= h;
    }

#### Design Pattern
**State pattern** — `ATM` delegates every operation to `currentState`. Each concrete state implements only the valid transitions and throws `IllegalStateException` (or calls `screen.display("Invalid operation")`) for invalid events. Adding a new state (e.g., `MaintenanceModeState`) requires only a new class implementing `ATMState` — no changes to `ATM` or any existing state class (Open/Closed Principle).

#### Key Trade-off
State objects are stateless (flyweight) except for `CardInsertedState` which must track `cardId` and `pinAttempts` for the current session. Stateless states can be singletons shared across all ATM instances. Stateful states are created fresh per session and garbage-collected on transition to `IDLE`.

**Interview talking point:** "If asked why you use the State pattern instead of if-else chains for ATM, answer: the State pattern makes adding new states or transitions trivial — you add a new State class without modifying existing ones (Open/Closed Principle). An if-else chain across all states becomes a maintenance nightmare when you add 'card blocked', 'maintenance mode', or 'network timeout' states — every new state requires touching every method. With State, the logic for each state is fully encapsulated in one class."

## Flashcards

| In LC 2040, how do you count pairs with product ≤ mid when nums1[i] is negative? | When nums1[i] < 0, dividing both sides of nums1[i]*nums2[j] ≤ mid by nums1[i] flips the inequality: nums2[j] ≥ ceil(mid / nums1[i]). | What are the lo and hi bounds for LC 875 (Koko Eating Bananas) and why? | lo=1 (Koko eats at least 1 banana/hour), hi=max(piles) (eating faster than the largest pile is wasteful — it provides no benefit and the answer cannot exceed max(piles)). The feasibility check is sum(ceil(pile/k)) ≤ h. |
| What are the lo and hi bounds for LC 1011 (Capacity To Ship) and why? | lo=max(weights): the ship must be able to carry the heaviest single package or shipping is impossible. hi=sum(weights): a ship with this capacity ships everything in exactly 1 day — always feasible. The feasibility check is the greedy day-counting simulation. |
| What is the Open/Closed Principle and how does the State pattern in ATM satisfy it? | Open/Closed: classes should be open for extension but closed for modification. State pattern satisfies it by encapsulating each state's behavior in its own class. Adding a new state (e.g., MaintenanceModeState) requires creating a new class implementing ATMState — no existing state classes or the ATM class itself need modification. |
| In the ATM State pattern, which state must be stateful and why? | CardInsertedState must be stateful — it stores cardId (needed for PIN validation with BankService) and pinAttempts (needed to decide whether to retry or capture/eject the card).
