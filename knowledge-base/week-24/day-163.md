## HLD (2 hours)

### Uber (Ride-Hailing)

**Requirements**

Functional:
- Rider requests a ride; matched to nearby available driver
- Real-time location tracking for both rider and driver during trip
- Fare calculation (distance + time + surge)
- Trip history for riders and drivers

Non-functional:
- 20M rides/day
- 5M active drivers sending location updates every 5 seconds
- Match latency < 1 second (p99)
- Location update throughput: 1M updates/sec

**Scale Estimation**
- Location updates: 5M drivers × 1 update/5sec = 1M updates/sec
- Match requests: 20M rides / 86,400 = 231/sec (low throughput; latency-sensitive, not throughput-sensitive)
- Trip storage: 20M trips/day × 1KB per trip = 20GB/day

**Location Service**

Why WebSocket (not HTTP):
- Driver app sends one GPS update every 5 seconds
- HTTP: TCP handshake + TLS + HTTP headers per request = ~500 bytes overhead per 50-byte payload
- WebSocket: one persistent connection; per-message overhead ~2 bytes; ~10x lower overhead
- At 1M updates/sec the difference is enormous in bandwidth and CPU cost

Location storage with Redis Geo:
- `GEOADD drivers longitude latitude driverId` — stores location in a geospatial index
- `GEORADIUS drivers lon lat 5 km ASC COUNT 20` — find up to 20 drivers within 5km, sorted by distance
- Driver location expires: `EXPIRE drivers:driverId 30` — if no update in 30 seconds, driver considered offline
- Each driver key has its own TTL; Redis Geo stores in a sorted set under the hood using geohash encoding

Geohash sharding:
- Divide map into geohash cells (e.g., 6-character geohash = ~1.2km x 0.6km cells)
- Use geohash prefix as shard key across Redis Cluster nodes
- Local queries hit one shard; cross-cell queries hit at most 2-4 neighboring cells

**Matching Service**

1. Rider requests ride → Matching Service receives request
2. `GEORADIUS` query → list of nearby available drivers (within 5km, max 20 candidates)
3. For each candidate: call Routing Service to get actual road ETA (not Euclidean distance)
4. Rank drivers by ETA; filter by vehicle type preference
5. Send ride offer to top-ranked driver via WebSocket
6. Driver has 10 seconds to accept or decline
7. On decline: mark driver as busy (skip for next N seconds), offer to next candidate
8. Repeat until accepted or no drivers available → show "no drivers nearby"

**Trip State Machine (State Pattern)**

```
REQUESTED → DRIVER_ASSIGNED → DRIVER_EN_ROUTE → ARRIVED → IN_PROGRESS → COMPLETED
         ↘                                                              ↗
          ────────────────────── CANCELLED ───────────────────────────
```

- Each state has allowed transitions; invalid transitions rejected
- Trip state stored in Redis (fast reads during active trip) + PostgreSQL (durable record)
- State machine: `TripState` interface with `handleEvent(event)` method; each concrete state handles allowed events and rejects invalid ones

**Surge Pricing (Strategy Pattern)**

- `PricingStrategy` interface: `calculateFare(distance, duration, context)`
- `NormalPricingStrategy`, `SurgePricingStrategy`, `PromoPricingStrategy` implement it
- Fare Service selects strategy at ride request time based on supply/demand ratio in the driver's geohash cell
- Surge factor recalculated every minute: `surgeMultiplier = requestRate / (availableDrivers * normalRate)`
- If surgeMultiplier > 1.2 → switch to SurgePricingStrategy for that geohash cell

**Real-time Tracking for Rider**

- Driver sends location updates via WebSocket to Location Service every 5 seconds during trip
- Location Service publishes to Kafka topic: `trip-location:{tripId}` → message = `{driverId, lat, lon, timestamp}`
- Rider app subscribes via SSE (Server-Sent Events) or WebSocket to their trip feed
- SSE sufficient here: unidirectional push (server → client); simpler than WebSocket for this use case
- Rider app renders driver's moving pin on map using the location stream

**Design Patterns Used:**
- State: trip lifecycle (REQUESTED → COMPLETED, CANCELLED)
- Strategy: pricing calculation (Normal, Surge, Promo)
- Observer: location updates propagated to rider via Kafka pub/sub

### BookMyShow (Movie Ticket Booking)

**Requirements**

Functional:
- Browse movies, shows, theaters, showtimes
- View interactive seat map for a showtime
- Select seats and hold temporarily during checkout
- Complete booking with payment; receive confirmation
- Cancel booking

Non-functional:
- 5M bookings/day (normal); 10x spike on popular release day = 580/sec peak
- Seats cannot be double-booked under any circumstances (hard constraint)
- Seat lock during checkout: 10 minutes TTL
- 99.99% availability

**Scale Estimation**
- Normal QPS: 5M / 86,400 approximately 58/sec writes; 10x peak = 580/sec
- Read QPS: much higher (seat map browsing, search) — cache aggressively

**Seat Locking (Critical Section)**

Redis atomic lock during checkout:
```
SET seat:{showId}:{seatId} {lockToken} NX EX 600
```
- `NX`: only set if key does not exist (atomic — no race condition)
- `EX 600`: TTL of 10 minutes; lock automatically releases if payment times out
- `lockToken` = UUID generated per checkout session; used to verify ownership before release

On payment success:
1. Verify lock token matches: `GET seat:{showId}:{seatId}` == lockToken
2. Mark seat BOOKED in PostgreSQL: `UPDATE seats SET status='BOOKED', userId=X WHERE id=Y AND status='AVAILABLE'`
3. Delete Redis lock: `DEL seat:{showId}:{seatId}`
4. Insert booking record; publish confirmation email event to Kafka

On payment failure or timeout:
1. Release Redis lock: `DEL seat:{showId}:{seatId}` (or let TTL expire)
2. Seat becomes available again automatically

Why Redis over DB-level lock for this:
- Redis `SET NX EX` is a single round-trip atomic operation; extremely fast
- DB-level `SELECT FOR UPDATE` holds a DB connection for the entire checkout duration (up to 10 min) — exhausts connection pool under spike

**Concurrency Handling**

Optimistic locking in DB (belt-and-suspenders):
```sql
UPDATE seats
SET status = 'BOOKED', version = version + 1
WHERE id = ? AND version = ? AND status = 'AVAILABLE';
```
- If 0 rows affected → concurrent booking detected → return conflict error to user
- Combined with Redis NX lock: Redis handles the spike; optimistic locking in DB handles the rare edge case where Redis lock is bypassed or expires just as payment completes

Pessimistic locking (`SELECT FOR UPDATE`):
- Acquires row-level DB lock; blocks all other transactions on those rows
- Acceptable only for very low concurrency (< 10 concurrent users per show)
- Not appropriate for a popular movie release with thousands of concurrent users

**Database Design**

PostgreSQL (relational joins needed across movies, theaters, shows, seats):
- `movies (id, title, duration, genre, rating)`
- `theaters (id, name, city, address)`
- `shows (id, movieId, theaterId, showtime, status)` — partition by date for archival
- `seats (id, showId, row, number, type, status: AVAILABLE/LOCKED/BOOKED)`
- `bookings (id, userId, showId, seatIds[], totalAmount, status, idempotencyKey)`

Cassandra for booking history:
- High write volume (5M bookings/day); partition by userId for fast "my bookings" queries
- Append-only pattern (no updates to historical bookings)

Seat availability cache:
- Redis sorted set or hash: `showId → {seatId: status}` for fast seat map rendering
- Async sync from PostgreSQL; invalidated on booking

**Waitlist (Observer Pattern)**

- User can join waitlist for a fully booked show: `INSERT INTO waitlist (showId, userId, timestamp)`
- On booking cancellation: Booking Service publishes `SEAT_RELEASED` event to Kafka
- Notification Service (observer) consumes event → queries waitlist for first user → sends push/email notification
- First notified user gets a 5-minute exclusive Redis lock on the released seat to complete booking
- If they don't complete in 5 minutes → lock released → next waitlisted user notified

**Queue for Spike Handling**
- On popular release day: SQS FIFO queue absorbs booking requests at the API layer
- Booking workers process at sustainable rate (e.g., 500 bookings/sec)
- Users see "processing your request" with estimated wait time
- Prevents DB overload while maintaining eventual booking completion guarantee

## STAR Interview Framework

**Situation:** "A popular movie release day causes a 10x traffic spike on BookMyShow — 580 concurrent booking requests per second for the same showtime. We had experienced double-booking incidents in the past where two users were charged for the same seat."

**Task:** "Design the seat booking system to handle the spike while guaranteeing no double-bookings and maintaining p99 booking latency under 2 seconds."

**Action:**
1. Redis `SET seatId lockToken NX EX 600` as the seat lock — atomic, single round-trip, 10-minute TTL prevents indefinite holds; eliminates double-booking at the critical section
2. Optimistic locking in PostgreSQL as a belt-and-suspenders check: `UPDATE ... WHERE status='AVAILABLE' AND version=N` — 0 rows affected signals conflict
3. SQS FIFO queue absorbs the spike at the API layer: booking workers consume at 500/sec sustained rate; users get wait time estimate rather than error
4. Observer pattern for waitlist: cancellations trigger Kafka events; Notification Service notifies first waitlisted user with 5-minute exclusive lock

**Result:** Zero double-bookings post-deployment. p99 booking latency stayed under 2 seconds even at 10x spike. Waitlist system converted approximately 30% of cancellations into confirmed bookings.
