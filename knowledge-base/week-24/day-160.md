### Docker & Kubernetes (Design Context)

**Container vs VM**
- Container: app + runtime + dependencies isolated in Linux namespaces (PID, net, mount, IPC); shares host kernel; starts in milliseconds
- VM: full OS virtualization via hypervisor; minutes to start; stronger isolation; higher overhead
- Interview framing: "We containerize each microservice so it runs identically in dev, staging, and production; eliminates environment drift"

**Key K8s Objects**

| Object | Purpose |
|--------|---------|
| Pod | One or more containers sharing network namespace and storage; smallest deployable unit; ephemeral |
| Deployment | Declares desired state (N replicas of a pod spec); manages rolling updates and rollbacks |
| Service | Stable DNS name + ClusterIP that load-balances across all matching pods; abstracts pod churn |
| Ingress | HTTP/HTTPS routing rules; maps domain/path to a Service; handled by an Ingress Controller (Nginx, Traefik) |
| ConfigMap | Externalizes non-secret configuration; mounted as env vars or files in pods |
| Secret | Base64-encoded sensitive values; not encrypted at rest by default — use Sealed Secrets or AWS KMS CSI driver for encryption at rest |
| HPA | Horizontal Pod Autoscaler; scales pod count based on CPU/memory or custom metrics (RPS via Prometheus adapter) |
| PVC/PV | Persistent Volume Claim; decouples pod lifetime from storage lifetime; used for stateful workloads |

**HPA in practice**
- Metrics sources: metrics-server (CPU/memory), KEDA (event-driven — scale on Kafka lag, SQS depth)
- Configuration: `minReplicas: 2, maxReplicas: 10, targetCPUUtilizationPercentage: 60`
- Interview answer: "Under peak load our API pods scale from 3 to 10 replicas automatically; we set target CPU at 60% to leave headroom for traffic spikes"

**In interviews:** "We containerize each service, define a Deployment with 3 replicas for HA, expose via Service for internal discovery, route external traffic through Ingress with TLS termination, and HPA scales to 10 replicas under peak load based on RPS metric from Prometheus"

### Service Mesh (Istio / Linkerd)

**What it is:** An infrastructure layer for service-to-service communication. A sidecar proxy (Envoy in Istio, linkerd-proxy in Linkerd) is injected alongside every pod. It intercepts all inbound and outbound traffic transparently.

**Capabilities unlocked without code changes:**
- **mTLS:** mutual TLS between all services — zero-trust networking; both sides authenticate; encrypts all east-west traffic
- **Circuit breaking:** automatically stops traffic to a failing downstream
- **Retry logic:** configurable per-route retry policies with timeout
- **Load balancing:** least-connections, round-robin at layer 7
- **Distributed tracing:** automatically instruments requests; exports spans to Jaeger or Zipkin
- **Traffic shaping:** canary deployments via weighted routing (10% to v2, 90% to v1) using VirtualService CRDs in Istio

**Trade-offs**
- Adds sidecar CPU/memory overhead (~50MB RAM per pod, ~2ms added latency)
- Increases operational complexity; requires Istio control plane (istiod) HA
- For most teams: Linkerd is simpler and lighter; Istio is more powerful but complex

**Interview framing:** "For inter-service security and observability we layer Istio on top of K8s. It gives us mTLS between all services and we get distributed traces in Jaeger automatically — zero changes to application code."

### Deployment Strategies

**Rolling Update (K8s default)**
- Replace pods one by one; at any point N-1 old pods + 1 new pod running simultaneously
- Zero downtime by default
- Limitation: slow rollback (must roll forward again); both old and new versions serve traffic simultaneously (requires backward compatibility)
- Use for: routine updates where old and new are API-compatible

**Blue-Green**
- Maintain two identical environments: Blue (live) and Green (staging the new version)
- Smoke test Green; then atomically switch load balancer from Blue to Green
- Instant rollback: flip the LB back to Blue
- Cost: 2x infrastructure during the transition window
- Use for: major releases, DB schema changes requiring atomic cutover, zero-risk requirement

**Canary**
- Route a small percentage of traffic (1-5%) to the new version; ramp up gradually (1% → 10% → 50% → 100%)
- Monitor error rate, latency, and business metrics at each stage
- Auto-rollback if error rate spikes above threshold (Flagger + Prometheus can automate this)
- Blast radius limited to X% of users during rollout
- Use for: high-risk releases where you want real production validation with limited exposure

**Feature Flags**
- Deploy code dark (feature disabled for all users); enable via config at runtime (LaunchDarkly, GrowthBook, Unleash)
- Decouple deployment from release: code ships weeks before feature is turned on
- Enable for internal users → beta users → % of users → 100%
- Instant disable without redeploy if issues found
- Use for: A/B testing, gradual rollouts, trunk-based development

**Shadow / Mirror Deployment**
- Duplicate production traffic to new version without serving the response to users
- Validate correctness and performance without any user impact
- Use before canary for high-risk services (payment processing, critical path)

**Choosing the right strategy:**

| Scenario | Strategy |
|----------|----------|
| Routine service update | Rolling |
| Major release, instant rollback required | Blue-Green |
| High-risk change, validate with real traffic | Canary |
| Decouple deploy from release | Feature Flag |
| Validate without user impact at all | Shadow |

### DNS & Anycast Routing

**Anycast**
- Same IP address announced from multiple Points of Presence (PoPs) via BGP
- BGP routing protocol directs each request to the topologically nearest PoP automatically
- Used by: Cloudflare (1.1.1.1), Google DNS (8.8.8.8), all major CDNs, DDoS scrubbing
- Benefit: sub-millisecond routing optimization, inherent DDoS resilience (attack traffic distributed across all PoPs)

**GeoDNS**
- DNS server returns different A records based on the geographic location of the DNS resolver
- US users get US IP → EU users get EU IP → APAC users get APAC IP
- Used for: routing users to nearest data center, compliance data residency requirements

**DNS TTL trade-offs**
- Low TTL (60s): enables fast failover but dramatically increases DNS query volume and resolver load
- High TTL (3600s): reduces DNS load and improves DNS cache hit rate, but failover takes up to TTL duration
- Standard practice: keep normal TTL high (300-3600s); lower TTL to 60s before a planned migration or failover test

### Chaos Engineering

**Principle:** Proactively inject failure in a controlled, hypothesis-driven way to discover weaknesses before they cause real incidents.

**Netflix Chaos Monkey:** Randomly terminates EC2 instances in production during business hours. Validates that auto-recovery, health checks, and failover work as expected. Forces engineers to build resilient systems.

**What to inject:**
- Network latency (add 200ms to all calls to dependency X)
- Packet loss (drop 5% of packets between service A and B)
- CPU stress (pin CPU at 90% on N instances)
- Node failure (terminate K8s node or EC2 instance)
- Dependency unavailability (block access to Redis, downstream API)
- Disk full (fill /var/log to capacity)
- DNS resolution failure (block DNS for a specific domain)

**Chaos Engineering Process**
1. Define steady state: measurable baseline (e.g., p99 latency < 200ms, error rate < 0.1%)
2. State hypothesis: "The system will maintain steady state when 10% of packets are dropped between API and DB"
3. Define blast radius: limit experiment to staging first, then 5% of production PoP
4. Run experiment; collect metrics
5. Compare to steady state: hypothesis confirmed or refuted
6. Fix weaknesses found; add monitoring; repeat

**Game Days:** Scheduled chaos experiments with full incident response team on standby. Combine tabletop (talk through scenarios) with live injection. Build organizational muscle memory for real incidents.

**Tools:**
- Gremlin: commercial chaos platform with fine-grained control
- Chaos Mesh: open-source, K8s-native
- AWS Fault Injection Simulator (FIS): injects failures at AWS infrastructure level
- Litmus: open-source K8s chaos engineering

### Reverse Proxy (Nginx)

**Capabilities:** SSL/TLS termination, static file serving, gzip compression, request buffering, rate limiting (limit_req_zone), upstream load balancing, connection pooling.

**Nginx vs API Gateway**
- Nginx: transport-layer aware, fast, simple; routes by host/path; no awareness of business logic
- API Gateway (Kong, AWS API GW): application-aware; handles auth (JWT validation), rate limiting per user, request transformation, response caching, API versioning, developer portal
- Use both: Nginx as edge (SSL termination, static assets) + API gateway behind it for service routing and auth

## STAR Interview Framework

**Situation:** "We needed to deploy a non-backward-compatible database schema change — adding a NOT NULL column — to a 50-million-row table during business hours with zero downtime and a 2-minute rollback SLO."

**Task:** "Design a deployment strategy that allows shipping the schema change safely without taking the system down or causing data inconsistency during the migration."

**Action:**
1. Added the column as NULLABLE with a DEFAULT value — backward compatible; existing application code still works; deployed old app version confirms no errors
2. Started a batched backfill job: `UPDATE table SET col = default WHERE col IS NULL LIMIT 1000` with 100ms sleep between batches to avoid lock contention and replication lag
3. Once backfill complete, added NOT NULL constraint using `ALTER TABLE ... SET NOT NULL` — fast on PostgreSQL 12+ because data is already populated (metadata-only change)
4. Deployed new app version (which writes the new column) as a canary at 10% traffic; monitored error rate for 15 minutes
5. Blue-green switch: flipped load balancer from v1 (old app) to v2 (new app) atomically once canary validated

**Result:** Zero downtime. Canary caught one edge case at 2% traffic (null handling in one code path); fixed before full rollout. Rollback executed in 30 seconds by flipping the load balancer back to the Blue environment.

## Flashcards

| Q | A |
|---|---|
| What does K8s HPA do and what metrics can it use? | Horizontal Pod Autoscaler scales pod count based on CPU/memory (via metrics-server) or custom metrics like RPS or queue depth (via KEDA or Prometheus adapter) |
| Blue-green vs canary deployment — key difference? | Blue-green = full traffic switch atomically (instant rollback, 2x infra cost briefly); canary = gradual traffic shift (limits blast radius to X% of users, no 2x infra cost, slower rollback) |
| What does a service mesh sidecar proxy add without code changes? | mTLS between all services, circuit breaking, retry policies, distributed tracing, and traffic shaping (canary weights via VirtualService) |
| Anycast vs GeoDNS — what is the difference? | Anycast: same IP announced from multiple PoPs, BGP routes to topologically nearest (sub-ms routing); GeoDNS: different IPs returned based on resolver geography (DNS resolution time overhead) |
| What is a feature flag and why use it? | Config-driven runtime toggle to enable/disable code for specific users; decouples deployment from release; enables gradual % rollout and instant disable without redeploy |
| What must you define before a Chaos Engineering experiment? | Steady state (measurable baseline), hypothesis, blast radius limit, rollback procedure, and success/failure criteria |
| What is shadow/mirror deployment and when is it used? | Duplicate production traffic to new version without serving the response; observe behavior without user impact; used before canary for high-risk services |
