# Day 162 — Classic System Design — Twitter Feed + Instagram
**Week 24 | Phase: HLD Mastery | Month 5**

## Focus
Twitter's home timeline is one of the most instructive system design problems: it forces you to grapple with asymmetric fan-out (celebrities vs regular users), cache design at scale, and the write-vs-read trade-off. Instagram layers media storage and content discovery on top. Both problems frequently appear in senior engineer interviews as primary or secondary 45-minute problems.

## HLD (2 hours)

### Twitter Feed (Home Timeline)

**Requirements**

Functional:
- Post a tweet (text, up to 280 characters; media via separate upload)
- Follow / unfollow users
- View home timeline (tweets from all accounts you follow, reverse-chronological)
- Like, retweet, reply
- Full-text search across tweets

Non-functional:
- 200M DAU
- 500M tweets/day
- Read:write ratio approximately 100:1 (extremely read-heavy)
- Feed load latency < 200ms p99
- Eventual consistency acceptable (seeing a tweet 1-2 seconds after post is fine)

**Scale Estimation**
- Write QPS: 500M / 86,400 approximately 5,800/sec
- Read QPS: 500M x 100 / 86,400 approximately 580,000/sec (read-dominated)
- Tweet storage: 500M tweets x 280B x 365 days x 5 years approximately 250TB for text; media on S3 separately
- Timeline cache: 200 tweets x 8B (tweet ID) x 200M users = 320GB total across Redis cluster

**Feed Delivery Models**

**Fan-out on Write (Push model)**
- When a user posts a tweet: immediately push tweet ID to every follower's timeline in Redis
- Timeline is a Redis sorted set (score = tweet timestamp): `ZADD userId:timeline timestamp tweetId`
- Feed read = single `ZREVRANGE userId:timeline 0 19` — extremely fast (one Redis lookup)
- Problem with celebrities: Lady Gaga (100M followers) posting a tweet = 100M Redis writes in milliseconds — write spike, hot path, Redis backpressure

**Fan-out on Read (Pull model)**
- At feed load time: fetch all followees' tweet IDs from DB → merge-sort by timestamp → return top N
- Cheap writes (just write the tweet to the poster's own timeline)
- Problem for power users: user following 500 accounts → 500 DB queries per feed refresh — expensive, slow
- Impractical at Twitter's scale for regular reads

**Hybrid Model (Twitter's actual approach)**
- Regular users (<= 1M followers): fan-out on write — their tweets are pushed to all follower timelines in Redis
- Celebrity accounts (> 1M followers): fan-out on read — celebrity tweets are not pre-pushed; lazily injected at read time
- At feed load: serve precomputed Redis cache + perform live query for celebrity accounts the user follows + merge results by timestamp
- Threshold: Twitter uses follower count as the cutoff; accounts are tagged as celebrities when crossing the threshold

**Storage Architecture**

Tweets:
- Cassandra, partitioned by userId, clustering key = tweetId (Snowflake — time-ordered)
- Append-only write pattern; no updates; Cassandra excels at this with its LSM-tree
- Wide rows: all tweets for a user in one partition → fast user timeline reads
- No joins needed: tweet object is self-contained (author, content, timestamp, engagement counts)

Social Graph:
- MySQL (or dedicated Graph service): userId → list of followerIds and followeeIds
- Adjacency list; paginated with cursor
- For very large graphs: separate Graph DB service backed by a distributed key-value store

Timeline Cache:
- Redis sorted set per user: key = `timeline:{userId}`, score = timestamp, member = tweetId
- `ZADD timeline:{userId} timestamp tweetId` (written by fan-out worker)
- `ZREVRANGE timeline:{userId} 0 19` (read — top 20 newest tweets)
- TTL on inactive users' timelines (e.g., no activity for 30 days → evict; rebuild on next login)
- Redis Cluster with consistent hashing distributes 320GB across nodes

**Deep Dives**

Timeline pagination:
- Cursor = last tweetId seen (Snowflake encodes timestamp)
- `ZRANGEBYSCORE timeline:{userId} -inf cursorScore LIMIT 20` — stable pagination even as new tweets arrive at top

Retweet:
- Store as a reference: `{tweetId, retweeterId, originalTweetId, timestamp}`
- Fan-out the retweet reference (not a copy of the tweet content) — saves storage
- At render time: dereference originalTweetId to get content

Trending topics:
- Sliding window count per hashtag: `ZINCRBY trending:minute:{window} 1 #hashtag`
- Top-K: `ZREVRANGE trending:minute:{window} 0 9`
- Background job aggregates windows: 1min → 5min → 1hour → 1day

Search:
- Elasticsearch cluster: index each tweet on creation
- Fields: content (analyzed, inverted index), userId, createdAt (range filter), engagementScore
- Relevance = recency × engagement (likes + retweets weighted by freshness)
- Typeahead for hashtag search: prefix queries on hashtag index

### Instagram (Photo Sharing)

**Key architectural differences from Twitter:**
- Media-heavy: photos and videos dominate; text is secondary
- Stories: ephemeral 24-hour content (separate from permanent posts)
- Explore/Reels: ML-ranked discovery beyond social graph
- Feed model: same fan-out hybrid as Twitter; Instagram targets 300ms p99 for feed

**Photo Upload Flow**

```
[Client]
   1. POST /upload/init → server issues presigned S3 URL
   2. PUT directly to S3 (bypasses app servers entirely)
   3. POST /upload/complete {s3Key}
        |
[Upload Service] ──► publishes to Kafka: photo-uploaded
        |
[Lambda / Worker]
   ├── Generate 3 thumbnail sizes (320px, 640px, 1080px)
   ├── Store thumbnails in S3 with deterministic keys
   └── Update metadata DB with CDN-prefixed URLs
```

Why presigned S3 URLs: client uploads directly to S3, bypassing the app server entirely. App server never touches large binary data. Dramatically reduces app server bandwidth and cost.

CDN for photos:
- CloudFront serves all photo requests; cache-control: `immutable, max-age=31536000`
- Content-addressed filenames: `/photos/userId/photoId_v1_640.jpg` — URL changes on new version → automatic cache invalidation

**Stories (Ephemeral Content)**
- Separate Cassandra table: partition key = userId, clustering key = storyId (Snowflake)
- `TTL = 86400` seconds on each row → Cassandra automatically deletes after 24 hours during compaction
- Story feed construction: same fan-out model, TTL-aware; expired stories not served even before compaction (check creation timestamp at read time)
- Story view order: priority queue sorted by closeness score (interaction frequency between viewer and poster + recency of interaction)

**Explore / Discovery**
- Not graph-based; pure ML ranking for content not from followed accounts
- Candidate generation: user interaction history → embedding lookup → approximate nearest-neighbor search in embedding space (FAISS) to find visually/topically similar content
- Scoring: ML model predicts engagement (like probability, save probability, comment probability)
- Filtering: remove already-seen, remove low-quality content, apply safety classifiers
- Ranked top 50-100 candidates shown in Explore grid

**Notifications for Instagram**
- New follower, like, comment, mention → Notification Service (Kafka fan-out per channel)
- Batch non-urgent notifications: "5 people liked your photo" rather than one push per event
- Real-time: push via APNs (iOS) and FCM (Android)

## STAR Interview Framework

**Situation:** "We need to design Twitter's home timeline for 200M DAU with 580K read QPS. The naive fan-out-on-write approach breaks for celebrity accounts (100M followers); the naive fan-out-on-read approach is too slow for power users following thousands of accounts."

**Task:** "Design a feed system that delivers the home timeline in under 200ms p99 while handling the celebrity fan-out problem at scale."

**Action:**
1. Fan-out on write for regular users (<= 1M followers): fan-out workers push tweetIds to follower Redis sorted-set timelines as tweets are posted; feed read is a single ZREVRANGE
2. Fan-out on read for celebrity accounts: at feed read time, live-query celebrity tweets from Cassandra and merge with precomputed Redis cache sorted by timestamp
3. Cassandra for tweet storage: partition by userId, Snowflake clustering key for time-ordered retrieval at 50K writes/sec per node
4. Redis sorted sets for timeline cache: ZADD tweetId score=timestamp; TTL evicts inactive users automatically

**Result:** 200ms p99 feed load achieved. Fan-out workers handle 5,800 write QPS without celebrity write spikes blocking the pipeline. Redis TTL reduces cache footprint by 40% by evicting inactive user timelines.

## Flashcards

| Q | A |
|---|---|
| Fan-out on write vs fan-out on read — when does each fail? | Fan-out on write fails for celebrities (100M Redis writes per tweet); fan-out on read fails for users following thousands (hundreds of DB queries + merge sort per feed load) |
| What Redis data structure stores a user's home timeline and what operations are used? | Sorted set: ZADD tweetId score=timestamp for writes; ZREVRANGE 0 N for top-N newest tweets; O(log N) insert, O(log N + K) range read |
| How does Twitter handle celebrity accounts in its feed system? | Celebrity tweets are not pre-pushed to follower timelines; at feed read time the system live-queries celebrity tweets and merges with the precomputed Redis sorted-set cache |
| Why Cassandra for tweet storage? | Append-only write pattern fits LSM-tree; partition by userId for fast user timeline reads; wide rows for time-series data; horizontal scaling; no joins needed; 50K writes/sec per node |
| How do Instagram stories expire after 24 hours? | Cassandra row-level TTL=86400 causes automatic deletion during compaction; story feed query also filters by creation timestamp as a secondary guard before TTL compaction runs |
| Why use presigned S3 URLs for photo upload? | Client uploads directly to S3 bypassing app servers; app server never handles large binary data; reduces bandwidth cost dramatically; S3 handles multi-GB file ingestion natively |
