# Day 161 — Classic System Design — URL Shortener + Notification System
**Week 24 | Phase: HLD Mastery | Month 5**

## Focus
URL shortener and notification system are two canonical interview problems that test different design axes. The shortener tests key-value storage, hashing, caching strategy, and redirect mechanics. The notification system tests async fan-out, channel abstraction, failure handling, and deduplication. Both appear frequently as warm-up or standalone 45-minute problems.

## HLD (2 hours)

### URL Shortener (bit.ly)

**Requirements**

Functional:
- POST /shorten → return shortUrl; accept optional custom alias; accept optional expiry timestamp
- GET /{code} → 302 redirect to original URL
- Analytics: track redirect count per URL

Non-functional:
- 100M new URLs per day (write)
- 10:1 read ratio → 1 billion redirects per day
- p99 redirect latency < 10ms (the core SLO)
- 99.99% availability

**Scale Estimation**
- Write QPS: 100M / 86,400 ≈ 1,200/sec
- Read QPS: 1B / 86,400 ≈ 11,600/sec
- Storage: 100M URLs/day × 500 bytes × 365 days × 5 years ≈ 90TB
- Cache sizing: 80/20 rule — 20% of URLs drive 80% of redirects; 20M URLs × 500B = 10GB → fits in Redis comfortably

**Short Code Generation**

Option A: Hash-based
- SHA256(originalUrl) → Base62 encode → take first 7 characters
- 62^7 = 3.5 trillion unique combinations
- Problem: hash collision possible (two different URLs → same first 7 chars); handle with check-and-retry (append salt and rehash on collision)

Option B: Distributed counter (preferred)
- Globally unique monotonic counter (Snowflake ID, or DB sequence via PostgreSQL SERIAL)
- Base62 encode the integer → predictable 7-character code with no collision risk
- Counter-based: easily distributed via Snowflake (no coordination needed for global uniqueness)

Base62 alphabet: `a-z A-Z 0-9` — 62 characters; URL-safe; avoids ambiguous characters (0/O, 1/l)

**Why 7 characters?** 62^6 = 56 billion, 62^7 = 3.5 trillion — comfortable margin for 100M/day × 10 years = ~365 billion URLs needed

**High-Level Architecture**

```
[Client] ──► [Load Balancer]
                  │
          ┌───────┴────────┐
          │                │
   [Shorten Service]  [Redirect Service]
          │                │
    [PostgreSQL/      [Redis Cache]
     DynamoDB]             │
                     [DynamoDB fallback]
                           │
                    [Kafka analytics]
                           │
                    [ClickHouse / Analytics DB]
```

**Write path (shorten):**
1. Check cache/DB for exact duplicate (same originalUrl already shortened)
2. If custom alias requested: check uniqueness in DB; reject if taken
3. Generate short code (Snowflake counter → Base62 encode)
4. Write to DB: `{shortCode, originalUrl, userId, createdAt, expiryTs}`
5. Populate Redis cache: `SET shortCode originalUrl EX 86400`
6. Return shortUrl

**Read path (redirect):**
1. Redirect Service receives GET /{code}
2. Check Redis cache: on hit → verify not expired → return 302 immediately
3. Cache miss: query DynamoDB → populate cache → return 302
4. Publish redirect event to Kafka asynchronously (no synchronous analytics overhead)

**DB Choice:** DynamoDB or Cassandra — simple key-value (shortCode → {originalUrl, expiryTs, userId}); no joins required; high read throughput; horizontal scaling; TTL support for expiry.

**302 vs 301 Redirect**

| | 302 Temporary | 301 Permanent |
|---|---|---|
| Browser behavior | Re-requests server every visit | Caches redirect; never re-requests |
| Server load | Higher (every redirect hits server) | Lower (cached in browser) |
| Analytics | Full tracking possible | Only first redirect tracked |
| Use when | Analytics needed (production) | Analytics not needed; purely forwarding |

For bit.ly and analytics-driven services: **always 302**.

**Deep Dives**

Custom aliases:
- Same DB table; check uniqueness with SELECT before INSERT; prioritize custom alias in lookup
- Rate limit custom alias creation per user (prevent squatting)

Expiry:
- Store `expiryTs` in DB row
- Redirect Service checks `expiryTs > NOW()` before responding; return 404 if expired
- Background Kafka Streams consumer or scheduled Lambda cleans up expired rows daily

Analytics:
- Publish redirect event to Kafka: `{shortCode, userId, timestamp, geo, userAgent}`
- Flink stream processor aggregates per-minute counts
- Persist to ClickHouse for fast OLAP queries
- Real-time counter in Redis: `INCR shortCode:redirectCount`

Rate limiting on shorten endpoint:
- Redis token bucket per userId: 100 URLs/day free tier; 10,000/day paid
- `INCR userId:date` with TTL set to end of day; reject when count exceeds limit

### Notification System

**Requirements**

Functional:
- Send email, SMS, and push notifications
- User can set channel preferences (enable/disable each channel) and opt out per category
- Template-based messages with variable substitution
- Schedule future notifications (e.g., send in 24 hours)
- At-least-once delivery guarantee

Non-functional:
- 10M notifications per day
- Real-time notifications delivered in < 5 seconds
- Deduplication required (retry-safe)

**Architecture**

```
[Business Services]
        │  (HTTP)
        ▼
[Notification API Service]
        │  validates request
        │  fetches user preferences (Redis)
        │  filters opted-out channels
        │
        ├──► [Kafka: email-topic]   ──► [Email Worker (SendGrid)]
        ├──► [Kafka: sms-topic]     ──► [SMS Worker (Twilio)]
        └──► [Kafka: push-topic]    ──► [Push Worker (FCM / APNs)]
```

**Key Components**

Notification Service:
- Validates inbound request (required fields, valid templateId, valid userId)
- Fetches user preferences from Redis cache (userId → {enabledChannels, optOutCategories})
- Filters out channels user has disabled or categories user has opted out of
- Renders template (substitutes {{variables}} from request payload)
- Publishes to per-channel Kafka topic with idempotency key in message header

User Preference Service:
- Source of truth in PostgreSQL; cached in Redis with 5-minute TTL
- Cache invalidation on user preference update (publish invalidation event)

Template Service:
- Store templates in DB: `{templateId, channel, subject, bodyTemplate, version}`
- Variable substitution: `Hello {{firstName}}, your order {{orderId}} has shipped`
- Render at notification dispatch time (not template definition time)

Channel Workers:
- Each worker group consumes from its Kafka topic (email-worker-group, sms-worker-group, push-worker-group)
- Calls third-party provider API (SendGrid, Twilio, FCM)
- Retry on failure: exponential backoff (1s → 2s → 4s → 8s → 16s); max 5 retries
- After max retries: publish to Dead Letter Queue (DLQ) topic
- DLQ processor: trigger PagerDuty alert if DLQ depth > 0; allow manual reprocess or discard

**Scheduling**
- Store scheduled notifications in DB: `{notificationId, payload, scheduledAt, status: PENDING}`
- Poller service: `SELECT * WHERE scheduledAt <= NOW() AND status = PENDING ORDER BY scheduledAt LIMIT 100`; runs every minute
- On retrieval: mark status = PROCESSING (optimistic lock with version); publish to Kafka; mark SENT on ack
- Alternative: Kafka with delay queue library (Kafka Timer Service or Redis sorted set as delay queue: score = scheduledAt timestamp)

**Deduplication**
- Idempotency key = `hash(userId + templateId + correlationId)`
- Store in Redis with TTL equal to dedup window (1 hour default)
- Worker checks key before calling provider: `SET idempotencyKey "sent" NX EX 3600`
- If key already exists → skip send, log as duplicate, ack Kafka message
- Set key immediately after successful send (not before — avoids false dedups on provider failure)

**Handling Provider Failures**
- Circuit breaker per provider: if SendGrid returns 5xx for 10 consecutive calls → open circuit → fail fast for 30s → half-open probe
- Fallback: if primary provider circuit open → route to backup provider (SendGrid → AWS SES)
- Priority queue: high-priority notifications (transactional) processed before marketing; separate Kafka topics with separate consumer groups

## STAR Interview Framework

**Situation:** "The product team needed a multi-channel notification system supporting email, SMS, and push for a fintech app. Users were receiving duplicate payment notifications due to Kafka consumer group rebalances during retries. Push notification delivery was also unreliable with no visibility into failures."

**Task:** "Design a notification system supporting 10M notifications per day with at-least-once delivery, deduplication, per-user channel preferences, and observable failure handling."

**Action:**
1. Modeled each channel as a separate Kafka topic with dedicated consumer worker groups — channel isolation prevents SMS backlog from blocking email delivery
2. Added idempotency key (hash of userId+templateId+correlationId) stored in Redis with 1-hour TTL; workers check before calling provider — eliminated all duplicates from rebalance retries
3. Added circuit breaker per provider with automatic fallback (SendGrid → AWS SES) — resolved unreliable push delivery
4. DLQ with PagerDuty alerting on depth > 0 gives full visibility into failures

**Result:** Sub-5-second delivery for real-time notifications. Zero duplicates in production since idempotency key rollout. DLQ catches approximately 0.01% failure rate from provider issues — all resolved within the hour via manual reprocess tooling.

## Flashcards

| Q | A |
|---|---|
| Why use Base62 for short code generation vs UUID? | Base62 produces shorter, URL-safe codes (7 chars = 3.5T combos vs UUID's 36 chars); URL-safe alphabet (no +, /, =); human-readable and typeable |
| 302 vs 301 in URL shortener — which to choose and why? | 302 (Temporary Redirect) for analytics — browser re-requests server every time, enabling full click tracking; 301 (Permanent) only if tracking is not needed and you want to minimize server load |
| How do you handle URL shortener at 10B redirects/day? | Redis cache (10GB covers 20M hot URLs = 80% of traffic); geographically distributed Redis clusters per region; read replicas for DynamoDB; CDN in front for the most popular shortcodes |
| How do you prevent duplicate notifications? | Idempotency key = hash(userId+templateId+correlationId) stored in Redis with TTL; worker atomically checks-and-sets before calling provider; skip send if key already exists |
| Why Kafka for notification fanout vs direct HTTP calls to providers? | Decouples producers from channel workers; retry and DLQ on failure without producer awareness; backpressure handling; adding a new channel = new consumer group, zero change to producer |
| How do you schedule a notification to send in 24 hours? | Store in DB with scheduledAt timestamp; minute-poller queries PENDING rows where scheduledAt <= NOW(); publish to Kafka on retrieval; OR use Redis sorted set as delay queue with score = scheduledAt epoch |
