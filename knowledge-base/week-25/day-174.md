## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: DP on Strings (Parsing + Sequence)
- Core idea: Define dp[i] as the number of ways (or best result) for processing s[0..i-1]. For each index i, look back over all valid substrings ending at i, parse them, and apply the constraint (e.g., numeric range [1,k]). Prune aggressively: stop on leading zeros or values exceeding the bound.
- Trigger condition: A string must be partitioned into segments satisfying numeric or lexicographic constraints, and the number of ways or optimal partition is required.
- Time complexity: O(n * log k) for LC 1416 (at most log10(k) characters per valid segment) | Space complexity: O(n)

### Weak Area Template (write from memory before solving):
```java
// String parsing DP (LC 1416 skeleton)
// dp[i] = number of ways to form valid array using s[0..i-1]
// Iterate i from 1 to n; try all j from i-1 down to max(0, i-10):
//   substring s.substring(j, i); if leading zero → break; parse to long; if > k → break; else dp[i] += dp[j]
// Return dp[n] % MOD

// Longest String Chain (LC 1048 skeleton)
Map<String, Integer> dp = new HashMap<>();
Arrays.sort(words, Comparator.comparingInt(String::length));
int result = 0;
for (String word : words) {
    int best = 0;
    for (int i = 0; i < word.length(); i++) {
        String pred = word.substring(0,i) + word.substring(i+1);
        best = Math.max(best, dp.getOrDefault(pred, 0) + 1);
    }
    dp.put(word, best);
    result = Math.max(result, best);
}
```

public int longestStrChain(String[] words) {
    Map<String, Integer> dp = new HashMap<>();
    Arrays.sort(words, Comparator.comparingInt(String::length));
    int result = 0;
    for (String word : words) {
        int best = 0;
        for (int i = 0; i < word.length(); i++) {
            String pred = word.substring(0, i) + word.substring(i + 1);
            best = Math.max(best, dp.getOrDefault(pred, 0) + 1);
        }
        dp.put(word, best);
        result = Math.max(result, best);
    }
    return result;
}

## System Design (30 min — End-to-End Mock)

### Topic: Amazon E-commerce — End-to-End Design

**Scale:** 300M products, 1.5M orders/day peak (Prime Day: ×10), 300M customers

**Core Services:**

1. **Product Catalog:** PostgreSQL for structured data; Elasticsearch for full-text search with faceted filters (category, price, rating, brand); DynamoDB for real-time inventory count
2. **Search:** query → Elasticsearch (relevance + filters) → ML ranking (CTR model, user personalization) → display results
3. **Cart:** Redis per-user cart (user:{id}:cart hash) with TTL; persisted to DynamoDB on checkout
4. **Checkout & Inventory:** atomic inventory deduction:
   ```sql
   UPDATE Products SET inventory = inventory - quantity
   WHERE product_id = ? AND inventory >= quantity
   ```
   If 0 rows affected → out of stock; retry or show error
5. **Order Pipeline (Saga pattern):**
   ORDER_PLACED → (Kafka) → Payment Service (charge card) → (Kafka) → Inventory Service (reserve) → (Kafka) → Fulfillment Service (pick/pack) → SHIPPED
   Compensating transactions: if Payment fails → cancel order; if Inventory fails → refund payment
6. **Recommendation:** collaborative filtering + "customers also bought" graph; batch computed nightly in Spark; served from DynamoDB with TTL refresh

**Prime Day strategy:** pre-warm caches, increase read replicas, circuit breakers on all downstream services, graceful degradation (show cached results if Elasticsearch is slow)

**Interview talking point:** "If asked how Amazon prevents selling items that are out of stock during a flash sale with 100K concurrent buyers, answer: atomic SQL UPDATE with a WHERE inventory >= quantity guard — only one transaction per unit of inventory succeeds; the database row-level lock serializes concurrent updates. For extreme scale (1M/sec), use Redis DECRBY with a floor check (WATCH + MULTI/EXEC) as a pre-gate before hitting the database."

## Flashcards

| Q | A |
|---|---|
| How do you set up the DP recurrence for LC 1416 Restore The Array? | dp[i] = ways to restore s[0..i-1]. For each i, try all substrings s[j..i-1] parsing to a number in [1,k]; if leading zero or value > k, stop. dp[i] += dp[j] for each valid j. Base: dp[0]=1. Answer: dp[n] mod 1e9+7. |
| In Longest String Chain, why must you sort by word length before running DP? | A predecessor of a word is always shorter by exactly 1 character. Sorting by length guarantees that when you compute dp[word], all potential predecessors have already been computed — DP processes shorter words before longer ones. |
| How does the 2D DP for Count Square Submatrices work? | dp[i][j] = size of the largest all-ones square with bottom-right corner at (i,j). If matrix[i][j]=1: dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1. The answer += dp[i][j] because a square of size k contributes k² — 1 counts to the total (dp accumulates this automatically). |
| What is the Saga pattern and when does Amazon use it? | Saga = a sequence of local transactions, each publishing an event for the next step; on failure, compensating transactions undo previous steps. Amazon uses it for order processing: Place → Charge → Reserve → Fulfill; if Reserve fails, a compensating event triggers a Refund. |
| How does Amazon's search handle 300M products with sub-100ms latency? | Elasticsearch clusters with sharded inverted indexes; product data is indexed by category/brand/attributes for fast filtering; search first retrieves candidate set (recall phase via BM25), then ML ranking model re-ranks top candidates (precision phase) — two-stage retrieval in <100ms total. |
