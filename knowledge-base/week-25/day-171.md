## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Binary Search on Partition
- Core idea: Instead of searching for a value directly, binary search on the partition index of the shorter array. Adjust lo/hi by comparing cross-partition maximums and minimums to ensure the partition invariant holds.
- Trigger condition: Finding a median or k-th element across two sorted arrays where merging both is too expensive (O(m+n)).
- Time complexity: O(log(min(m,n))) | Space complexity: O(1)

### Weak Area Template (write from memory before solving):
```java
// Binary search on partition (LC 4 skeleton)
// TODO: findMedianSortedArrays(int[] nums1, int[] nums2)
// Key: ensure nums1 is shorter; binary search on nums1 partition [0..m]
//   partitionX = (lo + hi) / 2
//   partitionY = (m + n + 1) / 2 - partitionX
//   maxLeftX = (partitionX==0) ? INT_MIN : nums1[partitionX-1]
//   minRightX = (partitionX==m) ? INT_MAX : nums1[partitionX]
//   (same for Y)
//   if maxLeftX <= minRightY && maxLeftY <= minRightX → found partition
//   else if maxLeftX > minRightY → hi = partitionX - 1
//   else lo = partitionX + 1
```

### Code Skeleton
```java
// LC 4 — Median of Two Sorted Arrays — cold solve (30-min timer — narrate before coding)
// TODO: double findMedianSortedArrays(int[] nums1, int[] nums2)
// Key: ensure nums1 is shorter; binary search partitionX in [0,m];
//      partitionY = (m+n+1)/2 - partitionX;
//      invariant: maxLeftX <= minRightY && maxLeftY <= minRightX

// LC 33 — Search in Rotated Sorted Array — revision (write from memory)
public int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return mid;
        if (nums[lo] <= nums[mid]) { // left half sorted
            if (nums[lo] <= target && target < nums[mid]) hi = mid - 1;
            else lo = mid + 1;
        } else { // right half sorted
            if (nums[mid] < target && target <= nums[hi]) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return -1;
}
```

## System Design (30 min — End-to-End Mock)

### Topic: Instagram — End-to-End Design

**Scale:** 1B users, 100M daily active, 100M posts/day, 500M stories/day
**Storage:** 10M photos/day × 500KB avg = 5TB/day; store in S3, serve via CDN

**Core Services:**

1. **Upload Service:** chunked upload → S3; enqueue thumbnail generation job → SQS → Worker
2. **Post Service:** PostgreSQL for metadata (post_id, user_id, s3_url, caption, created_at, like_count)
3. **Feed Service:** hybrid fan-out
   - Push (write): for users with <1M followers — write to follower feed tables on post creation
   - Pull (read): for celebrities (>1M followers) — query their latest posts at read time
   - Feed table: (user_id, post_id, created_at) — index on (user_id, created_at DESC)
4. **Story Service:** stories expire at 24h; TTL-based expiry in Cassandra (partition by user_id, cluster by created_at)
5. **Explore/Discovery:** collaborative filtering + CNN embeddings of images → ANN search

**Key numbers:**
- Feed read: 99% served from Redis cache (pre-computed feed for active users)
- CDN offloads: 95%+ of photo serving
- Stories write: Cassandra handles ~500M writes/day ≈ 5,800 writes/sec

**Interview talking point:** "If asked how Instagram serves 100M photos per day without crushing origin servers, answer: photos are stored once in S3 then distributed to 100+ CDN edge nodes globally. When a user in Tokyo views a photo, they get it from a Tokyo CDN POP — S3 is only hit on the first cache miss per region. CDN cache hit rate exceeds 95%, meaning origin handles only 5M requests/day instead of 100M."
