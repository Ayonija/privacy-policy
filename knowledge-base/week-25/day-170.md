## DSA (1.5 hours — OA Contest Mode)

### Pattern: Patience Sort / Binary Search on Tails (LIS/LNS Variants)
- Core idea: Maintain a `tails[]` array where `tails[i]` is the smallest possible tail element of any non-decreasing (or strictly increasing) subsequence of length `i+1`. For each new element, binary search for where it fits in `tails[]` to find the answer length and update the array. The length of the longest valid subsequence at position i equals the insertion index + 1.
- Trigger condition: "Longest non-decreasing/increasing subsequence ending at or including each element" or "obstacle course" style problems where you need per-element LIS length — patience sort gives O(N log N).
- Time complexity: O(N log N) | Space complexity: O(N)

### Contest Template (write from memory before solving):
```java
// Patience sort: tails[i] = smallest tail of any subsequence of length i+1
// For NON-DECREASING (allow equal): use upper_bound (first element > x)
// For STRICTLY INCREASING: use lower_bound (first element >= x)

int[] tails = new int[n];
int len = 0; // current length of tails array

for (int x : nums) {
    // NON-DECREASING: find first index where tails[idx] > x
    int lo = 0, hi = len;
    while (lo < hi) {
        int mid = (lo + hi) / 2;
        if (tails[mid] <= x) lo = mid + 1; // allow equal → move right
        else hi = mid;
    }
    // lo = insertion position = length of LNS ending with x
    tails[lo] = x;          // replace with x (maintains smallest possible tail)
    if (lo == len) len++;   // extended the longest subsequence
    result[i] = lo + 1;     // length of longest non-decreasing subseq ending at i
}
```

| 1 | Find the Longest Valid Obstacle Course at Each Position | 1964 | Hard (Cold Solve) | Patience sort (non-decreasing LIS per position) | Maintain tails[]; for each obstacle[i], binary search for first tails[j] > obstacle[i] (upper_bound, allows equal → non-decreasing); replace tails[j]=obstacle[i]; result[i] = j+1 |
| 2 | Insert Interval | 57 | Medium (Revision) | Interval merge (linear sweep) | REVISION: Append non-overlapping intervals before new one; merge all overlapping (merged.start=min, merged.end=max); append merged; append remaining. Three-phase sweep: before, overlap, after. |
| 3 | Remove Covered Intervals | 1288 | Medium (OA Timed) | Interval coverage (sort + greedy) | OA: Set 20-min timer. Sort by start ASC then end DESC; iterate: if intervals[i].end <= maxRight, covered (skip); else maxRight = intervals[i].end, count++. |

### Code Skeleton

```java
class Solution {

    // LC 1964 — Find the Longest Valid Obstacle Course at Each Position
    // (cold solve — 30-min timer, narrate approach before opening IDE)
    // TODO: public int[] longestObstacleCourseAtEachPosition(int[] obstacles)
    // Key: patience sort with NON-DECREASING rule (equal is allowed).
    //      upper_bound search: find first tails[j] > obstacles[i].
    //      Replace tails[j] = obstacles[i]; result[i] = j + 1.
    //      Non-decreasing means lo = mid+1 when tails[mid] <= x (not just < x).
    public int[] longestObstacleCourseAtEachPosition(int[] obstacles) {
        int n = obstacles.length;
        int[] tails = new int[n];
        int[] result = new int[n];
        int len = 0;

        for (int i = 0; i < n; i++) {
            int x = obstacles[i];
            // Binary search: first index where tails[idx] > x (upper bound)
            int lo = 0, hi = len;
            while (lo < hi) {
                int mid = (lo + hi) / 2;
                if (tails[mid] <= x) lo = mid + 1; // equal is OK (non-decreasing)
                else hi = mid;
            }
            tails[lo] = x;
            if (lo == len) len++;
            result[i] = lo + 1; // length of longest non-decreasing subseq ending here
        }
        return result;
    }

    // LC 57 — Insert Interval (revision — write from memory)
    public int[][] insert(int[][] intervals, int[] newInterval) {
        List<int[]> res = new ArrayList<>();
        int i = 0, n = intervals.length;
        // Phase 1: add all intervals entirely before newInterval
        while (i < n && intervals[i][1] < newInterval[0]) res.add(intervals[i++]);
        // Phase 2: merge all overlapping intervals into newInterval
        while (i < n && intervals[i][0] <= newInterval[1]) {
            newInterval[0] = Math.min(newInterval[0], intervals[i][0]);
            newInterval[1] = Math.max(newInterval[1], intervals[i][1]);
            i++;
        }
        res.add(newInterval);
        // Phase 3: add remaining intervals
        while (i < n) res.add(intervals[i++]);
        return res.toArray(new int[0][]);
    }

    // LC 1288 — Remove Covered Intervals (OA simulation — set 20-min timer before reading)
    // TODO: public int removeCoveredIntervals(int[][] intervals)
    // Key (unlock after 20 min if stuck): sort by start ASC, end DESC;
    //     track maxRight; if intervals[i][1] <= maxRight it's covered; else count it.
    public int removeCoveredIntervals(int[][] intervals) {
        // Sort: start ascending; for same start, end descending (longer interval first)
        Arrays.sort(intervals, (a, b) -> a[0] != b[0] ? a[0] - b[0] : b[1] - a[1]);
        int count = 0, maxRight = 0;
        for (int[] iv : intervals) {
            if (iv[1] > maxRight) { // not covered
                maxRight = iv[1];
                count++;
            }
            // else iv[1] <= maxRight → covered by a previous interval; skip
        }
        return count;
    }
}
```

#### Inventory Deduction (Prevent Oversell)

```sql
-- Atomic decrement with floor check — used at order placement time
UPDATE Products
SET inventory_count = inventory_count - :quantity
WHERE product_id = :product_id
  AND inventory_count >= :quantity;
-- If 0 rows affected → out of stock → rollback order + return error to user

-- Wrapped in transaction with order creation:
BEGIN;
  -- 1. Deduct inventory (check rowcount)
  UPDATE Products SET inventory_count = inventory_count - ? WHERE product_id = ? AND inventory_count >= ?;
  -- 2. Create Order
  INSERT INTO Orders ...;
  -- 3. Create OrderItems
  INSERT INTO OrderItems ...;
  -- 4. Create Payment record (PENDING)
  INSERT INTO Payments ...;
COMMIT;
```

#### Order State Machine

```
                    ┌─────────────────────────────────────────┐
                    │          payment failure                  │
                    ▼                                           │
  PLACED ──────────────────► CONFIRMED ──► SHIPPED ──► DELIVERED
    │         payment              │
    │          success             │
    │                              ▼
    └──────────────────────► CANCELLED (before SHIPPED)
                                   │
                                   ▼
                              REFUNDED (payment reversal triggered)

Transitions:
  PLACED      → CONFIRMED  : payment gateway callback with success
  PLACED      → CANCELLED  : payment gateway callback with failure
  CONFIRMED   → SHIPPED    : warehouse picks + labels order
  SHIPPED     → DELIVERED  : delivery agent marks delivered
  CONFIRMED   → CANCELLED  : buyer cancels before shipping (refund triggered)
  DELIVERED   → REFUNDED   : buyer raises refund within window (handled separately)
```

#### Key Design Decisions

```
1. Price snapshot in OrderItems:
   unit_price stores the price AT ORDER TIME — prevents historical orders from
   showing updated prices if the seller changes the product price later.

2. Address snapshot in Orders:
   delivery_address JSON stores the address at order time — prevents address
   changes from affecting in-flight orders.

3. UNIQUE on Payments.order_id:
   Ensures exactly one payment record per order — prevents double-charging bugs.

4. UNIQUE on Reviews.order_item_id:
   Ensures one review per purchased item — prevents review flooding and ties
   review authenticity to a real purchase.

5. CartItems PRIMARY KEY (cart_id, product_id):
   Prevents duplicate cart entries; updating quantity reuses the existing row.
```

**Design pattern:** Snapshot pattern — capture price and address at transaction time to make historical records immutable and correct even when source data changes.

**Trade-off:** Denormalizing `subtotal = quantity * unit_price` in OrderItems adds redundancy but eliminates recomputation on every invoice render. The constraint `subtotal = quantity * unit_price` can be enforced at the application layer or via a check constraint.

**Interview talking point:** "If asked how Amazon prevents selling items that are out of stock during a flash sale where thousands buy simultaneously, answer: use an atomic decrement with a floor check — `UPDATE Products SET inventory_count = inventory_count - 1 WHERE product_id = ? AND inventory_count > 0`. If 0 rows affected, the item is sold out. This uses database row-level locking, so concurrent transactions serialize on the same row, and only the transactions that see inventory_count > 0 succeed."

## Flashcards

| Q | A |
|---|---|
| What is the difference between lower_bound and upper_bound in patience sort, and which does each LIS variant use? | `lower_bound` finds the first index where `tails[i] >= x` → used for STRICTLY INCREASING LIS (replace at the first equal-or-greater tail). `upper_bound` finds the first index where `tails[i] > x` → used for NON-DECREASING LNS (allow equal, so only replace when strictly exceeded). |
| In LC 57 (Insert Interval), what are the three phases of the sweep? | Phase 1: copy all intervals that end before newInterval starts (`end < newInterval.start`). Phase 2: merge all overlapping intervals into newInterval (`start <= newInterval.end`). Phase 3: copy all remaining intervals. |
| Why sort by `end DESC` for the same `start` value in LC 1288 (Remove Covered Intervals)? | So that among intervals with the same start, the longest (largest end) is processed first — it will set `maxRight` to the maximum, causing all shorter intervals with the same start to be detected as covered in the same pass. |
| What SQL idiom prevents overselling in a flash sale at the database level? | `UPDATE Products SET inventory_count = inventory_count - 1 WHERE product_id = ? AND inventory_count > 0` — if `rowsAffected == 0`, the item is sold out. DB row-level locking serializes concurrent transactions on the same product row. |
| Why store `unit_price` and `delivery_address` as snapshots in Orders/OrderItems rather than FKs to the current values? | Prices and addresses change over time. A snapshot captures the exact state at transaction time, making historical order records accurate regardless of future updates to the product price or the user's address. |
