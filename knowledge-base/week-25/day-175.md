## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Backtracking with Aggressive Pruning
- Core idea: Always fill the topmost-leftmost uncovered cell or position to canonicalize the search space. Prune immediately when the current partial solution cannot possibly beat the best solution found so far. Try larger options first (largest square, biggest number) to find a good upper bound early, which makes subsequent pruning more aggressive.
- Trigger condition: Exhaustive search over combinatorial space where the number of paths is exponential, but most can be eliminated early via a bound or constraint check.
- Time complexity: Exponential worst case, but pruning reduces practical runtime dramatically | Space complexity: O(recursion depth)

### Weak Area Template (write from memory before solving):
```java
// Backtracking with pruning template
int result = Integer.MAX_VALUE;
void backtrack(int[][] grid, int filled, int count) {
    if (count >= result) return; // pruning: can't beat current best
    // find next uncovered cell (top-left first)
    int[] cell = findEmpty(grid);
    if (cell == null) { result = Math.min(result, count); return; }
    int r = cell[0], c = cell[1];
    // try all valid square sizes (largest first for better pruning)
    for (int size = Math.min(n-r, n-c); size >= 1; size--) {
        if (canPlace(grid, r, c, size)) {
            place(grid, r, c, size, 1); // mark filled
            backtrack(grid, filled + size*size, count + 1);
            place(grid, r, c, size, 0); // unmark
        }
    }
}
```

// LC 698 — Partition to K Equal Sum Subsets — revision (write from memory)
public boolean canPartitionKSubsets(int[] nums, int k) {
    int total = 0;
    for (int n : nums) total += n;
    if (total % k != 0) return false;
    int target = total / k;
    Arrays.sort(nums); // ascending; iterate in reverse during backtrack
    if (nums[nums.length - 1] > target) return false;
    int[] buckets = new int[k];
    return backtrack(nums, buckets, nums.length - 1, target, k);
}
private boolean backtrack(int[] nums, int[] buckets, int idx, int target, int k) {
    if (idx < 0) return true;
    Set<Integer> seen = new HashSet<>();
    for (int i = 0; i < k; i++) {
        if (seen.contains(buckets[i])) continue; // skip duplicate bucket sums
        if (buckets[i] + nums[idx] <= target) {
            seen.add(buckets[i]);
            buckets[i] += nums[idx];
            if (backtrack(nums, buckets, idx - 1, target, k)) return true;
            buckets[i] -= nums[idx];
        }
    }
    return false;
}

## System Design (30 min — End-to-End Mock)

### Topic: Airbnb — End-to-End Design

**Scale:** 7M+ listings, 2M+ guests/night, 150+ countries

**Core Services:**

1. **Listing Service:** PostgreSQL for structured listing data; Elasticsearch for geo-search + filters; S3 for photos; CDN for photo delivery
2. **Search:** geo-bounding-box filter in Elasticsearch → price/amenity filters → ML ranking (reviews, price, response rate, photos quality) → return listings
   Availability filter: separate availability service queries Cassandra; Elasticsearch doesn't know about bookings in real-time (propagation delay ~2s via Kafka consumer)
3. **Availability Calendar:** Cassandra — (listing_id, date) composite key; each date has status: AVAILABLE, BLOCKED, BOOKED
4. **Booking Service:**
   - HOLD: reserve dates in Cassandra (status=HELD, expires=+30min) — prevents double booking
   - PAYMENT: charge guest via Stripe/Braintree; idempotency key = booking_id
   - CONFIRM: update status=CONFIRMED; notify host via FCM/email; charge host payout to escrow
5. **Reviews:** PostgreSQL (review_id, listing_id, guest_id, host_id, rating 1-5, text, created_at); ratings aggregated via Kafka consumer updating avg_rating on listing
6. **Dynamic Pricing:** ML model on supply/demand, seasonality, local events → price suggestion; host sets floor/ceiling

**Key numbers:**
- Availability writes: 2M bookings/night × 3 nights avg = 6M Cassandra writes/day (trivial)
- Search latency: Elasticsearch geo query <50ms; availability join <10ms from Cassandra; ML ranking <20ms; total <100ms
- Double-booking prevention: Cassandra INSERT IF NOT EXISTS on (listing_id, date, status=HELD) — CAS (Compare And Swap) guarantees at-most-one HELD record per date

**Interview talking point:** "If asked how Airbnb prevents two guests from booking the same listing for the same date, answer: Cassandra lightweight transactions — INSERT INTO availability (listing_id, date, booking_id) VALUES (?, ?, ?) IF NOT EXISTS. If the INSERT returns 'applied=false', another booking already holds that date; the second request retries with the next available date or returns 'unavailable'."

## Flashcards

| Q | A |
|---|---|
| What is the key pruning condition in backtracking for LC 1240 Tiling a Rectangle? | Prune when current square count >= best found so far (count >= result). Also: always fill the topmost-leftmost uncovered cell to canonicalize the search space and eliminate symmetrical paths. |
| In LC 698 Partition to K Equal Sum Subsets, what two pruning optimizations reduce the search space most? | (1) Sort nums descending — larger numbers constrain placement early, failing faster. (2) Skip duplicate bucket sums — if placing num in bucket A (sum=5) fails, placing it in bucket B (also sum=5) will also fail; skip all buckets with identical current sums. |
| How does LC 494 Target Sum transform into a subset sum problem? | Let P = set of + numbers, N = set of - numbers. sum(P) - sum(N) = target AND sum(P) + sum(N) = total → 2*sum(P) = total + target → sum(P) = (total+target)/2. Find number of subsets summing to (total+target)/2 using 0/1 knapsack. |
| How does Airbnb prevent double-booking using Cassandra lightweight transactions? | Cassandra CAS: INSERT INTO availability (listing_id, date, booking_id) VALUES (?, ?, ?) IF NOT EXISTS. Cassandra uses Paxos to ensure only one INSERT succeeds across all replicas; second INSERT returns applied=false and the application handles the conflict. |
| What is the difference between Airbnb's search showing 'available' and the actual availability state in Cassandra? | Propagation lag (~2 seconds) — Elasticsearch is updated via a Kafka consumer after a booking is confirmed in Cassandra. During those 2 seconds, Elasticsearch may show a listing as available that is actually just-booked. The final availability check happens at HOLD time against Cassandra (the source of truth), not Elasticsearch. |
