## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Dijkstra on Grid / Multi-Source BFS
- Core idea: Replace the standard BFS FIFO queue with a min-heap when edge weights are non-uniform. The cost to move to a neighbor is not always 1 — it can be the cell's elevation, value, or some other function. Dijkstra always processes the globally cheapest unvisited node next.
- Trigger condition: Grid traversal where the path cost is the maximum (or sum) of cell values along the route, not just the number of steps.
- Time complexity: O(m*n * log(m*n)) | Space complexity: O(m*n)

### Weak Area Template (write from memory before solving):
```java
// Dijkstra on grid template
int[][] dist = new int[m][n]; // fill with MAX_VALUE
dist[0][0] = grid[0][0];
PriorityQueue<int[]> pq = new PriorityQueue<>(Comparator.comparingInt(a -> a[0]));
pq.offer(new int[]{grid[0][0], 0, 0});
int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
while (!pq.isEmpty()) {
    int[] cur = pq.poll();
    int d = cur[0], r = cur[1], c = cur[2];
    if (d > dist[r][c]) continue;
    for (int[] dir : dirs) {
        int nr = r+dir[0], nc = c+dir[1];
        if (nr>=0 && nr<m && nc>=0 && nc<n) {
            int newDist = Math.max(d, grid[nr][nc]); // for LC 778
            if (newDist < dist[nr][nc]) { dist[nr][nc] = newDist; pq.offer(new int[]{newDist,nr,nc}); }
        }
    }
}
```

// LC 934 — Shortest Bridge — revision (write from memory)
// Step 1: DFS from any 1-cell to find island 1, mark as 2, add all cells to BFS queue
// Step 2: Multi-source BFS expands outward; first contact with another 1-cell → return distance
public int shortestBridge(int[][] grid) {
    int n = grid.length;
    Queue<int[]> bfsQueue = new LinkedList<>();
    boolean found = false;
    int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
    // DFS to mark island 1
    outer: for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            if (grid[i][j] == 1) { dfs(grid, i, j, bfsQueue, dirs); found = true; break outer; }
    // BFS to reach island 2
    int steps = 0;
    while (!bfsQueue.isEmpty()) {
        int size = bfsQueue.size();
        while (size-- > 0) {
            int[] cur = bfsQueue.poll();
            for (int[] d : dirs) {
                int nr = cur[0]+d[0], nc = cur[1]+d[1];
                if (nr>=0 && nr<n && nc>=0 && nc<n) {
                    if (grid[nr][nc] == 1) return steps;
                    if (grid[nr][nc] == 0) { grid[nr][nc] = 2; bfsQueue.offer(new int[]{nr,nc}); }
                }
            }
        }
        steps++;
    }
    return -1;
}

## System Design (30 min — End-to-End Mock)

### Topic: WhatsApp — End-to-End Design

**Scale:** 2B users, 100B messages/day ≈ 1.15M messages/second
**Storage:** 100B messages × 100 bytes avg = 10TB/day (text only; media stored in S3 separately)

**Core Services:**

1. **Connection Service:** each client maintains a persistent WebSocket connection to a Chat Server; Chat Server is assigned via consistent hashing on user_id
2. **Message Flow:**
   Sender → Chat Server A → Kafka (message queue) → Chat Server B (if recipient connected) → recipient
   If recipient offline: Chat Server B stores message in Cassandra; delivers on reconnect
3. **Message Storage:** Cassandra (user_id, conversation_id, message_id DESC) — partition by conversation for fast history load
4. **Online Presence:** Redis pub/sub; user connects → set user:{id}:online=1 EX 30; heartbeat every 15s; subscribers get notified
5. **Group Messages:** Kafka topic per group; each consumer is a Chat Server holding group members; fan-out at Chat Server level (not per-message DB write)
6. **Media:** sender uploads to S3 presigned URL directly; share S3 URL in message; thumbnail generated async

**E2EE (high level):** keys generated on device; Signal Protocol: Diffie-Hellman ratchet generates new key per message; server stores only ciphertext

**Key numbers:**
- Chat Server handles ~50K concurrent connections
- 2B users / 50K = 40,000 Chat Servers

**Interview talking point:** "If asked how WhatsApp delivers 100B messages/day with just 50 engineers, answer: the architecture is simple — WebSocket connections + Kafka message queuing + Cassandra for storage. The key is that WhatsApp uses Erlang/Elixir (via Ejabberd) which handles millions of concurrent lightweight processes per server; each WebSocket connection is a tiny process — orders of magnitude more efficient than thread-per-connection Java servers."

## Flashcards

| Q | A |
|---|---|
| How does Dijkstra apply to LC 778 Swim in Rising Water? | dist[r][c] = minimum possible max-elevation on any path from (0,0) to (r,c); min-heap processes cells in order of current path cost; update neighbor if max(curr_cost, neighbor_elevation) < dist[neighbor]. |
| In LC 934 Shortest Bridge, how do you find and connect the two islands? | Step 1: DFS from any 1-cell to flood-fill island 1 (mark as 2, add all to BFS queue). Step 2: Multi-source BFS expands in 4 directions; first time a 0-cell at distance k reaches another 1-cell, return k. |
| What is WhatsApp's strategy for delivering messages to offline users? | Message is stored in Cassandra with status PENDING; when recipient reconnects, their Chat Server queries Cassandra for undelivered messages and pushes them, then marks DELIVERED. |
| How does WhatsApp's online presence system work at 2B user scale? | Redis key user:{id}:online with 30s TTL; client sends heartbeat every 15s to refresh TTL; presence subscribers use Redis pub/sub on user-specific channels; presence is eventually consistent (up to 30s stale). |
| What makes Dijkstra on a grid different from standard BFS? | Standard BFS assumes unit edge weights — each step costs 1. Dijkstra handles non-uniform weights — in LC 778, the "cost" to enter a cell is its elevation (not 1), so a min-heap replaces the FIFO queue to always process the cheapest unvisited cell first. |
