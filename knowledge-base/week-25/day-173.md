## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Binary Lifting (Sparse Table on Trees)
- Core idea: Precompute a table where up[node][j] = the 2^j-th ancestor of node. Built bottom-up: up[node][0] = parent, and up[node][j] = up[up[node][j-1]][j-1]. To find the k-th ancestor, decompose k in binary and jump through precomputed ancestors in O(log n).
- Trigger condition: Repeated k-th ancestor queries on a tree, or LCA (Lowest Common Ancestor) where O(n) per query is too slow.
- Time complexity: O(n log n) precompute, O(log n) per query | Space complexity: O(n log n)

### Weak Area Template (write from memory before solving):
```java
// Binary Lifting precomputation
int LOG = 16; // supports up to 2^16 = 65536 depth
int[][] up = new int[n][LOG];
// Build: up[node][0] = parent[node]; for j>=1: up[node][j] = up[up[node][j-1]][j-1]
// Query getKthAncestor(node, k):
//   for j = 0 to LOG-1: if (k >> j & 1) == 1: node = up[node][j]; if node == -1 return -1
//   return node
```

// LC 1302 — Deepest Leaves Sum — revision (write from memory)
public int deepestLeavesSum(TreeNode root) {
    int sum = 0;
    Queue<TreeNode> q = new LinkedList<>();
    q.offer(root);
    while (!q.isEmpty()) {
        sum = 0;
        int size = q.size();
        while (size-- > 0) {
            TreeNode node = q.poll();
            sum += node.val;
            if (node.left != null) q.offer(node.left);
            if (node.right != null) q.offer(node.right);
        }
    }
    return sum;
}

## System Design (30 min — End-to-End Mock)

### Topic: Google Drive / Dropbox — End-to-End Design

**Scale:** 1B users (Google), 700M files uploaded/day, avg file 1MB = 700TB/day

**Core Services:**

1. **Upload Service:** client splits file into 4MB chunks; compute SHA-256 hash per chunk; upload only chunks whose hash doesn't exist on server (deduplication + delta sync); presigned S3 URL per chunk
2. **Metadata Service:** PostgreSQL — Files(file_id, owner_id, name, size, version, s3_path, created_at, updated_at), FileChunks(file_id, chunk_index, chunk_hash, s3_chunk_url)
3. **Sync Service:** long-polling or WebSocket; on file change, server pushes change notification to all clients with access; client downloads only changed chunks
4. **Sharing Service:** FilePermissions(file_id, user_id, role: VIEWER/EDITOR/OWNER); shareable link generates signed token with expiry
5. **Version History:** each save creates a new version entry; restoration copies metadata pointer to old chunk set (chunks themselves never deleted until all versions GC'd)

**Conflict Resolution:**
- Last-write-wins (simple): timestamp comparison; works for most cases
- Operational Transform (Google Docs): real-time collaborative edits transformed against concurrent edits; complex but enables true collaborative editing

**Key numbers:**
- Chunk deduplication saves ~30% storage (duplicate files/chunks across users)
- Delta sync: uploading a 100MB file with 1MB change uploads only 1 chunk (1MB) not 100MB
- CDN for frequently accessed files; S3 for long-tail

**Interview talking point:** "If asked how Dropbox handles sync when two users edit the same file simultaneously, answer: Dropbox uses a simple last-write-wins strategy with conflict copies — if user A and user B both edit file.txt offline, when they sync, one version becomes file.txt and the other becomes 'file (user B's conflicted copy).txt'. True real-time collaborative editing (like Google Docs) requires Operational Transform or CRDTs — a fundamentally different architecture."

## Flashcards

| Q | A |
|---|---|
| How do you precompute binary lifting ancestors for a tree of n nodes? | Create up[n][LOG] where LOG=ceil(log2(maxDepth)). up[node][0]=parent. For j from 1 to LOG-1: up[node][j]=up[up[node][j-1]][j-1]. Precomputation: O(n log n). Each query: O(log n) by decomposing k in binary. |
| How do you find the deepest leaves sum in a binary tree in one pass? | BFS level-order — reset sum at start of each level, accumulate all node values; return sum after processing the last level. O(n) time O(w) space where w = max width. |
| In the pseudo-palindromic paths problem, how does a bitmask detect palindrome feasibility? | XOR-toggle a bit for each digit encountered (bit i = 1 if digit i appears odd times). At a leaf, a path is pseudo-palindromic if at most one bit is set: check bitmask & (bitmask-1) == 0. |
| How does Google Drive achieve delta sync to minimize upload bandwidth? | The client splits the file into fixed-size chunks (4MB) and computes a hash per chunk. On save, only chunks whose hash has changed are uploaded. A 100MB file with a 1MB edit uploads only 1 chunk (1MB) instead of 100MB. |
| What is the trade-off between last-write-wins and Operational Transform for file conflict resolution? | LWW is simple and fast but loses concurrent edits (one user's work is overwritten). OT preserves all concurrent edits by transforming operations against each other but requires a central sequencer and is complex to implement correctly — Google Docs uses OT; Dropbox uses LWW with conflict copies. |
