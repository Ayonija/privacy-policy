## DSA (2 hours)

### Pattern: Probability DP — Forward Propagation

State: `dp[state]` = probability of being in that state after some sequence of random events.

**Forward (probability flows forward):** Distribute current probability mass to future states.
`dp[next_state] += dp[curr_state] × transition_probability`

**Backward (expected value from state):** Compute expected cost/steps to complete from a state.
`dp[curr] = cost + Σ (prob × dp[next])`

**Trigger condition:**
- "What is the probability that [random process] ends with [condition]?"
- "Expected number of steps to reach a target state?"
- State space is manageable (polynomial in problem dimensions)

**Complexity:**
- Time: O(states × transitions)
- Space: O(states) — often optimize with rolling array

### Full Solution Walkthrough — Knight Probability in Chessboard (LC 688) — Revision

**Problem:** A knight starts at (row, column) on an n×n board. After k moves, each uniformly random from 8 valid knight moves, what is the probability the knight is still on the board?

**Forward DP:** `dp[r][c]` = probability of being at (r,c) after the current number of steps. Each step, distribute each cell's probability to its 8 neighbors (divide by 8 each).

```python
def knightProbability(n, k, row, column):
    moves = [(-2,-1),(-2,1),(-1,-2),(-1,2),(1,-2),(1,2),(2,-1),(2,1)]

    # dp[r][c] = probability of being at (r,c) currently
    dp = [[0.0] * n for _ in range(n)]
    dp[row][column] = 1.0   # start with probability 1 at starting position

    for _ in range(k):
        new_dp = [[0.0] * n for _ in range(n)]
        for r in range(n):
            for c in range(n):
                if dp[r][c] == 0:
                    continue   # skip cells with no probability mass
                for dr, dc in moves:
                    nr, nc = r + dr, c + dc
                    if 0 <= nr < n and 0 <= nc < n:
                        new_dp[nr][nc] += dp[r][c] / 8.0
                # Probability that falls off-board (nr/nc out of bounds) is just discarded
        dp = new_dp

    # Sum all remaining on-board probability
    return sum(dp[r][c] for r in range(n) for c in range(n))
```

**Why forward DP is natural here:** We start knowing the exact position (probability 1). Each step spreads probability outward. Off-board probability is simply discarded (not tracked). Backward DP would compute "probability of staying on board for remaining steps from (r,c)" — equivalent but requires computing over the full grid after k steps.

**Complexity:** O(k × n² × 8) = O(k × n²). Space: O(n²) with rolling array.

### Full Solution Walkthrough — New 21 Game (LC 837) — Hard

**Problem:** Player draws cards with values 1..maxPts uniformly at random. Stops when total score ≥ k. What is the probability the final score is ≤ n?

**Key observations:**
- dp[i] = probability of reaching EXACTLY score i
- For i ∈ [1, k-1]: player draws again from this state. dp[i] = (1/maxPts) × Σ dp[j] for j ∈ [i-maxPts, i-1]
- For i ∈ [k, n]: player STOPPED here (score ≥ k on previous draw, or started here). These are terminal: dp[i] = sum of transitions from [i-maxPts, i-1] ∩ [0, k-1] / maxPts
- Answer = Σ dp[i] for i ∈ [k, n]

**Sliding window optimization:** The sum `Σ dp[j] for j ∈ [i-maxPts, i-1]` is a fixed-size window of size maxPts. Maintain with O(1) updates.

```python
def new21Game(n, k, maxPts):
    # Edge cases:
    if k == 0:
        return 1.0    # already stopped at score 0, which is <= n (n >= 0)
    if n >= k + maxPts - 1:
        return 1.0    # any achievable score is <= n

    dp = [0.0] * (n + 1)
    dp[0] = 1.0         # start at score 0 with probability 1
    window_sum = 1.0    # sum of dp[max(0, i-maxPts) .. i-1], starts as dp[0] for i=1
    result = 0.0

    for i in range(1, n + 1):
        dp[i] = window_sum / maxPts   # probability of reaching score i

        if i < k:
            # Player continues drawing from score i → include dp[i] in window for future
            window_sum += dp[i]
        else:
            # Player stopped at score i (score >= k) → count toward result
            result += dp[i]

        # Slide window: remove dp[i - maxPts] which is now out of window range
        if i >= maxPts:
            window_sum -= dp[i - maxPts]

    return result
```

**Example trace for n=21, k=17, maxPts=10:**
- dp[0]=1.0, window_sum=1.0
- dp[1] = 1.0/10 = 0.1; i<k so window_sum += 0.1 = 1.1
- dp[2] = 1.1/10 = 0.11; window_sum = 1.21
- ...
- dp[10] = window_sum/10; when i=10>=maxPts=10: window_sum -= dp[0]=1.0
- dp[17]: i>=k=17, so result += dp[17]
- ...result sums dp[17..21]

**Window invariant:** Before computing dp[i], `window_sum` = Σ dp[j] for j in [max(0, i-maxPts), i-1]. This is exactly the set of scores from which a single draw can reach score i.

### Full Solution Walkthrough — Probability of Two Boxes (LC 1467) — Hard

**Problem:** There are n balls of each of k colors (given by `balls[i]`). Split ALL balls into two boxes of equal size (n = sum(balls)//2 each). What is the probability that both boxes have the same number of distinct colors?

**Approach:** Use memoized DFS over colors. For each color, decide how many go to box 1 (rest go to box 2). Count arrangements where both boxes have equal distinct colors, divided by total arrangements.

```python
def getProbability(balls):
    from math import comb
    from functools import lru_cache

    n = sum(balls) // 2   # target balls per box
    k = len(balls)

    @lru_cache(maxsize=None)
    def dp(idx, left_balls, left_colors, right_balls, right_colors):
        """
        Returns: weighted count of valid complete assignments from color idx onward.
        idx: current color index (0..k-1)
        left_balls: balls placed in box 1 so far
        left_colors: distinct colors in box 1 so far
        right_balls: balls placed in box 2 so far
        right_colors: distinct colors in box 2 so far
        """
        if idx == k:
            # All colors assigned; check if both boxes have equal distinct colors
            return 1 if left_colors == right_colors else 0
        result = 0
        for put_left in range(balls[idx] + 1):
            put_right = balls[idx] - put_left
            # Capacity check
            if left_balls + put_left > n or right_balls + put_right > n:
                continue
            # Number of ways to split balls[idx] into put_left and put_right
            ways = comb(balls[idx], put_left)
            new_lc = left_colors + (1 if put_left > 0 else 0)
            new_rc = right_colors + (1 if put_right > 0 else 0)
            result += ways * dp(
                idx + 1,
                left_balls + put_left, new_lc,
                right_balls + put_right, new_rc
            )
        return result

    # Total arrangements = (2n)! / product(balls[i]!) but we use combinatorial formula
    # Total = comb(2n, n) × ways to arrange within each box
    # We compute numerator (valid) / denominator (total) using the same counting basis
    from math import factorial
    total_ways = comb(sum(balls), n)   # ways to choose which n balls go to box 1
    # But we need to account for indistinguishable balls of same color
    # numerator from dp already uses comb(balls[i], put_left) which handles this
    # Total denominator = product over all possible splits = comb(sum, n) / product(1) = multinomial
    # Actually: total = comb(2n, n) when balls are distinguishable, but here same-color balls
    # are indistinguishable, so total = comb(2n,n) / prod(comb(balls[i], k_i) summed) 
    # Simpler: just compute dp(0,0,0,0,0) for valid and dp without color check for total

    def dp_total(idx, left_balls, right_balls):
        if idx == k:
            return 1
        result = 0
        for put_left in range(balls[idx] + 1):
            put_right = balls[idx] - put_left
            if left_balls + put_left > n or right_balls + put_right > n:
                continue
            result += comb(balls[idx], put_left) * dp_total(idx + 1, left_balls + put_left, right_balls + put_right)
        return result

    numerator = dp(0, 0, 0, 0, 0)
    denominator = dp_total(0, 0, 0)
    return numerator / denominator
```

**State space:** idx × left_balls × left_colors × right_balls × right_colors = k × n × k × n × k. With k≤8, n≤6×8=48: manageable.

## System Design (30 min)

### Topic: LRU + LFU Integration — Choosing Between Them in Production

**Full comparison:**

| Dimension | LRU | LFU | ARC |
|-----------|-----|-----|-----|
| Implementation | HashMap + DLL | 3 HashMaps | 4 lists + 2 ghost lists |
| Memory overhead | Low | Medium (3 maps) | High (4 lists) |
| New item eviction risk | Low (new items protected briefly) | High (freq=1, evicted first) | Low (T1 protects new) |
| Scan resistance | Poor (large scan evicts hot items) | Good (hot items have high freq) | Good (adaptive) |
| Optimal for | Temporal locality, recency | Stable frequency distribution | Unknown/mixed workloads |

**Redis approximated LRU:**
Redis doesn't maintain a full LRU list (too expensive at scale). Instead:
- On eviction: sample `maxmemory-samples` keys (default=5) at random
- Evict the one with the oldest `lru_clock` timestamp
- With 10 samples, approximates true LRU within ~1-2% of optimal eviction quality

```
maxmemory-policy = allkeys-lru    # evict any key
maxmemory-policy = volatile-lru   # only evict keys with TTL set
maxmemory-policy = allkeys-lfu    # Redis 4.0+: use approximated LFU
```

**When to use LFU in Redis (allkeys-lfu):** When your access pattern is Pareto-distributed (few hot items, many cold items that are accessed once or rarely). LFU protects the hot 1% from being evicted by the cold 99%.

**When to stick with LRU:** When workloads are bursty — a newly viral post should stay cached even if its historical frequency is low. LRU keeps it based on recency; LFU might evict it if it hasn't built up frequency yet.

**Interview talking point:**
*"For a user feed cache where content goes viral unpredictably, I'd use LRU — recent posts should stay regardless of historical frequency. For a product catalog where the top 1000 SKUs consistently drive 80% of views, LFU is better. Redis 4.0+ supports allkeys-lfu. In practice I'd benchmark with your actual access logs — replay them against both policies and compare hit rates."*
