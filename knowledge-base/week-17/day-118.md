## DSA (2 hours)

### Pattern: Bitmask for Character/Digit Set Membership + XOR Parity

**Core technique 1 — Character bitmask:**
Represent which characters (0-25 for a-z, 0-9 for digits) appear in a string as a 26-bit or 10-bit integer. Two strings share no character iff `mask1 & mask2 == 0`.

**Core technique 2 — XOR parity prefix:**
Track whether each character's count is odd or even using a rolling bitmask. `mask ^= (1 << char_index)` flips the parity bit. A substring [l+1..r] is palindrome-formable iff its parity mask is 0 (all even) or a power of 2 (exactly one odd). Parity mask of substring = `prefix[r] ⊕ prefix[l]`.

**Trigger condition:**
- "Do two strings share any character?" → bitmask AND test
- "Find longest palindrome-formable substring" → XOR prefix parity + hashmap
- "Maximum XOR pair with value constraint" → sliding window Trie

**Complexity:**
- Character bitmask precompute: O(n × |word|), test O(n²)
- Parity prefix approach: O(n × 10) for checking all single-bit masks per position

### Full Solution Walkthrough — Maximum Product of Word Lengths (LC 318) — Revision

**Problem:** Find the maximum value of `len(words[i]) * len(words[j])` where the two words share no common letters.

**Key insight:** Precompute a 26-bit mask for each word. Two words share no letters iff `mask[i] & mask[j] == 0` (no bit is set in both).

```python
def maxProduct(words):
    n = len(words)
    masks = [0] * n

    for i, word in enumerate(words):
        for c in word:
            masks[i] |= (1 << (ord(c) - ord('a')))
    # masks[i] has bit k set iff letter chr(ord('a')+k) appears in words[i]

    result = 0
    for i in range(n):
        for j in range(i + 1, n):
            if masks[i] & masks[j] == 0:   # no shared letters
                result = max(result, len(words[i]) * len(words[j]))

    return result
```

**Trace for words=["abcw","baz","foo","bar","xtfn","abcdef"]:**
- mask["abcw"] = bits for a,b,c,w = 0b10000000000000000000000111
- mask["xtfn"] = bits for x,t,f,n = ...
- "abcw" & "xtfn" = 0 (no shared) → product = 4×4 = 16 ✓

**Why this is O(n²):** Precompute masks in O(n×|word|), then all pairs in O(n²). No Trie needed — just AND-test is sufficient.

### Full Solution Walkthrough — Find Longest Awesome Substring (LC 1542) — Hard

**Problem:** Find the longest substring of a digit string (0-9) that can be rearranged into a palindrome.

**Palindrome condition:** A string can form a palindrome iff at most one character has an odd count.

**XOR prefix approach:** Maintain a rolling bitmask `mask` where bit d is 1 if digit d has appeared an odd number of times so far.
- Substring `(l, r]` has parity mask = `prefix[r] ⊕ prefix[l]`
- Valid substring iff this XOR is 0 (all even) OR has exactly one bit set (one odd)

```python
def longestAwesome(s):
    # first_seen[mask] = earliest index where this parity mask was first seen
    # We want the EARLIEST occurrence to maximize substring length
    first_seen = {0: -1}   # empty prefix has mask=0 (all parities even) at index -1
    mask = 0
    result = 0

    for i, c in enumerate(s):
        mask ^= (1 << int(c))   # flip parity of digit c

        # Case 1: parity mask is all-even (mask == 0, or mask seen before with same value)
        if mask in first_seen:
            result = max(result, i - first_seen[mask])
        else:
            first_seen[mask] = i   # record first occurrence of this mask

        # Case 2: exactly one digit has odd count → mask ^ (1 << d) was seen before
        for d in range(10):
            target = mask ^ (1 << d)   # what prefix mask would make XOR = single bit d?
            if target in first_seen:
                result = max(result, i - first_seen[target])

    return result
```

**Why store only the FIRST occurrence:** We want the longest substring, so we want the earliest possible left endpoint. If `first_seen[mask] = p1 < p2`, then using p1 as left gives length `i - p1 > i - p2`. We never update first_seen once a mask is recorded.

**Example trace for s="3242415":**
- i=0, c='3': mask=0b0001000. first_seen={0:-1, 8:0}. result=0
- i=1, c='2': mask=0b0001100. first_seen[12]=1. result=0
- i=2, c='4': mask=0b0011100. first_seen[28]=2. result=0
- i=3, c='2': mask=0b0001000. 8 in first_seen! result = max(0, 3-0) = 3
- ...eventually result = 5 for "24241" (rearranges to "42124" or palindrome "14241")

### Full Solution Walkthrough — Maximum Strong Pair XOR II (LC 2935) — Hard

**Problem:** A pair (x, y) is "strong" if `|x - y| ≤ min(x, y)`. Find the maximum XOR of a strong pair.

**Strong pair condition analysis:**
`|x - y| ≤ min(x, y)` where x, y > 0.
WLOG x ≤ y: then min(x,y)=x and |x-y|=y-x. Condition: y-x ≤ x → y ≤ 2x.
So strong pair with x ≤ y iff: `x ≤ y ≤ 2x`.

**Algorithm:** Sort nums. Use a two-pointer sliding window: for each `y = nums[r]`, maintain a window of valid `x` values where `x ≥ y/2` (i.e., `y ≤ 2x`). The window is `nums[l..r]` where `nums[l] >= y/2`.

```python
def maximumStrongPairXor(nums):
    nums.sort()
    n = len(nums)

    trie = {}

    def insert(num):
        node = trie
        for i in range(19, -1, -1):   # up to 2^20
            b = (num >> i) & 1
            if b not in node:
                node[b] = {}
            node = node[b]

    def query_max_xor(num):
        if not trie:
            return 0
        node = trie
        xor = 0
        for i in range(19, -1, -1):
            b = (num >> i) & 1
            want = 1 - b
            if want in node:
                xor |= (1 << i)
                node = node[want]
            else:
                if b in node:
                    node = node[b]
                else:
                    break
        return xor

    result = 0
    l = 0
    for r in range(n):
        y = nums[r]
        insert(y)   # always insert y (it can pair with itself if needed, but we check constraint)

        # Remove nums[l] from window if nums[l] < y/2 (i.e., y > 2*nums[l])
        while nums[l] * 2 < y:
            # Rebuild Trie without nums[l]
            # Note: dictionary-based Trie doesn't support efficient deletion
            # Solution: rebuild from scratch for window [l+1..r]
            l += 1
            trie.clear()
            for i in range(l, r + 1):
                insert(nums[i])

        result = max(result, query_max_xor(y))

    return result
```

**Note on Trie deletion:** Dictionary-based Tries don't support O(1) deletion. For production-quality code, use a Trie with reference counts. The rebuild approach above has O(n²) worst case. The optimal solution uses a Trie with a `count` field per node for O(32) deletion.

**Optimal delete-capable Trie:**
```python
class Trie:
    def __init__(self):
        self.children = [None, None]
        self.count = 0   # how many numbers pass through this node

    def insert(self, num):
        node = self
        for i in range(19, -1, -1):
            b = (num >> i) & 1
            if not node.children[b]:
                node.children[b] = Trie()
            node = node.children[b]
            node.count += 1

    def delete(self, num):
        node = self
        for i in range(19, -1, -1):
            b = (num >> i) & 1
            node = node.children[b]
            node.count -= 1

    def max_xor(self, num):
        node = self
        xor = 0
        for i in range(19, -1, -1):
            b = (num >> i) & 1
            want = 1 - b
            if node.children[want] and node.children[want].count > 0:
                xor |= (1 << i)
                node = node.children[want]
            elif node.children[b] and node.children[b].count > 0:
                node = node.children[b]
            else:
                break
        return xor
```

## System Design (30 min)

### Topic: Cache Sizing and Eviction Policy Selection

**Working set theory:** Optimal cache size = size of the working set (the set of keys that are actively accessed). Cache larger than the working set → wasted memory. Cache smaller → cache thrashing (low hit rate despite large cache).

**Miss penalty analysis:**
```
Effective latency = hit_rate × cache_latency + miss_rate × db_latency
Example: hit_rate=95%, cache RTT=0.1ms, DB RTT=5ms:
= 0.95 × 0.1 + 0.05 × 5 = 0.095 + 0.25 = 0.345ms
```
Increasing hit rate from 95% to 99%:
= 0.99 × 0.1 + 0.01 × 5 = 0.099 + 0.05 = 0.149ms
→ 2.3x latency improvement for 4% increase in hit rate.

**Choosing eviction policy:**

| Policy | When to Use | Trade-off |
|--------|-------------|-----------|
| LRU | Temporal locality: recent = likely future | Poor for scan workloads (large sequential scan evicts all hot items) |
| LFU | Stable frequency patterns: popular items accessed regularly | High miss rate for new items (start at freq=1, can be evicted immediately) |
| ARC (Adaptive) | Unknown workload mix | Balances recency + frequency; adapts automatically; higher memory overhead |
| CLOCK | Low-overhead LRU approximation | Used in OS page replacement; O(1) with single clock hand sweep |
| Random | Consistent performance under any workload | Not optimal for any specific pattern; no overhead; hard to reason about |

**Hotspot key problem:**
A single key (e.g., trending celebrity profile) receives 10% of all requests → one cache node handles 10% of all traffic = bottleneck.

Solutions:
1. **Local in-process cache per app server:** Each app server caches the hot key locally with a short TTL (1-5 seconds). 100 app servers × 1M QPS/server = 100x fan-out reduced to local reads.
2. **Key replication with random suffix:** Store `hot_key_0`, `hot_key_1`, ... `hot_key_N-1`. On read, pick a random suffix. Write to ALL N copies. Spreads read load across N cache nodes.

**Interview talking point:**
*"For an 8GB Redis node serving a product catalog, I'd start with LRU. If I see high miss rates on popular new items, I'd switch to ARC which adapts between LRU and LFU. For a celebrity profile hotspot at 1M QPS on a single key, I'd use in-process cache with a 1-second TTL on each app server — 50 servers means each server only fetches from Redis every ~50ms on average, reducing the per-key Redis load by 50x."*
