## DSA (2 hours)

### Pattern: XOR + Binary Trie for Maximum XOR

**Key XOR properties to internalize:**
- `a ⊕ a = 0` (self-cancellation)
- `a ⊕ 0 = a` (identity)
- XOR is commutative and associative
- `prefix[R] ⊕ prefix[L-1]` = XOR of subarray `nums[L..R]`
- To MAXIMIZE `x ⊕ y`: process bits MSB→LSB; greedily take the opposite bit at each level of a Trie

**Trigger condition:**
- "Find pair with maximum XOR"
- "Count subarrays/pairs with XOR in a range"
- "Maximum XOR subject to a value constraint" — signals offline query + Trie

**Complexity:**
- Trie insert/query: O(32) = O(1) per number for 32-bit integers
- O(n × 32) to build; O(n × 32) to answer all queries

### Full Solution Walkthrough — Single Number (LC 136) — Revision

**Problem:** Every element appears twice except one. Find the singleton in O(n) time and O(1) space.

```python
def singleNumber(nums):
    result = 0
    for n in nums:
        result ^= n
    return result
    # XOR all elements.
    # Pairs: a ⊕ a = 0 for each pair.
    # Lone element: 0 ⊕ x = x.
    # Final result = 0 ⊕ (all elements) = the single element.
```

**Verify:** [4, 1, 2, 1, 2]
- 0 ⊕ 4 = 4
- 4 ⊕ 1 = 5 (binary 101)
- 5 ⊕ 2 = 7 (111)
- 7 ⊕ 1 = 6 (110)
- 6 ⊕ 2 = 4 ← correct (4 is the singleton)

**XOR cancel proof:** For any pair (a, a): a ⊕ a = 0 because every bit in a XORed with itself is 0. XOR of all pairs = 0. XOR of 0 with the lone element x = x.

### Full Solution Walkthrough — Maximum XOR of Two Numbers (LC 421) — Hard (Key technique)

**Problem:** Find the maximum value of `nums[i] XOR nums[j]` for any two elements.

**Why Trie:** To maximize a XOR b, at each bit position from MSB to LSB, we want to choose the opposite bit. A binary Trie stores all numbers; for a query number, we greedily navigate the Trie to the opposite bit at each level.

```python
def findMaximumXOR(nums):
    # Build binary Trie (MSB first: bit 31 down to bit 0)
    trie = {}
    for num in nums:
        node = trie
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            if bit not in node:
                node[bit] = {}
            node = node[bit]

    max_xor = 0
    for num in nums:
        node = trie
        cur_xor = 0
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            want = 1 - bit   # Greedy: take opposite bit to maximize XOR
            if want in node:
                cur_xor |= (1 << i)
                node = node[want]
            else:
                node = node[bit]   # Opposite not available; take same bit (XOR bit = 0)
        max_xor = max(max_xor, cur_xor)

    return max_xor
```

**Greedy correctness:** XOR value is determined by the highest bit that differs. By choosing the opposite bit at each level MSB→LSB, we greedily set as many high-order bits as possible. The first "take same" decision sacrifices a bit — but there's no higher bit left to gain, so it's optimal.

**Trace for nums=[3,10,5,25,2,8]:**
- 25 (11001) XOR 5 (00101) = 28 (11100) — MSB 4 differs, then more
- 25 XOR 2 (00010) = 27 — max is actually 28 (25⊕5)
- Trie navigation for num=25: at each level, prefer opposite bit
  - Bit 4: 25 has 1, want 0 → go to 0 (node for numbers starting 0...) 
  - Build XOR greedily → confirms 28

### Full Solution Walkthrough — Maximum XOR With an Element From Array (LC 1707) — Hard

**Problem:** queries[i] = [xi, mi]. For each query, find the max XOR of xi with any element of nums where that element ≤ mi. Return -1 if no such element.

**Offline approach:** Sort queries by their limit mi. Sort nums. Use a two-pointer to incrementally insert eligible nums into the Trie as queries increase.

```python
def maximizeXor(nums, queries):
    nums.sort()
    # Index queries so we can return answers in original order
    indexed_queries = sorted(enumerate(queries), key=lambda x: x[1][1])  # sort by mi

    trie = {}

    def insert(num):
        node = trie
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            if bit not in node:
                node[bit] = {}
            node = node[bit]

    def query_max_xor(num):
        if not trie:
            return -1
        node = trie
        xor_val = 0
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            want = 1 - bit
            if want in node:
                xor_val |= (1 << i)
                node = node[want]
            elif bit in node:
                node = node[bit]
            else:
                return -1   # Trie is unexpectedly empty (shouldn't happen if trie non-empty)
        return xor_val

    result = [-1] * len(queries)
    j = 0   # pointer into sorted nums

    for qi, (xi, mi) in indexed_queries:
        # Insert all nums <= mi into the Trie
        while j < len(nums) and nums[j] <= mi:
            insert(nums[j])
            j += 1
        result[qi] = query_max_xor(xi)

    return result
```

**Why offline works:** If we sort both nums and queries by the same bound `mi`, then as we process queries in increasing order of `mi`, we only ever ADD nums to the Trie (never remove). This makes the two-pointer approach valid: nums are inserted exactly once, queries are answered in sorted order but results stored at original indices.

**Complexity:** O((n + q) log(n + q)) for sorting + O(n × 32 + q × 32) for Trie operations = O((n+q) × 32).

## System Design (30 min)

### Topic: Redis Deployment Modes — Sentinel vs. Cluster vs. Standalone

**Mode 1: Standalone**
- Single Redis instance
- No HA, no replication
- Use for: development, simple caching where downtime is acceptable
- Capacity: bounded by single node RAM

**Mode 2: Redis Sentinel**
- Master + one or more replicas + 3+ sentinel processes (need quorum for failover)
- Sentinels monitor the master; if master unreachable for `down-after-milliseconds` (default 30s), Sentinel promotes a replica
- Client connects to Sentinel first to discover current master address
- No horizontal scaling: data still lives on a single master (replicas are read-only)
- Use for: HA for single-node workloads, automatic failover, no need for sharding

```
           [Sentinel 1]
           [Sentinel 2]  ← quorum of 2 out of 3 needed to declare failure
           [Sentinel 3]
Master ──→ Replica 1
       ──→ Replica 2
```

**Mode 3: Redis Cluster**
- Shards data across multiple primary nodes (each owns a subset of 16384 slots)
- Each primary has replicas for HA
- Clients need cluster-aware library; server sends MOVED redirect if client contacts wrong node
- Horizontal scaling: add primaries to increase capacity and throughput
- Use for: data > single-node RAM, write throughput requires horizontal scale

**When to choose Cluster over Sentinel:**
1. Dataset exceeds single-node memory (e.g., > 256GB)
2. Write throughput exceeds single-node capacity (millions of writes/sec)
3. Need to scale out without downtime

**Replication lag issue in both modes:**
Redis replication is asynchronous. During failover:
- Writes to old primary after the last replicated write are LOST
- Mitigation: `WAIT numreplicas timeout` — blocks the write until numreplicas confirm receipt

**Interview talking point:**
*"For a session cache at 1M QPS with a 100GB dataset, I'd use Redis Sentinel — data fits on one node, and Sentinel gives automatic failover. At 10M QPS or 500GB+, I'd switch to Redis Cluster with 4-6 primary shards. I'd always monitor cluster_stats_messages_sent for gossip overhead and use hash tags {user_id} to keep related keys on the same shard when multi-key operations are needed."*

### XOR + Binary Trie — Maximum XOR (LC 421 / LC 1707)

**Situation:** "Maximum XOR of Two Numbers: find the pair with the highest XOR value from an array of n integers. Brute force checks all O(n²) pairs — for n=20,000, that's 400M comparisons and TLEs."

**Task:** "Use a binary Trie (MSB→LSB) to answer each XOR maximization query in O(32) time. Build the Trie once for O(n × 32) total, then query once per number for O(n × 32) total."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "XOR + Binary Trie. MSB→LSB processing is essential — XOR magnitude is dominated by the highest differing bit, so greedy from the top bit downward is globally optimal."
2. *Initialize:* "Build a Trie of all numbers, each number inserted as 32 bits from MSB to LSB. Each Trie node has children[0] and children[1]. Optionally add a count field for constrained queries."
3. *Core loop logic:* "For each num, query the Trie: at each bit level, if the opposite bit (1 - bit) exists in the Trie, take it (maximizes XOR at this bit). Otherwise take the same bit. Accumulate the XOR value as you descend."
4. *Convergence guarantee:* "Insert: O(32) per number. Query: O(32) per number. Total O(n × 32). The greedy choice at each bit level is locally and globally optimal — XOR value at bit k is worth 2^k, always dominates lower bits combined."
5. *Duplicate handling / edge case proactivity:* "For LC 1707 (constrained by limit mi): sort both nums and queries by limit value. Insert nums ≤ mi into the Trie before answering query i. Each number inserted at most once. Handle empty Trie: return -1."

**Result:** "O(n²) → O(n × 32). For n=20,000: 400M → 640K operations. LC 1707 offline queries run in O((n+q) × 32) total."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer Trie here |
|-------------|---------------------------|----------------------|
| Brute force O(n²) | n ≤ 1,000 | Infeasible for large n; Trie generalizes cleanly |
| Prefix XOR + hashmap | Count XOR = target problems | Trie needed for maximize/range-XOR; hashmap for count-XOR |
| Linear basis (XOR basis) | Maximum XOR over all subsets | Different problem — subset XOR, not pair XOR |

## Behavioral (30 min)

**Target LP:** Learn and Be Curious (Amazon) / Grow (Google)

**STAR Story: "Learning That Greedy XOR Maximization Fails Without a Trie"**

**Situation (20%):** Early in a project building a proximity-matching system for an internal tool, I designed a "similarity XOR score" between user preference vectors encoded as 32-bit integers. I implemented what I believed was an efficient O(n log n) approach: sort the values and pair adjacent elements (adjacent values in binary usually differ by fewer bits, so I assumed XOR would be high). The system passed all unit tests. After 2 weeks in production, a user filed a bug: the top matched pairs were obviously wrong — two users with wildly different preferences were being paired as "most similar."

**Task:** I owned the investigation. My hypothesis: my greedy pairing was wrong. I needed to identify exactly why and redesign the approach correctly.

**Action (60-70%):**
1. I constructed a small counterexample manually: with numbers [5, 3, 14, 7], the adjacent-sort approach paired (3,5) and (7,14), giving XOR scores 6 and 9. But the actual maximum XOR pair is (5,14)=11 and (3,7)=4. My greedy was suboptimal.
2. I researched the maximum XOR problem and found the binary Trie approach: insert all numbers MSB→LSB; for each number, greedily take the opposite bit at each level. This is O(n × 32) and provably optimal.
3. I re-implemented the matching system using the Trie approach, added tests with known-optimal answers (using brute-force O(n²) as ground truth for small test cases), and verified the new approach matched on all 10 test vectors.
4. I also wrote a post-mortem documenting why sorting is insufficient: adjacent values in sorted order are close in magnitude but not necessarily far in XOR space. The XOR-maximizing pair can be far apart numerically. The mental model "similar magnitude → low XOR" was backwards.
5. I shared the post-mortem with the team as a learning document. The principle I internalized: greedy correctness requires a proof, not intuition. For XOR problems specifically, the Trie gives a provable greedy at each bit level, not a heuristic.

**Result (10-20%):** After the redesign, the matching quality improved measurably: manual review of the top 100 matched pairs went from 60% "obviously wrong" to 8% (some edge cases with sparse preference vectors). The system has been running correctly in production for 6 months. More importantly, I now always ask: "Is this greedy provably optimal at each step, or just intuitive?" before committing to a greedy design.

**Interview tip:** Learn and Be Curious stories are strongest when you show you changed a MENTAL MODEL, not just a line of code. "I learned that greedy correctness requires a formal proof" is a principle. "I'll be more careful next time" is not.
