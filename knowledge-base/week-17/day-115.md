## DSA (2 hours)

### Pattern: Game Theory DP — Optimal Play Recurrence

**Core insight:** Each player maximizes their own outcome (or equivalently, minimizes the opponent's). When it is your turn, you choose the move that maximizes `(your gain) - (opponent's best gain from the resulting state)`.

**Two flavors:**
1. **Score-based:** `dp[state] = max advantage (your score minus opponent's score)` from this state onward. You win if `dp[initial] >= 0`.
2. **Win/lose (boolean):** `dp[state] = True` if the current player can force a win. True if ANY move leads to a losing state for the opponent.

**Trigger condition:**
- Two players alternate; both play optimally
- "Can Alice force a win?" or "What is Alice's maximum score?"
- Game always terminates; no draws or cycles (or handle separately)
- State space is small enough for memoization/tabulation

**Complexity:**
- Time: O(S × M) where S = number of states, M = moves per state
- Space: O(S) for dp array

### Full Solution Walkthrough — Predict the Winner (LC 486) — Revision

**Problem:** Two players alternately pick from either end of an array. Player 1 goes first. Can Player 1 guarantee a score >= Player 2?

**Key insight:** `dp(i, j)` = maximum score ADVANTAGE that the current player can achieve over the opponent from subarray `nums[i..j]`. A positive value means the current player wins.

```python
def predictTheWinner(nums):
    from functools import lru_cache
    n = len(nums)

    @lru_cache(maxsize=None)
    def dp(i, j):
        # Base case: only one number left
        if i == j:
            return nums[i]

        # Current player picks from either end and opponent plays optimally from the rest
        # Subtract dp(next_state) because that's the opponent's advantage
        pick_left  = nums[i] - dp(i + 1, j)
        pick_right = nums[j] - dp(i, j - 1)

        return max(pick_left, pick_right)

    return dp(0, n - 1) >= 0
```

**Trace for nums=[1, 5, 2]:**
- dp(0,0)=1, dp(1,1)=5, dp(2,2)=2
- dp(1,2) = max(5-dp(2,2), 2-dp(1,1)) = max(5-2, 2-5) = max(3,-3) = 3
- dp(0,1) = max(1-dp(1,1), 5-dp(0,0)) = max(1-5, 5-1) = max(-4, 4) = 4
- dp(0,2) = max(1-dp(1,2), 2-dp(0,1)) = max(1-3, 2-4) = max(-2,-2) = -2
- -2 < 0 → Player 1 cannot win → return False

**Why subtraction works:** When Player 1 picks `nums[i]`, the subproblem `dp(i+1, j)` is now Player 2's turn. `dp(i+1, j)` returns Player 2's advantage from that point. Player 1's net advantage = nums[i] - (Player 2's advantage) = `nums[i] - dp(i+1, j)`.

**Bottom-up equivalent:**
```python
def predictTheWinner_bottomup(nums):
    n = len(nums)
    dp = [[0] * n for _ in range(n)]
    for i in range(n):
        dp[i][i] = nums[i]
    for length in range(2, n + 1):
        for i in range(n - length + 1):
            j = i + length - 1
            dp[i][j] = max(nums[i] - dp[i+1][j], nums[j] - dp[i][j-1])
    return dp[0][n-1] >= 0
```

### Full Solution Walkthrough — Stone Game III (LC 1406) — Hard

**Problem:** Alice and Bob alternately take 1, 2, or 3 stones from the left end of a pile. Each stone has a value (can be negative). Maximize your score. Who wins?

```python
def stoneGameIII(stoneValue):
    n = len(stoneValue)
    # dp[i] = max advantage for the current player starting at index i
    dp = [float('-inf')] * (n + 1)
    dp[n] = 0   # no stones left: advantage = 0

    # Fill from right to left (current player looks ahead)
    for i in range(n - 1, -1, -1):
        total = 0
        for k in range(1, 4):   # take 1, 2, or 3 stones
            if i + k <= n:
                total += stoneValue[i + k - 1]   # value of stones taken
                # My advantage = what I take - opponent's advantage from remaining
                dp[i] = max(dp[i], total - dp[i + k])

    if dp[0] > 0:
        return "Alice"
    elif dp[0] < 0:
        return "Bob"
    else:
        return "Tie"
```

**Why `total - dp[i+k]`:** Current player takes stones i through i+k-1 (value = total). Opponent then faces position i+k with advantage dp[i+k]. Current player's net advantage = total - dp[i+k].

**Example trace for stoneValue=[1,2,3,7]:**
- dp[4]=0
- i=3: take 1: total=7, dp[3]=max(-inf, 7-0)=7
- i=2: take 1: 3-dp[3]=3-7=-4; take 2: 3+7-dp[4]=10. dp[2]=10
- i=1: take 1: 2-dp[2]=2-10=-8; take 2: 2+3-dp[3]=5-7=-2; take 3: 2+3+7-dp[4]=12. dp[1]=12
- i=0: take 1: 1-dp[1]=1-12=-11; take 2: 1+2-dp[2]=3-10=-7; take 3: 1+2+3-dp[3]=6-7=-1. dp[0]=-1
- dp[0]=-1 < 0 → Bob wins

### Full Solution Walkthrough — Stone Game IV (LC 1510) — Hard

**Problem:** Alice and Bob alternately remove a perfect-square number of stones. Player who cannot move loses. Given n stones, does Alice win?

**Sprague-Grundy intuition:** A position is LOSING (P-position) if ALL moves lead to WINNING positions (opponent always wins from there). A position is WINNING (N-position) if SOME move leads to a LOSING position (you can put opponent in a losing spot).

```python
def winnerSquareGame(n):
    # dp[i] = True if the current player can force a win with i stones
    dp = [False] * (n + 1)
    # dp[0] = False: no stones left, current player loses (can't move)

    for i in range(1, n + 1):
        j = 1
        while j * j <= i:
            if not dp[i - j * j]:
                # Moving to position i-j² puts opponent in a losing position
                dp[i] = True
                break
            j += 1
        # If no j found, all moves lead to winning positions for opponent → dp[i]=False

    return dp[n]
```

**P/N-position table for small n:**
| n | Squares available | dp[n] | Why |
|---|-------------------|-------|-----|
| 0 | — | False | No moves: current player loses |
| 1 | {1} | True | Take 1 → opponent faces 0 (loses) |
| 2 | {1} | False | Take 1 → opponent faces 1 (wins) |
| 3 | {1} | True | Take 1 → opponent faces 2 (loses) |
| 4 | {1,4} | True | Take 4 → opponent faces 0 (loses) |
| 5 | {1,4} | False | Take 1 → opponent 4 (True); Take 4 → opponent 1 (True) |
| 6 | {1,4} | True | Take 1 → opponent 5 (False) |
| 7 | {1,4} | False | Take 1 → 6 (True); Take 4 → 3 (True) |

Pattern of False: {0,2,5,7,10,13,15,...} — not a simple formula; DP needed.

## System Design (30 min)

### Topic: Cache Invalidation Strategies

Keeping the cache consistent with the database is the hardest problem in distributed systems ("There are only two hard things in computer science: cache invalidation and naming things").

**Strategy 1 — Write-through:**
- Write to cache AND database simultaneously on every write
- Cache is always consistent with DB
- Writes are slower (both must succeed)
- Best for: financial data, inventory counts — strong consistency required

**Strategy 2 — Write-back (write-behind):**
- Write to cache; flush to DB asynchronously (after a delay or batch)
- Writes are fast; risk of data loss if cache node fails before flush
- Best for: write-heavy workloads where eventual persistence is acceptable (e.g., analytics counters)

**Strategy 3 — Write-around:**
- Write directly to DB; bypass cache (don't populate cache on write)
- Cache is only populated on reads (on miss: load from DB, populate cache)
- Best for: write-once / rarely-read data (logs, audit trails)

**Strategy 4 — Cache-aside (lazy loading):**
- App checks cache on read; if miss → read from DB → write to cache
- Cache is only populated for items that are actually read
- Stale data window = TTL (or until explicit invalidation)
- Best for: read-heavy workloads with known access patterns

**Strategy 5 — Event-driven invalidation (CDC):**
- Database change triggers cache invalidation via Debezium (CDC) → Kafka → cache delete
- Cache reflects DB changes within milliseconds
- More complex: requires CDC pipeline (Debezium + Kafka/Redis pub-sub)
- Best for: when DB is the system of record and cache must stay fresh without TTL expiry

```
DB → Debezium (reads binlog/WAL) → Kafka topic → Consumer → Redis DEL key
```

**Interview talking point:**
*"For a product catalog that updates rarely but reads millions of times per second, I'd use cache-aside with a 5-minute TTL. For a user balance that updates frequently and must be accurate, I'd use write-through with no TTL. For user activity feeds that accept 1-2 second staleness, write-back with batch flushing reduces DB write pressure by 100x."*

### Game Theory DP — Score-based (Predict the Winner / Stone Game III)

**Situation:** "Predict the Winner: two players alternately pick from either end of an array. Can Player 1 guarantee a win? Brute force tries all possible play sequences — for n=20, that's 2^20 ≈ 1M paths, many of which are equivalent subproblems."

**Task:** "Model as interval DP where dp(i, j) = maximum score ADVANTAGE the current player can achieve from subarray nums[i..j]. The subtraction recurrence automatically accounts for optimal opponent play."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "Score-based game DP. Key insight: dp(i, j) encodes BOTH players' optimal play simultaneously by using advantage = your_score - opponent_score. No separate tracking of 'whose turn' is needed."
2. *Initialize:* "Base case: dp(i, i) = nums[i] — one element left, current player takes it, advantage = nums[i]. Memoize with lru_cache or fill 2D table diagonally."
3. *Core loop logic:* "dp(i, j) = max(nums[i] - dp(i+1, j), nums[j] - dp(i, j-1)). Subtraction reason: after Player 1 takes nums[i], it's Player 2's turn; dp(i+1, j) is Player 2's advantage from that state; Player 1's net = nums[i] - (Player 2's advantage)."
4. *Convergence guarantee:* "Subarray size strictly decreases each call. O(n²) states, O(1) transitions each = O(n²) total."
5. *Duplicate handling / edge case proactivity:* "n=1: dp(0,0)=nums[0] ≥ 0, so Player 1 always wins. n=2: Player 1 takes the larger of two — always wins. These edge cases fall out naturally without special handling."

**Result:** "O(n²) time and space. Player 1 wins iff dp(0, n-1) >= 0. For n=2000: 4M operations, easily within time limits."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer subtraction DP here |
|-------------|---------------------------|--------------------------------|
| Explicit turn-tracking (dp[i][j][turn]) | Asymmetric games where players have different move sets | Adds unnecessary complexity for symmetric games |
| Minimax with alpha-beta pruning | n > 20, need exact optimal | Harder to implement correctly; DP is sufficient here |

### Game Theory DP — Win/Lose Boolean (Stone Game IV)

**Situation:** "Stone Game IV: players alternately remove a perfect square number of stones. Last to move wins. Need to determine if the starting player wins for any n up to 100,000. Brute force game tree is exponential."

**Task:** "Use boolean DP: dp[i] = True if the current player wins when there are i stones. Compute for all i from 0 to n in O(n × √n) time."

**Action:** Walk the interviewer through these steps:
1. *Classify the pattern:* "Win/lose (P/N position) game DP. dp[i] = True (N-position: current player wins) if ANY move leads to a losing state for the opponent. dp[i] = False (P-position) if ALL moves lead to winning states for the opponent."
2. *Initialize:* "dp[0] = False — no moves available, current player loses (or wins-last-move variant; clarify with interviewer). Precompute all perfect squares ≤ n."
3. *Core loop logic:* "For i in range(1, n+1): for j² ≤ i: if dp[i - j²] == False: dp[i] = True; break. If no j² leads to a False state: dp[i] = False."
4. *Convergence guarantee:* "i - j² < i for all j ≥ 1, so smaller states are always computed first. O(n × √n) total — for n=100,000: 31M operations."
5. *Duplicate handling / edge case proactivity:* "The break optimization: once we find any move to a False state, dp[i] = True immediately. Don't enumerate all perfect squares unnecessarily."

**Result:** "O(n × √n) vs. exponential brute force. For n=100,000: ~31M operations vs. billions."

**Alternative Approaches & Trade-offs**

| Alternative | When you might consider it | Why prefer boolean DP here |
|-------------|---------------------------|----------------------------|
| Sprague-Grundy theorem | Multiple independent sub-games combined | Overkill for single-game problems; boolean DP is sufficient |
| Pattern detection (Sprague-Grundy values repeat) | Academic competitive programming | Unpredictable for new problems; compute from scratch |
