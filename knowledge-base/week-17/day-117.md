## DSA (2 hours)

### Pattern: Monotone Bit-Set Technique for AND/OR Subarrays

**Core property:** For any fixed right endpoint r, the set of distinct values of `arr[l..r]` (for all l ≤ r) under bitwise AND (or OR) has at most O(log V) distinct values. This is because:
- AND can only clear bits, never set them: value is non-increasing as l decreases
- At most 32 distinct transitions (each bit flipped at most once per right endpoint)

This lets us maintain a set of O(log V) distinct (value, count) pairs per step, enabling O(n log V) algorithms instead of O(n²).

**Trigger condition:**
- "For each right endpoint, maintain all distinct AND/OR values of subarrays ending here"
- Sliding window over AND values
- Closest to target queries over subarray AND

**Complexity:**
- Time: O(n log V) for monotone set approach
- Space: O(log V) per step (set of distinct values)

### Full Solution Walkthrough — Counting Bits (LC 338) — Revision

**Problem:** Return array `ans` of length n+1 where `ans[i]` = number of 1-bits in i.

**Two O(n) recurrences:**

```python
def countBits(n):
    dp = [0] * (n + 1)
    # Recurrence 1: dp[i] = dp[i >> 1] + (i & 1)
    # i >> 1 drops the last bit; (i & 1) adds back 1 if that bit was 1
    for i in range(1, n + 1):
        dp[i] = dp[i >> 1] + (i & 1)
    return dp
```

**Alternative — Brian Kernighan:**

```python
def countBits_bk(n):
    dp = [0] * (n + 1)
    # Recurrence 2: dp[i] = dp[i & (i-1)] + 1
    # i & (i-1) clears the LOWEST set bit of i
    # So popcount(i) = popcount(i with lowest bit removed) + 1
    for i in range(1, n + 1):
        dp[i] = dp[i & (i - 1)] + 1
    return dp
```

**Trace for i=6 (binary 110):**
- i >> 1 = 3 (binary 011); dp[3]=2. (i&1)=0. dp[6] = 2+0 = 2. ✓ (110 has 2 set bits)
- i & (i-1) = 6 & 5 = 110 & 101 = 100 = 4; dp[4]=1. dp[6] = 1+1 = 2. ✓

**Brian Kernighan for counting set bits in a single number (O(k) where k = popcount):**
```python
def popcount_bk(n):
    count = 0
    while n:
        n = n & (n - 1)   # clears lowest set bit
        count += 1
    return count
```
Used when k << 32 (sparse numbers); otherwise `bin(n).count('1')` is often fast enough.

### Full Solution Walkthrough — Mysterious Function Closest to Target (LC 1521) — Hard

**Problem:** The mysterious function is bitwise AND. Given arr and target, find the minimum `|AND(arr[l..r]) - target|` over all subarrays.

**Key property:** For a fixed right endpoint `r`, as we extend the subarray left (decrease l), AND can only decrease or stay the same. This creates at most 32 distinct AND values.

```python
def closestToTarget(arr, target):
    # prev = set of distinct AND values of all subarrays ending at the PREVIOUS position
    prev = set()
    result = float('inf')

    for num in arr:
        # All subarrays ending at current position:
        # - The subarray [i..i] (just num itself)
        # - Each previous subarray extended by num (AND with num)
        curr = {num}
        for v in prev:
            curr.add(v & num)
        # AND can only clear bits: curr has at most 32 distinct values

        for v in curr:
            result = min(result, abs(v - target))

        prev = curr   # This step's result becomes next step's "previous"

    return result
```

**Why O(n log V) total:** At each step, `curr` has at most 32 distinct values (each bit can only be cleared once). The inner loop over `prev` is O(log V). Total: O(n × log V).

**Example trace for arr=[9,12,3,7,15], target=5:**
- num=9: curr={9}. |9-5|=4. result=4. prev={9}
- num=12: curr={12, 9&12=8}={12,8}. |12-5|=7, |8-5|=3. result=3. prev={12,8}
- num=3: curr={3, 12&3=0, 8&3=0}={3,0}. |3-5|=2, |0-5|=5. result=2. prev={3,0}
- num=7: curr={7, 3&7=3, 0&7=0}={7,3,0}. |7-5|=2. result=2. prev={7,3,0}
- num=15: curr={15, 7,3,0}={15,7,3,0}. |7-5|=2. result stays 2.
- Return 2. (e.g., AND(9,12,3)=0... AND(3)=3 → |3-5|=2)

### Full Solution Walkthrough — Count Pairs With XOR in a Range (LC 1803) — Hard

**Problem:** Count pairs (i,j) with i < j where `low ≤ nums[i] ⊕ nums[j] ≤ high`.

**Approach:** Use Trie. Process nums left to right. For each new num, count how many PREVIOUSLY inserted nums produce XOR in [low, high] = `countLess(high+1) - countLess(low)`.

```python
def countPairs(nums, low, high):
    class TrieNode:
        def __init__(self):
            self.children = [None, None]
            self.count = 0   # how many numbers pass through this node

    root = TrieNode()

    def insert(num):
        node = root
        for i in range(14, -1, -1):   # nums[i] <= 2^14; use 15 bits
            b = (num >> i) & 1
            if not node.children[b]:
                node.children[b] = TrieNode()
            node = node.children[b]
            node.count += 1

    def count_less_than(num, limit):
        """Count numbers in Trie such that num XOR trie_number < limit."""
        node = root
        count = 0
        for i in range(14, -1, -1):
            if not node:
                break
            nb = (num >> i) & 1       # bit of query num
            lb = (limit >> i) & 1     # bit of limit

            if lb == 1:
                # To get XOR bit = 0 (< limit at this bit), go to child with same bit as nb
                # All those XOR values will have 0 here < 1 → all valid, count them
                if node.children[nb]:
                    count += node.children[nb].count
                # Continue along the path that gives XOR bit = 1 (equal to limit so far)
                node = node.children[1 - nb] if node.children[1 - nb] else None
            else:
                # lb == 0: to keep XOR ≤ limit at this bit, XOR bit must also be 0
                # Go to child giving XOR bit = 0 (same bit as nb)
                node = node.children[nb] if node.children[nb] else None

        return count

    result = 0
    for num in nums:
        result += count_less_than(num, high + 1) - count_less_than(num, low)
        insert(num)

    return result
```

**count_less_than logic explained:**
At each bit position, the `limit` bit tells us what XOR bit value would make the XOR strictly less than limit at this level:
- If `limit bit = 1`: XOR values with bit=0 here are all < limit (count them all). Then follow the path where XOR bit = 1 (equal to limit, continue deciding lower bits).
- If `limit bit = 0`: XOR values with bit=1 here are all > limit. Only follow path where XOR bit = 0.

## System Design (30 min)

### Topic: Cache Warming and the Thundering Herd Problem

**Cold start problem:** When a new cache node starts (deployment, failover), hit rate = 0%. Every request falls through to the database. For a system with 95% cache hit rate and 1M QPS, even a 2-minute cold start sends ~60M extra DB queries.

**Warming strategies:**

**Strategy 1 — Pre-population from DB:**
Before starting traffic, run a batch job that reads hot keys from DB and populates cache. Works for known-hot keys (e.g., top 1000 products by sales). Limitation: doesn't cover long-tail access.

**Strategy 2 — Shadow reads (dual-read warm-up):**
New cache node receives read traffic but ALSO reads from old node. Any miss is populated from the old node. The new node warms up by shadowing real traffic.

**Strategy 3 — Traffic replay from access logs:**
Replay the last N minutes of access logs against the new node. Approximates the working set quickly.

---

**Thundering herd (cache stampede) problem:** A popular cached key expires. Simultaneously, hundreds of requests arrive, all miss the cache, all hit the DB to rebuild the same value, all try to write the result back to cache.

**Solution 1 — Mutex/distributed lock:**
```python
import time

def get_with_lock(cache, db, key, ttl=300):
    val = cache.get(key)
    if val is not None:
        return val
    # Try to acquire lock for rebuilding this key
    lock_key = f"lock:{key}"
    if cache.set(lock_key, 1, nx=True, ex=5):   # NX = only if not exists
        # We won the lock: rebuild value
        val = db.query(key)
        cache.set(key, val, ex=ttl)
        cache.delete(lock_key)
        return val
    else:
        # Another process is rebuilding; wait briefly and retry
        time.sleep(0.05)
        return cache.get(key) or db.query(key)   # fallback to DB if still missing
```

**Solution 2 — Probabilistic early expiration (XFetch algorithm):**
Before key expires, random requests refresh it early with probability proportional to elapsed time since last refresh:
```
p = exp(-delta / beta)   where delta = ttl_remaining, beta = tuning param
refresh if rand() < p
```
Spreads the refresh across multiple requests before expiry, eliminating the simultaneous stampede.

**Solution 3 — Stale-while-revalidate:**
Serve the stale (expired) value immediately while asynchronously triggering a background refresh. Requires accepting brief staleness. Used by CDNs and browser caches (`Cache-Control: stale-while-revalidate=60`).

**Interview talking point:**
*"For a homepage widget cached for 5 minutes at 50K QPS, I'd use stale-while-revalidate: serve the cached value (even if just expired) while a background worker refreshes it. This gives zero latency impact on the user and eliminates stampedes. For a user's auth token (where staleness is unacceptable), I'd use a distributed lock with a 5-second lease — only one process rebuilds, others wait or fail-fast."*
