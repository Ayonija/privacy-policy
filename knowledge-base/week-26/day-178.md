## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: BFS with Implicit / Dynamic Adjacency + Group Pruning
- Core idea: Instead of a fixed adjacency list, compute neighbors on the fly during BFS. When a class of neighbors (e.g., all indices with the same value) can be enormous, process the group once and immediately remove it from the source map — this prevents re-enqueuing the same group from different entry points and avoids O(n²) worst-case processing.
- Trigger condition: "Minimum steps/jumps where you can move to i±1 or to any index sharing the same value" — any BFS where one neighbor type is a value-keyed group that collapses to a single BFS level.
- Time complexity: O(n) amortized (each index processed once, each group deleted once) | Space complexity: O(n)

### Template (write from memory before solving):
```java
// BFS with value-to-indices grouping (LC 1345 skeleton)
Map<Integer, List<Integer>> valToIdx = new HashMap<>();
for (int i = 0; i < n; i++) valToIdx.computeIfAbsent(arr[i], k->new ArrayList<>()).add(i);
Queue<Integer> q = new LinkedList<>(); q.offer(0);
int[] dist = new int[n]; Arrays.fill(dist, -1); dist[0] = 0;
while (!q.isEmpty()) {
    int i = q.poll();
    if (i == n-1) return dist[i];
    // try i-1, i+1
    // try all same-value indices (then REMOVE from map to avoid re-processing)
    List<Integer> sameVal = valToIdx.remove(arr[i]); // KEY: remove after use
    if (sameVal != null) for (int j : sameVal) { if (dist[j]==-1) { dist[j]=dist[i]+1; q.offer(j); } }
}
```

### Code Skeleton
```java
class Solution {

    // LC 1345 — Jump Game IV (cold solve — 30-min timer)
    // TODO: public int minJumps(int[] arr)
    // Key: BFS; value→indices map; DELETE group from map after processing to avoid TLE.
    public int minJumps(int[] arr) {
        int n = arr.length;
        if (n == 1) return 0;
        Map<Integer, List<Integer>> valToIdx = new HashMap<>();
        for (int i = 0; i < n; i++)
            valToIdx.computeIfAbsent(arr[i], k -> new ArrayList<>()).add(i);

        int[] dist = new int[n];
        Arrays.fill(dist, -1);
        dist[0] = 0;
        Queue<Integer> q = new LinkedList<>();
        q.offer(0);

        while (!q.isEmpty()) {
            int i = q.poll();
            // Neighbor: i-1
            if (i - 1 >= 0 && dist[i - 1] == -1) {
                dist[i - 1] = dist[i] + 1;
                if (i - 1 == n - 1) return dist[i - 1];
                q.offer(i - 1);
            }
            // Neighbor: i+1
            if (i + 1 < n && dist[i + 1] == -1) {
                dist[i + 1] = dist[i] + 1;
                if (i + 1 == n - 1) return dist[i + 1];
                q.offer(i + 1);
            }
            // Neighbors: all same-value indices — REMOVE after visiting to prevent re-enqueue
            List<Integer> sameVal = valToIdx.remove(arr[i]);
            if (sameVal != null) {
                for (int j : sameVal) {
                    if (dist[j] == -1) {
                        dist[j] = dist[i] + 1;
                        if (j == n - 1) return dist[j];
                        q.offer(j);
                    }
                }
            }
        }
        return -1;
    }

    // LC 797 — All Paths From Source to Target (revision — write from memory)
    public List<List<Integer>> allPathsSourceTarget(int[][] graph) {
        List<List<Integer>> result = new ArrayList<>();
        List<Integer> path = new ArrayList<>();
        path.add(0);
        dfs797(graph, 0, path, result);
        return result;
    }

    private void dfs797(int[][] graph, int node, List<Integer> path, List<List<Integer>> result) {
        if (node == graph.length - 1) {
            result.add(new ArrayList<>(path));
            return;
        }
        for (int next : graph[node]) {
            path.add(next);
            dfs797(graph, next, path, result); // no visited[] needed — DAG has no cycles
            path.remove(path.size() - 1);
        }
    }

    // LC 695 — Max Area of Island (OA timed — set 20-min timer before reading)
    // TODO: public int maxAreaOfIsland(int[][] grid)
    // Key (unlock after 20 min): DFS from each unvisited 1-cell; mark visited by setting
    //     cell to 0 in-place; count cells in component; track global max.
    public int maxAreaOfIsland(int[][] grid) {
        int max = 0;
        for (int r = 0; r < grid.length; r++)
            for (int c = 0; c < grid[0].length; c++)
                if (grid[r][c] == 1) max = Math.max(max, dfs695(grid, r, c));
        return max;
    }

    private int dfs695(int[][] grid, int r, int c) {
        if (r < 0 || r >= grid.length || c < 0 || c >= grid[0].length || grid[r][c] == 0) return 0;
        grid[r][c] = 0; // mark visited in-place
        return 1 + dfs695(grid, r+1, c) + dfs695(grid, r-1, c)
                 + dfs695(grid, r, c+1) + dfs695(grid, r, c-1);
    }
}
```

## System Design (30 min — End-to-End Mock)

### Topic: Google Maps / Navigation — End-to-End Design

```
Scale: 1B+ monthly users, 25M map updates/day, GPS probes from 2B Android devices

Core Services:
1. Map Data: road network as weighted directed graph (100B+ nodes = intersections, edges = road segments with distance/speed/restrictions); stored in custom columnar format (Protocol Buffers); sharded by geographic tile
2. Routing Engine: Contraction Hierarchies (CH) algorithm:
   - Preprocessing: contract unimportant nodes adding "shortcut edges" (offline, once per graph update)
   - Query: bidirectional Dijkstra using shortcuts; visits ~1000 nodes instead of billions; <100ms cross-country route
3. Real-Time Traffic: GPS probes from Android devices (anonymized) → Kafka → Speed Estimation Service → road segment speed updates → pushes to routing engine; lag ~2 minutes
4. Geocoding: address → lat/lng (forward); lat/lng → address (reverse); Elasticsearch with geo_point indexing + custom address normalization
5. Map Tiles: pre-rendered PNG/WebGL tiles at zoom levels 0-22; stored in object storage; served via CDN; tile server generates missing tiles on-demand with PostGIS rendering
6. Places / POI: PostgreSQL with PostGIS for location-aware queries; Elasticsearch for full-text search of place names; business data from Google My Business

ETA Calculation:
- Base ETA = Contraction Hierarchies route time
- Adjust for real-time traffic on each segment (speed probes)
- Adjust for historical patterns (Monday 8am vs Sunday 11am on I-405)
- Adjust for upcoming congestion prediction (ML model on probe density trends)
```

**Interview talking point:** "If asked how Google Maps routes you from NYC to LA in <100ms despite having billions of road nodes, answer: Contraction Hierarchies — a preprocessing step that contracts unimportant nodes (adding shortcut edges between their neighbors). During query, bidirectional Dijkstra only expands 'important' nodes (major highways); a cross-country route visits ~1000 contracted nodes instead of billions of raw intersections. The graph is precomputed offline once; queries reuse the contracted graph."

## Flashcards

| Q | A |
|---|---|
| In LC 1345 Jump Game IV, why must you delete a value's index group from the map after BFS processes it? | Without deletion, the BFS would re-enqueue all indices sharing the same value every time any of them is visited — O(n²) time. Deleting after first visit ensures the group is only processed once total; subsequent visits to same-value nodes just use i±1 neighbors. |
| In LC 797 All Paths From Source to Target, why is no visited array needed? | The input is a DAG (Directed Acyclic Graph) — there are no cycles, so DFS can never revisit a node on the current path. Every DFS path terminates at node n-1 or backtracks. A visited array would incorrectly prune valid paths that share intermediate nodes. |
| How does Google Maps' Contraction Hierarchies algorithm achieve sub-100ms routing at continental scale? | Offline preprocessing adds shortcut edges that skip unimportant nodes (e.g., a shortcut from highway entry to exit bypassing all intermediate intersections). Query-time bidirectional Dijkstra only expands high-importance nodes — the contracted graph has 1000× fewer nodes than the raw graph. |
| How does Google Maps incorporate real-time traffic into routing? | Anonymized GPS probes from Android devices are streamed via Kafka; a Speed Estimation service computes current speed per road segment every 2 minutes; these speeds update the edge weights in the routing engine. ETA re-estimates as you drive if conditions change. |
| What is the difference between geocoding and reverse geocoding, and how does Google implement both? | Geocoding: address text → lat/lng coordinates (uses Elasticsearch with address normalization + PostGIS). Reverse geocoding: lat/lng → nearest address (uses PostGIS ST_Distance spatial index to find nearest address node). Both are cached heavily since the same addresses are looked up repeatedly. |
