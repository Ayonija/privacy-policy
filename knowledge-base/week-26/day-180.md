## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Monotonic Stack + Stack Simulation + Stack-based String Processing
- Core idea: A monotonic stack maintains a strict ordering invariant (increasing or decreasing) on its elements. When that invariant is violated by an incoming element, pop and process everything that would break it — the popped element's "scope" extends exactly to the boundary of the new element. For string problems, a stack of (char, count) pairs efficiently collapses adjacent duplicates in a single pass.
- Trigger condition: "Largest rectangle / area under a histogram" — monotonic increasing stack on indices; "decode nested repeat string" — stack of (k, builtString); "remove k adjacent duplicates" — stack of (char, count).
- Time complexity: O(n) for all three | Space complexity: O(n)

### Template (write from memory before solving):
```java
// Monotonic stack — largest rectangle (LC 84)
Deque<Integer> stack = new ArrayDeque<>();
int maxArea = 0;
for (int i = 0; i <= heights.length; i++) {
    int h = (i == heights.length) ? 0 : heights[i];
    while (!stack.isEmpty() && h < heights[stack.peek()]) {
        int height = heights[stack.pop()];
        int width = stack.isEmpty() ? i : i - stack.peek() - 1;
        maxArea = Math.max(maxArea, height * width);
    }
    stack.push(i);
}
return maxArea;
```

### Code Skeleton
```java
class Solution {

    // LC 84 — Largest Rectangle in Histogram (cold solve — 30-min timer)
    // TODO: public int largestRectangleArea(int[] heights)
    // Key: monotonic increasing stack of indices; sentinel 0-height at end drains the stack;
    //      width = i - stack.peek() - 1 (or i if stack empty after pop).
    public int largestRectangleArea(int[] heights) {
        Deque<Integer> stack = new ArrayDeque<>();
        int maxArea = 0;
        for (int i = 0; i <= heights.length; i++) {
            int h = (i == heights.length) ? 0 : heights[i];
            while (!stack.isEmpty() && h < heights[stack.peek()]) {
                int height = heights[stack.pop()];
                int width = stack.isEmpty() ? i : i - stack.peek() - 1;
                maxArea = Math.max(maxArea, height * width);
            }
            stack.push(i);
        }
        return maxArea;
    }

    // LC 394 — Decode String (revision — write from memory)
    public String decodeString(String s) {
        Deque<Integer> countStack = new ArrayDeque<>();
        Deque<StringBuilder> strStack = new ArrayDeque<>();
        StringBuilder curr = new StringBuilder();
        int k = 0;
        for (char c : s.toCharArray()) {
            if (Character.isDigit(c)) {
                k = k * 10 + (c - '0');
            } else if (c == '[') {
                countStack.push(k);
                strStack.push(curr);
                curr = new StringBuilder();
                k = 0;
            } else if (c == ']') {
                int repeat = countStack.pop();
                StringBuilder prev = strStack.pop();
                String repeated = curr.toString().repeat(repeat);
                curr = prev.append(repeated);
            } else {
                curr.append(c);
            }
        }
        return curr.toString();
    }

    // LC 1209 — Remove All Adjacent Duplicates in String II (OA timed — set 20-min timer before reading)
    // TODO: public String removeDuplicates(String s, int k)
    // Key (unlock after 20 min): stack of int[]{char as int, count}; for each char,
    //     if stack top has same char increment count; if count == k pop;
    //     reconstruct string from remaining stack entries.
    public String removeDuplicates(String s, int k) {
        Deque<int[]> stack = new ArrayDeque<>(); // {char, count}
        for (char c : s.toCharArray()) {
            if (!stack.isEmpty() && stack.peek()[0] == c) {
                stack.peek()[1]++;
                if (stack.peek()[1] == k) stack.pop();
            } else {
                stack.push(new int[]{c, 1});
            }
        }
        StringBuilder sb = new StringBuilder();
        // Stack is in reverse order — collect into list first
        Deque<int[]> tmp = new ArrayDeque<>();
        while (!stack.isEmpty()) tmp.push(stack.pop());
        while (!tmp.isEmpty()) {
            int[] top = tmp.pop();
            for (int i = 0; i < top[1]; i++) sb.append((char) top[0]);
        }
        return sb.toString();
    }
}
```

## System Design (30 min — End-to-End Mock)

### Topic: 6-Month System Design Mastery — Grand Synthesis

```
The 5 Universal Principles That Govern Every System Design

1. SCALE FIRST, DESIGN SECOND
   - Write DAU, QPS, storage before drawing anything
   - 1K users = single server; 1M users = caching needed; 1B users = sharding + CDN + global distribution
   - "The numbers determine the architecture" — state this in every interview

2. THE THREE STORAGE CHOICES
   - Relational (PostgreSQL/MySQL): ACID, JOIN, < 10K writes/sec → auth, billing, orders
   - Wide-column (Cassandra/DynamoDB): horizontal scale, high write throughput, no JOIN → messages, activity feeds, time-series
   - Cache (Redis): sub-millisecond reads, ephemeral → sessions, rate limits, leaderboards, pre-computed feeds

3. ASYNC IS THE FORCE MULTIPLIER
   - Synchronous: simple but creates coupling and cascading failures
   - Async (Kafka/SQS): decouples services, absorbs traffic spikes, enables retry
   - Rule: if the response doesn't need to be immediate, make it async
   - Examples: email sending, push notifications, feed fan-out, video transcoding, analytics ingestion

4. THE CACHE IS ALWAYS THE ANSWER (AND THE PROBLEM)
   - Cache hit rate must be >90% or caching is not helping
   - Cache invalidation: TTL (simple but stale), write-through (consistent but slow writes), event-driven invalidation (Kafka event on data change → cache delete)
   - Thundering herd: when cache misses spike simultaneously → mutex lock + probabilistic early expiry

5. TRADE-OFFS ARE THE INTERVIEW
   - Consistency vs Availability: CP = ZooKeeper, PostgreSQL; AP = Cassandra, DynamoDB
   - Push vs Pull: push = lower latency, high write cost (fan-out); pull = simpler writes, higher read cost
   - Normalization vs Denormalization: normalized = storage efficient, JOIN-heavy; denormalized = fast reads, write amplification
   - Stateless vs Stateful: stateless = easy horizontal scale; stateful = complex but enables real-time (WebSocket chat)

6-Month Systems Mastered:
| System | Core Challenge | Key Pattern |
|--------|---------------|-------------|
| Twitter/X | Fan-out at scale | Hybrid push/pull feed |
| Uber/Lyft | Real-time matching | Redis Geo + WebSocket |
| YouTube/Netflix | Global video delivery | CDN + ABR streaming |
| Instagram | Photo feed at 1B users | S3 + CDN + hybrid fan-out |
| WhatsApp | 1.15M messages/sec | WebSocket + Kafka + Cassandra |
| Google Drive | Chunk deduplication | Delta sync + conflict resolution |
| Amazon | Flash sale inventory | Atomic SQL + Saga pattern |
| Airbnb | Availability at scale | Cassandra CAS + Elasticsearch |
| Zoom | Video at 300M users | SFU + simulcast + adaptive bitrate |
| Spotify | Recommendations at scale | Matrix factorization + ANN search |
| Google Maps | Cross-country routing in <100ms | Contraction Hierarchies |
```

**Interview talking point:** "If asked to name the one system design principle that matters most, answer: understand your scale before designing. The same feature — a user feed — requires completely different architectures at 10K users (single DB query), 10M users (cached pre-computed feed), and 1B users (hybrid push/pull with CDN). Scale drives every decision: database choice, caching strategy, async vs sync, sharding vs single node."
