## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Multi-Pattern Synthesis (Binary Search on Feasibility + Greedy Heap + Formula Optimisation)
- Core idea: Binary search on the answer when the feasibility check is efficiently computable — "can we achieve k?" is often easier than "what is the maximum k?" For greedy scheduling problems, translate the constraint into a mathematical formula when possible rather than simulating step-by-step. For heap-based greedy, maintain a size-bounded priority queue and swap out elements to enforce a budget constraint.
- Trigger condition: "Maximum number of X" or "can we assign k tasks" — binary search on k; "schedule tasks with cooldown" — formula first, simulate only if formula is non-obvious; "use k ladders optimally" — greedy min-heap of size k.
- Time complexity: LC 2071 O(n log n × log n) | LC 621 O(n) | LC 1642 O(n log k) | Space complexity: O(n), O(1), O(k)

### Template (write from memory before solving):
```java
// Binary search on feasibility (LC 2071 skeleton)
// canAssign(k, tasks, workers, pills, strength):
//   take k easiest tasks, k strongest workers
//   use TreeMap/deque for workers; iterate tasks hardest to easiest
//   for each task: if strongest worker can do it → assign;
//   else if pills remain → find weakest worker who can with pill → assign, use pill
//   else return false
// Binary search: lo=0, hi=min(tasks.len, workers.len); find max feasible k

// Task Scheduler (LC 621 revision):
int maxFreq = max frequency; int countMax = count of tasks with maxFreq;
return Math.max(tasks.length, (maxFreq-1)*(n+1) + countMax);
```

### Code Skeleton
```java
class Solution {

    // LC 2071 — Maximum Number of Tasks You Can Assign (cold solve — 30-min timer)
    // TODO: public int maxTaskAssign(int[] tasks, int[] workers, int pills, int strength)
    // Key: sort tasks ASC, workers ASC; binary search on k; feasibility check:
    //      take k smallest tasks, k largest workers; iterate tasks hardest→easiest;
    //      use TreeMap for workers (sorted); for each task, if strongest can do it → assign;
    //      else if pills remain, find weakest worker that can do it with pill → assign.
    public int maxTaskAssign(int[] tasks, int[] workers, int pills, int strength) {
        Arrays.sort(tasks);
        Arrays.sort(workers);
        int lo = 0, hi = Math.min(tasks.length, workers.length);
        while (lo < hi) {
            int mid = (lo + hi + 1) / 2;
            if (canAssign(tasks, workers, pills, strength, mid)) lo = mid;
            else hi = mid - 1;
        }
        return lo;
    }

    private boolean canAssign(int[] tasks, int[] workers, int pills, int strength, int k) {
        // k easiest tasks: tasks[0..k-1]; k strongest workers: workers[n-k..n-1]
        TreeMap<Integer, Integer> avail = new TreeMap<>();
        int n = workers.length;
        for (int i = n - k; i < n; i++)
            avail.merge(workers[i], 1, Integer::sum);
        int pillsLeft = pills;
        // Assign tasks from hardest to easiest among chosen k tasks
        for (int i = k - 1; i >= 0; i--) {
            int task = tasks[i];
            // Can the strongest available worker do this task without a pill?
            Integer strongest = avail.lastKey();
            if (strongest >= task) {
                avail.merge(strongest, -1, Integer::sum);
                if (avail.get(strongest) == 0) avail.remove(strongest);
            } else {
                // Use a pill: find weakest worker who can do task with pill
                if (pillsLeft == 0) return false;
                Integer withPill = avail.ceilingKey(task - strength);
                if (withPill == null) return false;
                avail.merge(withPill, -1, Integer::sum);
                if (avail.get(withPill) == 0) avail.remove(withPill);
                pillsLeft--;
            }
        }
        return true;
    }

    // LC 621 — Task Scheduler (revision — write from memory)
    public int leastInterval(char[] tasks, int n) {
        int[] freq = new int[26];
        for (char c : tasks) freq[c - 'A']++;
        int maxFreq = 0;
        for (int f : freq) maxFreq = Math.max(maxFreq, f);
        int countMax = 0;
        for (int f : freq) if (f == maxFreq) countMax++;
        return Math.max(tasks.length, (maxFreq - 1) * (n + 1) + countMax);
    }

    // LC 1642 — Furthest Building You Can Reach (OA timed — set 20-min timer before reading)
    // TODO: public int furthestBuilding(int[] heights, int bricks, int ladders)
    // Key (unlock after 20 min): min-heap of size = ladders; for each positive diff,
    //     push diff to heap; if heap.size() > ladders, pop min (use bricks for that jump);
    //     subtract popped value from bricks; if bricks < 0, stop at previous building.
    public int furthestBuilding(int[] heights, int bricks, int ladders) {
        PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // tracks ladder-assigned jumps
        for (int i = 0; i < heights.length - 1; i++) {
            int diff = heights[i + 1] - heights[i];
            if (diff <= 0) continue; // going down or same — free
            minHeap.offer(diff);
            if (minHeap.size() > ladders) {
                // Replace the smallest ladder-assigned jump with bricks
                bricks -= minHeap.poll();
                if (bricks < 0) return i;
            }
        }
        return heights.length - 1;
    }
}
```

## System Design (30 min — End-to-End Mock)

### Topic: Design Round Framework — Interview Synthesis

```
30-Minute System Design Interview Playbook:

Minute 0–3: Requirements
- Ask: "What are the core user actions?" (2-3 functional reqs)
- Ask: "What scale are we targeting?" (users, QPS, storage)
- State: "I'll assume [specific numbers] and revisit if needed"

Minute 3–8: Capacity Estimation
- Write down: DAU, peak QPS, storage per day, storage over 5 years
- Example: 100M DAU × 10 reads/day = 1B reads/day ÷ 86400 ≈ 11K reads/sec; peak 3× = 33K RPS

Minute 8–20: High-Level Design
- Draw boxes: Client → CDN → API Gateway → Services → DB/Cache/Queue
- Name each service, its responsibility, and its datastore
- Identify the critical path (most common user action) and trace it end-to-end

Minute 20–27: Deep Dive (pick 1-2 components)
- The interviewer will guide you; if not, pick the hardest component
- State the bottleneck, explain your design choice, give the trade-off
- Always have numbers: cache hit rate, replication lag, index cardinality

Minute 27–30: Summary + Trade-offs
- State what you would do differently with more time
- Name one alternative approach you considered and why you chose what you did

Systems to practice delivering in 30 min (pick any, time yourself):
[ ] Instagram (today's topic: feed, photo storage, stories)
[ ] WhatsApp (messaging, presence, group chat)
[ ] Google Drive (upload, sync, sharing, version history)
[ ] Amazon (catalog, search, checkout, inventory)
[ ] Airbnb (search, availability, booking)
[ ] Zoom (SFU, simulcast, recording)
[ ] Spotify (streaming, recommendations, offline)
[ ] Google Maps (routing, traffic, tiles)
```

**Interview talking point:** "If asked what the most common mistake candidates make in system design interviews, answer: jumping to the solution before establishing scale. Always write the numbers first — DAU, QPS, storage. A design for 1K users looks completely different from one for 1B users. The numbers determine whether you need sharding, CDN, caching, or just a single Postgres instance."

- **STAR prompt:** Tell me about a time you delivered a project under significant time or resource constraints. How did you prioritize and what was the outcome?
- **Leadership principle:** Deliver Results — the task assignment problem is itself a metaphor: given constrained resources (workers, pills), maximize output (tasks assigned). In interviews, constrained time demands the same greedy discipline: solve the highest-value sub-problems first.
