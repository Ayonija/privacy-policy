## DSA (1.5 hours — Weak Area Targeted Drill)

### Pattern: Merge Sort with Position Tracking / Inversion Count
- Core idea: During the merge step of merge sort, when a right-half element is placed before remaining left-half elements, those left-half elements each have one more "smaller element to the right." Track original indices alongside values so counts map back to the correct result position. This generalises to any "count elements satisfying a relative property" across an array.
- Trigger condition: "For each element, count how many elements to its right are smaller/larger" — any inversion-count generalisation where a naive O(n²) scan would TLE.
- Time complexity: O(n log n) | Space complexity: O(n)

### Template (write from memory before solving):
```java
// Merge sort counting inversions / smaller elements
int[] result = new int[n];
int[] indices = new int[n]; // track original indices
void mergeSort(int[] nums, int[] indices, int left, int right) {
    if (right - left <= 1) return;
    int mid = (left + right) / 2;
    mergeSort(nums, indices, left, mid);
    mergeSort(nums, indices, mid, right);
    // merge: for each left[i], count how many right[j] < left[i] were placed first
    int j = mid, count = 0;
    // ... standard merge with result[indices[i]] += count
}
```

### Code Skeleton
```java
class Solution {

    // LC 315 — Count of Smaller Numbers After Self (cold solve — 30-min timer)
    // TODO: public List<Integer> countSmaller(int[] nums)
    // Key: merge sort on (value, originalIndex) pairs; when right-half element is
    //      placed before remaining left-half elements, each of those left-half elements
    //      gains +1 to its count (that right element is smaller and to the right).
    private int[] result;
    private int[][] pairs; // pairs[i] = {nums[i], originalIndex}

    public List<Integer> countSmaller(int[] nums) {
        int n = nums.length;
        result = new int[n];
        pairs = new int[n][2];
        for (int i = 0; i < n; i++) pairs[i] = new int[]{nums[i], i};
        mergeSort(pairs, 0, n);
        List<Integer> ans = new ArrayList<>();
        for (int r : result) ans.add(r);
        return ans;
    }

    private void mergeSort(int[][] arr, int left, int right) {
        if (right - left <= 1) return;
        int mid = (left + right) / 2;
        mergeSort(arr, left, mid);
        mergeSort(arr, mid, right);
        merge315(arr, left, mid, right);
    }

    private void merge315(int[][] arr, int left, int mid, int right) {
        int i = left, j = mid;
        List<int[]> tmp = new ArrayList<>();
        while (i < mid && j < right) {
            if (arr[i][0] <= arr[j][0]) {
                // arr[j] elements placed so far (j - mid) are all < arr[i][0] and to its right
                result[arr[i][0 + 1]] += (j - mid); // BUG TRAP: arr[i][1] for original index
                // corrected: result[arr[i][1]] += (j - mid);
                tmp.add(arr[i++]);
            } else {
                tmp.add(arr[j++]);
            }
        }
        while (i < mid) { result[arr[i][1]] += (j - mid); tmp.add(arr[i++]); }
        while (j < right) tmp.add(arr[j++]);
        for (int k = 0; k < tmp.size(); k++) arr[left + k] = tmp.get(k);
    }

    // LC 406 — Queue Reconstruction by Height (revision — write from memory)
    public int[][] reconstructQueue(int[][] people) {
        // Sort: height DESC; if tie, k ASC
        Arrays.sort(people, (a, b) -> a[0] != b[0] ? b[0] - a[0] : a[1] - b[1]);
        List<int[]> list = new ArrayList<>();
        for (int[] p : people) list.add(p[1], p); // insert at index k
        return list.toArray(new int[people.length][]);
    }

    // LC 523 — Continuous Subarray Sum (OA timed — set 20-min timer before reading)
    // TODO: public boolean checkSubarraySum(int[] nums, int k)
    // Key (unlock after 20 min): prefix[i] % k; store first occurrence of each remainder
    //     in HashMap; if same remainder seen at index j where i - j >= 2, return true.
    public boolean checkSubarraySum(int[] nums, int k) {
        Map<Integer, Integer> map = new HashMap<>();
        map.put(0, -1); // remainder 0 seen at index -1 (before array starts)
        int prefixSum = 0;
        for (int i = 0; i < nums.length; i++) {
            prefixSum = (prefixSum + nums[i]) % k;
            if (map.containsKey(prefixSum)) {
                if (i - map.get(prefixSum) >= 2) return true;
            } else {
                map.put(prefixSum, i); // only store first occurrence
            }
        }
        return false;
    }
}
```

## System Design (30 min — End-to-End Mock)

### Topic: Zoom / Video Conferencing — End-to-End Design

```
Scale: 300M daily participants, 3.3 trillion minutes in 2020, peak 10M concurrent meetings

Core Architecture:
1. Signaling Server: WebSocket-based; handles session setup, participant join/leave events, ICE candidate exchange for WebRTC NAT traversal
2. Media Architecture — SFU (Selective Forwarding Unit):
   - Each participant uploads 1 video stream to the SFU
   - SFU forwards each stream to all other participants (selectively — simulcast layers for bandwidth adaptation)
   - vs MCU (Multipoint Control Unit): MCU mixes all streams → 1 composite stream per participant; lower client bandwidth but massive server CPU
   - Zoom uses SFU: 10-person call → 1 upload stream per person (not 9); server just forwards
3. Adaptive Bitrate: each participant publishes 3 simulcast layers (360p/720p/1080p); SFU selects layer per subscriber based on subscriber's bandwidth
4. Recording: SFU writes raw streams to Kafka; Recording Worker consumes, mixes with FFmpeg, uploads final MP4 to S3; transcript via Whisper ASR
5. Screen Share: treated as a separate high-resolution, low-framerate video track (1080p @ 5fps vs camera 720p @ 30fps)

Key numbers:
- SFU vs P2P: for 4 participants, P2P = 4×3=12 streams total; SFU = 4 upload + 4×3=12 forward = 4 upload streams, 3× reduction per user
- Jitter buffer: 60-200ms buffer absorbs network jitter; >200ms feels like lag
- Audio priority: if bandwidth constrained, drop video quality first; audio is 50Kbps vs video 500Kbps-2Mbps
```

**Interview talking point:** "If asked why Zoom chose SFU over P2P WebRTC for groups > 4, answer: in P2P, each participant uploads n-1 copies of their video — for a 10-person call, each person uploads 9 streams, requiring ~18 Mbps upload. SFU requires only 1 upload stream per participant (~2 Mbps); the SFU handles the fan-out. Most home connections can't sustain 9 simultaneous uploads, so SFU is the only viable architecture for groups beyond 4."

## Flashcards

| Q | A |
|---|---|
| How does merge sort count elements smaller than self to the right (LC 315)? | During merge: when right-half element is chosen over left-half elements, increment a counter; that counter = number of right-half elements placed before each left-half element = elements smaller than left-half element appearing to its right |
| In LC 406 Queue Reconstruction, why sort by height DESC then k ASC? | Taller people don't "see" shorter people in front, so inserting at index k among already-placed taller people gives the correct count. Sorting DESC height ensures all placed people are ≥ current height when we process each person. |
| How do you detect a continuous subarray sum divisible by k using prefix sums? | Compute prefix[i] = sum(0..i) % k. If prefix[i] == prefix[j] and i-j >= 2, then sum(j+1..i) is divisible by k. Store first occurrence of each remainder in a HashMap; check on each step. |
| What is the core difference between SFU and MCU in video conferencing? | SFU (Selective Forwarding Unit): each participant uploads 1 stream; SFU forwards each stream to all others; server does no transcoding. MCU (Multipoint Control Unit): server mixes all streams into one composite stream per participant; server CPU-intensive but clients receive only 1 stream. |
| In Zoom's adaptive bitrate, what are simulcast layers and how does the SFU use them? | Each camera publishes 3 resolution/bitrate encodings simultaneously (e.g., 360p/720p/1080p). The SFU independently selects which layer to forward to each subscriber based on their available bandwidth — low-bandwidth subscribers get 360p while others get 1080p, all from the same upload. |
