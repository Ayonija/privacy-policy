# Day 146 — Observer Pattern (Behavioral)
**Week 22 | Phase: LLD Mastery | Month 4**

## Focus
The Observer pattern lets a subject automatically notify all registered observers when its state changes — no polling, no tight coupling. The subject only knows the observer interface, not the concrete observers. Master it for any question involving notifications, event listeners, pub-sub, or keeping multiple views in sync with one model.

---

## Core Concept

### Analogy
Instagram follow: you follow a creator, and the moment they post, you get notified — without you checking their profile repeatedly. The creator (subject) broadcasts to all followers (observers). Unfollowing removes you from the list. The creator doesn't know or care who you are — only that you're a follower.

### What It Does
- Subject maintains a list of observers
- When subject state changes, it iterates the list and calls each observer's `notify()` method
- Observers register/unregister themselves; subject never hard-codes them
- Decouples the "what changed" (subject) from "what happens next" (observers)

### When to Reach For It (Spot Triggers)
- "Notify many objects when one changes"
- "Pub-sub / event listeners / callbacks"
- "Keep views in sync with model" (classic MVC — View is Observer, Model is Subject)
- "Users subscribe to updates" (price drops, stock alerts, score tickers)
- "Decouple producer from consumers"

## Pseudocode

```
// Observer interface
interface Subscriber {
    notify(post: Post): void
}

// Subject
class Channel {
    private subscribers: List<Subscriber> = []
    private latestPost: Post

    subscribe(s: Subscriber) {
        subscribers.add(s)
    }

    unsubscribe(s: Subscriber) {
        subscribers.remove(s)
    }

    publish(post: Post) {
        this.latestPost = post
        for s in subscribers:
            s.notify(post)           // push model — sends data with notification
    }
}

// Concrete observers
class EmailNotifier implements Subscriber {
    notify(post: Post) { sendEmail(post.authorEmail, post.content) }
}

class PushNotifier implements Subscriber {
    notify(post: Post) { sendPush(post.userId, post.content) }
}

// Client
channel = new Channel()
channel.subscribe(new EmailNotifier())
channel.subscribe(new PushNotifier())
channel.publish(new Post("New drop!"))   // both notifiers fire automatically
```

## Push vs Pull Model

| Model | Mechanism | When to Use |
|---|---|---|
| **Push** | Subject sends data with the notification: `notify(data)` | Data is small and always relevant to observers |
| **Pull** | Subject sends no data: `notify()`; observer calls `subject.getState()` to fetch | Data is large or observers only need a subset; avoids over-sending |

**Rule:** Push is simpler and more common. Pull is better when different observers need different slices of state, or state is large.

---

## Thread-Safety Note

```
// Naive notify loop is NOT thread-safe if observers subscribe/unsubscribe concurrently
publish(post: Post) {
    // WRONG — iterating while another thread calls subscribe()
    for s in subscribers: s.notify(post)
}

// Safe approach — snapshot the list before iterating
publish(post: Post) {
    snapshot = new List(subscribers)   // copy under lock
    for s in snapshot: s.notify(post)  // iterate snapshot — mutations don't affect this loop
}
```

Key rule: always notify on a **snapshot** of the subscriber list, not the live list.

## Observer at Scale — Modern Equivalents

| Scale | Technology | Observer Concept |
|---|---|---|
| In-process (single JVM) | Java `EventListener`, Spring `ApplicationEvents` | Direct method call on observers |
| Microservices | Kafka topics, RabbitMQ exchanges | Consumers = observers; topic = subject |
| Cloud | AWS SNS + SQS, Google Pub/Sub | Subscribers = observers; SNS topic = subject |
| Frontend | DOM `addEventListener`, React state/effect | Components = observers; DOM/state = subject |

**Key insight:** Kafka, SNS, and DOM events are all the Observer pattern at different scales. The vocabulary changes, the structure doesn't.

## Don't Confuse

| Pattern | Structure | Key Difference |
|---|---|---|
| Observer | One subject broadcasts to many watchers; watchers don't know each other | Subject holds observer list; direct broadcast |
| Mediator | Many objects communicate through one central hub | Hub (mediator) holds all participants; routes messages between them |

**One-line test:** "Do observers know each other?" — if No, and subject broadcasts directly → Observer. If all communication is routed through one central object → Mediator.

## STAR Interview Framework

**Situation:** "Design a notification system for an e-commerce platform where users subscribe to product price drops. When a product's price changes, all subscribed users must be notified via their preferred channels (email, push, SMS)."

**Task:** "Build the notification module so adding a new channel (e.g., WhatsApp) requires zero changes to the product service, and user subscriptions are managed independently."

**Action:**
1. Define `PriceObserver` interface: `onPriceChange(product: Product, newPrice: Money): void`
2. `ProductService` (subject) maintains a `Map<productId, List<PriceObserver>>` for subscriptions
3. `EmailNotifier`, `PushNotifier`, `SMSNotifier` each implement `PriceObserver`
4. `subscribe(userId, productId, channel)` registers the appropriate notifier for that product
5. On `updatePrice(productId, newPrice)`: iterate the subscriber list for that product, call `notify()` on each — push model
6. Snapshot subscriber list before iterating for thread safety
7. To add WhatsApp: write `WhatsAppNotifier implements PriceObserver`, register it — `ProductService` unchanged

**Result:** Adding a new notification channel reduced from a 2-week sprint (modifying product service) to a single-class addition. Channels tested independently. Notification fan-out decoupled from price update latency.
