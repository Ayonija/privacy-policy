## DSA (1.5 hours)

### Pattern Revision: Linked List Reversal (Iterative) + Fast/Slow Pointer + Monotonic Stack

**Iterative List Reversal template (reproduce from memory before reading the problem):**

```java
// Iterative list reversal template
ListNode prev = null, curr = head;
while (curr != null) {
    ListNode next = curr.next;
    curr.next = prev;
    prev = curr;
    curr = next;
}
// prev = new head
```

**Fast/Slow Pointer template (reproduce from memory before reading the problem):**

```java
// Fast/Slow Pointer — find middle node
ListNode slow = head, fast = head;
while (fast != null && fast.next != null) {
    slow = slow.next;
    fast = fast.next.next;
}
// slow is now at middle (or just past middle for even-length lists)
```

**Monotonic Stack template (next greater element, reproduce from memory):**

```java
// Monotonic stack (next greater element) template
Deque<Integer> stack = new ArrayDeque<>();
int[] result = new int[nums.length];
Arrays.fill(result, -1);
for (int i = 0; i < nums.length; i++) {
    while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) result[stack.pop()] = nums[i];
    stack.push(i);
}
```

### Code Skeleton

```java
class Solution {

    // LC 25 — Reverse Nodes in K-Group (cold solve — delete this stub and rewrite from scratch)
    // TODO: ListNode reverseKGroup(ListNode head, int k)
    // Key:
    //   Step 1: Check if at least k nodes remain (walk k steps; if null before k: return head unchanged)
    //   Step 2: Reverse k nodes iteratively (prev=null, curr=head, repeat k times)
    //   Step 3: After reversing, original head is now the tail of this chunk.
    //           head.next = reverseKGroup(curr, k)  (curr is the start of the remaining list)
    //   Step 4: Return prev (new head of reversed chunk)
    public ListNode reverseKGroup(ListNode head, int k) {
        ListNode check = head;
        for (int i = 0; i < k; i++) {
            if (check == null) return head;   // fewer than k nodes remain
            check = check.next;
        }
        // Reverse k nodes
        ListNode prev = null, curr = head;
        for (int i = 0; i < k; i++) {
            ListNode next = curr.next;
            curr.next = prev;
            prev = curr;
            curr = next;
        }
        head.next = reverseKGroup(curr, k);   // head is now the tail of reversed chunk
        return prev;                           // prev is new head of reversed chunk
    }

    // LC 739 — Daily Temperatures (revision — write from memory)
    public int[] dailyTemperatures(int[] temperatures) {
        int n = temperatures.length;
        int[] result = new int[n];
        Deque<Integer> stack = new ArrayDeque<>();  // stores indices
        for (int i = 0; i < n; i++) {
            while (!stack.isEmpty() && temperatures[i] > temperatures[stack.peek()]) {
                int idx = stack.pop();
                result[idx] = i - idx;
            }
            stack.push(i);
        }
        return result;
    }

    // LC 19 — Remove Nth Node From End (OA timed — set 20-minute timer)
    // TODO: ListNode removeNthFromEnd(ListNode head, int n)
    // Key: dummy node before head simplifies edge case where head itself is removed
    //   fast = dummy; move fast n+1 steps forward
    //   move both slow and fast until fast == null
    //   slow.next = slow.next.next
    public ListNode removeNthFromEnd(ListNode head, int n) {
        ListNode dummy = new ListNode(0);
        dummy.next = head;
        ListNode fast = dummy, slow = dummy;
        for (int i = 0; i <= n; i++) fast = fast.next;   // n+1 steps ahead
        while (fast != null) { slow = slow.next; fast = fast.next; }
        slow.next = slow.next.next;
        return dummy.next;
    }
}
```

## System Design (30 min)

### Topic: Caching Strategies — Component Deep-Dive

#### Cache-Aside (Lazy Loading)

```
Read path:
  App → check cache → HIT: return cached value
                    → MISS: read DB → populate cache → return value

Write path:
  App → write DB → (optionally) invalidate or update cache key
```

- **Pro:** Only requested data is cached; cache is never polluted with unread data.
- **Con:** First read after a miss is slow (DB read + cache write). Cache may hold stale data if DB is written without invalidating cache.

#### Write-Through

```
Write path:
  App → write cache AND write DB synchronously → return success
```

- **Pro:** Cache and DB always consistent; reads after writes are always fast.
- **Con:** Every write pays double latency (cache + DB). Cache fills with data that may never be read.

#### Write-Behind (Write-Back)

```
Write path:
  App → write cache → return success (fast)
  Background process → async flush cache writes to DB
```

- **Pro:** Writes are very fast; batching reduces DB load.
- **Con:** Risk of data loss if cache crashes before flush. Complexity: need durable write log.

#### TTL and Eviction Policies

| Policy | Behavior | Use when |
|--------|----------|----------|
| LRU (Least Recently Used) | Evict the key that was accessed least recently | General-purpose; works well when recent access predicts future access |
| LFU (Least Frequently Used) | Evict the key accessed fewest times | Long-lived hot keys; avoids evicting items that are always popular |
| TTL-based | Expire keys after fixed time regardless of access | Data with known staleness window (session tokens, rate limit counters) |

**Redis eviction policies:**
- `allkeys-lru`: evict any key using LRU (use for general caches).
- `volatile-lru`: evict only TTL-tagged keys using LRU (use to protect keys without TTL).
- `noeviction`: reject writes when memory full (use for durable data where loss is unacceptable).

#### Cache Stampede (Thundering Herd) Prevention

**Problem:** A popular cache key expires. Simultaneously, hundreds of threads detect a miss and all query the DB. The DB is overwhelmed.

**Solution 1 — Distributed mutex (Redis `SET NX`):**

```
On cache miss:
  Try to acquire lock: SET lock:key 1 NX EX 5
  If lock acquired: read DB, populate cache, release lock
  If lock not acquired: spin-wait briefly, then read from cache (populated by winner)
```

**Solution 2 — Probabilistic early expiry:**
- Before TTL expires, some fraction of reads probabilistically trigger a refresh slightly early, spreading the regeneration cost over time.

**Interview talking point:** "If asked how to prevent a thundering herd when a popular cache key expires, answer: use a distributed mutex (Redis SET NX) so only one process regenerates; others wait and read the freshly populated key."

## Flashcards

| Q | A |
|---|---|
| In LC 25 Reverse Nodes in K-Group, after reversing k nodes, what does the original head node become and what should its `next` pointer be set to? | The original head becomes the tail of the reversed chunk. Its `next` should point to the result of recursively calling `reverseKGroup` on the remaining list starting at `curr`. |
| In Daily Temperatures (LC 739), what invariant does the monotonic stack maintain? | The stack always holds indices of days in decreasing order of temperature. When a warmer day is found at index i, all indices with temperature less than `temperatures[i]` are popped and their wait time is computed as `i - poppedIndex`. |
| Why do you move the fast pointer n+1 steps (not n) in LC 19 Remove Nth From End? | The slow pointer must stop at the node just before the one to delete. Moving fast n+1 steps creates a gap of n+1 between fast and slow, so when fast reaches null, slow is at the predecessor of the target node. |
| What is the key risk of write-behind (write-back) caching? | If the cache crashes before the async flush reaches the database, writes that were acknowledged to the client are permanently lost. Mitigation: use a durable write-ahead log or write-behind queue on persistent storage. |
| What is cache stampede (thundering herd) and what is the Redis-native solution? | When a hot cache key expires simultaneously for many clients, all miss and query the DB at once, overwhelming it. Redis solution: `SET lock:key 1 NX EX 5` — only one process wins the lock and regenerates the key; others wait and read the freshly populated value. |
