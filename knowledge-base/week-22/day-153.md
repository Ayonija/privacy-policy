## DSA (1.5 hours)

### Pattern Revision: Post-order DFS (Arm Pattern) + BFS Level-Order + BST Invariants + LCA

**Post-order DFS arm pattern template (reproduce from memory before reading the problem):**

```java
// Post-order DFS arm pattern
int maxResult = 0;
private int dfs(TreeNode node) {
    if (node == null) return 0;
    int left = dfs(node.left), right = dfs(node.right);
    maxResult = Math.max(maxResult, left + right); // combine arms at this node
    return 1 + Math.max(left, right);              // return best single arm upward
}
```

**BFS level-order template (reproduce from memory before reading the problem):**

```java
// BFS level order template
Queue<TreeNode> q = new LinkedList<>();
q.offer(root);
List<List<Integer>> result = new ArrayList<>();
while (!q.isEmpty()) {
    int size = q.size();
    List<Integer> level = new ArrayList<>();
    for (int i = 0; i < size; i++) {
        TreeNode node = q.poll();
        level.add(node.val);
        if (node.left != null) q.offer(node.left);
        if (node.right != null) q.offer(node.right);
    }
    result.add(level);
}
```

**LCA template (reproduce from memory before reading the problem):**

```java
// LCA (Lowest Common Ancestor) for binary tree
public TreeNode lowestCommonAncestor(TreeNode root, TreeNode p, TreeNode q) {
    if (root == null || root == p || root == q) return root;
    TreeNode left = lowestCommonAncestor(root.left, p, q);
    TreeNode right = lowestCommonAncestor(root.right, p, q);
    if (left != null && right != null) return root; // p and q are in different subtrees
    return left != null ? left : right;
}
```

### Code Skeleton

```java
class Solution {

    // LC 2246 — Longest Path With Different Adjacent Characters (cold solve — delete this stub and rewrite from scratch)
    // TODO: int longestPath(int[] parent, String s)
    // Key:
    //   Build adjacency list: for i in [1..n-1]: graph[parent[i]].add(i)
    //   DFS(node): iterate over children; only count child arm if s.charAt(child) != s.charAt(node)
    //     Collect valid child arms in a list; sort descending; take top 2
    //     Update global max = top1 + top2 + 1 (or just top1 + 1 if only one valid child)
    //     Return top1 + 1 (best arm from this node upward)
    int globalMax = 1;
    public int longestPath(int[] parent, String s) {
        int n = parent.length;
        List<List<Integer>> graph = new ArrayList<>();
        for (int i = 0; i < n; i++) graph.add(new ArrayList<>());
        for (int i = 1; i < n; i++) graph.get(parent[i]).add(i);
        dfs2246(0, graph, s);
        return globalMax;
    }
    private int dfs2246(int node, List<List<Integer>> graph, String s) {
        int top1 = 0, top2 = 0;
        for (int child : graph.get(node)) {
            int childArm = dfs2246(child, graph, s);
            if (s.charAt(child) == s.charAt(node)) continue; // skip same-character child
            if (childArm > top1) { top2 = top1; top1 = childArm; }
            else if (childArm > top2) top2 = childArm;
        }
        globalMax = Math.max(globalMax, top1 + top2 + 1);
        return top1 + 1;
    }

    // LC 543 — Diameter of Binary Tree (revision — write from memory)
    int diameter = 0;
    public int diameterOfBinaryTree(TreeNode root) {
        dfs543(root);
        return diameter;
    }
    private int dfs543(TreeNode node) {
        if (node == null) return 0;
        int left = dfs543(node.left), right = dfs543(node.right);
        diameter = Math.max(diameter, left + right);
        return 1 + Math.max(left, right);
    }

    // LC 102 — Binary Tree Level Order Traversal (OA timed — set 20-minute timer)
    // TODO: List<List<Integer>> levelOrder(TreeNode root)
    // Key: use BFS; snapshot size = q.size() at start of each level; drain exactly size nodes
    public List<List<Integer>> levelOrder(TreeNode root) {
        List<List<Integer>> result = new ArrayList<>();
        if (root == null) return result;
        Queue<TreeNode> q = new LinkedList<>();
        q.offer(root);
        while (!q.isEmpty()) {
            int size = q.size();
            List<Integer> level = new ArrayList<>();
            for (int i = 0; i < size; i++) {
                TreeNode node = q.poll();
                level.add(node.val);
                if (node.left != null) q.offer(node.left);
                if (node.right != null) q.offer(node.right);
            }
            result.add(level);
        }
        return result;
    }
}
```

## System Design (30 min)

### Topic: Database Design — SQL vs NoSQL, Sharding, Replication — Component Deep-Dive

#### SQL vs NoSQL Decision Framework

```
Decision tree — which database?

Need ACID transactions, JOINs, foreign key constraints?
  → SQL (PostgreSQL / MySQL)
  → Use for: user auth, billing, subscriptions, order management

Need horizontal write scale, flexible schema, high write throughput?
  → NoSQL
  → Cassandra: time-series, activity feeds, write-heavy (partition by known key)
  → DynamoDB: key-value + document, predictable single-digit ms latency
  → MongoDB: document store with rich query (but careful: not horizontally sharded by default)
```

**SQL limits:** Vertical scale tops out around 32 cores / 256 GB RAM on a single primary. Beyond that: read replicas for reads, sharding for writes.

**NoSQL (Cassandra/DynamoDB):** Horizontal scale by design. Partition key determines which node owns the data. No joins — denormalize data to query patterns.

#### Sharding Strategies

| Strategy | How it works | Pros | Cons |
|----------|-------------|------|------|
| Range sharding | Shard by key range (e.g., user_id 1–1M on shard 1) | Simple; range queries efficient | Hot spots if data is not uniformly distributed |
| Hash sharding | Hash(key) % num_shards → shard assignment | Even distribution regardless of key space | Range queries require hitting all shards |
| Directory-based | Lookup table maps each key to its shard | Flexible; can move data without rehashing | Single point of failure for lookup service; adds latency |

**Consistent hashing** (used in Cassandra/DynamoDB): Keys and nodes live on a ring; key routes to nearest clockwise node. Adding/removing nodes only remaps a fraction of keys.

#### Replication

```
Primary-Replica replication:
  Client writes → Primary → async replication → Replica 1, Replica 2
  Client reads (read-heavy) → Replica 1 or Replica 2

Replication lag: typically 10–100 ms async
Read replicas: help read-heavy load ONLY; do not help write throughput
Sharding: required when write throughput exceeds what one primary can handle (> ~10K writes/sec)
```

**Leader election when primary fails:** Raft (etcd, CockroachDB) or Paxos (Chubby/Zookeeper). Automatic failover in managed services (Amazon RDS Multi-AZ, Cassandra with gossip protocol).

**Cassandra replication factor:** RF=3 means 3 copies of each partition across nodes. QUORUM reads/writes require (RF/2)+1 = 2 responses → survives 1 node failure.

**Interview talking point:** "If asked when to shard vs just use a read replica, answer: shard when write throughput exceeds what one primary can handle (> ~10K writes/sec); read replicas only help read-heavy loads — they don't help if writes are the bottleneck."

## Flashcards

| Q | A |
|---|---|
| In the post-order DFS arm pattern, what is the difference between the value returned from `dfs(node)` and the value used to update `maxResult`? | `dfs(node)` returns `1 + max(left, right)` — the longest single arm from this node upward (for the parent to extend). `maxResult` is updated with `left + right` — the path that passes through this node as the peak, which cannot extend further upward. |
| In LC 2246, why do you skip a child arm if the child's character equals the parent's character? | The problem requires adjacent characters in the path to be different. A child with the same character as its parent cannot be part of the same path, so its arm length contribution is set to 0 (skipped entirely). |
| In BFS level-order traversal, why must you snapshot `int size = q.size()` at the start of each level loop? | Because you add new children to the queue inside the same loop. If you check `q.isEmpty()` or `q.size()` dynamically, you would process next-level nodes as part of the current level. Snapshotting size isolates exactly the current level. |
| When should you shard a database instead of just adding read replicas? | When write throughput exceeds what a single primary can handle (roughly > 10K writes/sec). Read replicas copy the primary's data and only help read load — they do not reduce write load on the primary. Sharding distributes writes across multiple primaries. |
| In Cassandra with replication factor 3 and QUORUM consistency, how many node failures can the cluster survive for reads and writes? | QUORUM requires (3/2)+1 = 2 responses. With 3 nodes and needing 2, the cluster can survive 1 node failure and still satisfy QUORUM reads and writes. |
