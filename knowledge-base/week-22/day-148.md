# Day 148 — Backtracking Hard: Grid & Trie-Augmented Backtracking
**Week 22 | Phase 3 — Mock Interviews | Month 5**

## Focus
Apply backtracking to grid problems and augment with a Trie data structure to prune invalid paths early — a critical optimization that transforms Word Search II from TLE to accepted.

---

## DSA (1.5 hours)

### Pattern: Grid Backtracking (Counting Paths with Obstacles)
- Core idea: DFS through a grid, visiting each cell exactly once (mark visited, recurse, unmark). For Unique Paths III, every non-obstacle cell must be visited exactly once before reaching the end. Track remaining cells to visit as a countdown.
- Trigger condition: "Count paths through a grid visiting every non-obstacle cell exactly once" or "find all paths with a specific property." The must-visit-all constraint makes DP optimal but DFS backtracking also works.
- Time complexity: O(4^(m×n)) worst case (each cell tries 4 directions) | Space complexity: O(m×n) recursion depth

### Pattern: Trie-Augmented Backtracking (Multi-Word Search)
- Core idea: Build a Trie from the word list. DFS through the grid; at each cell, traverse the Trie simultaneously. If the current Trie node has no child matching the current character → prune immediately (no word in the dictionary starts with this prefix). If the Trie node marks end of word → collect result, but continue DFS (other words may extend further). Remove found words from Trie to avoid duplicates.
- Trigger condition: "Find all words from a dictionary that exist as paths in a grid." Without Trie, you'd DFS separately per word — O(W × m × n × 4^L). With Trie, all words share a single DFS: O(m × n × 4^L × L).
- Time complexity: O(m × n × 4^L) where L = max word length | Space complexity: O(total characters in all words) for Trie

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Unique Paths III | 980 | Hard (Cold Solve) | Grid Backtracking | Count non-obstacle cells; DFS visits each exactly once; base case: reach end cell with `remaining == 0` |
| 2 | Word Search II | 212 | Hard (Cold Solve) | Trie-Augmented Backtracking | Build Trie from words; single DFS through grid; prune when Trie child doesn't exist; remove found words to avoid dups |
| 3 | Word Search | 79 | Medium (Revision) | Grid Backtracking | REVISION: DFS with in-place marking (`board[r][c] = '#'` to avoid re-visit); restore after DFS returns |

    //      3. When reaching end cell: if remaining == 0, count++; return.
    //      4. Explore 4 directions, skip out-of-bounds and obstacles (-1) and visited.

    // LC 212 — Word Search II (cold solve — delete this stub and rewrite from scratch)
    // TODO: List<String> findWords(char[][] board, String[] words)
    // Key: Build Trie from all words (each node: Map<Character, TrieNode> children, String word=null)
    //      DFS(r, c, trieNode):
    //        if out of bounds or visited or board[r][c] not in trieNode.children: return
    //        next = trieNode.children.get(board[r][c])
    //        if next.word != null: result.add(next.word); next.word = null (avoid duplicate)
    //        mark visited (board[r][c] = '#')
    //        DFS all 4 directions
    //        restore board[r][c]
    //      Optimization: prune Trie nodes with no children (remove leaf nodes after collecting word)

    // LC 79 — Word Search (revision — write from memory)
    public boolean exist(char[][] board, String word) {
        int m = board.length, n = board[0].length;
        for (int r = 0; r < m; r++)
            for (int c = 0; c < n; c++)
                if (dfs(board, word, r, c, 0)) return true;
        return false;
    }
    private boolean dfs(char[][] board, String word, int r, int c, int idx) {
        if (idx == word.length()) return true;
        if (r < 0 || r >= board.length || c < 0 || c >= board[0].length) return false;
        if (board[r][c] != word.charAt(idx)) return false;
        char temp = board[r][c];
        board[r][c] = '#';
        boolean found = dfs(board, word, r+1, c, idx+1) || dfs(board, word, r-1, c, idx+1)
                     || dfs(board, word, r, c+1, idx+1) || dfs(board, word, r, c-1, idx+1);
        board[r][c] = temp;
        return found;
    }
}

#### Data Flow Patterns by Use Case

| Use Case | Flow |
|----------|------|
| Upload | Client → API GW → Upload Svc → S3 → Kafka → Transcode Workers → CDN Origin |
| Playback | Client → API GW → Streaming Svc → Signed CDN URL → CDN Edge (segments) |
| Search | Client → API GW → Search Svc → Elasticsearch → Ranking ML → Response |
| Recommendation | Client → API GW → Reco Svc → Redis (embeddings) → Faiss ANN → Ranking → Response |
| Comment | Client → API GW → Social Svc → Cassandra (write) → Kafka (async notif) |
| Like | Client → API GW → Social Svc → Kafka → Redis INCR → Cassandra (batch flush) |
| Notification | Kafka event → Notif Svc → FCM/APNs + Cassandra feed |

#### Key Trade-offs to Articulate

| Decision | Choice | Reasoning |
|----------|--------|-----------|
| Video metadata storage | Cassandra | Write-heavy, partition by video_id, eventual consistency OK |
| User account data | MySQL/Postgres | Strong consistency needed for auth/billing/subscriptions |
| View counts | Redis + Kafka batch | High write throughput; exact count not needed in real-time |
| CDN caching | Immutable segments + short-TTL manifests | Segments never change; manifests must reflect current segments |
| Search index | Elasticsearch | Full-text, tag search, faceting; BM25 + engagement scoring |
| Fan-out for feeds | Hybrid push/pull | Pure push breaks for celebrity channels |
| Transcode | Asynchronous via Kafka | Raw encoding takes hours; cannot block HTTP response |

**Interview talking point:** "If asked to walk through the end-to-end architecture in 5 minutes, answer: Start with upload (chunked → S3 → async transcode → CDN origin), then streaming (CDN edge → HLS/DASH → ABR client), then discovery (search via Elasticsearch, recommendations via two-tower + ANN), then engagement (comments in Cassandra, likes via Kafka, notifications via FCM). Database choices: Cassandra for write-heavy video data, MySQL for user accounts, Redis for hot data, Elasticsearch for search. All services communicate async via Kafka for decoupling."

## Flashcards

| Q | A |
|---|---|
| How do you track visited cells in Unique Paths III without a separate visited array? | Modify the grid in-place: set `grid[r][c] = -1` (or another sentinel) when visiting, restore to original value when backtracking. |
| Why does Trie-augmented backtracking make Word Search II efficient? | All words share a single DFS pass. The Trie prunes paths early: if no word has the current prefix, that DFS branch is abandoned immediately without exploring further. |
| What does removing a found word from the Trie accomplish? | Prevents the same word from being added to the result multiple times if it appears at multiple starting positions in the grid. |
| In the full YouTube HLD, why is Cassandra chosen for video metadata over MySQL? | Video metadata is write-heavy (new uploads, view count updates, comment writes) and can tolerate eventual consistency. Cassandra scales writes horizontally by partitioning on video_id. |
| What are the two CDN caching TTL strategies for video content and why? | Encoded segments: 1-year TTL (immutable; URL includes content hash). Manifest files: 5–30 seconds TTL (must reflect current segment list after live/new uploads). |
