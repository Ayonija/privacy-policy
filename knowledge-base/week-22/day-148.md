# Day 148 — Command + Template Method (Behavioral)
**Week 22 | Phase: LLD Mastery | Month 4**

## Focus
Two behavioral patterns that frequently appear together in practice. Command turns a request into a standalone object — enabling undo, queuing, and logging. Template Method fixes an algorithm's skeleton in a base class and lets subclasses fill in specific steps. Both address "how do I control the shape of an operation" — Command defers and reverses it, Template Method locks the sequence and varies the details.

---

## Part 1 — Command Pattern

### Analogy
A restaurant order slip (parchi): the waiter writes down your order, hands it to the kitchen, and the kitchen executes it later. The slip is the command — it captures the request as an object. It can be queued, handed to a different cook, or cancelled before execution.

### What It Does
- Turns a request into a standalone object (`Command`) with `execute()` and optionally `undo()`
- Decouples the object that **issues** the request (Invoker) from the object that **fulfills** it (Receiver)
- Enables: undo/redo (history stack), queuing (task queue), logging (audit trail), retry

### Components

| Role | Description | Example |
|---|---|---|
| Command (interface) | Declares `execute()` and `undo()` | `EditCommand` |
| ConcreteCommand | Implements execute/undo; holds reference to Receiver | `PasteCommand` |
| Receiver | Knows how to actually perform the action | `Document` |
| Invoker | Holds and executes commands; maintains history | Editor's history stack |
| Client | Creates ConcreteCommand and wires Receiver | Application startup |

### Pseudocode

```
// Command interface
interface Command {
    execute(): void
    undo(): void
}

// Receiver — knows how to do the actual work
class Document {
    private content: String = ""

    paste(text: String) { content += text }
    deleteLast(n: int) { content = content.substring(0, content.length - n) }
    getContent(): String { return content }
}

// Concrete Command
class PasteCommand implements Command {
    private doc: Document
    private text: String

    constructor(doc: Document, text: String) {
        this.doc = doc
        this.text = text
    }

    execute() { doc.paste(text) }
    undo() { doc.deleteLast(text.length) }
}

// Invoker — holds history stack
class EditorHistory {
    private history: Stack<Command> = new Stack()

    executeCommand(cmd: Command) {
        cmd.execute()
        history.push(cmd)
    }

    undo() {
        if (!history.isEmpty()) {
            history.pop().undo()
        }
    }
}

// Client
doc = new Document()
history = new EditorHistory()

cmd = new PasteCommand(doc, "Hello World")
history.executeCommand(cmd)   // doc: "Hello World"
history.undo()                 // doc: "" — PasteCommand.undo() reverses it
```

### Spot Triggers (Command)
- "Support undo/redo"
- "Queue or schedule requests" (task queue, job scheduler)
- "Decouple the caller from the doer"
- "Log and replay operations" (DB transaction log, audit trail)

### Real-World Uses (Command)

| Context | Command | Receiver |
|---|---|---|
| Text editor | `PasteCommand`, `DeleteCommand` | `Document` |
| DB transactions | `InsertCommand`, `UpdateCommand` | DB engine — rollback via `undo()` |
| Remote controls | `TurnOnCommand`, `TurnOffCommand` | `TV`, `Fan`, `Light` |
| Task queues | `SendEmailCommand`, `ResizeImageCommand` | Email service, image service |
| GUI buttons | Action bound to button click | Any backend service |

### Don't Confuse: Command vs Strategy
- **Command** wraps a **request** as an object — the key capability is `undo()` and queuing. Has a receiver.
- **Strategy** wraps an **algorithm choice** — no undo, no receiver. Just `run(data)`.
- Test: "Do I need to reverse or queue it?" → Command. "Do I need to swap which algorithm runs?" → Strategy.

## Part 2 — Template Method Pattern

### Analogy
Maggi recipe: boil water, add noodles, add masala, cook 2 minutes. The steps are fixed. The only variable step is "add masala" — you can swap in a spicy or mild version. The recipe (base class) fixes the sequence; you override only what varies.

### What It Does
- Base class defines the **algorithm skeleton** in a `final` template method
- Abstract steps are declared in the base class, implemented by subclasses
- The skeleton cannot be reordered by subclasses — only specific steps are overridden
- Avoids duplicating the shared skeleton across subclasses

### Pseudocode

```
// Abstract base class — defines the skeleton
abstract class Report {
    // Template method — fixed skeleton, NOT overridable
    final generate() {
        data = fetch()
        formatted = format(data)
        export(formatted)
    }

    // Steps subclasses must implement
    abstract fetch(): Data
    abstract format(data: Data): FormattedData

    // Hook method — optional override (has a default)
    export(formatted: FormattedData) {
        print(formatted)   // default behavior
    }
}

// Concrete subclasses — override only the varying steps
class PdfReport extends Report {
    fetch(): Data { return database.query("SELECT ...") }
    format(data: Data): FormattedData { return pdfRenderer.render(data) }
    export(formatted: FormattedData) { pdfLib.save(formatted, "report.pdf") }
}

class CsvReport extends Report {
    fetch(): Data { return database.query("SELECT ...") }
    format(data: Data): FormattedData { return csvSerializer.serialize(data) }
    // export() not overridden — uses base class print()
}

// Client
new PdfReport().generate()   // fetch → format → export (PDF)
new CsvReport().generate()   // fetch → format → export (print)
```

### Hook Methods
A **hook** is an optional override point in the base class that has a default implementation. Subclasses CAN override it but don't have to. Example: `export()` above — `CsvReport` skips it, `PdfReport` overrides it. Hooks prevent the "must implement everything" burden of abstract methods.

### Spot Triggers (Template Method)
- "Fixed sequence where some steps vary by type"
- "Same workflow, different details per variant"
- "Avoid duplicating the algorithm skeleton across subclasses"
- "Framework hooks" (JUnit `setUp/tearDown`, Spring `AbstractController`)

### Real-World Uses (Template Method)

| Context | Template Method | Variable Steps |
|---|---|---|
| Data parsers | `parse()` = read → tokenize → validate → store | `tokenize()` varies by format |
| Report generators | `generate()` = fetch → format → export | `format()` + `export()` vary |
| JUnit | Test lifecycle = setUp → test → tearDown | `setUp()` and `tearDown()` |
| Game turn loops | `takeTurn()` = draw card → play → score → end turn | `play()` varies per player type |
| Spring `AbstractController` | `handleRequest()` skeleton | Subclass fills in business logic |

### Don't Confuse: Template Method vs Strategy

| Dimension | Template Method | Strategy |
|---|---|---|
| Mechanism | Inheritance — subclass fills steps | Composition — algorithm injected |
| When decided | Compile-time (subclass chosen at instantiation) | Runtime (strategy swapped via setter) |
| Flexibility | Steps are fixed; skeleton cannot reorder | Whole algorithm swappable |
| Use when | Steps are logically part of one algorithm | Multiple completely independent algorithms |

**One-line test:** "Is the variation a step inside a fixed process?" → Template Method. "Is the variation the entire algorithm?" → Strategy.

## STAR Interview Framework

**Situation:** "Design an export system that generates reports in PDF and CSV formats. Both formats share the same pipeline: fetch data from the DB, format it, then output it. The fetch logic is identical; only formatting and output differ."

**Task:** "Eliminate duplication of the pipeline while keeping each format's specifics isolated and testable."

**Action:**
1. Define abstract `Report` with a `final generate()` template method: `fetch()` → `format()` → `export()`
2. Implement `fetch()` once in the base class (shared DB query — no duplication)
3. Declare `format()` and `export()` as abstract — each subclass implements its own
4. `PdfReport` overrides `format()` with PDF rendering and `export()` with file save
5. `CsvReport` overrides `format()` with CSV serialization; uses default `export()` (print)
6. Adding `ExcelReport`: write one subclass, override `format()` — skeleton and `fetch()` are untouched

**Result:** Zero duplication of the pipeline. Each format class is ~40 lines. Adding Excel export took 30 minutes vs the previous approach of copy-pasting a 200-line export function and risk of divergence.
