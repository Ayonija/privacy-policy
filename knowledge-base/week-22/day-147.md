# Day 147 — State Pattern (Behavioral)
**Week 22 | Phase: LLD Mastery | Month 4**

## Focus
The State pattern lets an object alter its behavior when its internal state changes — the object appears to switch classes. It replaces giant if-else or switch-on-status chains with a clean state machine where each state knows its own transitions and valid actions. Master it for any interview question involving lifecycle states, workflow engines, or status-driven behavior.

---

## Core Concept

### Analogy
A traffic signal: same physical object, but red means stop, green means go, yellow means slow down. The signal doesn't need a `switch(color)` block everywhere — each color (state) knows its own behavior and which color comes next.

### What It Does
- Object holds a `state` reference instead of a status field
- All behavior for a given status lives inside the corresponding State class
- State transitions are encapsulated inside the State objects — each state knows `next()`
- Eliminates the `if status == PLACED then ... else if status == SHIPPED then ...` sprawl

### When to Reach For It (Spot Triggers)
- Same object has 3+ statuses, each with different behavior
- Behavior depends entirely on current status: `if (status == X) { ... }`
- Status transitions form a defined graph (state machine)
- "Object changes behavior as its lifecycle progresses"

## Pseudocode

```
// State interface
interface OrderState {
    next(order: Order): void
    cancel(order: Order): void
    getLabel(): String
}

// Concrete states
class PlacedState implements OrderState {
    next(order: Order) { order.setState(new ShippedState()) }
    cancel(order: Order) { order.setState(new CancelledState()) }
    getLabel(): String { return "PLACED" }
}

class ShippedState implements OrderState {
    next(order: Order) { order.setState(new DeliveredState()) }
    cancel(order: Order) { throw new IllegalStateException("Cannot cancel — already shipped") }
    getLabel(): String { return "SHIPPED" }
}

class DeliveredState implements OrderState {
    next(order: Order) { /* terminal — no further transition */ }
    cancel(order: Order) { order.setState(new ReturnedState()) }
    getLabel(): String { return "DELIVERED" }
}

// Context — holds current state
class Order {
    private state: OrderState = new PlacedState()

    setState(s: OrderState) { this.state = s }
    advance() { state.next(this) }
    cancel() { state.cancel(this) }
    getStatus(): String { return state.getLabel() }
}

// Client — just calls actions, never touches state logic
order = new Order()
order.advance()   // PLACED → SHIPPED
order.advance()   // SHIPPED → DELIVERED
order.cancel()    // DELIVERED → RETURNED
```

### Order Lifecycle
```
[PLACED] --advance()--> [SHIPPED] --advance()--> [DELIVERED] --cancel()--> [RETURNED]
   |                                                                            |
cancel()                                                              (terminal state)
   |
[CANCELLED]

  |                           |
  |                       refund()
  |                           |
  +---------------------------+  (back to IDLE)
```

## State vs Giant If-Else — Comparison

```
// Without State pattern — code that grows unboundedly
class Order {
    status: String

    advance() {
        if (status == "PLACED") { status = "SHIPPED" }
        else if (status == "SHIPPED") { status = "DELIVERED" }
        else if (status == "DELIVERED") { /* nothing */ }
        // add RETURN_REQUESTED: touch this method, touch cancel(), touch getLabel()...
    }

    cancel() {
        if (status == "PLACED") { status = "CANCELLED" }
        else if (status == "SHIPPED") { throw new Error("...") }
        else if (status == "DELIVERED") { status = "RETURNED" }
        // repeat for every new status...
    }
}
```

**With State:** adding `RETURN_REQUESTED` = write one new class. The existing state classes and `Order` are unchanged.

## Real-World Uses

| Domain | States |
|---|---|
| Order lifecycle | PLACED → SHIPPED → DELIVERED → RETURNED |
| Vending machine | IDLE → HAS_MONEY → DISPENSING → IDLE |
| TCP connection | LISTEN → SYN_SENT → SYN_RECEIVED → ESTABLISHED → CLOSE_WAIT → CLOSED |
| Game character | IDLE → RUNNING → JUMPING → ATTACKING → DEAD |
| Loan application | SUBMITTED → UNDER_REVIEW → APPROVED → DISBURSED → CLOSED |
| Document workflow | DRAFT → REVIEW → APPROVED → PUBLISHED |

## Don't Confuse

| Pattern | Who triggers the change? | How? |
|---|---|---|
| State | The **object itself** transitions based on its own lifecycle | `state.next(this)` called internally; caller just calls `order.advance()` |
| Strategy | The **external caller** swaps the algorithm | Caller calls `context.setStrategy(new X())` explicitly |

**One-line test:** "Who makes the switch?" — if the object manages its own transitions → State. If the caller picks and injects the behavior → Strategy.

## STAR Interview Framework

**Situation:** "Design a vending machine that handles idle, money-inserted, and dispensing states. Each state has different valid actions: in idle you can insert money but not select; in has-money you can select or refund; in dispensing you wait for the product to release before accepting new input."

**Task:** "Model the state machine so that adding a new state (e.g., OUT_OF_STOCK) requires zero changes to existing state classes."

**Action:**
1. Define `VendingMachineState` interface: `insertMoney()`, `selectProduct(id)`, `dispense()`, `refund()`
2. Implement `IdleState`, `HasMoneyState`, `DispensingState` — each handles its valid actions and throws on invalid ones
3. `VendingMachine` holds `currentState`; all public methods delegate to `currentState.method(this)`
4. State transitions: `IdleState.insertMoney()` → `machine.setState(new HasMoneyState(amount))`
5. `HasMoneyState.selectProduct()` validates inventory, calculates change, transitions to `DispensingState`
6. `DispensingState.dispense()` releases product, returns change, transitions back to `IdleState`
7. Add `OutOfStockState`: one new class; `HasMoneyState.selectProduct()` checks stock and transitions there if empty — no other classes change

**Result:** Each state is testable in isolation. Adding out-of-stock handling took one class + one transition check — compared to modifying 4 methods in a monolithic switch block. Bug surface per state reduced to its own class.
