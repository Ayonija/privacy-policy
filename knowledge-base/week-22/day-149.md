## DSA (1.5 hours)

### Pattern: Exhaustive Expression Evaluation Backtracking
- Core idea: 24 Game — for any subset of N numbers, try every permutation and every combination of operators (+, -, ×, ÷) between them. Since parenthesization matters, represent intermediate results as fractions (avoid floating-point precision issues) and recursively combine two numbers at a time. When only one number remains, check if it equals 24.
- Trigger condition: "Given N numbers, can you form a target value using arithmetic operators and any parenthesization?" The key insight is reducing the problem: pick any two numbers, combine them with any operator, and recurse on the resulting N-1 numbers.
- Time complexity: O(N! × 4^N) — N! permutations × 4 operators per pair, but N ≤ 4 so the constant is bounded | Space complexity: O(N) recursion depth

### Pattern: Trie-Constrained Grid Partition (Word Squares)
- Core idea: Build a Trie from the word list. Use the constraint that word[i][j] == word[j][i] (the i-th character of the j-th word equals the j-th character of the i-th word) to prune. When adding the k-th word to the square, its prefix is fully determined by column k of the words already placed. Use Trie prefix lookup to find all words starting with that prefix.
- Trigger condition: "Build an N×N grid where each row and column spells a valid word from the dictionary." The symmetry constraint `square[i][j] = square[j][i]` allows aggressive Trie-based pruning.
- Time complexity: O(W × L × 26^L) worst case with Trie pruning | Space complexity: O(W × L) for Trie

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | 24 Game | 679 | Hard (Cold Solve) | Exhaustive Expression Evaluation | Pick any 2 numbers, apply all 4 operators (and reversed operands for - and /), recurse on N-1 numbers; use double with epsilon comparison |
| 2 | Word Squares | 425 | Hard (Cold Solve) | Trie-Constrained Grid Partition | `square[i][j] = square[j][i]`; prefix for k-th word = column k of placed words; Trie lookup narrows candidates |
| 3 | Combination Sum | 39 | Medium (Revision) | Combination Backtracking | REVISION: Allow repeating same element; start next recursion from `i` (not `i+1`); subtract from target each level |

    // LC 39 — Combination Sum (revision — write from memory)
    public List<List<Integer>> combinationSum(int[] candidates, int target) {
        List<List<Integer>> result = new ArrayList<>();
        backtrack(candidates, target, 0, new ArrayList<>(), result);
        return result;
    }
    private void backtrack(int[] candidates, int remain, int start, List<Integer> path, List<List<Integer>> result) {
        if (remain == 0) { result.add(new ArrayList<>(path)); return; }
        for (int i = start; i < candidates.length; i++) {
            if (candidates[i] > remain) continue;
            path.add(candidates[i]);
            backtrack(candidates, remain - candidates[i], i, path, result);
            path.remove(path.size() - 1);
        }
    }
}

#### MOCK PROMPT
"Design a video streaming platform like YouTube. Users can upload, search, and stream videos. The platform serves 2 billion users and streams 1 billion hours of video per day."

---

#### STEP-BY-STEP ANSWER FRAMEWORK (45 minutes)

**Minutes 0–5: Clarify Requirements**

*Functional:*
- Users can upload videos (chunked, any size)
- Users can search by title/tags/transcript
- Users can stream videos with adaptive quality
- Users can subscribe to channels and see feeds
- Users can comment, like, share

*Non-Functional:*
- Availability: 99.99% (video streaming must not go down)
- Latency: < 2s time-to-first-frame globally
- Durability: uploaded video never lost
- Scale: 2B users, 500 hrs uploaded/min, 1B hrs streamed/day

*Out of scope:* live streaming, monetization, ad system (unless asked)

**Minutes 5–10: Scale Estimates**
```
Upload:   500 hrs/min = 8.3 hrs/sec ≈ 30 GB/sec raw video input
Transcode: each upload → 15-20 renditions, ~10-50 CPU-hours total
Storage:  1 hr at avg 1 GB/hr = 500 GB/min new storage = 720 TB/day
Streaming: 1B hrs/day = 1B × 3600 sec × avg 1 Mbps / 8 = 450 PB/day egress
Read:write ratio: 10,000:1 (streaming >> uploading)
Peak concurrent streams: ~10-15M at prime time
```

**Minutes 10–20: Core Architecture (Draw and Explain)**

Explain the three major paths:

*Path 1: Upload*
```
Client → API Gateway → Upload Service
→ Chunked write to S3 (presigned URLs, client uploads directly)
→ Kafka: video.uploaded event
→ Transcode Coordinator → 15-20 FFmpeg workers
→ CDN Origin (encoded segments + HLS manifests)
→ Metadata Service: mark video PUBLISHED
```

*Path 2: Streaming*
```
Client opens video → API Gateway → Streaming Service
→ Returns signed CDN URL for manifest.m3u8
→ Client fetches manifest from CDN edge (CloudFront / Netflix OCA)
→ Client downloads segments (6-second HLS chunks)
→ ABR algorithm adjusts quality every 30s based on bandwidth
→ 95%+ CDN cache hit rate; origin served only on miss
```

*Path 3: Discovery*
```
Search: Client → Search Service → Elasticsearch (BM25 + engagement score)
Recommendation: Client → Reco Service → Redis (user embeddings)
  → Faiss ANN → top 500 candidates → Ranking model → top 20
  → Refresh daily (batch training) + hourly (incremental)
```

**Minutes 20–30: Deep Dive (Pick 1–2 areas)**

*Interviewer pick: "Deep dive on the transcoding pipeline."*
- Chunked S3 multipart upload → S3 triggers Kafka message.
- Transcode Coordinator: fan out 15-20 worker tasks (one per rendition) to SQS.
- Workers run FFmpeg; output segments to S3 CDN origin.
- Segment naming: `video_id/quality/seg_NNNN.ts` (immutable, content-addressed).
- Status machine: QUEUED → TRANSCODING → QC_CHECK → PUBLISHED / FAILED.
- Retries: max 3, then dead-letter queue for manual intervention.
- ETA: 1 hour of 4K video ≈ 2-8 CPU-hours → use spot/preemptible instances.

*Interviewer pick: "How do you handle trending?"*
- Kafka ingest: 1M view events/sec peak.
- Flink stream processor: 1-minute tumbling windows per video_id.
- Redis ZINCRBY: sliding window buckets (1 bucket per hour, 6 buckets ≈ 6-hour window).
- ZUNIONSTORE for top-K trending page.
- Refresh trending page every 60 seconds.

**Minutes 30–40: Trade-offs and Failure Modes**

| Trade-off | Decision | Why |
|-----------|----------|-----|
| Cassandra vs. MySQL for video metadata | Cassandra | Write-heavy, partition-friendly, tunable consistency |
| Fan-out push vs. pull for subscription feeds | Hybrid | Push for small channels, pull for celebrities |
| Exact vs. approximate view counts | Approximate (Redis) | True exact count at 1B events/day is too expensive in real-time |
| HLS vs. DASH | Both (HLS for Apple, DASH for others) | Browser/device support requirements |
| Single origin vs. multi-region | Multi-region origin + CDN | Latency; single origin is bottleneck at global scale |

**Failure scenarios:**
- CDN POP down → clients retry next closest POP (DNS failover).
- Transcode worker dies mid-job → SQS message redelivered (visibility timeout); retry.
- Cassandra node down → replica reads (replication factor = 3, QUORUM consistency).
- Kafka lag spike → add consumers; Kafka retains messages for 7 days.

**Minutes 40–45: Numbers to Memorize for This Interview**

Say these aloud without looking:
- 2B logged-in YouTube users
- 500 hours uploaded per minute
- 1B hours watched per day
- 15–20 transcoded renditions per video
- Netflix OCA: 17,000+ appliances in ISPs worldwide
- ABR: 6-second segments, bandwidth probe every 30s
- Recommendation: 75-80% of Netflix watches come from recommendations
- Cache hit rate: 95%+ on CDN edge
- TTFF target: < 2 seconds

**Interview talking point:** "If asked to design YouTube from scratch in 45 minutes, lead with: requirements → scale math → three core paths (upload, streaming, discovery) → one deep dive → trade-off table. State scale numbers fluently — 500 hrs/min upload and 1B hrs/day streaming anchor every architectural decision."

## Flashcards

| Q | A |
|---|---|
| How do you handle the 24 Game recursion when two numbers remain? | Pick any 2 remaining numbers, apply all operators (a+b, a-b, b-a, a×b, a÷b, b÷a where divisor≠0); add result to list; recurse on list of 1. If result equals 24 (within 1e-6), return true. |
| What symmetry constraint makes Word Squares prunable? | `square[i][j] == square[j][i]` — the k-th character of the j-th word equals the j-th character of the k-th word. This means the prefix of each new word is determined by the column of already-placed words. |
| What is the Combination Sum backtracking rule to allow element reuse? | When recursing after picking `candidates[i]`, pass `i` (not `i+1`) as the start index. This allows picking the same element again. |
| In the YouTube mock interview, what three paths should you describe first? | Upload path (client → S3 → transcode → CDN origin), Streaming path (client → CDN edge → ABR), Discovery path (search via Elasticsearch, recommendations via two-tower). |
| Why use epsilon comparison (`< 1e-6`) instead of `== 24` in the 24 Game? | Floating-point arithmetic on doubles accumulates rounding errors. Division especially: `8 / (3 - 8/3) = 24.000000000000004`. Epsilon comparison handles this correctly. |
