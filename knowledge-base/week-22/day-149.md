## Mix-Up 1: Strategy vs State (Most Common)

### The Core Confusion
Both have a context that holds an interchangeable behavior object. The structural code looks similar. The conceptual difference is decisive.

### The Diagnostic Test
**"Who controls the switch?"**
- External caller makes the switch → **Strategy**
- Object manages its own transitions → **State**

### Side-by-Side Pseudocode

```
// STRATEGY — caller injects the algorithm from outside
class Sorter {
    private strategy: SortStrategy    // injected by caller

    setStrategy(s: SortStrategy) { this.strategy = s }
    sort(data) { return strategy.sort(data) }
}

// Caller decides:
sorter.setStrategy(new QuickSort())     // explicit external injection
sorter.sort(data)

sorter.setStrategy(new MergeSort())     // caller switches it again
sorter.sort(data)
// Sorter has NO IDEA which it'll get — it never calls setStrategy() on itself
```

```
// STATE — object manages its own transitions
class Order {
    private state: OrderState = new PlacedState()

    setState(s: OrderState) { this.state = s }   // called by states, not by caller
    advance() { state.next(this) }               // caller just says "go forward"
}

class PlacedState implements OrderState {
    next(order: Order) {
        order.setState(new ShippedState())   // STATE transitions itself — caller never does this
    }
}
// Caller:
order.advance()   // no knowledge of which state comes next
```

### Structural Tell

| Dimension | Strategy | State |
|---|---|---|
| Who calls `setX()` | External caller explicitly | State classes call `context.setState()` internally |
| Context knows next behavior? | No — whatever caller injects | No — but state objects decide |
| Transitions defined where? | Not applicable — caller chooses | Inside state classes themselves |

## Mix-Up 2: Observer vs Mediator

### The Core Confusion
Both decouple communicating objects. Observer is one-to-many broadcast. Mediator is many-to-many through a hub.

### The Diagnostic Test
**"Do the participants know each other at all?"**
- Subject broadcasts; watchers don't interact with each other → **Observer**
- All participants communicate only through one central hub; none speak directly → **Mediator**

### Observer Recap
```
// Subject broadcasts; observers are passive receivers
class ProductService {
    private observers: List<PriceObserver>
    updatePrice(newPrice) {
        for o in observers: o.onPriceChange(newPrice)   // broadcast
    }
}
// Observers know nothing about each other
```

### Mediator Pseudocode
```
// Mediator — all communication routed through here
interface ChatMediator {
    sendMessage(from: User, message: String): void
    register(user: User): void
}

class ChatRoom implements ChatMediator {
    private users: List<User>

    register(user: User) { users.add(user) }

    sendMessage(from: User, message: String) {
        for u in users:
            if u != from:
                u.receive(message)   // mediator routes; users never call each other directly
    }
}

class User {
    private mediator: ChatMediator
    send(message: String) { mediator.sendMessage(this, message) }   // goes through hub
    receive(message: String) { /* display */ }
}
```

**Mediator analogy:** Air traffic control. Planes don't talk to each other — all communication goes through the ATC tower. The tower knows all planes; planes only know the tower.

### Structural Tell

| Dimension | Observer | Mediator |
|---|---|---|
| Subject knows observers? | Knows the interface, not the concrete type | Mediator knows all participants by name |
| Observers know each other? | No | No — but they all know the mediator |
| Communication direction | One-way broadcast (subject → observers) | Bidirectional routing (any → mediator → any) |
| Typical use | Notification, event listeners, pub-sub | Chat rooms, GUI coordination, flight control |

## Mix-Up 3: Command vs Strategy

### The Diagnostic Test
**"Do I need to reverse it or queue it?"**
- Yes → **Command** (has `undo()`, has a Receiver, sits in a history stack)
- No, just swap the algorithm → **Strategy** (has only `run(data)`, no receiver, no history)

### Side-by-Side

```
// COMMAND — wraps a request, has undo
class PasteCommand {
    private doc: Document          // holds a Receiver
    execute() { doc.paste(text) }
    undo() { doc.deleteLast(n) }   // can reverse
}
history.push(cmd)   // can be queued and stored
```

```
// STRATEGY — wraps an algorithm choice
class QuickSort implements SortStrategy {
    sort(data): data { /* algorithm */ }   // no receiver, no undo
}
sorter.setStrategy(new QuickSort())   // just plugged in
```

## Mix-Up 4: Decorator vs Template Method

### The Diagnostic Test
**"Compile-time structure or runtime wrapping?"**
- Runtime — wrap objects dynamically, stack multiple wrappers → **Decorator**
- Compile-time — fixed sequence, subclass overrides steps → **Template Method**

| Dimension | Decorator | Template Method |
|---|---|---|
| Mechanism | Composition, wraps at runtime | Inheritance, subclass fills steps |
| Stackable? | Yes — wrap A in B in C | No — one subclass per variant |
| When variation decided | Runtime | At instantiation (class chosen) |
| What varies | Additional behavior wrapped around core | Specific steps inside a fixed skeleton |

## The Full Confusion-Killer Table

| Pair | Pattern A | Pattern B | One-Line Tell |
|---|---|---|---|
| Strategy vs State | External caller injects algo | Object transitions itself | "Who switches?" — caller = Strategy, self = State |
| Observer vs Mediator | Subject broadcasts to passive watchers | All route through central hub | "Do participants talk to each other?" — no direct talk = Mediator |
| Command vs Strategy | Wraps a request (has undo/queue/receiver) | Wraps an algorithm choice (no undo) | "Need undo/queue?" → Command. "Need to swap algo?" → Strategy |
| Decorator vs Template Method | Runtime composition, stackable wrappers | Compile-time inheritance, fixed skeleton | "Runtime wrapping?" → Decorator. "Fixed sequence, vary steps?" → Template Method |
| Decorator vs Proxy | Adds behavior (enhancement) | Controls access (protection/lazy-load) | "Adding?" → Decorator. "Controlling?" → Proxy |

## STAR Interview Framework

**Situation:** "Walk me through how you'd distinguish Strategy from State in a real system. The interviewer is specifically testing whether you know when to use each one."

**Task:** "Give a precise structural explanation, not just a verbal definition. Show the code difference."

**Action:**
1. Start with the diagnostic: "The tell is who controls the switch."
2. Strategy example: Payment checkout — the user selects UPI, the checkout UI calls `paymentService.setStrategy(new UPIStrategy())`. The service never decides its own payment method.
3. State example: Order processing — `order.advance()` is called. Inside, `PlacedState.next()` calls `order.setState(new ShippedState())`. The caller never knows which state comes next — the order manages its own lifecycle.
4. Structural difference: In Strategy, the context's `setX()` is called by external client code. In State, the context's `setState()` is called by the State objects themselves.
5. Code smell signals: if you're calling `setAlgorithm()` from outside → Strategy. If you're calling `setState()` from inside state objects → State.

**Result:** Interviewer sees you know the structural distinction, not just the verbal one. Demonstrating the pseudocode difference signals you can actually implement it correctly.
