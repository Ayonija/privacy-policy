## Machine Coding 1: Vending Machine

### Pattern Map
- **State** â€” machine behavior depends on current state (IDLE / HAS_MONEY / DISPENSING)
- **Strategy** â€” payment processing (CASH / CARD / UPI) is swappable

### Entities

| Entity | Responsibility |
|---|---|
| `VendingMachine` | Context: holds current state + product inventory |
| `VendingMachineState` | Interface: `insertMoney(amount)`, `selectProduct(id)`, `dispense()`, `refund()` |
| `IdleState` | Accepts money; rejects select and dispense |
| `HasMoneyState` | Accepts select and refund; rejects more money; validates stock and funds |
| `DispensingState` | Executes dispense, returns change, resets to Idle |
| `Product` | id, name, price, stockCount |
| `PaymentStrategy` | Interface: `processPayment(amount): PaymentResult` |
| `CashPayment`, `CardPayment`, `UPIPayment` | Concrete payment strategies |

### State Pseudocode

```
interface VendingMachineState {
    insertMoney(machine: VendingMachine, amount: Money): void
    selectProduct(machine: VendingMachine, productId: String): void
    dispense(machine: VendingMachine): void
    refund(machine: VendingMachine): void
}

class IdleState implements VendingMachineState {
    insertMoney(machine, amount) {
        machine.setInsertedAmount(amount)
        machine.setState(new HasMoneyState())
    }
    selectProduct(machine, id) { throw "Insert money first" }
    dispense(machine) { throw "Insert money first" }
    refund(machine) { /* nothing to refund */ }
}

class HasMoneyState implements VendingMachineState {
    insertMoney(machine, amount) { throw "Already has money" }

    selectProduct(machine, id) {
        product = machine.getProduct(id)
        if (product == null) { throw "Product not found" }
        if (product.stock == 0) { throw "Out of stock" }
        if (machine.getInsertedAmount() < product.price) { throw "Insufficient funds" }
        machine.setSelectedProduct(product)
        machine.setState(new DispensingState())
    }

    refund(machine) {
        machine.returnMoney(machine.getInsertedAmount())
        machine.setInsertedAmount(0)
        machine.setState(new IdleState())
    }
    dispense(machine) { throw "Select a product first" }
}

class DispensingState implements VendingMachineState {
    insertMoney(machine, amount) { throw "Currently dispensing" }
    selectProduct(machine, id) { throw "Currently dispensing" }
    refund(machine) { throw "Currently dispensing" }

    dispense(machine) {
        product = machine.getSelectedProduct()
        product.stock -= 1
        change = machine.getInsertedAmount() - product.price
        machine.returnMoney(change)
        machine.setInsertedAmount(0)
        machine.setSelectedProduct(null)
        machine.setState(new IdleState())
    }
}
```

### Edge Cases
| Scenario | Handling |
|---|---|
| Insufficient funds | `HasMoneyState.selectProduct()` throws before transitioning |
| Out of stock | Check `product.stock == 0` in `HasMoneyState.selectProduct()` |
| Change calculation | `change = insertedAmount - product.price`; return via `machine.returnMoney()` |
| Refund | `HasMoneyState.refund()` returns full inserted amount, resets to Idle |
| Concurrent access | Lock on `VendingMachine` object for all state-mutating methods |

## Machine Coding 2: Splitwise

### Pattern Map
- **Strategy** â€” split type (EQUAL / EXACT / PERCENT) is swappable per expense
- **Observer** â€” when expense is added, all group members' balances update automatically

### Entities

| Entity | Responsibility |
|---|---|
| `User` | id, name, email |
| `Group` | groupId, members: List<User>, expenses: List<Expense> |
| `Expense` | amount, paidBy: User, split: SplitStrategy, participants: List<User> |
| `SplitStrategy` | Interface: `calculateSplit(amount, participants): Map<User, Money>` |
| `EqualSplit` | Divides amount equally: `amount / n` per person |
| `ExactSplit` | Takes explicit amounts per person; validates sum == total |
| `PercentSplit` | Takes percentages per person; validates sum == 100% |
| `BalanceSheet` | `Map<(userId, userId), Money>` â€” who owes whom |

### Core Logic Pseudocode

```
class Expense {
    amount: Money
    paidBy: User
    splitStrategy: SplitStrategy
    participants: List<User>
    observers: List<BalanceObserver>

    constructor(amount, paidBy, strategy, participants) {
        this.amount = amount
        this.paidBy = paidBy
        this.splitStrategy = strategy
        this.participants = participants
    }
}

class ExpenseService {
    private balanceSheet: BalanceSheet

    addExpense(expense: Expense) {
        shares = expense.splitStrategy.calculateSplit(expense.amount, expense.participants)

        for (user, share) in shares:
            if user != expense.paidBy:
                // user owes expense.paidBy the share amount
                balanceSheet.addDebt(debtor=user, creditor=expense.paidBy, amount=share)

        notifyMembers(expense)   // Observer: update all UI/notifications
    }
}

// Strategy implementations
class EqualSplit implements SplitStrategy {
    calculateSplit(amount, participants): Map<User, Money> {
        share = amount / participants.size()
        return {p: share for p in participants}
    }
}

class ExactSplit implements SplitStrategy {
    private exactAmounts: Map<User, Money>

    calculateSplit(amount, participants): Map<User, Money> {
        if (sum(exactAmounts.values()) != amount) throw "Amounts don't sum to total"
        return exactAmounts
    }
}

class PercentSplit implements SplitStrategy {
    private percentages: Map<User, Float>

    calculateSplit(amount, participants): Map<User, Money> {
        if (sum(percentages.values()) != 100.0) throw "Percentages don't sum to 100"
        return {p: amount * percentages[p] / 100 for p in participants}
    }
}
```

| Scenario | Handling |
|---|---|
| Percent doesn't sum to 100 | `PercentSplit.calculateSplit()` throws `InvalidSplitException` |
| Exact amounts don't sum to total | `ExactSplit.calculateSplit()` throws |
| Circular debts (Aâ†’Bâ†’Câ†’A) | Net balance computation resolves circular debts automatically |

## Common LLD Interview Mistakes

| Mistake | Why It Fails | Fix |
|---|---|---|
| Forcing patterns everywhere | Unnecessary complexity; shows you don't know when NOT to use a pattern | Ask "what problem am I solving?" before reaching for a pattern |
| Making everything Singleton | Hard to test (global state), thread-safety issues, hides dependencies | Singleton only when exactly one instance is a hard constraint (DB pool, config) |
| God class violating SRP | One class does everything; impossible to test or extend | Split by responsibility: each class has one reason to change |
| Confusing Strategy and State | Wrong pattern = wrong structure = bugs when adding features | Use the "who switches?" diagnostic test first |
| Coding to concrete classes | `new ArrayList<>()` in business logic, not `List<>` | Code to interfaces; dependencies injected |
| Memorizing names, not the WHEN | Can name patterns but can't say when to use them | For every pattern, know the exact interview trigger phrase |

## STAR Interview Framework

**Situation:** "Design Splitwise end-to-end. Users can create groups, add expenses with different split types (equal, exact, percent), track who owes whom, and simplify debts."

**Task:** "Cover the full design: entities, patterns used, core algorithms, and at least two edge cases."

**Action:**
1. Entities: User, Group, Expense, SplitStrategy (interface), BalanceSheet
2. Strategy pattern: `SplitStrategy` interface with `EqualSplit`, `ExactSplit`, `PercentSplit` â€” adding new split type is one class
3. Observer pattern: `addExpense()` notifies all group members to update their balance views and send notifications
4. `addExpense()` flow: validate split â†’ `calculateSplit()` â†’ update `BalanceSheet` â†’ `notifyMembers()`
5. `BalanceSheet.addDebt()` nets off bidirectional debts: if A owes B $10 and B owes A $6, result is A owes B $4
6. Simplify debts: compute net balance per user â†’ greedy two-pointer pairing max creditor with max debtor â†’ minimum transactions
7. Edge cases: percent validation (sum must equal 100), exact validation (sum must equal total), circular debt resolution via net balance

**Result:** New split type = one class, zero changes to `ExpenseService`. Balance sheet always reflects net positions â€” no circular debt accumulation. Simplification is O(n log n).
