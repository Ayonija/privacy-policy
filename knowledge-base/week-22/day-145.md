# Day 145 — Strategy Pattern (Behavioral)
**Week 22 | Phase: LLD Mastery | Month 4**

## Focus
The Strategy pattern defines a family of interchangeable algorithms and lets you swap them at runtime from outside the object. The key insight is composition over inheritance: the Sorter never changes, only the plugged-in algorithm does. Master it for any interview question involving "multiple ways to do the same thing."

---

## Core Concept

### Analogy
Google Maps: the destination is fixed, but you choose car, bike, or walk. Each route is a different algorithm; Maps hands off to whichever you select. The same trip, different strategy — and Maps never rewrites itself to add a new route type.

### What It Does
- Defines a family of algorithms, encapsulates each one, makes them interchangeable
- The algorithm is injected from outside; the host object delegates to it
- Adding a new algorithm = write one new class; the host is never touched

### When to Reach For It (Spot Triggers)
- "Multiple ways to do the same thing" (sort, compress, pay, price)
- "Switch algorithm at runtime based on user choice or config"
- "Giant if-else picking which method to run" — Strategy collapses it to a single `strategy.execute()`
- "Support pluggable behavior without subclassing the host"

## Pseudocode

```
// Strategy interface
interface SortStrategy {
    sort(data: int[]): int[]
}

// Concrete strategies
class QuickSort implements SortStrategy {
    sort(data: int[]): int[] { /* quick sort implementation */ }
}

class MergeSort implements SortStrategy {
    sort(data: int[]): int[] { /* merge sort implementation */ }
}

// Context — holds a strategy reference
class Sorter {
    private strategy: SortStrategy

    setStrategy(s: SortStrategy) {
        this.strategy = s
    }

    sort(data: int[]): int[] {
        return strategy.sort(data)   // pure delegation
    }
}

// Client — picks and injects from outside
sorter = new Sorter()
sorter.setStrategy(new QuickSort())
sorter.sort([5, 3, 1, 4])   // uses QuickSort

sorter.setStrategy(new MergeSort())
sorter.sort([5, 3, 1, 4])   // now uses MergeSort — Sorter unchanged
```

Key points:
- `Sorter` has zero knowledge of which algorithm it holds
- Swap is done by the **caller**, not by `Sorter`
- New algorithm (e.g., `HeapSort`) = new class only, zero changes to `Sorter`

---

## Composition Over Inheritance

| Approach | What happens when you add HeapSort |
|---|---|
| Strategy (composition) | Write `HeapSort implements SortStrategy` — done. `Sorter` unchanged. |
| Inheritance | Write `HeapSorter extends Sorter` — new subclass, and you cannot mix-and-match later. |

**Rule:** If you find yourself writing `QuickSorter extends Sorter`, `MergeSorter extends Sorter` — you are doing it wrong. Flatten to Strategy.

## Real-World Uses

| Domain | Strategies |
|---|---|
| Payment processing | `CreditCardPayment`, `UPIPayment`, `WalletPayment` — same `pay(amount)` interface |
| Sorting / compression | QuickSort vs MergeSort; GZIP vs LZ4 |
| Pricing rules | `FlatRatePricing`, `TieredPricing`, `DiscountPricing` |
| Route planners | Car, bike, walk, transit — same `getRoute(from, to)` interface |
| Tax calculators | India GST vs US Sales Tax vs EU VAT |

---

## Don't Confuse

| Pattern | Who controls the switch? | Change mechanism |
|---|---|---|
| Strategy | **External caller** injects the algorithm | Caller calls `setStrategy()` before invoking |
| State | **Object itself** transitions to the next state | Object's internal state changes behavior automatically |

**One-line test:** Ask "Who picks the algorithm?" — if it's the caller → Strategy. If it's the object reacting to its own status → State.

## STAR Interview Framework

**Situation:** "Design a payment processing system for an e-commerce platform supporting credit card, UPI, and digital wallet payments. The business wants to add new payment methods (BNPL, crypto) with zero changes to the checkout flow."

**Task:** "Architect the payment module so each method is independently pluggable and the checkout service is completely decoupled from payment-method specifics."

**Action:**
1. Define `PaymentStrategy` interface: `pay(amount: Money, context: PaymentContext): PaymentResult`
2. Implement `CreditCardStrategy`, `UPIStrategy`, `WalletStrategy` — each encapsulates its own validation, gateway call, and error handling
3. `CheckoutService` holds a `PaymentStrategy` field; payment method injected at runtime based on user selection
4. Add `BNPLStrategy` later: write one class, register it in the factory — `CheckoutService` is untouched
5. Tested each strategy in isolation; integration tests mock the strategy interface

**Result:** Adding a new payment method went from a 3-sprint checkout refactor to a single-class addition. Payment-related bugs isolated to individual strategy classes — MTTR dropped from days to hours. Checkout service code complexity frozen.

## Flashcards

| Q | A |
|---|---|
| What is the core intent of the Strategy pattern? | Defines a family of interchangeable algorithms and lets the caller swap them at runtime via composition |
| In Strategy, who controls which algorithm is used? | The external caller — it injects the strategy into the context object |
| What is the structural sign that you need Strategy? | A giant if-else chain picking which algorithm to run; or a parallel hierarchy of subclasses per algorithm variant |
| Strategy vs State — one-line distinction | Strategy: caller picks the algorithm from outside. State: object transitions its own behavior based on internal status. |
| Spot trigger for Strategy in an interview | "Multiple ways to do the same thing", "swap algorithm at runtime", "support new payment methods without changing checkout" |
| Why is Strategy preferred over inheritance for algorithm variation? | New algorithm = one new class, context unchanged; inheritance requires a new subclass per variant and forbids runtime swapping |
