## 1. Chain of Responsibility

### Analogy
Support ticket routing: L1 support tries to handle it. If they can't, they escalate to L2. If L2 can't, it goes to L3. Each level either handles it or passes it along — no single handler knows the full chain.

### What It Does
- Passes a request along a chain of handlers; each decides to handle or pass on
- Decouples sender from receiver — sender doesn't know which handler will process the request
- Chain can be constructed dynamically at runtime

### Pseudocode
```
abstract class SupportHandler {
    private next: SupportHandler

    setNext(handler: SupportHandler): SupportHandler {
        next = handler
        return handler   // allows chaining: l1.setNext(l2).setNext(l3)
    }

    handle(ticket: Ticket) {
        if (canHandle(ticket)) {
            process(ticket)
        } else if (next != null) {
            next.handle(ticket)   // pass along
        }
    }

    abstract canHandle(ticket: Ticket): boolean
    abstract process(ticket: Ticket): void
}

class L1Support extends SupportHandler {
    canHandle(t) { return t.severity == "LOW" }
    process(t) { resolveBasicIssue(t) }
}
```

### Real Uses
- Log level filtering (DEBUG → INFO → WARN → ERROR handlers)
- HTTP middleware pipelines (auth → rate-limit → logging → handler)
- Exception handling chains
- Approval workflows (manager → director → VP)

## 2. Iterator

### Analogy
A `for-each` loop over any collection — you walk through elements one by one without caring whether the collection is an array, linked list, tree, or database cursor.

### What It Does
- Provides a standard way to traverse a collection without exposing its internal structure
- Separates traversal logic from the collection itself
- All Java/Python collections implement this: `Iterable` / `__iter__`

### Pseudocode
```
interface Iterator<T> {
    hasNext(): boolean
    next(): T
}

class LinkedListIterator<T> implements Iterator<T> {
    private current: Node<T>

    constructor(head: Node<T>) { current = head }

    hasNext(): boolean { return current != null }
    next(): T {
        val = current.value
        current = current.next
        return val
    }
}

// Client
iter = new LinkedListIterator(list.head)
while (iter.hasNext()) {
    process(iter.next())
}
```

### Real Uses
- All Java `Iterable` implementations (`ArrayList`, `LinkedList`, `TreeSet`)
- Database cursor / ResultSet iteration
- File line-by-line readers
- Custom tree traversal (in-order, BFS, DFS as different iterators)

## 3. Mediator

### Analogy
Air traffic control: planes never talk to each other directly. Every communication goes through ATC. ATC knows all planes; planes only know ATC. Remove ATC and you have chaos.

### What It Does
- Centralizes complex communication between many objects through one hub
- Reduces the number of direct connections from O(n²) to O(n)
- Each participant only knows the mediator interface, not other participants

### Key Difference from Observer

| Observer | Mediator |
|---|---|
| Subject broadcasts; observers are passive | Mediator routes bidirectional messages between equals |
| Subject doesn't know who observes | Mediator knows all participants |
| One-to-many (one subject, many watchers) | Many-to-many (any can send/receive via hub) |

### Real Uses
- Chat room (users send via room, room routes to recipients)
- GUI component coordination (button click → mediator → update label + log + enable other button)
- Event bus (Spring `ApplicationEventPublisher`)
- Air traffic control systems

## 4. Memento

### Analogy
Save game: you snapshot your game state before a boss fight. If you die, you restore the save. The game engine doesn't expose your raw state to the outside — the save file is opaque.

### What It Does
- Captures an object's internal state in a snapshot (Memento) without violating encapsulation
- Allows restoring the object to that snapshot later
- The Memento is opaque — only the Originator can read its contents

### Pseudocode
```
// Originator — creates and restores from Memento
class TextEditor {
    private content: String

    type(text: String) { content += text }

    // Save current state
    save(): Memento { return new Memento(content) }

    // Restore from snapshot
    restore(m: Memento) { content = m.getState() }
}

// Memento — opaque snapshot
class Memento {
    private state: String
    constructor(state) { this.state = state }
    getState(): String { return state }   // only Originator should call this
}

// Caretaker — stores snapshots, doesn't read them
class UndoHistory {
    private history: Stack<Memento>

    save(editor: TextEditor) { history.push(editor.save()) }
    undo(editor: TextEditor) { editor.restore(history.pop()) }
}
```

### Memento vs Command for Undo

| Memento | Command |
|---|---|
| Saves full object state snapshot | Records the inverse operation |
| Easier for simple objects | Better for granular, composable actions |
| Memory-heavy for large objects | Operation-heavy but memory-efficient |
| Use when state is the whole picture | Use when operations are well-defined and reversible |

### Real Uses
- Text editor undo (save full state snapshot)
- Browser history (page state snapshots)
- Database transaction savepoints
- Game checkpoints

## 5. Visitor

### Analogy
A tax inspector visits different types of buildings — residential, commercial, industrial — and applies different tax rules to each without the buildings changing. The new inspection rule (new tax year) adds a new Visitor, not new methods to buildings.

### What It Does
- Adds a new operation to a class hierarchy without modifying those classes
- Double dispatch: `element.accept(visitor)` → `visitor.visit(element)`
- Useful when you have a stable class hierarchy but frequently-changing operations

### Pseudocode
```
interface DocumentElement {
    accept(visitor: DocumentVisitor): void
}

class Paragraph implements DocumentElement {
    accept(v: DocumentVisitor) { v.visitParagraph(this) }
}

class Table implements DocumentElement {
    accept(v: DocumentVisitor) { v.visitTable(this) }
}

interface DocumentVisitor {
    visitParagraph(p: Paragraph): void
    visitTable(t: Table): void
}

class HtmlExporter implements DocumentVisitor {
    visitParagraph(p) { render("<p>" + p.text + "</p>") }
    visitTable(t) { render("<table>" + t.toHtml() + "</table>") }
}

class PdfExporter implements DocumentVisitor {
    visitParagraph(p) { pdf.addParagraph(p.text) }
    visitTable(t) { pdf.addTable(t.data) }
}

// Client
elements = [new Paragraph(), new Table()]
visitor = new HtmlExporter()
for e in elements: e.accept(visitor)
```

### Real Uses
- Compiler AST operations (type checking, code generation, optimization passes as different visitors)
- Document export (HTML/PDF/Word as different visitors on same document model)
- Report generation on domain model

## 6. Interpreter

### Analogy
A calculator that reads `"3 + 4 * 2"` and evaluates it. Build a tiny grammar, represent each rule as a class, and evaluate by composing expression objects.

### What It Does
- Defines a grammar for a language and provides an interpreter to evaluate sentences in that grammar
- Each grammar rule → one class; expressions compose into a tree

### Pseudocode
```
interface Expression {
    interpret(context: Map<String, Integer>): int
}

class NumberExpression implements Expression {
    private value: int
    interpret(ctx) { return value }
}

class AddExpression implements Expression {
    private left, right: Expression
    interpret(ctx) { return left.interpret(ctx) + right.interpret(ctx) }
}

// "1 + 2" as expression tree
expr = new AddExpression(new NumberExpression(1), new NumberExpression(2))
expr.interpret({})   // returns 3
```

### Real Uses
- Scripting / rule engines (evaluate user-defined rules)
- SQL parsers, regex engines
- Expression evaluators in spreadsheets
- Configuration DSL interpreters

## 7. Abstract Factory (Deeper Cut)

### The "Family Consistency" Problem
Factory Method creates one product. Abstract Factory creates **families** of related products that must be used together consistently.

```
// Without Abstract Factory — risk of mixing incompatible components
button = new WindowsButton()
checkbox = new MacCheckbox()   // mixing platforms — visual inconsistency
```

```
// With Abstract Factory — guaranteed consistency
interface UIFactory {
    createButton(): Button
    createCheckbox(): Checkbox
}

class WindowsFactory implements UIFactory {
    createButton(): Button { return new WindowsButton() }
    createCheckbox(): Checkbox { return new WindowsCheckbox() }
}

class MacFactory implements UIFactory {
    createButton(): Button { return new MacButton() }
    createCheckbox(): Checkbox { return new MacCheckbox() }
}

// Client receives a factory — never mixes products from different families
app = new Application(new WindowsFactory())
app.build()   // always gets Windows button + Windows checkbox together
```

**Key insight:** Abstract Factory = multiple Factory Methods under one roof, guaranteeing cross-product consistency.

### Real Uses
- Cross-platform UI toolkits (AWT, Swing themes)
- Database driver factories (MySQL vs PostgreSQL — connection, statement, result set all consistent)
- Cloud provider SDKs (AWS vs GCP — each has consistent storage, compute, messaging clients)

## 8. Prototype (Deeper Cut)

### Analogy
Ctrl+C Ctrl+V: clone an existing object instead of building from scratch. The copy knows how to copy itself — you don't need to know its internal structure.

### What It Does
- Creates new objects by copying (cloning) an existing instance
- Useful when object creation is expensive (DB load, complex initialization)
- Useful when new objects differ only slightly from existing ones

### Pseudocode
```
interface Cloneable {
    clone(): Cloneable
}

class GameCharacter implements Cloneable {
    name: String
    level: int
    equipment: List<Item>

    clone(): GameCharacter {
        copy = new GameCharacter()
        copy.name = this.name
        copy.level = this.level
        copy.equipment = new List(this.equipment)   // deep copy
        return copy
    }
}

// Client — clone template, then modify
template = loadFromDB("warrior_template")   // expensive load once
warrior1 = template.clone()
warrior1.name = "Ragnar"   // only differs by name
```

### Real Uses
- Game character templates (clone base character, customize per player)
- Document templates (clone base doc, fill in specifics)
- Spring prototype bean scope (`@Scope("prototype")` — new instance per request)
- Config objects with slight per-environment variations

## STAR Interview Framework

**Situation:** "Describe a scenario where Chain of Responsibility fits better than a sequence of if-else checks. Walk through the structure."

**Task:** "Show you understand both the pattern and when to reach for it over simpler alternatives."

**Action:**
1. Scenario: HTTP request processing — auth check → rate limit check → input validation → business logic handler
2. Without CoR: one monolithic method with 4 nested ifs; adding a new step means modifying it; each step is hard to test in isolation
3. With CoR: each step is its own handler class; chain configured at startup; new step = new handler class registered in the chain
4. Each handler: `if (canHandle(req)) { process(); } else { next.handle(req); }`
5. Real example: Spring `FilterChain` / Servlet filters — each filter is a handler; the chain is built by the container

**Result:** Adding a new middleware step is one class, one registration line. Each handler is unit-testable. Chain order is configurable at runtime — same handlers, different order for different environments.
