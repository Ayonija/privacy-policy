## DSA (1.5 hours)

### Pattern: DP on Intervals with Group Tracking (Remove Boxes)
- Core idea: `dp[l][r][k]` = max points from boxes `[l..r]` with `k` extra boxes identical to `boxes[l]` attached to its left. If `boxes[l] == boxes[m]` for some `m` in `(l..r)`, you can "save" box m to extend the left group: `dp[l][r][k] = dp[l+1][m-1][0] + dp[m][r][k+1]`. Otherwise remove the left group: `dp[l][r][k] = (k+1)² + dp[l+1][r][0]`.
- Trigger condition: Removing a contiguous group of identical elements yields points proportional to group size squared. Elements can be brought together by removing intervening elements. This extra grouping dimension `k` is the hallmark.
- Time complexity: O(n⁴) | Space complexity: O(n³)

### Pattern: Backtracking on Unknown Environment (Robot Room Cleaner)
- Core idea: The robot cannot see the full grid. Use DFS with direction tracking: define a virtual coordinate system (`(row, col)`) to track the robot's absolute position. Use a visited set for (row, col). To backtrack (return to caller), go back one step: turn 180°, move, turn 180° again.
- Trigger condition: "Navigate an unknown environment" or "explore a grid without seeing it." The key is simulating DFS with explicit undo (reverse movement) since you cannot jump back to previous positions.
- Time complexity: O(N - O) where N = total cells, O = obstacles (visit each reachable cell once) | Space complexity: O(N) for visited set

    //        robot.clean()

    //          nextDir = (dir + turn) % 4
    //          nextRow = row + dirs[nextDir][0]; nextCol = col + dirs[nextDir][1]
    //          if (nextRow,nextCol) not visited and robot.move():
    //            dfs(nextRow, nextCol, nextDir, visited)
    //            goBack(): robot.turnRight(); robot.turnRight(); robot.move(); robot.turnRight(); robot.turnRight()
    //          robot.turnRight()  // try next direction

    // LC 131 — Palindrome Partitioning (revision — write from memory)
    public List<List<String>> partition(String s) {
        int n = s.length();
        boolean[][] isPalin = new boolean[n][n];
        for (int i = n - 1; i >= 0; i--)
            for (int j = i; j < n; j++)
                isPalin[i][j] = s.charAt(i) == s.charAt(j) && (j - i <= 2 || isPalin[i+1][j-1]);
        List<List<String>> result = new ArrayList<>();
        backtrack(s, 0, isPalin, new ArrayList<>(), result);
        return result;
    }
    private void backtrack(String s, int start, boolean[][] isPalin, List<String> path, List<List<String>> result) {
        if (start == s.length()) { result.add(new ArrayList<>(path)); return; }
        for (int end = start; end < s.length(); end++) {
            if (isPalin[start][end]) {
                path.add(s.substring(start, end + 1));
                backtrack(s, end + 1, isPalin, path, result);
                path.remove(path.size() - 1);
            }
        }
    }
}

#### Architecture Trade-offs Cheat Sheet

```
DECISION TREE — "Which database for YouTube?"

Is the data strongly consistent? (auth, billing, subscriptions)
  → YES: MySQL / PostgreSQL

Is it write-heavy with flexible schema? (video metadata, comments, watch history)
  → YES: Cassandra (partition by video_id or user_id)

Does it need full-text search? (video search, autocomplete)
  → YES: Elasticsearch

Does it need low-latency reads at high throughput? (sessions, trending, embeddings)
  → YES: Redis

Does it need scalable key-value with predictable latency? (user preferences, history)
  → YES: DynamoDB

Is it event-driven async communication? (upload done, view events, notif fan-out)
  → YES: Kafka (Kafka Streams / Flink for stream processing)
```

#### Critical Failure Modes & Mitigations

| Failure | Impact | Mitigation |
|---------|--------|------------|
| CDN POP down | Regional users can't stream | DNS failover to next POP; multi-CDN strategy |
| Transcode worker crash | Video stuck in processing | SQS visibility timeout → redelivery; max 3 retries then dead-letter |
| Kafka lag spike | View events delayed, trending stale | Scale consumer group; Kafka retains 7 days |
| Elasticsearch node down | Search unavailable | 2 replicas per shard; routing to healthy nodes |
| Cassandra node down | Comments/metadata reads degrade | RF=3, QUORUM reads/writes survive 1 node failure |
| Recommendation model stale | Homepage shows old content | Daily batch retrain + hourly incremental update |

#### Full System Summary Table

| Concern | Solution | Why |
|---------|----------|-----|
| Upload at scale | S3 multipart + presigned URLs | Client uploads directly to S3; no backend bandwidth cost |
| Transcoding | Async Kafka → spot instances + FFmpeg | Encoding is CPU-intensive; cannot block HTTP; spot = cheap |
| Global delivery | CDN edge + Netflix OCA in ISPs | 95%+ cache hit rate; < 50ms to nearest edge |
| ABR streaming | HLS/DASH with 6s segments | Adapt quality per segment; fast response to network change |
| Recommendations | Two-tower NN + ANN (Faiss) | 500 candidates in < 100ms; ranking in < 20ms |
| Search | Elasticsearch + BM25 + ML ranking | Full-text + engagement signals; < 100ms query |
| Trending | Kafka → Flink → Redis ZINCRBY sliding window | High-throughput approximate counting |
| Comments | Cassandra (partition: video_id) | Write-heavy; keyset pagination; counter columns for likes |
| Likes | Kafka + Redis INCR → Cassandra flush | Absorbs write spikes; eventual consistency OK |
| Notifications | Kafka fan-out → FCM/APNs | Async delivery; Cassandra in-app feed for durability |
| Auth | JWT (15min) + refresh token in Redis | Stateless validation; true revocation via Redis delete |

**Interview talking point:** "If asked what the hardest problem in YouTube's architecture is, answer: The fan-out problem — when a channel with 100M subscribers uploads a video, you need to write a notification to 100M feeds. Solution: hybrid push/pull (push for < 1M subscribers, pull for celebrities), async Kafka fan-out over minutes (not seconds), and Redis caching of pre-computed feeds for active users. No single database write can handle 100M fan-out synchronously."
