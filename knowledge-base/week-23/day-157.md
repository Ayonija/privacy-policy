## DSA (1.5 hours — Revision Mode)

### Pattern Revision: Hashing & Greedy (HashMap Frequency, Interval Scheduling)
- Core idea: HashMap frequency maps reduce counting and lookup to O(1), enabling linear-time solutions for problems involving duplicates, complements, or running sums. Greedy interval scheduling exploits the insight that always choosing the interval with the earliest end time maximizes the number of non-overlapping intervals — this is proven optimal by an exchange argument.
- Trigger condition: Use HashMap when the problem involves counting occurrences, checking existence, or tracking last-seen indices. Use greedy interval scheduling when asked to maximize scheduled intervals, minimize removed intervals, or assign meeting rooms.
- Time complexity: HashMap operations O(1) average | Sort-based greedy O(N log N) | Merge intervals O(N log N) sort + O(N) scan
- Space complexity: O(N) for HashMap or result list

### Template (write from memory before solving):
```java
// Merge Intervals
Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
List<int[]> result = new ArrayList<>();
result.add(intervals[0]);
for (int i = 1; i < intervals.length; i++) {
    int[] last = result.get(result.size() - 1);
    if (intervals[i][0] <= last[1]) last[1] = Math.max(last[1], intervals[i][1]); // overlap: extend
    else result.add(intervals[i]); // no overlap: new interval
}

// Non-overlapping Intervals (count to remove)
Arrays.sort(intervals, (a, b) -> a[1] - b[1]); // sort by END (greedy: keep earliest ending)
int count = 0, end = Integer.MIN_VALUE;
for (int[] interval : intervals) {
    if (interval[0] >= end) end = interval[1]; // no overlap: take this interval
    else count++; // overlap: remove this one (it ends later since sorted by end)
}
// count = number of intervals removed

// HashMap frequency pattern
Map<Integer, Integer> freq = new HashMap<>();
for (int num : nums) freq.merge(num, 1, Integer::sum);
// OR: freq.put(num, freq.getOrDefault(num, 0) + 1);
```

## System Design (30 min — HLD Component Review)

### Topic: Monitoring, Observability & Alerting (Three Pillars)

Observability is the ability to understand a system's internal state from its external outputs. The three pillars are metrics, logs, and traces — each answers a different question.

**Pillar 1 — Metrics (What is happening? At what rate?):**
- Numerical time-series: QPS, p50/p99 latency, error rate, CPU %, memory usage, GC pause time.
- Collection: Prometheus (pull-based scraping from /metrics endpoints every 15s) + Grafana (dashboards).
- SLI (Service Level Indicator) = the actual measured metric (e.g., p99 latency = 185ms).
- SLO (Service Level Objective) = the target (e.g., p99 < 200ms for 99.9% of requests).
- SLA (Service Level Agreement) = contractual commitment with penalty clauses (e.g., 99.9% uptime or credit issued).
- Error budget = 1 - SLO. If SLO = 99.9%, error budget = 0.1% = 43.8 min/month of allowed downtime.

**Pillar 2 — Logs (What exactly happened? With what context?):**
- Structured logs (JSON with fields: timestamp, level, service, traceId, userId, message) vs unstructured text.
- Log levels: DEBUG (verbose, 1% sampling in production) / INFO (normal operations) / WARN (recoverable issues) / ERROR (action needed).
- Centralization: ELK stack (Elasticsearch for search, Logstash for ingestion/transform, Kibana for UI) or Splunk.
- Log sampling: DEBUG-level logs sampled at 1% in production to reduce volume without losing signal.
- Retention: hot (7 days, fast SSD), warm (30 days, slower storage), cold (1 year, S3/Glacier).

**Pillar 3 — Traces (Which service caused the slowdown? Where in the call chain?):**
- Distributed request tracing: each request gets a unique trace ID propagated in HTTP headers (X-Trace-ID).
- Each service creates a span measuring its processing time; spans are linked by parent-child relationships.
- Tools: Jaeger, Zipkin, AWS X-Ray, Datadog APM.
- Sampling: 100% in development; 1–5% in production (tail-based sampling: always capture slow/error traces).

**Alerting:**
- Alert on SLO burn rate (how fast you're consuming your error budget), not just raw thresholds. Example: "if error rate would exhaust monthly budget in 2 hours, page immediately."
- On-call routing: PagerDuty / OpsGenie — escalation policies, alert deduplication, runbook links.
- Avoid alert fatigue: every alert should be actionable and have a runbook.

**Key numbers:**
- Prometheus scrape interval: 15 seconds
- Log write to Elasticsearch: ~50ms per batch
- Distributed trace overhead: ~1–3ms per span, ~0.1% CPU overhead with 1% sampling

**Interview talking point:** "If asked how you would debug a sudden p99 latency spike in a microservices system, answer: start with metrics to confirm the spike and identify which service is slow (Grafana dashboard, filter by service name). Then check distributed traces for that time window to find the slow span — which specific DB query, external API call, or service-to-service call added latency. Then pull structured logs from that service at that time to find the root cause — slow DB query due to missing index, GC pause, or external API timeout. Metrics tell you something is wrong. Traces tell you where. Logs tell you why."

## Flashcards

| Q | A |
|---|---|
| What is the key insight for the Non-overlapping Intervals greedy solution? | Sort by END time (not start). Always keep the interval with the earliest end — it leaves the most room for future intervals. Count every interval you skip as one removal. This is the Activity Selection Problem. |
| How does Course Schedule III use a max-heap? | Sort by deadline. Maintain a max-heap of chosen course durations and a running total. If adding a course exceeds its deadline, swap it with the longest course in the heap (if current course is shorter) — same count, smaller total duration. |
| What is the difference between SLI, SLO, and SLA? | SLI = actual measured metric (e.g., p99 latency today). SLO = target (e.g., p99 < 200ms). SLA = contract with customer/penalty if violated. |
| Name the three pillars of observability and the primary tool for each. | Metrics → Prometheus + Grafana. Logs → ELK stack (Elasticsearch + Logstash + Kibana). Traces → Jaeger or Zipkin. |
| Why sort by end time (not start time) for the greedy interval scheduling algorithm? | Sorting by end maximizes future flexibility — taking the earliest-ending interval leaves the largest remaining time window for subsequent intervals. Sorting by start does not guarantee this. |
