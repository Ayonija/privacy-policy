## Full-Text Search

### Inverted Index
- Forward index: document → list of words it contains
- **Inverted index**: word → list of documents containing it (like a book's back-index)
- At query time: look up word in inverted index → get document IDs → score + rank → return top N
- Enables O(1) lookup per term regardless of corpus size

### Index Pipeline (at write time)
1. **Tokenization**: split text into tokens ("Hello World" → ["Hello", "World"])
2. **Normalization**: lowercase, unicode normalization
3. **Stop-word removal**: filter "the", "a", "is" (high frequency, low information)
4. **Stemming / Lemmatization**: "running" → "run"; "better" → "good" (for relevance)
5. Write token → document mapping to inverted index

### Relevance Scoring
- **TF-IDF** (Term Frequency × Inverse Document Frequency):
  - TF: how often the term appears in this document (more = more relevant)
  - IDF: log(total_docs / docs_containing_term) — rare terms are more significant
  - Score = TF × IDF for each term; sum across query terms
- **BM25** (Best Match 25): improved TF-IDF; accounts for document length normalization; default in Elasticsearch since v5

### Elasticsearch
- Distributed: data sharded across nodes; each shard is a Lucene index
- Near real-time: ~1 second delay after indexing before document is searchable (refresh interval)
- Replicas: each shard has configurable replica count; replicas serve reads
- Mapping: schema for each field; text (analyzed) vs keyword (exact match) vs integer
- Use for: product search, log analysis, autocomplete, full-text search at scale

## Typeahead / Autocomplete Design

### Trie (Small Dataset / In-Memory)
- Prefix tree; each path from root to node represents a prefix
- O(prefix_length) lookup; fast for small vocabularies
- Storage: all N-gram prefixes; memory-intensive for large corpora

### Elasticsearch Prefix Query (Large Scale)
- `prefix` or `wildcard` query on keyword field; or use `search_as_you_type` mapping
- Distributed; handles billions of documents; handles updates automatically
- Latency: ~10–50ms for prefix queries at scale

### Redis Sorted Set (Top-K Completions)
- Precompute top-K completions per prefix; store in Redis sorted set
- Key: `autocomplete:{prefix}`; members: suggestions; score: search frequency
- O(log N) insertion; O(log N + K) range query for top-K
- Pre-warm: offline job computes prefix → top-K mapping and loads into Redis
- Update: increment frequency counter; async job rebuilds top-K for affected prefixes
- Used by: Twitter typeahead, many search-as-you-type features

## Storage: Block vs Object vs File

### Block Storage
- Raw storage presented as a disk device; OS manages the filesystem on top
- **Low latency**: direct I/O, no abstraction overhead
- **Use cases**: databases (need low-latency random I/O), VM boot volumes, any workload that manages its own filesystem
- **Examples**: AWS EBS, GCP Persistent Disk
- Access: mount as device; only one instance at a time (unless multi-attach, limited)

### Object Storage
- Flat namespace of objects; each object has a globally unique key + metadata + data
- **Cheap**: ~$0.02/GB/month vs ~$0.10/GB for block
- **Scalable**: unlimited objects; no filesystem limits
- **Eventually consistent** (historically); S3 now provides strong read-after-write consistency
- **Use cases**: images, videos, backups, static website assets, ML training data, logs
- **Examples**: AWS S3, GCS, Azure Blob Storage
- Access: HTTP API (GET/PUT/DELETE); pre-signed URLs for temporary access

### File Storage (NFS / EFS)
- POSIX filesystem interface; mountable by multiple instances simultaneously
- Shared filesystem: multiple servers read/write same directory
- **Use cases**: shared config, legacy apps expecting NFS, ML training where multiple nodes read same dataset
- **Examples**: AWS EFS, Azure Files, NFS
- Higher latency than block; more flexible than object

### Decision Table

| Need | Use |
|---|---|
| Database storage | Block (EBS) |
| Image/video storage | Object (S3) |
| Multiple servers need shared filesystem | File (EFS) |
| Static website assets | Object (S3) + CDN |
| ML training data | Object (S3) or File (EFS) |
| VM boot disk | Block (EBS) |

## Large File Upload (Chunked / Multipart)

### The Problem
- Single HTTP POST for a 5GB file: connection drops at 4.9GB → entire upload lost
- Cannot parallelize; saturates a single connection

### Multipart Upload (S3 Pattern)
1. Client calls `CreateMultipartUpload` → gets upload ID
2. Client splits file into N chunks (minimum 5MB for S3; typical 50–100MB)
3. Client uploads each chunk in parallel: `UploadPart(upload_id, part_number, data)` → get ETag
4. Client calls `CompleteMultipartUpload(upload_id, [part ETags])` → S3 assembles
5. On failure: retry only the failed chunk; other chunks already committed

### Resume Support
- Client tracks which chunk ETags were received
- On resume: skip chunks with confirmed ETags; re-upload only missing ones
- `ListParts` API returns confirmed parts for a given upload ID

### Pre-Signed URLs
- Server generates temporary S3 URLs (valid 15min – 7 days)
- Client uploads directly to S3 (bypasses your server) — saves bandwidth and server load
- Common pattern: client requests upload URL from API → API returns pre-signed PUT URL → client uploads directly to S3

## Authentication vs Authorization

- **Authentication (AuthN)**: verifying identity — "Who are you?" (login, token verification)
- **Authorization (AuthZ)**: verifying permissions — "What are you allowed to do?" (RBAC, ACL)
- AuthN happens before AuthZ; authorization is meaningless without knowing who the user is

---

## OAuth 2.0

### Core Purpose
Delegated authorization: allow a third party to access resources on behalf of a user, without sharing credentials.

### Four Flows

| Flow | Use Case | How It Works |
|---|---|---|
| Authorization Code | Web apps (server-side) | User logs in at auth server; auth code returned to redirect URI; server exchanges code for tokens |
| Authorization Code + PKCE | Mobile / SPA apps | Same as auth code but uses code verifier/challenge instead of client secret (client secret can't be kept secret in mobile) |
| Client Credentials | Server-to-server (no user involved) | Client presents client_id + secret; gets access token directly |
| Implicit | Deprecated | Token returned directly in redirect; insecure — replaced by PKCE |

### Token Types
- **Access token**: short-lived (15min–1hr); used to call APIs; bearer token
- **Refresh token**: long-lived (days/weeks); used to get new access token; stored securely server-side or HttpOnly cookie
- **ID token**: JWT containing user identity claims; used by client, not API

## JWT (JSON Web Token)

### Structure
```
header.payload.signature
```
- **Header**: algorithm (HS256, RS256), token type
- **Payload**: claims — sub (user ID), iat (issued at), exp (expiry), custom claims
- **Signature**: HMAC or RSA signature over header + payload

### Benefits
- **Stateless**: server verifies signature without DB lookup → horizontal scale
- **Self-contained**: user info in payload — no session table

### Risks & Mitigations
- Token theft: store in HttpOnly cookie (not localStorage) — not accessible via JS, CSRF-protected
- Expiry: keep access token TTL short (15–60 min); refresh token to renew
- Algorithm confusion: always specify and validate expected algorithm; reject `alg: none`
- Revocation: JWT is stateless — revocation requires a denylist (Redis set of revoked JTIs)

## TLS / HTTPS

### Handshake (Simplified TLS 1.3)
1. Client sends supported cipher suites + random nonce
2. Server sends certificate (public key) + selected cipher
3. Client verifies certificate against trusted CA
4. Key exchange: ECDHE generates shared session key (asymmetric used only for handshake)
5. All subsequent data encrypted with symmetric session key (AES-GCM)

### Why It Matters in System Design
- Terminate TLS at load balancer (not each service) — reduces CPU on internal services
- HSTS header: force HTTPS even if user types HTTP
- Certificate rotation: automated via Let's Encrypt / ACM

---

## API Key Management

- Never put API keys in URLs — they appear in logs and browser history; use Authorization header
- Rotate keys on compromise; provide key rotation without downtime (two active keys during rotation window)
- Rate limit by API key (not just IP) — IPs can be shared (NAT)
- Hash API keys in DB (like passwords) — if DB is breached, keys are not exposed

## STAR Interview Framework

**Situation:** "Design user authentication for a consumer app with 10M users, supporting email/password and social login (Google, Facebook). Tokens must be stateless for horizontal scaling."

**Task:** "Choose authentication mechanism, token strategy, and social login integration."

**Action:**
1. Email/password: bcrypt (cost factor 12) hash stored in DB; never plaintext or MD5/SHA1
2. Social login: OAuth 2.0 Authorization Code + PKCE for mobile; Authorization Code for web; exchange authorization code for ID token; extract email/sub claim; upsert user in DB
3. JWT access tokens (15-minute TTL, RS256 signed); refresh tokens stored in DB with device fingerprint (long-lived, 30-day TTL)
4. Access token in memory; refresh token in HttpOnly cookie — eliminates XSS token theft risk
5. Refresh token rotation: each use issues a new refresh token; old one invalidated — detects token theft via reuse detection
6. Token revocation list in Redis: store revoked JTIs until expiry — enables logout-from-all-devices

**Result:** Auth service handles 50K auth calls/sec stateless. Social login accounts for 70% of sign-ups (lower friction). Zero credential-stuffing successes post-deployment (rate limiting on login endpoint by IP + account). Refresh token reuse detected and automatically invalidated user sessions in 3 theft incidents.

## Flashcards

| Q | A |
|---|---|
| What is an inverted index? | A mapping from each term to the list of documents containing it — enables O(1) per-term lookup regardless of corpus size |
| BM25 vs TF-IDF: what does BM25 improve? | BM25 normalizes for document length — a long document with many term occurrences is not scored higher than a short document with the same density |
| Block vs object storage: primary difference? | Block = raw device, low latency, used for DBs; Object = flat HTTP-accessible blob store, cheap, used for media/backups |
| OAuth 2.0: when to use PKCE? | Mobile apps and SPAs where client secret cannot be kept secret; PKCE replaces the client secret with a one-time code verifier/challenge |
| JWT storage: why HttpOnly cookie over localStorage? | HttpOnly cookies are inaccessible to JavaScript — prevents XSS attacks from stealing tokens; localStorage is fully accessible to any JS on the page |
| What is refresh token rotation? | Each refresh token use issues a new refresh token and invalidates the old one; reuse of an old token signals theft and triggers session revocation |
| Multipart upload: how does resume work? | Client calls ListParts to get confirmed ETags; re-uploads only missing parts; calls CompleteMultipartUpload with all ETags when done |
| AuthN vs AuthZ: one-line distinction | AuthN: verifying identity ("who are you?"); AuthZ: verifying permissions ("what can you do?") — AuthN must precede AuthZ |
