## Client-Server Architecture

### Core Model
- Client initiates requests; server responds — never reversed in basic HTTP
- Stateless HTTP preferred: no session pinning, any server handles any request, horizontal scale is trivial
- Stateful connections (WebSockets) are exceptions, not defaults — justify them explicitly

### Scaling Implications
- Stateless servers: add more behind a load balancer, done
- Stateful servers: need sticky sessions or externalized session store (Redis) — adds operational complexity

## Monolith vs Microservices

| Dimension | Monolith | Microservices |
|---|---|---|
| Deployment | Single deployable unit | Independent per service |
| Development | Simple local dev, easy debugging | Complex local env, distributed tracing needed |
| Coupling | Tight — change one thing, risk breaking others | Loose — services evolve independently |
| Scaling | Scale the whole app | Scale only the hot service |
| Fault isolation | One crash takes everything down | Failures can be contained |
| Operational cost | Low | High — service mesh, CI/CD per service, observability |
| Data | Shared DB, easy joins | Each service owns its DB — cross-service queries expensive |

### Decision Framework
- Start with monolith: faster iteration, simpler ops, single codebase
- Split to microservices when:
  - Team size > ~8–10 people (Conway's Law: architecture mirrors org)
  - Scale demands differ dramatically between components
  - Different deployment cadences or tech stacks needed
  - Independent failure domains required
- Use **Strangler Fig Pattern** to migrate: new features as services, strangle old monolith incrementally — never big-bang rewrite

## Sync vs Async Communication

| | Sync (REST / gRPC) | Async (Queues / Events) |
|---|---|---|
| Response | Immediate | Eventually |
| Coupling | Tight — caller waits | Loose — producer/consumer decoupled |
| Throughput | Limited by slowest service in chain | High — queue absorbs bursts |
| Consistency | Strong | Eventual |
| Complexity | Simple | Queue infra, idempotency, DLQ handling |

### When to Use Async
- Background jobs (email send, report generation, image resizing)
- Fan-out notifications (1 event → N subscribers)
- Batch processing pipelines
- When producer speed >> consumer speed (rate mismatch)
- Any operation where caller does not need the result to proceed

### When to Use Sync
- User-facing reads that must return data immediately
- Operations where you must confirm success before proceeding (payment auth)
- Simple request/response with tight latency SLO

## Latency vs Throughput vs Availability

### Latency
- Time for a single request to complete end-to-end
- Always report p50/p95/p99 — mean hides tail latency; p99 is what your worst users experience
- Typical targets: p99 < 100ms (interactive), p99 < 500ms (acceptable), p99 > 1s (problem)
- Reducing latency: caching, DB indexing, CDN, fewer network hops, async offloading

### Throughput
- Requests per second (RPS) or transactions per second (TPS) the system can sustain
- Increase throughput: horizontal scale, async processing, connection pooling, read replicas
- Latency and throughput often trade off — higher concurrency can increase latency under load

### Availability
- % of time system correctly serves requests
- 99.9% = 8.76 hours downtime/year
- 99.99% = 52.6 minutes downtime/year
- 99.999% = 5.26 minutes downtime/year
- Achieved via: redundancy, failover, health checks, load balancers, multi-AZ deployments

## CAP Theorem

### The Three Properties
- **Consistency (C)**: every read returns the most recent write or an error
- **Availability (A)**: every request receives a response (not necessarily latest data)
- **Partition Tolerance (P)**: system continues operating despite network partition (dropped/delayed messages between nodes)

### The Constraint
- Network partitions are unavoidable in distributed systems — P is not optional
- Therefore: under partition, choose between C (refuse to serve stale data) or A (serve stale data rather than fail)
- This is a tradeoff in practice, not a binary — many systems tune the dial

### Practical Examples
| System | Type | Trade-off |
|---|---|---|
| Zookeeper, HBase, etcd | CP | Refuses requests during partition; no stale reads |
| Cassandra, DynamoDB, CouchDB | AP | Serves reads during partition; may return stale data |
| Traditional RDBMS (single node) | CA | Not distributed; partition tolerance doesn't apply |

### Interview Framing
- Never just say "CP" or "AP" — explain what you sacrifice and why it's acceptable for the use case
- "For a financial ledger, I choose CP — returning stale balances is worse than returning an error. For a social media feed, I choose AP — a slightly stale feed is fine; a failure is not."

---

## PACELC

Extends CAP to cover the non-partition case:
- **PAC**: if there is a Partition, choose between Availability and Consistency (same as CAP)
- **ELC**: Else (no partition), tradeoff between Latency and Consistency

| | PAC choice | ELC choice |
|---|---|---|
| Cassandra | A | L (low latency, tunable consistency) |
| DynamoDB | A | L |
| Spanner/CockroachDB | C | C (pays latency for strong consistency globally) |

PACELC is more realistic: partitions are rare, but the latency vs consistency tradeoff exists all the time.

## SLA / SLO / SLI

| Term | Definition | Who Sets It | Example |
|---|---|---|---|
| SLI (Service Level Indicator) | The actual measured metric | Engineering | 99.95% of requests succeed; p99 latency = 120ms |
| SLO (Service Level Objective) | Internal target for the SLI | Product + Engineering | 99.9% success rate; p99 < 200ms |
| SLA (Service Level Agreement) | External contract with consequences | Business | 99.5% uptime or 10% credit issued |

- SLO is stricter than SLA — you want to catch degradation before the SLA is breached
- Error budget = 100% − SLO; if budget is exhausted, freeze new feature releases, focus on reliability
- Interview: always quantify — "I'd set an SLO of 99.9% availability and p99 < 200ms for this service"

## STAR Interview Framework

**Situation:** "Your team is building a new product from scratch. The CTO asks you to justify starting with a monolith vs microservices."

**Task:** "Provide a decision framework and make a concrete recommendation with justification."

**Action:**
1. Assessed team size (6 engineers), timeline (3-month MVP), and domain complexity (bounded contexts not yet clear)
2. Chose monolith: faster iteration, single deployable, no distributed systems overhead during discovery phase
3. Applied Strangler Fig plan: designed module boundaries inside the monolith matching anticipated service splits
4. Set trigger criteria: split when any module has > 2 teams touching it, or when a module's scale requirements diverge 10x from others
5. Documented CAP and async/sync boundaries explicitly so future splits would not require data model rewrites

**Result:** MVP shipped in 10 weeks. 8 months later, split the payments module to microservice (2-week migration, zero downtime) using pre-designed boundaries. The other modules stayed monolith — right choice for their scale.

## Flashcards

| Q | A |
|---|---|
| Why is stateless HTTP preferred for horizontal scaling? | No session pinning — any server handles any request; adding instances behind a load balancer requires no coordination |
| What triggers a monolith-to-microservices split? | Team size > ~10, dramatically different scale needs per component, independent deployment cadences, or clear bounded contexts |
| What is the Strangler Fig Pattern? | Incrementally replace monolith by routing new features to services; old code is strangled over time — avoids big-bang rewrite |
| CAP Theorem: what is always unavoidable in distributed systems? | Partition Tolerance (P) — network partitions happen; you must choose between C and A when they do |
| 99.99% availability = how many minutes of downtime per year? | ~52.6 minutes per year |
| What does p99 latency mean? | 99% of requests complete within this time — it represents the worst-case experience for the vast majority of users |
| PACELC vs CAP: what does PACELC add? | CAP only covers partition scenarios; PACELC also models the Latency vs Consistency tradeoff in normal (non-partition) operation |
| SLO vs SLA — which is stricter and why? | SLO is stricter — it's the internal target that gives a buffer before the external SLA is breached; catching issues at SLO prevents SLA violations |
