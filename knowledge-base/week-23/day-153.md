# Day 153 — Databases: SQL vs NoSQL, Sharding, Replication, Indexing
**Week 23 | Phase: HLD Mastery | Month 5**

## Focus
Database selection and design are the most impactful decisions in any system design interview. One wrong choice cascades into scale failures, consistency bugs, and unmaintainable queries. This day builds a precise decision framework covering storage engines, distribution strategies, and query optimization primitives.

---

## SQL vs NoSQL Decision Framework

### Choose SQL When
- Data is relational with complex joins
- ACID transactions required (financial data, inventory, bookings)
- Schema is stable and well-defined
- Complex ad-hoc queries and aggregations
- Mature tooling: ORM, migrations, query planner

### Choose NoSQL When
- Extreme write throughput (>100K writes/sec)
- Flexible or evolving schema (user-generated content, product catalogs)
- Horizontal scale is the primary requirement from day one
- Access patterns are simple and well-known (key lookups, range scans)
- Eventual consistency is acceptable

### Rule of Thumb
Start SQL. Switch to NoSQL when you hit scale limits or need schema flexibility that SQL cannot provide cleanly. Premature NoSQL adoption trades SQL safety for operational complexity without the scale benefit.

| Use Case | Recommended |
|---|---|
| User accounts, financial transactions | SQL (PostgreSQL, MySQL) |
| Session data, rate limit counters | Redis (key-value) |
| Product catalog, user profiles | MongoDB/DynamoDB (document) |
| Time-series metrics, IoT data | InfluxDB, Cassandra (wide-column) |
| Social graph, recommendations | Neo4j (graph) |

## ACID vs BASE

### ACID (Relational DBs)
- **Atomicity**: transaction is all-or-nothing
- **Consistency**: transaction brings DB from one valid state to another
- **Isolation**: concurrent transactions behave as if sequential
- **Durability**: committed data survives crashes (written to WAL/disk)

### BASE (NoSQL)
- **Basically Available**: system responds even during partial failures
- **Soft state**: state may change over time without input (due to eventual consistency)
- **Eventually consistent**: reads will eventually return latest write after all replicas converge

Interview: "I need ACID for anything involving money or inventory; BASE is fine for social feeds or analytics."

## Database Sharding & Partitioning

### What It Is
Horizontal sharding splits rows across multiple machines (shards). Each shard owns a disjoint subset of data.

### Shard Key Selection Criteria
- **High cardinality**: enough distinct values to distribute evenly
- **Even distribution**: avoid hotspot shards (e.g., shard by user_id not by country)
- **Access locality**: queries should hit as few shards as possible

### Range vs Hash Sharding

| | Range Sharding | Hash Sharding |
|---|---|---|
| Distribution | Groups by value range | Uniform spread via hash function |
| Range queries | Efficient (single shard scan) | Scattered across all shards |
| Hotspots | Hot ranges possible (e.g., recent timestamps) | Unlikely — hash distributes evenly |
| Use for | Time-series data where range scans dominate | General key-value, user data |

### Common Sharding Problems
- **Cross-shard queries**: joins across shards require scatter-gather; expensive
- **Rebalancing**: adding a shard requires migrating data; consistent hashing minimizes this
- **Hotspot shards**: one shard receives disproportionate load — add virtual nodes or re-shard

---

## Consistent Hashing

- Arrange hash space as a ring (0 to 2^32)
- Each node owns a portion of the ring
- Key maps to the nearest node clockwise on the ring
- Adding/removing a node: only K/N keys need to remap (K = keys, N = nodes) — far better than modulo hashing
- **Virtual nodes**: each physical node has multiple positions on the ring — ensures even distribution even with heterogeneous node capacities
- Used by: Cassandra, DynamoDB, Amazon's internal systems

## Replication

### Master-Slave (Primary-Replica)
- All writes go to master (primary); replicas serve reads
- Replication lag: replicas may be seconds behind master — reads may return stale data
- Read replicas dramatically increase read throughput (typical ratio: 1 primary, 5 replicas)
- On primary failure: promote a replica; gap in replication = potential data loss

### Master-Master (Multi-Primary)
- Both nodes accept writes; conflicts resolved via timestamps or application logic
- Enables geographic write distribution
- Conflict resolution is hard — last-write-wins loses data; avoid unless truly needed

### Sync vs Async Replication
| | Sync | Async |
|---|---|---|
| Durability | No data loss — write confirmed only after replica acknowledges | Potential data loss if primary crashes before replica catches up |
| Write latency | Higher — waits for replica | Lower — fire and forget |
| Use for | Financial systems, primary data of record | Analytics replicas, read replicas where slight lag is acceptable |

## Indexing Strategies

### B-Tree Index (Default)
- Balanced tree structure; O(log n) lookups
- Excellent for: equality lookups, range queries, ORDER BY, BETWEEN
- Default index type in PostgreSQL, MySQL

### Hash Index
- O(1) equality lookup via hash function
- No support for range queries or ordering
- Use only when all queries are exact-match (e.g., session cache)

### Composite Index
- Index on multiple columns: (A, B, C)
- **Leftmost prefix rule**: index is used for queries on A, A+B, or A+B+C — not B alone or C alone
- Order matters: put high-selectivity, equality-first columns leftmost
- Example: index on (user_id, created_at) serves queries filtering by user_id with date ranges

### Covering Index
- Index contains all columns a query needs — no table heap lookup required
- Dramatically faster for frequently run queries on large tables
- Trade-off: larger index, slower writes

### When NOT to Index
- Low-cardinality columns (e.g., boolean, enum with 3 values) — optimizer won't use it
- Write-heavy tables — every write must update all indexes; too many indexes kill write throughput
- Small tables — full scan is faster than index lookup below ~100K rows in most cases

## NewSQL

- ACID compliance + horizontal scaling + distributed SQL queries
- Best of both worlds but higher operational complexity
- **CockroachDB**: PostgreSQL-compatible, auto-sharding, multi-region, ACID
- **Google Spanner**: globally distributed, external consistency, TrueTime clock
- **TiDB**: MySQL-compatible, HTAP (handles both transactional and analytical workloads)
- Use when: you need SQL semantics but must scale writes horizontally across regions

## STAR Interview Framework

**Situation:** "Design the database layer for a social network expecting 100M users, 500M posts, and 10B likes. The main access patterns are: user profile reads, user feed reads, and post like counts."

**Task:** "Select appropriate databases, design the sharding strategy, and handle the read-heavy access pattern."

**Action:**
1. User profiles: PostgreSQL (relational, ACID for account changes); sharded by user_id using consistent hashing across 16 shards
2. Feed data (posts, timelines): Cassandra — high write throughput, wide-column model fits time-ordered feed queries, AP consistency acceptable for feed
3. Like counts: Redis counters (INCR is atomic); async background job persists to DB; approximate counts fine for display
4. Read replicas: 1 primary + 5 replicas per shard for user profile reads; 99% of traffic is reads
5. Composite index on (user_id, created_at DESC) for feed queries — covering index with content column for zero heap lookups

**Result:** System handles 50K writes/sec and 500K reads/sec. Feed p99 latency under 50ms. Profile reads under 20ms via read replicas. Like counts served from Redis at sub-millisecond latency.

## Flashcards

| Q | A |
|---|---|
| When should you choose SQL over NoSQL? | Relational data with joins, ACID transactions required, stable schema, complex queries — financial data, user accounts, inventory |
| What is the leftmost prefix rule for composite indexes? | Index on (A, B, C) is usable for queries filtering on A, A+B, or A+B+C — not B or C alone |
| What problem does consistent hashing solve? | When a node is added/removed, only K/N keys need to remap instead of rehashing everything (K=keys, N=nodes) |
| Range vs hash sharding — when to use each? | Range: efficient range scans, risk of hot ranges (use for time-series); Hash: even distribution, no range scan support (use for user data) |
| What is replication lag and why does it matter? | Async replicas may be behind primary by seconds — reads from replicas can return stale data; matters for consistency-sensitive reads |
| What is a covering index? | An index that contains all columns a query needs — the query is answered entirely from the index with no table heap access |
| ACID: what does Isolation guarantee? | Concurrent transactions behave as if they executed serially — one does not see the intermediate state of another |
| What are the three problems with database sharding? | Cross-shard queries (scatter-gather), hotspot shards (uneven key distribution), and rebalancing complexity when adding nodes |
