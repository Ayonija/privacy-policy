# Day 155 — Messaging & Event-Driven Systems
**Week 23 | Phase: HLD Mastery | Month 5**

## Focus
Messaging infrastructure is what transforms a brittle synchronous system into a resilient, scalable one. The decisions here — queue vs stream, delivery guarantee, saga vs 2PC — directly determine whether your architecture survives partial failures at scale. Get the semantics wrong and you get either silent data loss or duplicate processing; both are production disasters.

---

## Message Queues (Point-to-Point)

### Core Model
- One producer sends a message; exactly one consumer processes it
- Message is deleted from queue after successful acknowledgment
- Retry: if consumer fails, message returns to queue for redelivery

### RabbitMQ
- AMQP protocol; rich routing via exchanges and bindings
- **Exchange types**: Direct (exact routing key), Topic (wildcard routing), Fanout (broadcast), Headers
- Supports message priority, dead-letter routing, delayed messages
- Best for: complex routing rules, heterogeneous consumers, when you need per-message flexibility

### AWS SQS
- Managed, serverless, scales infinitely
- **Standard queue**: at-least-once delivery, best-effort ordering, highest throughput
- **FIFO queue**: exactly-once processing, strict order, 300 TPS (or 3K with batching)
- Visibility timeout: message hidden from other consumers while being processed; if not deleted within timeout, becomes visible again
- DLQ: built-in after N failed delivery attempts

### When to Use Queues
- Task distribution: one job processed by one worker
- Background jobs: send email, resize image, generate report
- Rate leveling: smooth out traffic spikes; queue absorbs burst, workers drain at steady rate
- Retry with backoff: queue handles retry scheduling

## Event Streaming (Kafka)

### Core Model
- **Durable log**: messages (events) written to ordered, immutable log; persisted for configurable retention (default 7 days)
- **Topic**: logical stream; partitioned for scale
- **Partition**: ordered sequence of records; parallelism unit
- **Consumer group**: multiple consumers share partitions; each partition assigned to exactly one consumer in the group at a time
- **Offset**: consumer's position in a partition; consumers commit offsets to track progress

### Kafka Architecture

```
Producers → [Topic: orders]
               Partition 0: [e1, e2, e3, ...]
               Partition 1: [e4, e5, e6, ...]
               Partition 2: [e7, e8, e9, ...]
          ↓
Consumer Group A (order-service): reads all partitions
Consumer Group B (analytics-service): independently reads all partitions
Consumer Group C (audit-service): independently reads all partitions
```

### Kafka vs Queue: Decision Table

| Dimension | Queue (RabbitMQ/SQS) | Kafka |
|---|---|---|
| Message consumption | Once — deleted after ACK | Multiple times — consumers read independently |
| Retention | Until processed | Configurable time window (days/weeks) |
| Ordering | Per-queue (FIFO only in SQS FIFO) | Per-partition; within partition is strictly ordered |
| Replay | No | Yes — seek to any offset |
| Throughput | Moderate | Millions of messages/sec |
| Consumer independence | Shared — one message, one consumer | Each consumer group maintains its own offset |
| Best for | Task distribution, background jobs | Event sourcing, audit logs, multiple downstream consumers, stream processing |

### Kafka Use Cases
- Event sourcing: every state change is an event; replay to rebuild state
- Audit logs: immutable record of all events; replay for compliance
- Multiple downstream consumers: analytics + search + ML all consume the same order events
- Stream processing: Kafka Streams or Flink processes events as they arrive
- Change Data Capture (CDC): DB changes (via Debezium) published to Kafka

## Pub/Sub Pattern

- Publisher emits events without knowledge of subscribers
- Event bus routes events to all registered subscribers
- Decouples producers from consumers completely

### Implementations
- **AWS SNS**: managed pub/sub; fan-out to SQS, Lambda, HTTP, email, SMS
- **Google Cloud Pub/Sub**: managed, high-throughput, push or pull delivery
- **Redis Pub/Sub**: in-memory, no persistence, at-most-once — use only for ephemeral notifications
- **Kafka topics**: can be used as pub/sub with multiple consumer groups

### SNS + SQS Fan-Out Pattern
```
SNS Topic (OrderPlaced)
  → SQS Queue (inventory-service)
  → SQS Queue (email-service)
  → SQS Queue (analytics-service)
```
Each downstream service processes at its own pace with independent retries.

## Dead Letter Queues (DLQ)

- Message routed to DLQ after N failed processing attempts
- Isolates "poison messages" (malformed, unexpected data) from blocking the main queue
- Enables: manual inspection, root cause analysis, reprocessing after fix
- Always configure DLQ in production; a queue without DLQ means failed messages disappear silently

### Best Practices
- Alert on DLQ depth > 0 in production
- Store enough context in the message to diagnose failure
- Build tooling for DLQ reprocessing — you will need it

## Delivery Semantics

| Semantic | Behavior | Risk | Requirement |
|---|---|---|---|
| At-most-once | Send once; no retry on failure | Data loss | Low-value events where loss is tolerable |
| At-least-once | Retry until ACK received | Duplicate messages | Idempotent consumers required |
| Exactly-once | Delivered and processed exactly once | Complex, high overhead | Kafka transactions + idempotent producer, or dedup logic |

### Idempotent Consumer Design
- Assign each message a unique ID
- Before processing: check if ID already processed (in DB or Redis)
- If already processed: skip; if not: process + record ID atomically
- Critical for: payment processing, order creation, any state mutation

### Kafka Exactly-Once Semantics (EOS)
- Requires: idempotent producer (sequence numbers) + transactional API
- Producer deduplicates retries; consumer offset commit and DB write happen in same transaction
- Performance cost: ~10-20% lower throughput vs at-least-once

### Interview: Always Clarify
"What delivery guarantee does your system need? At-least-once with idempotent consumers is the most practical choice for most systems — it's simpler than exactly-once and only requires idempotency logic at the consumer."

## Event Sourcing

### Core Concept
- Store the sequence of events that led to current state, not the current state itself
- Current state = replay all events from the beginning (or from a snapshot)
- Immutable log — events are never updated or deleted

### Benefits
- Complete audit trail: every state change is recorded
- Time travel: reconstruct state at any point in history
- Debugging: replay event sequence to reproduce bugs
- Event replay: feed events to new services without re-engineering data pipelines

### Drawbacks
- Eventual consistency by default
- Query complexity: cannot directly query current state without a read model
- Snapshot management: long event log → must snapshot periodically for performance

## CQRS (Command Query Responsibility Segregation)

### Core Concept
- **Command side (write model)**: handles state changes; optimized for writes; enforces business rules
- **Query side (read model)**: optimized for reads; denormalized views for fast queries
- Sync via events: write model emits events; read model consumes and materializes

```
Client → Command → Write Model (validates + persists events) → Event Bus
                                                                     ↓
Client ← Query  ← Read Model (materialized view, optimized for read queries)
```

### When to Use CQRS
- Read and write workloads have very different scale/shape requirements
- Complex domain with many read models needed from same data
- Often paired with event sourcing — events are the sync mechanism

### When Not to Use CQRS
- Simple CRUD apps — adds complexity without benefit
- Small teams without clear domain boundaries

## STAR Interview Framework

**Situation:** "Design an order processing system for an e-commerce platform that must handle 100K orders per second, guarantee no orders are lost, and notify inventory, payments, and notifications services for each order."

**Task:** "Choose the messaging architecture, delivery guarantees, and failure handling strategy."

**Action:**
1. Order service writes to DB and publishes `OrderPlaced` event to Kafka topic (orders) atomically via outbox pattern — guarantees event is not lost if Kafka is down
2. Kafka topic with 50 partitions for 100K/sec throughput; partition by order_id for per-order ordering
3. Three consumer groups: inventory-service, payment-service, notification-service — each processes independently at own pace
4. At-least-once delivery; all consumers made idempotent using order_id as dedup key checked in Redis
5. DLQ configured: orders failing 3× routed to DLQ; alerting triggers on DLQ depth > 10; ops team has reprocessing tooling
6. Outbox pattern: DB transaction writes order + outbox record atomically; relay process polls outbox and publishes to Kafka; deletes after confirm

**Result:** System handles 120K orders/sec in load tests. Zero order loss. Payment service processes at 80K/sec without blocking order ingestion. DLQ caught 0.01% malformed orders during rollout — reprocessed after fix with no data loss.

## Flashcards

| Q | A |
|---|---|
| Kafka vs Queue: why can Kafka serve multiple consumers for the same events? | Kafka is a durable log; messages are not deleted after consumption; each consumer group maintains its own offset and reads independently |
| What is the practical difference between at-least-once and exactly-once? | At-least-once can produce duplicates — requires idempotent consumers; exactly-once prevents duplicates but has ~10-20% throughput overhead via transactions |
| What is a Dead Letter Queue used for? | Isolates messages that fail processing N times; enables manual inspection, root cause analysis, and reprocessing after fix |
| Event Sourcing: what is the current state? | The result of replaying all events from the log (or from the latest snapshot); state is never stored directly |
| CQRS: what is the purpose of separating the write model from the read model? | Write model is optimized for business rule enforcement; read model is denormalized and optimized for fast query patterns — they can scale independently |
| What is the outbox pattern and why is it needed? | Write event to an outbox table in the same DB transaction as the business write; relay process publishes to Kafka — guarantees no event is lost even if Kafka is down at write time |
| SNS + SQS fan-out: why use SQS queues as SNS subscribers instead of Lambda directly? | SQS decouples processing rate and provides retry/DLQ — Lambda subscribed directly has no queue buffer and can overwhelm downstream systems on bursts |
| Kafka partition: what does it guarantee and what does it not? | Within a partition: strict ordering; across partitions: no ordering guarantee — partition by a meaningful key (e.g., user_id) to preserve per-entity ordering |
