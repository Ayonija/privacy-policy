## DSA (1.5 hours — Revision Mode)

### Pattern Revision: Advanced Graphs (Dijkstra, Bellman-Ford, 0-1 BFS, MST)
- Core idea: Dijkstra greedily expands the closest unvisited node using a min-heap, guaranteeing shortest paths in graphs with non-negative weights. Bellman-Ford relaxes all edges N-1 times, handling negative weights and detecting negative cycles. 0-1 BFS uses a deque where cost-0 edges push to front and cost-1 edges push to back, giving O(V+E) for binary-weight graphs.
- Trigger condition: Use Dijkstra when weights ≥ 0 and you need single-source shortest paths. Use Bellman-Ford when weights can be negative or you need exactly K relaxations. Use 0-1 BFS when edge weights are only 0 or 1.
- Time complexity: Dijkstra O((V+E) log V) | Bellman-Ford O(V·E) | 0-1 BFS O(V+E) | Prim/Kruskal MST O(E log E)
- Space complexity: O(V+E) for adjacency list + dist array

### Template (write from memory before solving):
```java
// Dijkstra — single-source shortest path (non-negative weights)
int[] dist = new int[n]; Arrays.fill(dist, Integer.MAX_VALUE); dist[src] = 0;
PriorityQueue<int[]> pq = new PriorityQueue<>(Comparator.comparingInt(a -> a[0]));
pq.offer(new int[]{0, src});
while (!pq.isEmpty()) {
    int[] curr = pq.poll(); int d = curr[0], u = curr[1];
    if (d > dist[u]) continue; // stale entry — skip
    for (int[] edge : graph.get(u)) {
        int v = edge[0], w = edge[1];
        if (dist[u] + w < dist[v]) { dist[v] = dist[u] + w; pq.offer(new int[]{dist[v], v}); }
    }
}

// 0-1 BFS — edge weights are only 0 or 1
int[] dist01 = new int[n]; Arrays.fill(dist01, Integer.MAX_VALUE); dist01[src] = 0;
Deque<int[]> dq = new ArrayDeque<>(); dq.offerFirst(new int[]{0, src});
while (!dq.isEmpty()) {
    int[] curr = dq.pollFirst(); int d = curr[0], u = curr[1];
    if (d > dist01[u]) continue;
    for (int[] edge : graph.get(u)) {
        int v = edge[0], w = edge[1];
        if (dist01[u] + w < dist01[v]) {
            dist01[v] = dist01[u] + w;
            if (w == 0) dq.offerFirst(new int[]{dist01[v], v});
            else dq.offerLast(new int[]{dist01[v], v});
        }
    }
}

// Bellman-Ford — relax K+1 times for "at most K stops" variant
int[] prices = new int[n]; Arrays.fill(prices, Integer.MAX_VALUE); prices[src] = 0;
for (int i = 0; i <= K; i++) {
    int[] temp = Arrays.copyOf(prices, n); // snapshot: prevent chaining in same round
    for (int[] flight : flights) { // [from, to, cost]
        int from = flight[0], to = flight[1], cost = flight[2];
        if (prices[from] != Integer.MAX_VALUE && prices[from] + cost < temp[to])
            temp[to] = prices[from] + cost;
    }
    prices = temp;
}
```

## System Design (30 min — HLD Component Review)

### Topic: CAP Theorem in Practice + Distributed Systems Fundamentals

**CAP Theorem:** A distributed system can only guarantee 2 of 3 properties simultaneously:
- **Consistency (C):** Every read receives the most recent write or an error — all nodes see the same data at the same time.
- **Availability (A):** Every request receives a response (not necessarily the most recent data) — no timeouts, no errors.
- **Partition Tolerance (P):** The system continues operating even when network partitions split nodes. In practice, network partitions always happen, so P is non-negotiable — the real choice is C vs A during a partition.

**CP Systems (sacrifice Availability):**
- ZooKeeper, HBase, etcd
- On a partition, refuse requests rather than serve stale data
- Use when: distributed coordination, leader election, configuration management, payment processing

**AP Systems (sacrifice Consistency):**
- Cassandra, DynamoDB, CouchDB
- On a partition, serve possibly stale data; converge to the correct state eventually
- Eventual consistency: all nodes will agree eventually if writes stop
- Use when: social feeds, shopping carts, analytics, any read-heavy workload where stale-by-seconds is acceptable

**Consistency models (weakest to strongest):**
1. Eventual consistency — replicas converge eventually; no ordering guarantees
2. Monotonic reads — you never read data older than what you already read
3. Read-your-writes — after you write, you always read your own write
4. Causal consistency — causally related operations appear in order to all nodes
5. Linearizability — strongest; every operation appears instantaneous; total ordering

**Vector clocks:**
- Each node maintains a counter per node in the cluster
- Used to detect causality (which write happened before which) and conflicts without a global clock
- Conflict detected when neither vector clock dominates the other → need application-level merge (e.g., "last write wins" or user resolution)

**Key numbers to memorize:**
- Network round trip within same datacenter: ~0.5ms
- Network round trip cross-region: ~50–150ms
- ZooKeeper leader election on failure: ~200ms
- Cassandra eventual consistency convergence: typically < 1 second

**Interview talking point:** "If asked which to choose for a payment system vs a social feed, answer: payment → CP (ZooKeeper/Postgres with strong consistency; losing a transaction is worse than being unavailable for 30 seconds — the user will retry). Social feed → AP (Cassandra/DynamoDB; seeing a like 2 seconds late is acceptable; never refusing a read is more important than perfect consistency)."
