# Day 154 — Caching: Strategies, Redis, CDN, Cache Stampede
**Week 23 | Phase: HLD Mastery | Month 5**

## Focus
Caching is the single highest-leverage optimization in most read-heavy systems, capable of reducing DB load by 90%+ and cutting latency from hundreds of milliseconds to single digits. The challenge is not adding a cache — it is choosing the right write strategy, handling invalidation correctly, and preventing stampede failures that can take down the DB.

---

## Cache Write Strategies

### Cache-Aside (Lazy Loading) — Most Common
- App checks cache → cache miss → read from DB → write to cache → return data
- Cache is populated on demand; cold cache means initial requests are slow (cache miss penalty)
- Stale data risk: DB updated by another path, cache not invalidated
- Best for: read-heavy workloads where slight staleness is acceptable

```
read(key):
  value = cache.get(key)
  if value == null:
    value = db.read(key)
    cache.set(key, value, TTL)
  return value
```

### Write-Through
- Write to cache AND DB simultaneously in the same operation
- Cache always consistent with DB — no stale reads
- Higher write latency (two writes per operation)
- Wasted cache space: write all data, but only hot data gets read
- Best for: write then frequently read patterns; consistency-critical data

### Write-Back (Write-Behind)
- Write to cache only; async background job flushes to DB
- Lowest write latency — fastest write path
- Risk: data loss if cache crashes before flush
- Best for: non-critical high-frequency writes (metrics, counters, analytics)

### Write-Around
- Write directly to DB, bypass cache
- Cache populated only on reads (like cache-aside)
- Best for: write-once, read-rarely data (logs, audit trails) — no point caching what won't be read

### Decision Matrix

| Strategy | Read perf | Write perf | Consistency | Data loss risk |
|---|---|---|---|---|
| Cache-aside | High (after warm-up) | Normal | Eventual | None |
| Write-through | High | Lower (2 writes) | Strong | None |
| Write-back | High | Highest | Eventual | Yes (on crash) |
| Write-around | Normal (first read slow) | Normal | Strong | None |

## Eviction Policies

### LRU (Least Recently Used)
- Evict item not accessed for the longest time
- Works well for temporal locality — recently accessed data likely to be accessed again
- General-purpose default; good for most web workloads

### LFU (Least Frequently Used)
- Evict item with fewest total accesses
- Better for popularity-based data (trending content, top products)
- Expensive to track: requires maintaining access counts; LRU is simpler

### TTL (Time to Live)
- Expire data after a fixed duration regardless of access frequency
- Essential for session data, rate limit windows, OTP codes, any time-sensitive data
- Combine with LRU: TTL sets expiry ceiling; LRU handles eviction within budget

### Other Policies
- **FIFO**: evict oldest-inserted item; rarely optimal
- **Random**: simple to implement; surprisingly effective in some workloads
- **ARC (Adaptive Replacement Cache)**: maintains two lists (recency + frequency); adapts dynamically; used in ZFS

## Redis vs Memcached

| Dimension | Redis | Memcached |
|---|---|---|
| Data structures | String, List, Set, Sorted Set, Hash, HyperLogLog, Stream | String only |
| Persistence | RDB snapshots + AOF log | None |
| Pub/Sub | Yes | No |
| Scripting | Lua scripts for atomicity | No |
| Clustering | Redis Cluster (sharding built in) | Client-side sharding only |
| Multi-threading | Single-threaded (I/O async) | Multi-threaded |
| Memory efficiency | Slightly higher overhead | Slightly leaner |

### When to Choose Memcached
- Pure caching, no persistence needed
- Multi-threaded CPU utilization matters
- Simpler ops with no need for data structures

### When to Choose Redis (Most Cases)
- Need sorted sets (leaderboards), lists (activity feeds), sets (unique visitors)
- Need pub/sub (live notifications)
- Need atomic operations via Lua (rate limiting, distributed locks)
- Need persistence (cache survives restarts)
- Need Redis Sentinel or Cluster for HA

## CDN (Content Delivery Network)

### What It Does
- Globally distributed edge nodes cache content close to end users
- Requests served from nearest edge, not origin server — reduces latency and origin load

### Pull CDN
- Edge fetches content from origin on first request; caches it with TTL
- Origin load = cache miss rate × total requests (much lower than direct origin traffic)
- Simple: no pre-loading logic needed
- Examples: Cloudflare, AWS CloudFront (default behavior)

### Push CDN
- You proactively push content to edge nodes before users request it
- Use for: large media files, release deployments, content with known high demand
- More control over what's at the edge; requires push/invalidation automation

### What to Cache at CDN
- Always: images, videos, fonts, JS/CSS bundles, static HTML
- Carefully: API responses (only truly cacheable, public, non-personalized data)
- Never: authenticated per-user data, real-time prices, personalized content

### Cache-Control Headers
- `Cache-Control: max-age=31536000, immutable` — versioned static assets (hash in filename)
- `Cache-Control: no-cache` — must revalidate with origin before serving
- `Cache-Control: private` — browser only, not CDN

## Cache Invalidation — The Hard Problem

"There are only two hard things in computer science: cache invalidation and naming things." — Phil Karlton

### TTL-Based Invalidation
- Set expiry; cache serves stale until TTL expires
- Simple to implement; stale window = TTL duration
- Problem: can't immediately invalidate when data changes

### Event-Driven Invalidation (Cache-Aside Delete)
- DB write triggers cache.delete(key)
- Next read fetches fresh data from DB
- Problem: race condition — delete happens before new write is visible to all DB replicas

### Write-Through Invalidation
- Always update cache on write — no explicit invalidation needed
- Eliminates staleness but all writes touch cache; not ideal for write-heavy paths

### Versioned Cache Keys
- Embed version/hash in key: `user:123:v5`
- "Invalidate" by incrementing version — old key naturally expires via TTL
- Useful for distributed caches where coordinated delete is expensive

## Cache Stampede / Thundering Herd

### The Problem
1. Popular cache key expires
2. Thousands of requests simultaneously get cache miss
3. All go to DB simultaneously
4. DB is overwhelmed, latency spikes, potential cascade failure

### Solutions

**Mutex / Lock on Cache Miss**
- First miss acquires lock and fetches from DB; others wait
- Avoids DB overload; adds lock contention overhead
- Implementation: Redis `SET NX EX` (atomic set-if-not-exists with expiry)

**Probabilistic Early Expiration (XFetch)**
- Simulate expiry before actual TTL; probability of early re-fetch increases as TTL approaches 0
- Spreads recomputation across time window without coordination
- Formula: re-fetch if `currentTime − delta × beta × log(rand()) > expiry`

**Background Refresh**
- Background job pre-emptively refreshes cache before expiry
- Cache never goes cold for popular keys
- Works well when content update frequency is predictable

**Request Coalescing**
- Queue duplicate in-flight requests for same cache key; make one DB call, fan out result to all waiters
- Common in CDN edge servers

## STAR Interview Framework

**Situation:** "Design a caching layer for a leaderboard showing the top 1000 players by score, accessed by 10M users daily. Scores update in real-time from game results."

**Task:** "Ensure the leaderboard loads in < 50ms p99 while handling real-time score updates without serving stale data for more than 10 seconds."

**Action:**
1. Redis Sorted Set: `ZADD leaderboard score user_id` for O(log N) updates; `ZREVRANGE` for top-1000 reads
2. Cache-aside at application layer: check Redis → serve top-1000 JSON blob; regenerate from Redis Sorted Set if blob expires
3. TTL of 10 seconds on serialized top-1000 blob; Redis Sorted Set itself always current
4. Write path: game result service writes to DB + calls `ZADD` on Redis Sorted Set atomically (Lua script for multi-key atomicity)
5. Stampede protection: mutex lock on regeneration of serialized blob; probabilistic early expiration at 8s TTL to pre-warm before 10s expiry

**Result:** Leaderboard reads served from Redis at 2ms p99. DB never queried for leaderboard reads. Score updates reflected within 10 seconds. Zero thundering herd incidents during peak load (100K concurrent users post-tournament).

## Flashcards

| Q | A |
|---|---|
| Cache-aside: describe the read path | Check cache → miss → read DB → write to cache with TTL → return; cache populated lazily on demand |
| Write-through vs write-back: key tradeoff | Write-through: consistent, higher write latency; Write-back: lowest write latency, risk of data loss on cache crash |
| When does write-around make sense? | Write-once, read-rarely data like logs or audit trails — no benefit caching what won't be read |
| LRU vs LFU eviction: when to choose LFU? | LFU when data has clear popularity tiers (trending content, top-N lists); LRU for general temporal locality |
| What is a cache stampede and one solution? | Many requests hit DB simultaneously when a popular key expires; solve with mutex lock on miss, probabilistic early expiration, or background refresh |
| Redis sorted set: what operation is used for a leaderboard? | ZADD (update score), ZREVRANGE (top N by score descending) — both O(log N) |
| Pull vs Push CDN: when to use Push? | Push CDN when content is large, demand is predictable, or you need guaranteed edge presence before first user request |
| Cache invalidation race condition: what is it? | Delete-before-write: cache is deleted, but the new write hasn't propagated to all replicas yet — a read between delete and replication gets stale data back and re-caches it |
