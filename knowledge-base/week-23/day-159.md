## DSA (1.5 hours — Revision Mode)

### Pattern Revision: DP Part 2 (Interval DP, Game Theory DP, LCS)
- Core idea: Interval DP solves subproblems over contiguous subarrays of increasing length — outer loop iterates over lengths, inner loops fix start and end, innermost loop tries all split points. Game theory DP (minimax) models two-player zero-sum games: dp[i] = the best score the current player can achieve from position i, computed as the maximum of all choices minus what the opponent will optimally collect from the remaining subproblem.
- Trigger condition: Use interval DP when the optimal solution for a range depends on optimal solutions for all sub-ranges (Matrix Chain Multiplication, Burst Balloons, Optimal BST). Use game theory DP when two players alternate turns and each plays optimally.
- Time complexity: Interval DP O(N³) | Game Theory DP (1D) O(N) | LCS O(M×N)
- Space complexity: Interval DP O(N²) | Game Theory DP O(N) | LCS O(M×N) or O(N) optimized

### Template (write from memory before solving):
```java
// Interval DP — generic template (e.g., minimum cost to merge ranges)
int[][] dp = new int[n][n];
// Base case: single elements (len = 1) usually 0 or precomputed
for (int len = 2; len <= n; len++) {           // increasing subproblem size
    for (int i = 0; i + len - 1 < n; i++) {   // all start positions
        int j = i + len - 1;                   // end position
        dp[i][j] = Integer.MAX_VALUE;          // initialize (use 0 for max problems)
        for (int k = i; k < j; k++)            // try all split points
            dp[i][j] = Math.min(dp[i][j], dp[i][k] + dp[k+1][j] + cost(i, k, j));
    }
}
// Answer: dp[0][n-1]

// Game Theory DP — Stone Game III style (player takes 1, 2, or 3 stones)
// dp[i] = max stones current player can collect from stones[i..n-1]
int[] dp_game = new int[n + 1]; // dp[n] = 0 (base case)
for (int i = n - 1; i >= 0; i--) {
    dp_game[i] = Integer.MIN_VALUE;
    int take = 0;
    for (int t = 1; t <= 3 && i + t - 1 < n; t++) {
        take += stones[i + t - 1];
        dp_game[i] = Math.max(dp_game[i], take - dp_game[i + t]);
        // current player takes 'take'; opponent plays optimally from i+t
    }
}
// If dp[0] > 0 → Alice wins; == 0 → Tie; < 0 → Bob wins
```

## System Design (30 min — HLD Component Review)

### Topic: Microservices Architecture Patterns

**Service Discovery:**
- Problem: with dynamic scaling and container orchestration (Kubernetes), service IPs change constantly.
- DNS-based discovery: AWS ELB / Route53 — DNS A records point to load balancer; automatic health-check-based routing; works for most cases.
- Client-side discovery (Consul/Eureka): services register on startup with their IP:port; deregister on shutdown (or heartbeat timeout); clients query registry to get live instance list and load-balance themselves.
- Health checks: periodic HTTP /health endpoint checks; unhealthy instances removed from registry automatically.

**Circuit Breaker (Hystrix / Resilience4j):**
- States: CLOSED (normal, all requests pass through) → OPEN (after N consecutive failures, fail fast without calling downstream) → HALF-OPEN (after timeout, send one probe request; if success → CLOSED; if failure → OPEN again).
- Why: prevents a slow or failing downstream service from cascading timeouts to all upstream callers. With circuit breakers, callers fail fast (< 1ms) instead of waiting 30s for timeout.
- Fallback: return cached response, default value, or degraded response when circuit is OPEN.

**API Gateway vs BFF (Backend for Frontend):**
- API Gateway: single entry point for all client types; handles auth, rate limiting, SSL termination, routing, request aggregation.
- BFF: separate backend per client type (mobile BFF, web BFF, TV BFF); each tailors the response shape to its client's needs. Reduces over-fetching (mobile doesn't need 50 fields the web uses).
- When to use BFF: when mobile and web have significantly different data needs; when you want to avoid putting client-specific logic in the shared API Gateway.

**Saga Pattern for Distributed Transactions:**
- Problem: a single business operation spans multiple services (e.g., create order → reserve inventory → charge payment → send confirmation).
- Choreography saga: each service emits an event when it completes; the next service listens and reacts. No central coordinator. Risk: hard to track overall state.
- Orchestration saga: a central Saga Orchestrator sends commands to each service and listens for replies. Easier to trace and debug.
- Compensating transactions: if step N fails, steps N-1 through 1 must be "undone" with compensating actions (e.g., release reserved inventory if payment fails).

**Service Mesh (Istio / Linkerd):**
- A sidecar proxy (Envoy) is injected into every pod. All inter-service traffic flows through sidecars.
- Handles: mTLS (mutual TLS between services), retries, circuit breaking, distributed tracing, traffic splitting — without any application code changes.
- Adds ~5ms latency per hop (sidecar overhead). Worth it at scale (100+ services).

**Key numbers:**
- Circuit breaker failure threshold: typically 5 failures in 10 seconds before OPEN
- Service mesh sidecar overhead: ~5ms per request, ~3% CPU overhead
- Saga compensating transaction SLA: typically same or less time than original transaction

**Interview talking point:** "If asked how Netflix handles a downstream service failure without taking down the homepage, answer: each service call is wrapped in a circuit breaker with a fallback — for example, the recommendation service circuit breaker returns a cached or generic list instead of failing the homepage render. When the circuit opens, calls fail in < 1ms instead of timing out after 30 seconds, which prevents thread pool exhaustion in the caller. Netflix calls this 'graceful degradation': show something useful rather than an error page."
