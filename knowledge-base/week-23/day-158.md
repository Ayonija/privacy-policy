# Day 158 — Observability + Reliability Fundamentals
**Week 23 | Phase: HLD Mastery | Month 5**

## Focus
A system that cannot be observed cannot be debugged. A system with no failure-handling strategy will cascade. This day covers the three pillars of observability, the reliability patterns that distinguish senior-level designs, and the distributed transaction alternatives that come up repeatedly in system design interviews.

---

## The Three Pillars of Observability

Observability answers: "What is the system doing right now, and why is it doing that?"

| Pillar | Answers | Tools |
|---|---|---|
| Logs | What happened step-by-step in one request? | ELK, Loki + Grafana, Splunk |
| Metrics | What is the system's aggregate health over time? | Prometheus, Datadog, CloudWatch |
| Traces | How did this single request flow through all services? | Jaeger, Zipkin, OpenTelemetry |

---

## Logging

### Structured Logging
- Log as JSON, not free-form strings: `{"level":"ERROR","service":"order-service","request_id":"abc123","error":"DB timeout","latency_ms":5002}`
- Machine-parseable: can be queried, filtered, aggregated in log platforms
- Unstructured: `ERROR: DB timeout for order service` — hard to parse at scale

### Centralized Logging
- **ELK Stack**: Logstash (ingest + transform) → Elasticsearch (index + store) → Kibana (visualize)
- **Loki + Grafana**: Prometheus-style labels for logs; cheaper than ELK (no full-text index); good for log lines filterable by labels
- Agents: Filebeat, Fluentd, Fluent Bit ship logs from hosts to central store

### Log Levels
- DEBUG: verbose; never in production for high-throughput services
- INFO: normal operations; requests, state changes
- WARN: unexpected but recoverable; degraded fallback used
- ERROR: operation failed; requires investigation; alert candidates

### Correlation ID (Request ID)
- Assign UUID at request entry point (API Gateway or load balancer)
- Thread through all downstream service calls via header (e.g., `X-Request-ID`)
- Include in every log line: enables reconstructing full request journey across services
- Essential for debugging distributed systems

## Metrics & Monitoring

### RED Method (Service Health)
- **Rate**: requests per second — is the service handling expected load?
- **Errors**: error rate (%) — is the service failing requests?
- **Duration**: request latency (p50, p95, p99) — is the service slow?
- Use for: all user-facing and service-to-service endpoints

### USE Method (Resource Health)
- **Utilization**: how busy is the resource? (CPU %, disk I/O %)
- **Saturation**: how much work is queued/waiting? (run queue length, I/O wait)
- **Errors**: error events on the resource (disk errors, network drops)
- Use for: infrastructure resources (CPU, memory, disk, network)

### Prometheus + Grafana
- Prometheus: **pull-based** scraping; periodically fetches `/metrics` from each service
- PromQL: query language for aggregations, rates, percentiles
- Grafana: visualization dashboards on top of Prometheus and other sources
- Alertmanager: routes alerts (PagerDuty, Slack, email) when alert rules fire

### Alerting Philosophy
- **Alert on SLO breach**, not raw metrics: "p99 latency > 500ms for 5 minutes" is actionable; "CPU > 80%" often is not
- Avoid alert fatigue: every alert must be actionable; noisy alerts get ignored
- Symptoms over causes: alert on "user requests failing" not "database connection pool exhausted" — the DB issue is the cause, the user impact is what matters

## Distributed Tracing

### Concepts
- **Trace**: complete journey of a single request through all services — spans a call tree
- **Span**: one service's work within a trace; has start time, duration, metadata, status
- **Parent-child relationship**: Span in Service B was called by Span in Service A → parent-child
- **Trace ID**: unique ID assigned to the entire trace; propagated via HTTP headers

### Header Propagation
- **B3 propagation** (Zipkin): `X-B3-TraceId`, `X-B3-SpanId`, `X-B3-ParentSpanId`
- **W3C TraceContext**: `traceparent`, `tracestate` — standardized, increasingly adopted
- Must propagate headers in every outbound call; if you drop the header, the trace breaks

### Tools
- **Jaeger**: CNCF project; open source; Uber-origin; self-hosted or cloud
- **Zipkin**: Twitter-origin; lightweight; B3 headers
- **OpenTelemetry (OTel)**: vendor-neutral SDK + specification; instrument once, export to any backend (Jaeger, Datadog, Honeycomb)
- Best practice: instrument with OpenTelemetry; swap backend without code changes

## Circuit Breaker

### Problem
Service A calls Service B. Service B is degraded (high latency, partial failures). Without circuit breaker: Service A threads block waiting for B; thread pool exhausts; Service A also fails. Cascade failure.

### States

```
CLOSED (normal) ──failure threshold exceeded──► OPEN (fast-fail)
    ▲                                                │
    │                                           probe timer
    │                                                │
    └────── success threshold met ──────── HALF-OPEN (test recovery)
```

- **CLOSED**: requests pass through; failures counted; opens if failure rate > threshold within window
- **OPEN**: all requests immediately return error (fast-fail); no calls to B; gives B time to recover
- **HALF-OPEN**: allows a small probe percentage of requests through; if they succeed → CLOSED; if they fail → back to OPEN

### Implementation Libraries
- **Resilience4j** (Java): functional, lightweight, modern; replaces Hystrix
- **Hystrix** (Java): Netflix, now deprecated but widely referenced
- **Polly** (.NET): circuit breaker + retry + timeout
- Python: `pybreaker`, `circuitbreaker`

### Configuration Parameters
- `failureRateThreshold`: e.g., 50% failure rate in last 10 requests triggers OPEN
- `waitDurationInOpenState`: e.g., 60 seconds in OPEN before probing
- `permittedNumberOfCallsInHalfOpenState`: e.g., 5 probe calls to decide CLOSED vs OPEN

## Retry with Exponential Backoff + Jitter

### Basic Exponential Backoff
- Retry delays: 1s → 2s → 4s → 8s → 16s (2^n seconds)
- Max retries: cap at 3–5; stop retrying if circuit is broken
- **Only retry idempotent operations**: GET, PUT, DELETE — not POST (may create duplicates)

### Why Jitter
- Without jitter: all failed requests retry at the same time (t=1s, t=2s, ...) → synchronized thundering herd on recovery
- With jitter: `delay = random(0, 2^n seconds)` → spread retries over time window → gentler recovery load

### Full Algorithm
```
for attempt in 1..MAX_RETRIES:
  try:
    result = call_service()
    return result
  except RetriableError:
    if attempt == MAX_RETRIES: raise
    base_delay = 2^attempt seconds
    sleep(random(0, base_delay))  // jitter
```

## Idempotency

### Definition
Calling the same operation N times produces the same result as calling it once. State mutation is applied exactly once.

### Idempotency Key Pattern
- Client generates a UUID and includes it as `Idempotency-Key: <uuid>` header
- Server: before processing, check if key exists in idempotency store (Redis/DB)
  - If exists: return the stored response (replay)
  - If not: process request → store (key, response) atomically → return response
- Expiry: idempotency keys expire after 24–72 hours

### Critical Applications
- Payment APIs: prevent double-charges on retries
- Order creation: prevent duplicate orders
- Any state mutation that may be retried by client or message consumer

## Saga Pattern (Distributed Transactions)

### The Problem
2PC (Two-Phase Commit) is slow, blocks resources, and doesn't work across microservices with independent DBs. Saga provides an alternative without distributed locking.

### Core Concept
- Distributed transaction = sequence of local transactions
- Each step has a **compensating transaction** (rollback action)
- If step N fails: execute compensating transactions for steps N-1, N-2, ... in reverse order

### Choreography Saga
- No central coordinator; each service listens for events and emits next event
- Service A completes → emits event → Service B listens → completes → emits event → ...
- On failure: Service B emits failure event → Service A listens → executes compensation
- Pros: decoupled, no single point of failure
- Cons: hard to trace, implicit flow, difficult to reason about failure paths

### Orchestration Saga
- Central orchestrator (saga orchestrator service) directs each step explicitly
- Orchestrator calls Service A → waits → calls Service B → if failure → calls rollback on A
- Pros: explicit flow, easy to trace, single place to understand the business process
- Cons: orchestrator is a dependency; can become a bottleneck or single point of failure

### Example: Order Creation Saga
```
1. Order Service: Create order (PENDING)
   → fail: compensate: mark order CANCELLED

2. Payment Service: Reserve funds
   → fail: compensate: release funds, cancel order

3. Inventory Service: Reserve items
   → fail: compensate: release inventory, refund funds, cancel order

4. Order Service: Confirm order (CONFIRMED)
```

## Bulkhead Pattern

- Isolate failures by allocating separate thread pools (or connection pools) per downstream dependency
- If Service B is slow: its thread pool fills; Service A's calls to Service C are unaffected — separate pool
- Prevents one slow downstream from starving all threads in Service A
- Named after ship bulkheads: one compartment flooding doesn't sink the ship

---

## Graceful Degradation

- Return a degraded but valid response instead of failing completely
- Examples:
  - Recommendations service down → show "popular items" static list, not empty page
  - Search service slow → show cached previous results + "results may be stale" notice
  - Payment gateway timeout → queue payment for async retry, show "payment processing"
- Feature flags: disable non-critical features (social feed, recommendations) under load; keep core flow (checkout, auth) running

## STAR Interview Framework

**Situation:** "You are designing the checkout flow for an e-commerce platform using microservices: Order Service, Payment Service, Inventory Service. A payment can succeed but inventory reservation can fail, leaving the user charged with no order. How do you make this resilient?"

**Task:** "Design the distributed transaction, failure handling, and observability for the checkout flow."

**Action:**
1. Saga pattern (orchestration): Order Orchestrator directs the saga; each step has a compensating transaction defined upfront
2. Step sequence: CreateOrder → ReserveInventory → ChargePayment → ConfirmOrder; each step writes to its own DB atomically
3. Compensation: if ChargePayment fails → ReleaseInventory → MarkOrderFailed; if ConfirmOrder fails → RefundPayment → ReleaseInventory → MarkOrderFailed
4. Idempotency keys on all saga steps — if orchestrator crashes and restarts mid-saga, replayed steps are safe
5. Circuit breakers on Payment and Inventory service clients; if payment gateway is OPEN → fast-fail checkout with user-friendly message rather than hanging
6. Observability: correlation ID threaded through all saga steps; distributed trace visualizes full saga; metrics on saga completion rate + compensation rate; alert if compensation rate > 0.1% (indicates systematic failure)

**Result:** Checkout saga handles 10K orders/min with zero partial-charge incidents post-deployment. Compensation rate held at 0.02% (mostly inventory out-of-stock). Circuit breaker on payment gateway prevented 3 cascade failures during gateway degradation — checkout returned clear error in < 100ms vs previous 30-second timeouts.
