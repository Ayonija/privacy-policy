## DSA (1.5 hours — Revision Mode)

### Pattern Revision: DP Part 1 (1D DP, 2D DP, LCS, LIS with Patience Sort)
- Core idea: 1D DP builds a table where dp[i] represents the answer for the first i elements or the target value i. 2D DP (LCS) compares two sequences character by character, building dp[i][j] as the answer for prefixes of length i and j. LIS with patience sort replaces the naive O(N²) DP with binary search on a "tails" array, achieving O(N log N): tails[i] is the smallest tail element of all increasing subsequences of length i+1.
- Trigger condition: Use 1D DP when the problem has an optimal substructure over a single index or value. Use 2D DP when comparing two sequences. Use patience sort LIS when asked for longest increasing subsequence with strict time constraints (N up to 10^5).
- Time complexity: Coin Change O(amount × coins) | LCS O(M×N) | LIS naive O(N²) | LIS patience sort O(N log N)
- Space complexity: Coin Change O(amount) | LCS O(M×N) or O(min(M,N)) optimized | LIS O(N)

### Template (write from memory before solving):
```java
// Coin Change — 1D DP
int[] dp = new int[amount + 1];
Arrays.fill(dp, amount + 1); // sentinel for "impossible" (larger than any valid answer)
dp[0] = 0;
for (int i = 1; i <= amount; i++)
    for (int coin : coins)
        if (coin <= i) dp[i] = Math.min(dp[i], dp[i - coin] + 1);
return dp[amount] > amount ? -1 : dp[amount];

// LCS — 2D DP
int[][] dp = new int[m + 1][n + 1];
for (int i = 1; i <= m; i++)
    for (int j = 1; j <= n; j++)
        dp[i][j] = (text1.charAt(i-1) == text2.charAt(j-1)) ? dp[i-1][j-1] + 1
                                                               : Math.max(dp[i-1][j], dp[i][j-1]);
return dp[m][n];

// LIS — Patience Sort (O(N log N))
List<Integer> tails = new ArrayList<>();
for (int num : nums) {
    int pos = Collections.binarySearch(tails, num);
    if (pos < 0) pos = -(pos + 1); // insertion point
    if (pos == tails.size()) tails.add(num); // new longest subsequence
    else tails.set(pos, num); // replace to keep tails as small as possible
}
return tails.size(); // LIS length
// Note: tails is NOT the actual LIS — it is a monotonically increasing array used only for length calculation
```

## System Design (30 min — HLD Component Review)

### Topic: Authentication & Authorization Patterns

**JWT (JSON Web Token):**
- Three parts: `header.payload.signature` (Base64URL encoded, dot-separated).
- Header: algorithm (HS256 or RS256). Payload: claims (sub, iat, exp, roles). Signature: HMAC or RSA of header+payload.
- Stateless: server validates signature without any DB lookup — scales horizontally.
- Access token TTL: 15 minutes (short so compromise is limited). Refresh token TTL: 30 days, stored in HttpOnly cookie (not localStorage — prevents XSS theft).
- Trade-off: cannot invalidate a JWT before expiry without a blocklist.

**OAuth2 Flows — know which to use when:**
- Authorization Code (+ PKCE for SPAs): web apps and mobile; most secure; auth server issues code, app exchanges for token.
- Client Credentials: service-to-service (machine-to-machine); no user involved; client sends client_id + client_secret.
- Device Flow: TVs, CLIs, IoT — device shows user a code; user authenticates on a separate device.
- Implicit: deprecated (tokens exposed in URL fragment — use Authorization Code + PKCE instead).

**RBAC (Role-Based Access Control):**
- Users are assigned roles; roles have permissions (e.g., ADMIN, EDITOR, VIEWER).
- Simple to implement and reason about. Scalable for most enterprise apps.
- Use when: number of distinct permission patterns is small and bounded.
- Example: admin panel with roles like "content_editor" and "analytics_viewer".

**ABAC (Attribute-Based Access Control):**
- Policies evaluate user attributes + resource attributes + environment conditions.
- Example: "a doctor can read a patient record if doctor.department == patient.primaryDepartment AND time is business hours".
- Flexible but complex; harder to audit and debug.
- Use when: fine-grained, context-dependent access is required (multi-tenant SaaS, healthcare, finance).

**Refresh Token Rotation (security hardening):**
- On each use of a refresh token, invalidate it and issue a new one.
- If an old (already-used) refresh token is presented, it indicates token theft → invalidate the entire token family → force re-login.
- This limits the blast radius of a stolen refresh token to one use window.

**JWT Invalidation strategies:**
1. Blocklist in Redis: `SADD blocklist <jti>` on logout; `SISMEMBER blocklist <jti>` on every request. Cost: 1 sub-ms Redis call per request.
2. Short-lived access tokens (15 min): don't invalidate at all; let them expire. Only blocklist refresh tokens (issued less frequently).
3. Version counter in DB: store `token_version` per user; embed version in JWT; on logout, increment DB version; reject tokens with stale version. Cost: 1 DB read per request.

**Key numbers:**
- JWT validation (signature check): < 1ms CPU
- Redis SISMEMBER (blocklist check): < 0.5ms
- PKCE code verifier: 43–128 character random string, SHA256 hashed to code_challenge

**Interview talking point:** "If asked how to invalidate a JWT before its expiry, answer: maintain a token blocklist in Redis — store the JWT ID (jti claim) in a Redis set on logout and check SISMEMBER on every request. Cost: one sub-millisecond Redis lookup per request. The better production approach: keep access tokens short-lived (15 min) so you only need to blocklist refresh tokens — this limits blast radius to 15 min without per-request Redis overhead on the hot path."

## Flashcards

| Q | A |
|---|---|
| Why do you sort equal-width envelopes by height DESC in Russian Doll Envelopes? | So that two envelopes of the same width cannot both appear in the LIS — since heights within the same width are decreasing, the LIS algorithm can only pick at most one per width group |
| What does the "tails" array in patience sort LIS actually represent? | tails[i] is the smallest possible tail element of all increasing subsequences of length i+1 seen so far; it is NOT the actual LIS sequence, just a structure for binary search |
| What is the coin change DP recurrence and its base case? | dp[0] = 0; dp[i] = min(dp[i - coin] + 1) for all coin <= i; initialize dp[1..amount] = amount+1 (sentinel for impossible) |
| What is the difference between RBAC and ABAC? When do you use each? | RBAC: users have roles, roles have permissions — simple, bounded. ABAC: policies on user+resource+environment attributes — flexible, complex. Use RBAC for internal tools; ABAC for multi-tenant SaaS with per-tenant rules |
| How does refresh token rotation detect token theft? | On each refresh, old token is invalidated and a new one is issued. If the old (already-invalidated) token is used again, it signals reuse — the server invalidates the entire token family, forcing re-login |
