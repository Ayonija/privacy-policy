## DSA (1.5 hours — Revision Mode)

### Pattern Revision: Heaps & Tries (Top-K, Streaming Median, Prefix Trees)
- Core idea: A min-heap maintains the K largest elements seen so far — whenever size exceeds K, pop the minimum, keeping only the top K. Two heaps (max-heap for lower half, min-heap for upper half) partition the stream so the median is always at the top(s). A Trie stores strings character by character in a 26-ary tree; each node has children[26] and an isEnd flag, enabling O(L) insert/search/prefix-check where L is word length.
- Trigger condition: Use a heap for "top K", "K closest", "K most frequent", or "streaming median". Use a Trie for prefix matching, autocomplete, or word existence queries on a large dictionary.
- Time complexity: Heap insert/pop O(log K) | Top-K O(N log K) | Two-heap addNum O(log N) | Trie insert/search O(L)
- Space complexity: Heap O(K) | Trie O(N·L) worst case where N = number of words, L = average length

### Template (write from memory before solving):
```java
// Two-Heap streaming median
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder()); // lower half
PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // upper half
void addNum(int num) {
    maxHeap.offer(num);
    minHeap.offer(maxHeap.poll()); // balance: push max of lower half up
    if (maxHeap.size() < minHeap.size()) maxHeap.offer(minHeap.poll()); // maxHeap always >= minHeap size
}
double findMedian() {
    return maxHeap.size() > minHeap.size() ? maxHeap.peek() : (maxHeap.peek() + minHeap.peek()) / 2.0;
}

PriorityQueue<Integer> minPQ = new PriorityQueue<>();
for (int num : nums) {
    minPQ.offer(num);
    if (minPQ.size() > k) minPQ.poll(); // evict the smallest; heap retains top K largest
}
```

```java
    public int[] topKFrequent(int[] nums, int k) {
        Map<Integer, Integer> freq = new HashMap<>();
        for (int n : nums) freq.merge(n, 1, Integer::sum);
        PriorityQueue<Integer> pq = new PriorityQueue<>(Comparator.comparingInt(freq::get));
        for (int key : freq.keySet()) {
            pq.offer(key);
            if (pq.size() > k) pq.poll();
        }
        int[] res = new int[k];
        for (int i = k - 1; i >= 0; i--) res[i] = pq.poll();
        return res;

class Trie {
    private TrieNode root = new TrieNode();
    static class TrieNode { TrieNode[] children = new TrieNode[26]; boolean isEnd; }
    public void insert(String word) {
        TrieNode node = root;
        for (char c : word.toCharArray()) {
            int i = c - 'a';
            if (node.children[i] == null) node.children[i] = new TrieNode();
            node = node.children[i];
        }
        node.isEnd = true;
    }
    public boolean search(String word) {
        TrieNode node = root;
        for (char c : word.toCharArray()) { node = node.children[c - 'a']; if (node == null) return false; }
        return node.isEnd;
    }
    public boolean startsWith(String prefix) {
        TrieNode node = root;
        for (char c : prefix.toCharArray()) { node = node.children[c - 'a']; if (node == null) return false; }
        return true;
    }
}
```

## System Design (30 min — HLD Component Review)

### Topic: Rate Limiting Algorithms

Rate limiting protects your service from abuse, ensures fair usage, and prevents resource exhaustion. Know all five algorithms — interviewers often ask you to compare them.

**Token Bucket:**
- Bucket holds up to B tokens; refills at rate R tokens/sec. Each request consumes 1 token; if empty, request is rejected or queued.
- Allows short bursts up to B (bucket capacity) while enforcing average rate R.
- Implementation: Redis with Lua script (atomically check + decrement token count + set expiry).
- Use when: API quotas, upload limits — YouTube upload quota is token bucket (allows uploading 3 videos quickly while enforcing daily limit).

**Leaky Bucket:**
- Requests enter a fixed-size queue; processed at a constant rate. Excess requests are dropped.
- Smooths traffic output — no burst tolerance. Good for protecting downstream services that need steady input.
- Use when: network traffic shaping, video streaming buffer management.

**Fixed Window Counter:**
- Increment a counter per `[user:window_start]` key; reset every window (e.g., every 60 seconds).
- Problem: up to 2× the rate limit can slip through at the window boundary (e.g., 100 requests at 11:59:59 + 100 at 12:00:00 = 200 in 1 second, not 60).
- Implementation: Redis `INCR` + `EXPIRE`. Simplest to implement.

**Sliding Window Log:**
- Store timestamp of every request in a Redis sorted set per user. Count entries in the last 60 seconds.
- Accurate: no boundary burst problem. Memory cost: O(requests per window) — expensive at scale.
- Use when: strict accuracy is required and request volume is low.

**Sliding Window Counter (hybrid):**
- Track current window count + previous window count. Estimate rate as: `current_count + prev_count × (1 - elapsed_fraction)`.
- ~3% error vs perfect sliding window. O(1) memory. Best balance of accuracy and efficiency.
- Used by: Cloudflare, Redis rate limiting libraries.

**Key numbers:**
- Redis INCR latency: < 0.5ms
- Token bucket check + decrement via Lua: ~1ms including network round trip
- Sliding window log memory: ~100 bytes per request entry in sorted set

**Interview talking point:** "If asked which rate limiting algorithm YouTube uses for upload quotas, answer: token bucket — it allows short bursts (uploading 3 videos in quick succession) while enforcing an average rate limit over time. Fixed windows are simpler but allow 2× the quota at window edges, which is unacceptable for quota enforcement. Leaky bucket would reject bursts, which degrades user experience."

## Flashcards

| Q | A |
|---|---|
| How do you maintain a streaming median with two heaps? | maxHeap (lower half) + minHeap (upper half); always push to maxHeap first, then balance by pushing maxHeap.poll() to minHeap; if minHeap grows larger, push back to maxHeap; median = maxHeap.peek() (odd) or average of both tops (even) |
| What is the key difference between token bucket and leaky bucket? | Token bucket allows bursts up to bucket capacity B while enforcing average rate R. Leaky bucket processes at a constant rate with no burst tolerance — excess requests are dropped immediately |
| What is the boundary burst problem with fixed window counters? | At window boundary (e.g., 11:59:59 → 12:00:00), a user can make full-quota requests at end of old window + full-quota at start of new window, effectively getting 2× the rate limit in a short period |
| How does a Trie differ from a HashMap<String, Boolean> for prefix queries? | Trie supports O(L) prefix lookup (startsWith) natively. HashMap would require iterating all keys or hashing prefixes separately. Trie also shares common prefixes, saving memory for large dictionaries |
| What is the time complexity of The Skyline Problem solution and why? | O(N log N) — sorting events is O(N log N); each event does one TreeMap add/remove which is O(log N); total N events = O(N log N) overall |
