## REST vs GraphQL vs gRPC

### REST (Representational State Transfer)
- Resource-based URLs (`/users/123/orders`); standard HTTP verbs (GET, POST, PUT, DELETE, PATCH)
- Stateless; JSON over HTTP/1.1; human-readable; cacheable with standard HTTP headers
- **Over-fetching**: API returns more fields than client needs
- **Under-fetching**: one call doesn't return enough data; N+1 calls required
- Versioning: `/v1/users`, `/v2/users` or header-based
- Best for: public APIs, external partners, browser clients, any place where wide compatibility matters

### GraphQL
- Client declares exactly what data it needs in the query — no over/under-fetching
- Single endpoint (`POST /graphql`); query language in the body
- Introspectable schema: self-documenting API
- **Caching complexity**: unique queries are hard to cache at HTTP layer; use persisted queries or application-level cache
- **N+1 problem**: naive resolvers hit DB once per item; solved with DataLoader (batch + deduplicate DB calls)
- Best for: mobile clients (minimize payload), BFF (Backend for Frontend), complex read requirements with many entity types

### gRPC
- Binary Protobuf encoding: 3–10x smaller payload than JSON; faster serialization
- Strongly typed: .proto schema is the contract; code generated for client + server
- HTTP/2: multiplexing, header compression, bidirectional streaming
- **4 call types**: Unary, Server Streaming, Client Streaming, Bidirectional Streaming
- Interop: harder outside Google ecosystem; browsers need gRPC-Web proxy
- Best for: internal microservice-to-service communication; low-latency, high-throughput internal APIs; streaming data

### Decision Framework

| Scenario | Protocol |
|---|---|
| Public-facing external API | REST |
| Mobile client with bandwidth constraints | GraphQL |
| Internal microservice calls | gRPC |
| Flexible client data requirements | GraphQL |
| Streaming (real-time sensor data, file transfer) | gRPC streaming |
| Browser-to-server | REST or GraphQL (gRPC needs proxy) |

## API Gateway

### What It Does
- Single entry point for all client requests to the backend
- Responsibilities: routing, authentication/authorization, rate limiting, SSL termination, load balancing, request/response transformation, logging

### Why Use It
- Clients don't need to know about internal service topology
- Cross-cutting concerns implemented once at gateway, not in every service
- A/B testing, canary deployments, protocol translation (REST in, gRPC out)

### Implementations
- **AWS API Gateway**: fully managed; Lambda integration; REST, HTTP, WebSocket APIs
- **Kong**: open-source; plugin ecosystem; runs on Nginx; self-hosted or managed
- **Nginx / Envoy**: used as API gateway; more config, more control

### BFF (Backend for Frontend) Pattern
- Separate gateway per client type: mobile BFF, web BFF, third-party partner BFF
- Each BFF optimized for its client's data shape and protocol needs
- Prevents one "generic" API from becoming a lowest-common-denominator design
- Mobile BFF: compact responses, aggressive caching; Web BFF: richer data, server-side rendering support

## Rate Limiting & Throttling

### Token Bucket
- Bucket holds tokens; tokens added at fixed rate R; bucket capacity is B (max burst size)
- Each request consumes 1 token; if bucket empty, request rejected
- **Allows bursting** up to B requests instantaneously; smooths over time at rate R
- Used by: AWS, Stripe, most production systems

### Leaky Bucket
- Requests enter bucket; processed at fixed rate R regardless of input rate
- Excess buffered (up to bucket size) or dropped
- **Smooths output rate** — good for protecting downstream services
- Does not allow bursting

### Fixed Window Counter
- Count requests per time window (e.g., 100 req/minute)
- Simple: `INCR key; EXPIRE key 60`
- Problem: boundary spike — 100 requests at 00:59, 100 more at 01:00 = 200 requests in 2 seconds; both windows valid but rate is double the limit

### Sliding Window (Log)
- Track exact timestamps of all requests in a sorted set
- Count requests in the window `[now − 60s, now]`
- Eliminates boundary spike problem; most accurate
- Storage: O(requests) per user — expensive at scale

### Sliding Window (Counter Approximation)
- Hybrid: current window count + previous window count × (overlap fraction)
- Good approximation of true sliding window; O(1) storage per user
- Used by Cloudflare, Redis-based rate limiters

### Redis Implementation
```lua
-- Atomic token bucket in Redis via Lua script
local tokens = redis.call('GET', key) or BUCKET_SIZE
if tokens >= 1 then
    redis.call('SET', key, tokens - 1)
    redis.call('EXPIRE', key, WINDOW)
    return 1  -- allowed
end
return 0  -- rejected
```

### Rate Limit Response
- HTTP 429 Too Many Requests
- Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`

## Pagination

### Offset Pagination
```sql
SELECT * FROM posts ORDER BY created_at DESC LIMIT 20 OFFSET 200
```
- Simple to implement and explain
- **Problem at scale**: DB must scan and discard OFFSET rows — O(OFFSET) cost; page 1000 is slow
- **Problem**: items can shift between pages if new data is inserted (inconsistent page views)
- Use for: small datasets, admin panels where random page access is needed

### Cursor Pagination
```sql
SELECT * FROM posts WHERE id < :last_seen_id ORDER BY id DESC LIMIT 20
```
- Cursor = opaque pointer to last-seen item (usually encoded ID or timestamp)
- O(log N) per page — uses index; constant cost regardless of page depth
- Stable: new inserts don't shift existing pages
- **Limitation**: cannot jump to arbitrary page; only next/previous
- Use for: infinite scroll feeds, timelines, any deep pagination

### Keyset Pagination (Multi-Column Cursor)
```sql
WHERE (created_at, id) < (:cursor_time, :cursor_id) ORDER BY created_at DESC, id DESC
```
- Handles ties in sort key (multiple rows with same timestamp)
- Requires composite index on (created_at, id)

## Real-Time Communication

### Polling
- Client sends HTTP request every N seconds to check for updates
- Simple; no persistent connections; high unnecessary load; N second latency
- Use only as fallback or for very low-frequency updates

### Long-Polling
- Client sends request; server holds it open until data is available or timeout
- Reduced unnecessary requests vs polling; still HTTP overhead per "push"
- Complex server-side: must manage holding connections; scaling is tricky

### SSE (Server-Sent Events)
- HTTP/1.1 persistent connection; server pushes events to client over text/event-stream
- **Unidirectional**: server → client only
- Auto-reconnect built into browser EventSource API
- Works over HTTP (no protocol upgrade); simpler than WebSockets
- Best for: live feeds, dashboards, notifications, stock tickers, progress updates

### WebSockets
- Full-duplex, persistent TCP connection via HTTP upgrade handshake
- **Bidirectional**: client and server can send at any time
- Low latency: no HTTP overhead per message after handshake
- Stateful: connection pinned to a server; requires sticky sessions or external state (Redis pub/sub)
- Best for: chat, multiplayer gaming, collaborative editing, live trading platforms

### Decision Framework

| Scenario | Protocol |
|---|---|
| Live score updates, news feed | SSE |
| Chat application | WebSocket |
| Real-time collaboration (Google Docs) | WebSocket |
| Progress bar for file upload | SSE |
| Multiplayer game state sync | WebSocket |
| Simple notification bell count | Polling (if low frequency) |

## STAR Interview Framework

**Situation:** "Design a rate limiter service for a public API platform. Each API key is allowed 1000 requests per minute. The system must handle 50K API keys and 10M requests/minute total."

**Task:** "Design the rate limiter: choose the algorithm, storage, and integration point."

**Action:**
1. Chose sliding window counter (approximation): accurate enough for 99.9% of cases, O(1) Redis storage per key, avoids fixed window boundary spikes
2. Redis Cluster: key = `ratelimit:{api_key}:{window_minute}`; two buckets per key (current + previous minute); atomic Lua script reads both, computes approximation, increments, sets expiry
3. Integration at API Gateway layer (Kong plugin): before routing to any backend service; returns HTTP 429 with Retry-After header on rejection
4. Rate limit state stored in Redis (not gateway memory) — horizontally scalable gateway nodes; all share same Redis state
5. Per-endpoint limits: premium keys get 10K/min; Lua script checks tier from a Redis Hash loaded at key registration
6. Monitoring: track rejection rate per key; alert if legitimate user is rate-limited (indicates tier misconfiguration)

**Result:** Rate limiter adds < 1ms p99 overhead. Handles 10M req/min across 50K API keys on a 3-node Redis Cluster. Eliminated scraper abuse that was causing 40% of DB load. Zero false positives for legitimate users in 6 months post-launch.

## Flashcards

| Q | A |
|---|---|
| REST over-fetching vs under-fetching: what are they? | Over-fetching: API returns more fields than needed; under-fetching: one call is insufficient, N+1 calls required — both solved by GraphQL |
| gRPC: why is it faster than REST/JSON? | Binary Protobuf encoding (3–10x smaller than JSON), HTTP/2 multiplexing, strongly typed with generated code |
| Token bucket vs leaky bucket: key difference | Token bucket allows bursting up to bucket capacity; leaky bucket enforces a constant output rate — no bursting |
| Fixed window rate limit: what is the boundary spike problem? | Requests at end of window + requests at start of next window can both be within limit but double the intended rate in a short period |
| Cursor pagination vs offset: why is cursor preferred for feeds? | Cursor uses an index seek (O(log N)) regardless of page depth; offset requires scanning OFFSET rows (O(N)); cursor is also stable under insertions |
| SSE vs WebSocket: when to choose SSE? | SSE for unidirectional server-to-client push (live feeds, notifications); WebSocket for bidirectional real-time (chat, gaming) |
| BFF pattern: what problem does it solve? | One generic API becomes a lowest-common-denominator; BFF provides a dedicated API optimized for each client's specific data shape and protocol needs |
| Rate limit HTTP status code and key response headers? | HTTP 429; headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, Retry-After |
