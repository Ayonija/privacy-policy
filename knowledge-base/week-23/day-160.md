## DSA (1.5 hours — Revision Mode)

### Pattern Revision: Backtracking (DFS + Pruning + Memoization)
- Core idea: Backtracking is a systematic DFS through the decision tree of all possible states. At each step, make a choice, recurse, then undo the choice (backtrack). Prune branches early when a partial solution cannot possibly lead to a valid complete solution. When the same subproblem (same index or same remaining state) can be reached via multiple paths, add memoization to avoid recomputation.
- Trigger condition: Use backtracking when the problem requires generating all valid combinations, permutations, or partitions, and the search space can be pruned. Add memoization when the same index/state is visited multiple times (convert from exponential to polynomial).
- Time complexity: Permutations O(N!) | Subsets O(2^N) | Word Break II O(N² × dict_size) with memo | Letter Case Permutation O(2^L) where L = number of letters
- Space complexity: O(N) recursion stack + O(result size) + O(N) memo table

### Template (write from memory before solving):
```java
// Generic backtracking template
List<List<Integer>> result = new ArrayList<>();
void backtrack(int start, List<Integer> path) {
    if (/* base case: valid complete solution */) {
        result.add(new ArrayList<>(path)); // snapshot — never add 'path' directly
        return;
    }
    for (int i = start; i < candidates.length; i++) {
        if (/* pruning condition */) continue; // e.g., candidates[i] > remaining
        path.add(candidates[i]);
        backtrack(i + 1, path); // i+1 for no reuse; i for reuse allowed
        path.remove(path.size() - 1); // undo choice
    }
}

// Permutations variant (boolean used[] array)
void permute(int[] nums, boolean[] used, List<Integer> path, List<List<Integer>> result) {
    if (path.size() == nums.length) { result.add(new ArrayList<>(path)); return; }
    for (int i = 0; i < nums.length; i++) {
        if (used[i]) continue;
        used[i] = true;
        path.add(nums[i]);
        permute(nums, used, path, result);
        path.remove(path.size() - 1);
        used[i] = false;
    }
}

// Word Break II — DFS + memoization
Map<Integer, List<String>> memo = new HashMap<>();
List<String> dfs(String s, Set<String> dict, int start) {
    if (memo.containsKey(start)) return memo.get(start);
    List<String> result = new ArrayList<>();
    if (start == s.length()) { result.add(""); return result; }
    for (int end = start + 1; end <= s.length(); end++) {
        String word = s.substring(start, end);
        if (dict.contains(word)) {
            for (String rest : dfs(s, dict, end))
                result.add(word + (rest.isEmpty() ? "" : " " + rest));
        }
    }
    memo.put(start, result);
    return result;
}
```

### Extended Component Notes (30-second deep dives)

**Redis Cache:**
- Eviction policies: LRU (Least Recently Used, default for caches), LFU (Least Frequently Used), TTL-based.
- Persistence: RDB snapshots (point-in-time, smaller) vs AOF append-only log (durable, larger). Both for production.
- Cluster mode: shards data across nodes with consistent hashing; each shard has a primary + replica.
- Pub/Sub: lightweight; no persistence; use Kafka for durable messaging.

**Kafka:**
- Topics are divided into partitions; each partition is an ordered log; consumer groups read partitions in parallel.
- Ordering: guaranteed within a partition; none across partitions. Use the same partition key (e.g., userId) to order per-user events.
- Consumer lag = messages produced - messages consumed; alert when lag grows unboundedly.
- Kafka Streams / Flink for stream processing with exactly-once semantics.

**PostgreSQL:**
- Indexing: B-tree (default, equality + range), Hash (equality only), GIN (full-text, JSONB), GiST (geo).
- Connection pooling: PgBouncer in transaction mode reduces connections from thousands to dozens.
- VACUUM: reclaims dead tuples from MVCC; run autovacuum; monitor bloat.
- Read replicas via streaming replication; replica lag is typically < 1 second on fast networks.

**Cassandra:**
- Partition key determines which node stores the data (consistent hashing on token ring).
- Clustering key determines sort order within a partition.
- Wide row design: store time-series as rows with clustering key = timestamp; efficient range scans.
- Avoid hot partitions: if one user generates 90% of writes, that node becomes a bottleneck.

**CDN:**
- Cache-Control headers control TTL at the edge. `max-age=31536000, immutable` for versioned static assets.
- Cache invalidation: purge by URL, by tag, or by surrogate key. Cloudflare / Fastly support tag-based purge.
- Origin shield: single point of contact for CDN edge nodes → origin; reduces origin load dramatically.

**Elasticsearch:**
- Write path: document → in-memory buffer → segment (flushed every 1 sec) → merge (background).
- Near real-time (NRT): documents visible in search within ~1 second of indexing.
- Avoid storing full document if only specific fields are searched — use _source: false to reduce storage.

### Interview Talking Points — One Per Component

**Redis:** "If asked about cache stampede: use probabilistic early expiration or a distributed lock (SET NX EX) to let only one request rebuild the cache while others serve the stale value."

**Kafka:** "If asked how to handle exactly-once delivery: use Kafka transactional API (producer + idempotent writes) or store consumer offset and business result in the same DB transaction."

**PostgreSQL:** "If asked when to switch to NoSQL: when write throughput consistently exceeds 10K/sec after vertical scaling, or when storage per table exceeds 1TB and sharding adds too much complexity."

**Cassandra:** "If asked about Cassandra's CAP position: Cassandra is AP by default (ONE consistency level). Use QUORUM for CP-like behavior at the cost of ~2× write latency."

**CDN:** "If asked how Netflix serves video at scale without overloading S3: Netflix Open Connect Appliances (OCA) are CDN servers placed directly inside ISP networks. 95%+ of traffic is served from the ISP's local OCA, never hitting Netflix's origin."

**Load Balancer:** "If asked L4 vs L7: L4 (TCP/UDP level) is faster (< 0.1ms overhead) and can handle any protocol but has no application context. L7 (HTTP level) adds ~1ms but enables URL-based routing, cookie-based sticky sessions, and SSL termination."

**Elasticsearch:** "If asked how to handle indexing spikes: Elasticsearch uses immutable segments. Bulk index during off-peak hours, use index aliases to swap indices atomically, and set refresh_interval=30s during heavy indexing to reduce segment creation overhead."

**API Gateway:** "If asked the difference between an API Gateway and a Service Mesh: Gateway manages north-south traffic (external clients to internal services) — auth, rate limiting, SSL. Service mesh (Istio) manages east-west traffic (service to service) — mTLS, retries, circuit breaking, observability. Both can coexist."

**SQS:** "If asked about at-least-once delivery and how to handle duplicates: use idempotency keys. Store processed message IDs in a Redis set or DB unique constraint. Processing the same message twice produces the same result."

**S3:** "If asked how to handle large file uploads without routing through your backend: generate a presigned URL server-side (valid 15 min), return it to the client, client uploads directly to S3. Your backend is never in the data path."
