## DSA (1.5 hours)

### Pattern: Preorder Serialization + Stack Reconstruction (Revisit)
- For LC 297 cold solve, the key discipline: serialize with a consistent delimiter so null markers and multi-digit numbers don't collide. During deserialization, use a queue (not index tracking) to consume tokens cleanly.

### Pattern: Topological Sort from Character Constraints (Revisit)
- For LC 269 cold solve, the hardest part is the edge case: `"abc"` before `"ab"` is an invalid input. Check `w1.length() > w2.length() && w1.startsWith(w2)` before extracting any edge.

    // LC 560 — Subarray Sum Equals K (revision — write from memory)
    public int subarraySum(int[] nums, int k) {
        Map<Integer, Integer> prefixCount = new HashMap<>();
        prefixCount.put(0, 1);  // empty prefix
        int sum = 0, count = 0;
        for (int num : nums) {
            sum += num;
            count += prefixCount.getOrDefault(sum - k, 0);

## System Design (1 hour)

### Topic: Uber / Lyft — Full System Design Integration (Mock Interview Format)

**Step 1 — Clarify Requirements (2 min in real interview)**

Functional:
- Rider requests a ride (origin, destination)
- System matches nearest available driver
- Real-time location tracking of driver during trip
- Dynamic pricing with surge based on supply/demand
- Payment on trip completion
- Push notifications at each state transition

Non-functional:
- 500M trips/year (~15M trips/day, ~175 trips/sec average, peak ~3×)
- Match latency ≤ 5 seconds; ETA accuracy ≤ ±30%
- High availability (99.99%); location data eventual consistency acceptable
- 5M active drivers; 100M registered riders

**Step 3 — Deep Dives (pick 2 in a real interview)**

**Deep Dive A — Ride Request to Match (Critical Path):**
1. Rider POSTs ride request → API Gateway → Ride Request Service
2. Ride Request Service calls Pricing Service for fare estimate → stored with `fare_lock_id` in Redis (2-min TTL)
3. Matching Service queries Redis Geo: `GEOSEARCH drivers:{city} FROMLONLAT lng lat BYRADIUS 2 km ASC COUNT 10`
4. For each candidate driver (sorted by distance): attempt Redis NX lock (`SET driver:{id}:status MATCHED NX PX 30000`)
5. First driver to be locked → notify via Driver Gateway WebSocket + push (FCM/APNs)
6. Driver accepts → Trip Service creates trip record in PostgreSQL (status = DRIVER_ACCEPTED, version = 1)
7. Kafka event `trip.driver_accepted` → Notification Service → push to rider

**Deep Dive B — Location Update Pipeline:**
1. Driver GPS ping every 4s → WebSocket → Driver Gateway
2. Gateway publishes to Kafka `driver-location` (partitioned by driver_id)
3. Location Worker consumes: `GEOADD drivers:{city} lng lat driver_id` + update Redis hash `driver:{id}:last_ping`
4. Cassandra write: `INSERT INTO location_history (driver_id, ts, lat, lng)`
5. If driver IN_TRIP: WebSocket push of driver position to rider app (via Rider Gateway)
6. Staleness checker (background): if `last_ping` > 30s ago → GEOREM + mark STALE

## Flashcards

| Q | A |
|---|---|
| In LC 297, why use "," as a delimiter rather than just concatenating values? | Without a delimiter, multi-digit numbers and null markers ("123#" vs "12" + "3#") become ambiguous; a delimiter makes tokenization unambiguous |
| In LC 269, what is the single edge case that makes the input invalid before any graph construction? | A word that is a proper prefix of the next word but appears after it (e.g., "abc" before "ab") — violates the sort invariant |
| Name the six deep-dive components of Uber's system design. | Ride Matching (Redis Geo + NX lock), Location Pipeline (WebSocket → Kafka → Redis), Pricing/Surge (supply/demand per H3 zone), ETA (Contraction Hierarchies + route cache), Payment (idempotent charge + payout), Trip State Machine (PostgreSQL + Kafka events) |
| In Uber's critical match path, what prevents two riders from getting the same driver? | Redis NX lock with TTL: `SET driver:{id}:status MATCHED NX PX 30000` — atomic set-if-not-exists; only one matching service instance wins |
| What is the Uber ETA service's approach to sub-100ms route computation? | Contraction Hierarchies (CH) on a preprocessed road graph; bidirectional Dijkstra visits O(s) important nodes (100-1000) instead of millions; combined with zone-pair route cache |
