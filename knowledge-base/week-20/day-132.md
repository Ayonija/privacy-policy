## LLD: SOLID — S, O, L

### S — Single Responsibility Principle (SRP)
**Smell it fixes:** God class — one class that handles everything (data, formatting, persistence, email sending). Any requirement change touches this one class; unrelated bugs collide.

**Principle:** A class should have only one reason to change.

**Analogy (chai dukaan):** The chai dukaan owner makes chai — that's his one job. He does not also design the menu, manage accounts, and deliver orders. If you split duties, each person is an expert and changes to billing don't affect chai-making.

**Bad code:**
```java
class OrderManager {
    void placeOrder(Order o) { /* validate, persist to DB */ }
    void sendConfirmationEmail(Order o) { /* format + send email */ }
    void generateInvoicePDF(Order o) { /* render PDF */ }
    void logOrder(Order o) { /* write to log file */ }
}
// Problem: a change to email template forces a retest of DB logic and PDF generation
```

**Fixed code:**
```java
class OrderService     { void placeOrder(Order o) { /* DB + validation */ } }
class EmailService     { void sendConfirmation(Order o) { /* email only */ } }
class InvoiceService   { void generatePDF(Order o) { /* PDF only */ } }
class OrderLogger      { void log(Order o) { /* logging only */ } }
// Each class has exactly one reason to change
```

**Interview questions:**
- "What is SRP?" → One reason to change; one axis of responsibility.
- "How do you find SRP violations?" → Look for classes with methods that belong to different conceptual roles; look for "and" in a class name (UserManagerAndLogger).

### O — Open/Closed Principle (OCP)
**Smell it fixes:** Every time a new type is added, you modify an existing class's if-else or switch-case chain — risk of breaking existing behavior on every change.

**Principle:** Open for extension, closed for modification. Add new behavior by adding new code, not changing existing code.

**Analogy (plug socket):** A power socket doesn't change when you plug in a new device. The socket (existing code) is closed for modification; each new device (extension) is a new plug — no rewiring required.

**Bad code:**
```java
class DiscountCalculator {
    double calculate(String customerType, double price) {
        if (customerType.equals("REGULAR"))  return price * 0.95;
        if (customerType.equals("PREMIUM"))  return price * 0.80;
        // adding VIP requires editing this class — risk of regression
        return price;
    }
}
```

**Fixed code (plugin pattern):**
```java
interface DiscountStrategy {
    double apply(double price);
}

class RegularDiscount implements DiscountStrategy {
    public double apply(double price) { return price * 0.95; }
}

class PremiumDiscount implements DiscountStrategy {
    public double apply(double price) { return price * 0.80; }
}

class VIPDiscount implements DiscountStrategy {      // NEW type = NEW class only
    public double apply(double price) { return price * 0.70; }
}

class DiscountCalculator {
    double calculate(DiscountStrategy strategy, double price) {
        return strategy.apply(price); // never changes
    }
}
```

**Interview questions:**
- "What is OCP?" → Add new behavior without modifying existing code.
- "How do you implement OCP?" → Strategy pattern, polymorphism, plugin registries.

### L — Liskov Substitution Principle (LSP)
**Smell it fixes:** Subclass breaks parent's expected behavior — callers that work with the parent type throw exceptions or produce wrong results when given the subclass.

**Principle:** If `S` is a subtype of `T`, objects of type `S` must be substitutable for objects of type `T` without altering correctness.

**Analogy (Penguin / Bird):** If `Bird` declares `fly()`, and you pass a `Penguin` wherever a `Bird` is expected, the program breaks — Penguins cannot fly. The inheritance relationship violates LSP even though a Penguin IS-A Bird biologically.

**Bad code:**
```java
class Bird {
    void fly() { System.out.println("Flying..."); }
}

class Penguin extends Bird {
    @Override
    void fly() { throw new UnsupportedOperationException("Penguins can't fly!"); }
}

// Caller breaks at runtime
void makeItFly(Bird b) {
    b.fly(); // crashes when b is Penguin — contract violated
}
```

**Fixed code:**
```java
interface Bird { void eat(); } // only behavior ALL birds share

interface FlyingBird extends Bird { void fly(); }

class Sparrow implements FlyingBird {
    public void eat() { }
    public void fly() { System.out.println("Flying"); }
}

class Penguin implements Bird {
    public void eat() { }
    // no fly() — correct contract for a Penguin
}
```

**LSP rules of thumb:**
- Preconditions in the subclass must not be stronger than in the parent.
- Postconditions in the subclass must not be weaker than in the parent.
- If `override` throws an exception the parent never threw → LSP violation.

**Interview questions:**
- "What is LSP?" → Subclass must honor the parent's behavioral contract.
- "How do you detect LSP violations?" → `throw UnsupportedOperationException`, or `instanceof` checks in callers.

## STAR: "Tell me about a time you refactored code that violated SRP"

**Situation:** "At [company], our `UserService` class was 1,200 lines long. It handled user registration, password hashing, session management, email verification, audit logging, and PDF export of account statements. Every sprint, at least one unrelated bug would appear because a change to the PDF export would break session handling tests."

**Task:** "I proposed and led a refactoring effort to decompose the class — goal: zero cross-concern breakages in a sprint."

**Action:**
1. *Mapped responsibilities:* I read through the entire class and listed distinct reasons it would need to change — I found six (registration, auth, session, email, audit, PDF).
2. *Extracted in priority order:* I started with the most-changed class: `EmailService` was extracted first because it changed every week for marketing campaigns and was causing the most test noise.
3. *Introduced interfaces:* Each extracted service got an interface so call sites could be updated without knowing the implementation; mocks became trivial to write.
4. *Moved incrementally:* I extracted one service per PR, ran full regression after each, and verified no existing tests broke.

**Result:** "The original 1,200-line class shrank to a 90-line orchestrator that delegated to six focused services. Sprint velocity improved — no more cross-concern test failures. The `EmailService` alone had 38 tests added that were previously impossible to write in isolation."

| Q | A |
|---|---|
| What is SRP and what smell does it fix? | One reason to change; fixes god classes that mix unrelated responsibilities |
| What is OCP? Give the plug socket analogy. | Open for extension, closed for modification; socket (existing code) never changes — new device (new class) plugs in |
| What is the code signal of an OCP violation? | A switch-case or if-else chain that must be edited every time a new type is added |
