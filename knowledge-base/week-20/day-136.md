## LLD: Builder Pattern

### What it Solves: Telescoping Constructor Anti-Pattern
When an object has many optional fields, constructors multiply:
```java
// Telescoping constructors — nightmare to read and call correctly
Pizza(String size) { ... }
Pizza(String size, boolean cheese) { ... }
Pizza(String size, boolean cheese, String topping1) { ... }
Pizza(String size, boolean cheese, String topping1, String topping2) { ... }
// After 5+ parameters, callers guess which arg goes where
```
Builder replaces this with a readable, step-by-step assembly.

---

### Analogy: Subway Sandwich
You build your sandwich one step at a time: choose bread → add veggies → add sauce → choose toast → `.build()`. Each step is optional. The sandwich is only assembled (and validated) when you call `build()`. You get exactly what you asked for, in a readable sequence.

---

### Core Pseudocode: PizzaBuilder

```java
class Pizza {
    private final String size;       // required
    private final boolean cheese;    // optional
    private final boolean pepperoni; // optional
    private final List<String> toppings;

    private Pizza(Builder builder) {
        this.size      = builder.size;
        this.cheese    = builder.cheese;
        this.pepperoni = builder.pepperoni;
        this.toppings  = Collections.unmodifiableList(builder.toppings);
        // Fields set once here; object is immutable after build()
    }

    public static class Builder {
        private final String size;       // required — passed in constructor
        private boolean cheese    = false;
        private boolean pepperoni = false;
        private List<String> toppings = new ArrayList<>();

        public Builder(String size) { this.size = size; } // required field forced upfront

        public Builder cheese()              { this.cheese    = true; return this; }
        public Builder pepperoni()           { this.pepperoni = true; return this; }
        public Builder addTopping(String t)  { this.toppings.add(t); return this; }

        public Pizza build() {
            // Validate here before handing out the object
            if (size == null || size.isBlank()) throw new IllegalStateException("Size is required");
            return new Pizza(this);
        }
    }
}

// Caller — fluent, readable
Pizza myPizza = new Pizza.Builder("LARGE")
    .cheese()
    .addTopping("mushrooms")
    .addTopping("onions")
    .build();
```

**Key mechanics:**
- Each setter returns `this` — enables fluent chaining.
- The `Pizza` constructor is `private` — only `Builder.build()` can create a `Pizza`.
- Fields are `final` in `Pizza` — object is immutable after `build()`.
- Validation lives in `build()` — not scattered across constructors.

---

### Immutable Object via Builder
The pattern enforces immutability cleanly:
1. All `Pizza` fields are `private final`.
2. The only path to set them is `build()`.
3. No setters on `Pizza` itself — once built, state cannot change.
4. Pass immutable copies of collections (`unmodifiableList`).

This is how Java's `HttpRequest`, Guava's `ImmutableList.Builder`, and Lombok's `@Builder` work internally.

---

### Real-World Uses

| Use Case | Why Builder fits |
|---|---|
| HTTP request builder | Many optional headers, timeout, retry policy — all optional; object should be immutable once sent |
| SQL query builder | Clauses (WHERE, JOIN, ORDER BY) are optional combinations; query only valid when fully constructed |
| Config/settings object | Large number of optional properties with sensible defaults; validation at build time |
| DTO / test data setup | Test fixtures with 10+ fields where only 2-3 are relevant per test — readable construction |

**Java standard library examples:** `StringBuilder`, `HttpClient.newBuilder()`, `ProcessBuilder`, `AlertDialog.Builder` (Android).

---

### Fluent Chaining — Each Setter Returns `this`
```java
// Without fluent chaining (verbose)
Builder b = new Pizza.Builder("MEDIUM");
b.cheese();
b.pepperoni();
Pizza p = b.build();

// With fluent chaining (each method returns `this`)
Pizza p = new Pizza.Builder("MEDIUM").cheese().pepperoni().build();
```
The pattern: every setter modifies state on the builder and returns `this` so the next call chains directly. `build()` is the terminal operation.

---

### Don't Confuse: Builder vs. Factory

| | Builder | Factory |
|---|---|---|
| **Problem solved** | Assembling one complex object step by step | Deciding which type of object to create |
| **Caller interaction** | Multiple chained calls → `.build()` | One call with a type hint → object returned |
| **Output** | One fully configured instance of a specific type | Instance of the right subtype |
| **Object complexity** | High — many optional parts | Low to medium — type selection is the concern |
| **Analogy** | Subway: bread → veggies → sauce → build | Chai tapri: "ek chai" → kitchen decides type |
| **Machine coding trigger** | "Build a configurable X with many options" | "Create the right type of X based on input" |

## STAR: "Design a configurable HTTP client with optional headers, timeouts, retries"

**Situation:** "Our internal HTTP client was initialized with a 12-argument constructor. New engineers regularly passed arguments in the wrong order, causing subtle bugs in production — wrong timeout on a critical path, missing auth header on a public endpoint."

**Task:** "Replace the constructor with a readable, validated, immutable configuration pattern."

**Action:**
1. *Identified the anti-pattern:* The 12-argument constructor was the textbook telescoping constructor — no way to know which `int` was connection timeout vs. read timeout at the call site.
2. *Applied Builder:* Created `HttpClientConfig.Builder` with named setter methods for each option. Required fields (`baseUrl`) were forced in the `Builder` constructor; everything else defaulted to safe values (3s timeout, 3 retries, no custom headers).
3. *Validation at `build()`:* Added assertions — timeout must be positive, baseUrl must be a valid URI, retries must be 0–10. Errors surface at construction time, not during first HTTP call at 2 AM.
4. *Immutability:* `HttpClientConfig` fields were all `private final`; headers were copied into an `unmodifiableMap`. Once built, the config could be safely shared across threads.

**Result:** "Call-site readability went from `new HttpClient(url, null, 3000, 5000, 3, true, false, ...)` to `new HttpClient.Builder(url).readTimeout(5000).retries(3).build()`. Two misconfiguration production incidents in the prior quarter dropped to zero. Onboarding time for new engineers on the HTTP module halved."

## Flashcards

| Q | A |
|---|---|
| What anti-pattern does Builder solve? | Telescoping constructor — too many constructor overloads for optional combinations of fields |
| How does fluent chaining work in Builder? | Each setter modifies builder state and returns `this`; calls chain; `build()` is the terminal operation |
| How does Builder enforce immutability? | Product class has `private final` fields set only in `build()`; product has no setters; collections wrapped in unmodifiable wrappers |
| Where should validation live in a Builder? | In `build()` — validates all fields before creating the object; errors surface at construction, not at use time |
| Name three real-world Java Builder uses | `HttpClient.newBuilder()`, `StringBuilder`, `ProcessBuilder` (also: SQL query builders, test fixture builders) |
