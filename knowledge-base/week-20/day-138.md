# Day 138 — Graphs Hard: Largest Color Value & 3D Rain Water
**Week 20 | Phase 3 — Mock Interviews | Month 5**

## Focus
Two algorithmically distinct hard graph problems — finding the maximum frequency of any color on any path in a directed graph using topological sort + DP (LC 1857) and computing trapped water in a 3D elevation map using BFS with a min-heap (LC 407) — stretch beyond standard patterns toward the hardest problems seen at Google.

## DSA (1.5 hours)

### Pattern: Topological Sort + Color Frequency DP on DAG
- For LC 1857, `dp[node][c]` = maximum count of color c on any path ending at node. Process nodes in topological order (Kahn's BFS). For each neighbor: `dp[neighbor][c] = max(dp[neighbor][c], dp[node][c] + (colors[neighbor] == c ? 1 : 0))`. If a cycle exists (some nodes never reach in-degree 0), return -1. Answer is `max(dp[node][colors[node] - 'a'])` over all nodes.
- Trigger condition: "Maximum frequency of any label/color on any path in a directed graph — combine topo sort with per-node DP over 26 colors."
- Time complexity: O(V + E + 26·V) | Space complexity: O(V · 26)

### Pattern: BFS Min-Heap Boundary Expansion (3D Trapping Rain Water)
- For LC 407, initialize a min-heap with all boundary cells. Use a visited array. While the heap is non-empty: pop the minimum-height cell; for each unvisited neighbor, `water += max(0, current_min_height - neighbor_height)`; push the neighbor with `max(current_min_height, neighbor_height)` into the heap (water cannot escape below the current boundary).
- Trigger condition: "Trapping rain water in a 2D grid — the boundary 'surrounds' the interior; water trapped at each cell is bounded by the minimum-height boundary cell that can reach it."
- Time complexity: O(mn · log(mn)) | Space complexity: O(mn)

### Code Skeleton

```java
class Solution {

    // LC 1857 — Largest Color Value in a Directed Graph
    public int largestPathValue(String colors, int[][] edges) {
        int n = colors.length();
        List<List<Integer>> adj = new ArrayList<>();
        int[] inDegree = new int[n];
        for (int i = 0; i < n; i++) adj.add(new ArrayList<>());

        for (int[] e : edges) {
            adj.get(e[0]).add(e[1]);
            inDegree[e[1]]++;
        }

        // dp[node][c] = max count of color c on any path ending at node
        int[][] dp = new int[n][26];
        Deque<Integer> queue = new ArrayDeque<>();

        for (int i = 0; i < n; i++) {
            if (inDegree[i] == 0) {
                queue.offer(i);
                dp[i][colors.charAt(i) - 'a'] = 1;
            }
        }

        int processed = 0, result = 0;
        while (!queue.isEmpty()) {
            int node = queue.poll();
            processed++;
            result = Math.max(result, dp[node][colors.charAt(node) - 'a']);

            for (int next : adj.get(node)) {
                for (int c = 0; c < 26; c++) {
                    int add = (colors.charAt(next) - 'a' == c) ? 1 : 0;
                    dp[next][c] = Math.max(dp[next][c], dp[node][c] + add);
                }
                if (--inDegree[next] == 0) queue.offer(next);
            }
        }

        return processed == n ? result : -1;  // -1 if cycle (not all nodes processed)
    }


    // LC 407 — Trapping Rain Water II
    public int trapRainWater(int[][] heightMap) {
        if (heightMap == null || heightMap.length < 3 || heightMap[0].length < 3) return 0;
        int m = heightMap.length, n = heightMap[0].length;
        boolean[][] visited = new boolean[m][n];
        // Min-heap: [height, row, col]
        PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[0] - b[0]);

        // Add all boundary cells to heap
        for (int r = 0; r < m; r++) {
            pq.offer(new int[]{heightMap[r][0], r, 0});
            pq.offer(new int[]{heightMap[r][n-1], r, n-1});
            visited[r][0] = visited[r][n-1] = true;
        }
        for (int c = 1; c < n - 1; c++) {
            pq.offer(new int[]{heightMap[0][c], 0, c});
            pq.offer(new int[]{heightMap[m-1][c], m-1, c});
            visited[0][c] = visited[m-1][c] = true;
        }

        int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
        int water = 0;

        while (!pq.isEmpty()) {
            int[] curr = pq.poll();
            int h = curr[0], r = curr[1], c = curr[2];
            for (int[] d : dirs) {
                int nr = r + d[0], nc = c + d[1];
                if (nr < 0 || nr >= m || nc < 0 || nc >= n || visited[nr][nc]) continue;
                visited[nr][nc] = true;
                water += Math.max(0, h - heightMap[nr][nc]);
                pq.offer(new int[]{Math.max(h, heightMap[nr][nc]), nr, nc});
            }
        }
        return water;
    }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Payment Service

**Core components:**
- **Fare calculation** — Trip Service signals completion → Pricing Service computes final fare (actual distance + time, apply surge if fare-locked window expired)
- **Payment Service** — charges the rider's stored payment method; integrates with Stripe/Braintree/Adyen as payment gateway; stores tokenized card references (never raw PAN)
- **Idempotency** — each charge request carries a unique `payment_intent_id` (trip_id + attempt_number); the payment gateway deduplicates retries using this key; prevents double-charges on network retries
- **Driver payout** — after trip completion, a payout record is created; payouts are batched daily (or weekly) and sent via ACH/Stripe Connect; Uber takes platform fee (25-30%), driver receives remainder
- **Fraud detection** — ML model scores each transaction in real time (device fingerprint, location, velocity, card history); high-score transactions are flagged for review or declined
- **Dispute handling** — rider disputes charge → CS team reviews trip data (GPS trace, duration, fare breakdown); refund issued via payment gateway reversal

**Key trade-offs:**
- **Synchronous charge vs async:** synchronous gives immediate confirmation but blocks trip completion API on payment gateway latency (~200-500ms); async sends the charge in background after trip ends — Uber uses async with retry, showing "processing payment" UI to rider
- **Stored card token vs redirect:** storing a tokenized reference (from gateway vault) allows one-click payment without PCI scope for Uber's servers; redirect (e.g., 3D Secure) adds security but degrades UX
- **Payout frequency:** daily payouts reduce driver cash flow anxiety but increase ACH transaction count (cost); weekly payouts reduce cost but drivers prefer daily — Uber offers both (daily with Instant Pay surcharge, weekly free)

**Interview talking point:** "If asked how Uber prevents double-charging a rider if the payment service crashes mid-charge, answer: Each charge request has an idempotency key (`trip_id + attempt_seq`) sent to the payment gateway. If Uber retries after a timeout, the gateway recognizes the same idempotency key and returns the original result (success or failure) without processing a second charge. Uber persists the `payment_intent_id` to DB before calling the gateway, so the idempotency key is recoverable even after a crash."

## Flashcards

| Q | A |
|---|---|
| In LC 1857, how do you detect a cycle using topological sort? | If `processed < n` after Kahn's BFS completes (some nodes never reached in-degree 0), a cycle exists — return -1 |
| In LC 1857, why initialize `dp[node][colors[node]-'a'] = 1` only when the node enters the queue? | The node's own color contributes 1 to the count of paths ending at it; initializing before entering the queue would be wrong for nodes that depend on predecessors |
| In LC 407, why push `max(current_boundary_height, neighbor_height)` rather than `neighbor_height`? | Water can only escape by flowing over the boundary; if the neighbor is shorter than the current boundary, the effective "seal height" is the boundary — pushing max ensures inward expansion respects the water level |
| In LC 407, why initialize the heap with boundary cells rather than starting from the interior? | Water flows outward to the boundary; computing from the boundary inward naturally captures all cells that are "sealed" by higher boundary walls |
| In Uber's Payment Service, how does idempotency prevent double charges on retry? | Each charge sends a unique `payment_intent_id` to the gateway; the gateway deduplicates by this key and returns the original result without reprocessing |
