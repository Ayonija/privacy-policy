## LLD: The 4 Pillars of OOP

### 1. Encapsulation
**Definition:** Bundle data (fields) and behavior (methods) into a single unit (class), and restrict direct access to the data.

**Analogy:** An ATM — you interact through a screen and keypad (public interface); you cannot touch the cash mechanism or internal state directly.

**What it solves:** Prevents invalid state; external code cannot corrupt internal data. Changes to internals don't break callers.

**Code example:**
```java
// Bad — field exposed
class BankAccount {
    public double balance; // anyone can set this to -9999
}

// Good — encapsulated
class BankAccount {
    private double balance;

    public void deposit(double amount) {
        if (amount > 0) balance += amount; // validation lives here
    }

    public double getBalance() { return balance; }
}
```

**Common interview questions:**
- "What is encapsulation? Give an example." → ATM + private field with getter/setter
- "Why not just make all fields public?" → No validation, no invariant enforcement, change breaks all callers

---

### 2. Abstraction
**Definition:** Expose only what is necessary; hide implementation details. The caller knows *what* to do, not *how* it is done.

**Analogy:** Steering wheel — you turn it; you do not know the hydraulic/rack-and-pinion mechanics underneath.

**What it solves:** Reduces cognitive load; lets you swap implementations without changing callers. Enforced in Java via abstract classes and interfaces.

**Code example:**
```java
// Abstraction via interface
interface Shape {
    double area(); // caller knows what to call, not how it's computed
}

class Circle implements Shape {
    private double radius;
    Circle(double r) { this.radius = r; }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

class Square implements Shape {
    private double side;
    Square(double s) { this.side = s; }

    @Override
    public double area() { return side * side; }
}
```

**Common interview questions:**
- "What is abstraction?" → Hiding implementation, exposing contract
- "Difference between abstract class and interface in Java?" → Abstract class: partial implementation, single inheritance; Interface: pure contract, multiple inheritance

---

### 3. Inheritance
**Definition:** A child class acquires properties and behavior of a parent class, allowing code reuse and hierarchical relationships.

**Analogy:** Parent–child — a child inherits traits (hair color, eye color) from a parent but can override them or add new ones.

**What it solves:** Code reuse; establishes an IS-A relationship between types.

**Code example:**
```java
class Animal {
    String name;
    void eat() { System.out.println(name + " eats."); }
}

class Dog extends Animal {
    void bark() { System.out.println(name + " barks."); } // Dog IS-A Animal
}

// Usage
Dog d = new Dog();
d.name = "Rex";
d.eat();  // inherited
d.bark(); // own behavior
```

**Common interview questions:**
- "What is the difference between IS-A and HAS-A?" → Inheritance = IS-A; Composition = HAS-A. Prefer composition when the IS-A relationship might break (Penguin IS-A Bird but cannot fly).
- "Why is multiple inheritance not allowed in Java for classes?" → Diamond problem — ambiguous method resolution; Java solves it with interfaces.

### 4. Polymorphism
**Definition:** One interface, many implementations. Objects of different types respond to the same message (method call) in type-specific ways.

**Analogy:** A `run` command to a human, a dog, and a robot all trigger movement — but the mechanism differs per entity.

**What it solves:** Open–closed behavior — callers work against the abstraction; adding a new type doesn't require changing the caller.

**Code example:**
```java
class Animal {
    void sound() { System.out.println("..."); }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Woof"); } // runtime polymorphism
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

// Caller doesn't need to know the concrete type
Animal[] animals = { new Dog(), new Cat() };
for (Animal a : animals) a.sound(); // dynamic dispatch at runtime
```

### Abstraction vs. Encapsulation (Most Confused Pair)

| | Abstraction | Encapsulation |
|---|---|---|
| **What** | Hiding *implementation details* | Hiding *data* |
| **Why** | Reduce complexity, expose contract | Protect state, enforce invariants |
| **How** | Interfaces, abstract classes | Private fields, public methods |
| **Analogy** | Steering wheel (hide hydraulics) | ATM (hide cash mechanism) |
| **Level** | Design level | Class level |

Memory hook: **Abstraction = what a thing *does***; **Encapsulation = what a thing *holds***.

---

### Overriding vs. Overloading (Polymorphism Types)

| | Method Overriding | Method Overloading |
|---|---|---|
| **Type** | Runtime (dynamic) polymorphism | Compile-time (static) polymorphism |
| **Where** | Subclass redefines parent method | Same class, different parameter signatures |
| **Binding** | Late binding — resolved at runtime | Early binding — resolved at compile time |
| **Annotation** | `@Override` in Java | No annotation needed |
| **Example** | `Dog.sound()` overrides `Animal.sound()` | `add(int a, int b)` vs `add(double a, double b)` |

## STAR: "Explain OOP with an example from your work"

**Situation:** "In a previous project, we had a payment processing module where all payment logic — credit card, PayPal, UPI — was crammed into one monolithic class with public fields, if-else chains, and no shared contract."

**Task:** "I was asked to refactor it to support adding new payment methods without touching existing code."

**Action:**
1. *Encapsulation:* Made all payment-specific fields private (`cardNumber`, `apiKey`) with controlled access methods — no external code could accidentally wipe transaction state.
2. *Abstraction:* Defined a `PaymentProcessor` interface with a single `process(amount)` method — callers only knew the contract, not the provider's HTTP calls.
3. *Inheritance:* `DigitalPaymentProcessor` abstract class held shared retry logic inherited by `PayPalProcessor` and `UPIProcessor`, avoiding duplication.
4. *Polymorphism:* The checkout service held a `PaymentProcessor` reference; swapping providers at runtime required zero checkout-side changes.

**Result:** "Adding a new payment method (Stripe) took one afternoon — implement the interface, register the class. Zero changes to the checkout service. Payment bugs dropped because shared retry logic lived in one place."

| Q | A |
|---|---|
| What does encapsulation protect and how? | Protects object state by making fields private; exposes behavior through public methods with validation |
| What is abstraction's core purpose? | Hide *how* something works; expose only *what* it does via interface/abstract class |
