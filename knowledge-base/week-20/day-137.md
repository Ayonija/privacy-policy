# Day 137 — Graphs Hard: Multi-Level Topo Sort & Parallel Course Scheduling
**Week 20 | Phase 3 — Mock Interviews | Month 5**

## Focus
Two advanced topological sort problems — ordering items within and across groups using a two-level DAG (LC 1203) and computing the minimum time to complete all courses with parallelism via DP on a topological order (LC 2050) — both require composing multiple graph algorithms.

---

## DSA (1.5 hours)

### Pattern: Two-Level Topological Sort (Items Within Groups)
- For LC 1203, treat items with `group == -1` as belonging to unique singleton groups. Build two graphs: (1) item dependency graph, (2) group dependency graph (derived from item dependencies that cross groups). Run topological sort on groups; within each group's set, run topological sort on items. If either sort detects a cycle, return [].
- Trigger condition: "Order items with both item-level and group-level dependencies — sort groups first, then items within groups."
- Time complexity: O(n + m) | Space complexity: O(n + m)

### Pattern: Topological Sort + DP on DAG (Longest Path = Critical Path)
- For LC 2050, build a DAG of courses. Process nodes in topological order (Kahn's BFS); for each course, `time[course] = duration[course] + max(time[prereq] for all prereqs)`. The answer is max over all time[course]. This is the "critical path" problem — longest weighted path in a DAG.
- Trigger condition: "Minimum time to complete all tasks with prerequisite dependencies and parallel execution — longest path in a DAG."
- Time complexity: O(V + E) | Space complexity: O(V + E)

### Code Skeleton

```java
class Solution {

    // LC 2050 — Parallel Courses III (easier to show clean DP pattern first)
    public int minimumTime(int n, int[][] relations, int[] time) {
        List<List<Integer>> adj = new ArrayList<>();
        int[] inDegree = new int[n + 1];
        int[] earliest = new int[n + 1];  // earliest completion time

        for (int i = 0; i <= n; i++) adj.add(new ArrayList<>());
        for (int[] rel : relations) {
            adj.get(rel[0]).add(rel[1]);
            inDegree[rel[1]]++;
        }

        // Initialize: courses with no prerequisites can start at time 0
        Deque<Integer> queue = new ArrayDeque<>();
        for (int i = 1; i <= n; i++) {
            if (inDegree[i] == 0) {
                queue.offer(i);
                earliest[i] = time[i - 1];
            }
        }

        while (!queue.isEmpty()) {
            int course = queue.poll();
            for (int next : adj.get(course)) {
                // next can start only after course finishes
                earliest[next] = Math.max(earliest[next], earliest[course] + time[next - 1]);
                if (--inDegree[next] == 0) queue.offer(next);
            }
        }

        int result = 0;
        for (int i = 1; i <= n; i++) result = Math.max(result, earliest[i]);
        return result;
    }


    // LC 1203 — Sort Items by Groups (helper method pattern)
    public int[] sortItems(int n, int m, int[] group, int[][] beforeItems) {
        // Step 1: assign unique group IDs to singleton items (group == -1)
        int groupId = m;
        for (int i = 0; i < n; i++) {
            if (group[i] == -1) group[i] = groupId++;
        }
        int totalGroups = groupId;

        // Step 2: build item graph and group graph
        List<List<Integer>> itemAdj = new ArrayList<>();
        List<List<Integer>> groupAdj = new ArrayList<>();
        int[] itemDeg = new int[n];
        int[] groupDeg = new int[totalGroups];

        for (int i = 0; i < n; i++) itemAdj.add(new ArrayList<>());
        for (int i = 0; i < totalGroups; i++) groupAdj.add(new ArrayList<>());

        for (int i = 0; i < n; i++) {
            for (int pre : beforeItems[i]) {
                itemAdj.get(pre).add(i);
                itemDeg[i]++;
                if (group[pre] != group[i]) {
                    groupAdj.get(group[pre]).add(group[i]);
                    groupDeg[group[i]]++;
                }
            }
        }

        // Step 3: topo sort items and groups
        List<Integer> itemOrder = topoSort(itemAdj, itemDeg, n);
        List<Integer> groupOrder = topoSort(groupAdj, groupDeg, totalGroups);
        if (itemOrder.isEmpty() || groupOrder.isEmpty()) return new int[]{};

        // Step 4: group items by group ID, then order groups, then flatten
        Map<Integer, List<Integer>> groupToItems = new HashMap<>();
        for (int item : itemOrder) {
            groupToItems.computeIfAbsent(group[item], k -> new ArrayList<>()).add(item);
        }

        List<Integer> result = new ArrayList<>();
        for (int g : groupOrder) {
            result.addAll(groupToItems.getOrDefault(g, new ArrayList<>()));
        }

        return result.stream().mapToInt(Integer::intValue).toArray();
    }

    private List<Integer> topoSort(List<List<Integer>> adj, int[] inDeg, int n) {
        Deque<Integer> queue = new ArrayDeque<>();
        for (int i = 0; i < n; i++) if (inDeg[i] == 0) queue.offer(i);
        List<Integer> order = new ArrayList<>();
        while (!queue.isEmpty()) {
            int node = queue.poll();
            order.add(node);
            for (int next : adj.get(node)) {
                if (--inDeg[next] == 0) queue.offer(next);
            }
        }
        return order.size() == n ? order : new ArrayList<>();  // empty = cycle
    }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Notification Service

**Core components:**
- **Trigger events:** trip state changes (driver accepted, driver arrived, trip started, trip completed), fare estimates, promo offers, safety check-ins
- **Notification Service** — consumes from Kafka `trip-events`; determines notification type and channel (push, SMS, in-app); calls APNs (iOS) or FCM (Android) for push
- **Channel routing:** push first; fall back to SMS if push fails (device offline or no permission); in-app is always updated via WebSocket/SSE
- **Deduplication:** rider should not receive 10 "driver is arriving" pushes if the event fires multiple times; use Redis key `notif:{trip_id}:{event_type}` with `SET NX EX 60` to ensure at-most-one delivery per event type per minute
- **Delivery tracking:** each notification has a `delivery_id`; APNs/FCM return delivery receipts; unacknowledged notifications are retried with exponential backoff up to 3 times; dead-letter queue for permanent failures
- **Localization:** notification text must match rider's locale stored in the User Service; fetched at send time from a cache

**Key trade-offs:**
- **At-least-once vs at-most-once delivery:** APNs/FCM guarantee at-least-once delivery; for trip status it's better to risk duplicate than miss — use idempotency on the client (app deduplicates by `event_type + trip_id`)
- **SMS fallback cost vs reliability:** SMS is 10-100x more expensive than push; Uber uses SMS only for critical events (trip completed + fare) and for markets with low push reliability
- **Batching vs real-time:** for marketing/promo notifications, batch overnight; for trip status, real-time is mandatory (latency SLA < 5s)

**Interview talking point:** "If asked how Uber ensures a rider is notified within 5 seconds of their driver arriving, answer: When the driver marks 'arrived' in the app, the Driver App Backend sends an HTTP POST to the Trip Service, which (1) updates the trip state, (2) publishes to Kafka `trip-events` in < 10ms. The Notification Service consumer processes the event, looks up the rider's push token from the User Service cache, and calls FCM/APNs in < 100ms. FCM targets < 1s delivery to device. Total: < 5s end-to-end."

## Flashcards

| Q | A |
|---|---|
| In LC 1203, why assign unique group IDs to items with `group == -1`? | These items have no group — treating each as a singleton group allows the group-level topo sort to handle their ordering constraints without special-casing |
| In LC 1203, how is a group-level dependency derived from item-level dependencies? | If item `pre` is in group A and item `next` is in group B (A ≠ B) and `pre` must come before `next`, add edge A → B in the group graph |
| In LC 2050, what does `earliest[course]` represent? | The earliest possible completion time of `course`, accounting for the longest chain of prerequisites |
| In LC 2050, why is the answer `max(earliest[i])` rather than the sum? | Courses run in parallel — multiple independent chains execute simultaneously; the bottleneck is the slowest (longest) chain |
| In Uber's notification service, what does the Redis `SET NX EX 60` key prevent? | Sending duplicate notifications for the same event type within 60 seconds — if another notification for the same `(trip_id, event_type)` was sent recently, the SET NX fails and the duplicate is dropped |
