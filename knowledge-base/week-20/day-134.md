## LLD: Singleton Pattern

### What it Does
Ensures a class has **exactly one instance** throughout the application's lifetime and provides a global access point to that instance.

### Analogy: College Principal
A college has exactly one principal. Anyone who needs to speak to the principal reaches the same person — there's never a second principal instantiated. The `getPrincipal()` method always returns the same object, no matter how many times it's called.

### When to Use
- Logger (one log writer, avoiding interleaved/duplicated output)
- Configuration manager (one source of truth for app config)
- Database connection pool (one pool shared by all services)
- In-memory cache (one shared cache to avoid stale reads across instances)
- Thread pool manager

---

### Implementation Progression

**Step 1 — Lazy Init (broken under concurrency):**
```java
class Logger {
    private static Logger instance;          // null at startup

    private Logger() {}                      // prevent external instantiation

    public static Logger getInstance() {
        if (instance == null) {              // RACE CONDITION HERE
            instance = new Logger();         // two threads can both pass the null check
        }
        return instance;
    }
}
```
**Thread-safety trap:** Thread A and Thread B both read `instance == null` as true before either creates the object. Both enter the block. Two `Logger` objects are created — Singleton is broken.

---

**Fix 1 — Synchronized Method (correct, slow):**
```java
public static synchronized Logger getInstance() {
    if (instance == null) {
        instance = new Logger();
    }
    return instance;
}
// Every call acquires a lock — 100x slower after the instance exists
```

---

**Fix 2 — Double-Checked Locking (correct, fast):**
```java
class Logger {
    private static volatile Logger instance; // volatile ensures visibility across CPUs

    private Logger() {}

    public static Logger getInstance() {
        if (instance == null) {              // fast path: no lock if already created
            synchronized (Logger.class) {
                if (instance == null) {      // re-check inside lock
                    instance = new Logger();
                }
            }
        }
        return instance;
    }
}
// volatile prevents instruction reordering; second null-check prevents duplicate creation
```
**Why `volatile`:** Without it, the CPU can partially construct the object and write the reference before the constructor finishes — another thread reads a non-null but not-yet-usable instance.

---

**Fix 3 — Eager Initialization (simplest, safe):**
```java
class Logger {
    private static final Logger INSTANCE = new Logger(); // JVM class-loading is thread-safe

    private Logger() {}

    public static Logger getInstance() { return INSTANCE; }
}
// Downside: created even if never used; acceptable when the object is lightweight
```

---

**Fix 4 — Enum Singleton (best practice in Java):**
```java
enum Logger {
    INSTANCE;

    public void log(String msg) { System.out.println(msg); }
}
// Usage: Logger.INSTANCE.log("Hello");
// JVM guarantees one instance per enum constant; serialization-safe; reflection-safe
```
Enum Singleton is the **recommended approach in Java** (per Effective Java, Item 3). It is immune to serialization attacks and reflection-based instantiation that can break class-based Singletons.

### Why Singleton ≠ Global Variable

| | Global Variable | Singleton |
|---|---|---|
| **Testability** | Cannot be substituted in tests | Can be mocked if accessed via interface |
| **Initialization control** | Set at compile time or immediately | Lazy or eager, with lifecycle control |
| **Access control** | Any code reads/writes directly | Controlled access through one method |
| **Subclassable/mockable** | No | Yes — extract interface, inject mock in tests |

In production code, inject Singleton through DIP (constructor injection) to preserve testability.

### Machine Coding Trigger
"There should be only one X" → Singleton for X.

Examples:
- "There should be only one database connection pool"
- "There should be only one configuration loader"
- "There should be only one instance managing the parking lot"

---

### Don't Confuse
| Pattern | Answers the question |
|---|---|
| **Singleton** | *How many* instances should exist? (exactly one) |
| **Factory** | *Which type* of object should be created? |

Singleton manages instance count. Factory delegates object creation. They are orthogonal — a Factory can itself be a Singleton (e.g., a `VehicleFactory` that is the single factory instance).

## STAR: "Design a thread-safe logger for a distributed system"

**Situation:** "In a microservices deployment, each service had its own logger instance. Log entries from the same request were scattered across different files with no correlation. The team needed a centralized, thread-safe logging component."

**Task:** "Design a logger that is instantiated once per process, handles concurrent writes without interleaving, and supports structured log entries for downstream aggregation."

**Action:**
1. *Singleton:* Applied double-checked locking with `volatile` to guarantee exactly one `Logger` instance per JVM process — multiple threads calling `Logger.getInstance()` always receive the same object.
2. *Thread safety for writes:* Used a `BlockingQueue<LogEntry>` — calling threads enqueue entries non-blockingly; a single dedicated background thread drains the queue and writes to disk/stdout, eliminating interleaved output.
3. *Structured logging:* Each `LogEntry` carried a `correlationId` (injected from thread-local context), `level`, `timestamp`, and `message` — serialized to JSON for Splunk/Elasticsearch ingestion.
4. *Graceful shutdown:* Registered a JVM shutdown hook to drain the queue before process exit — no log entries dropped on SIGTERM.

**Result:** "Log correlation rates went from 34% to 99% for multi-service request traces. The singleton approach eliminated duplicate logger instances that had been consuming 120 MB of heap. The async queue absorbed bursts of 50k log entries/second without blocking calling threads."

## Flashcards

| Q | A |
|---|---|
| What is the Singleton pattern? | Ensures exactly one instance of a class exists; provides a global access point |
| What is the thread-safety trap in lazy Singleton initialization? | Two threads can both read `instance == null` as true before either creates the object, resulting in two instances |
| What does `volatile` do in double-checked locking? | Forces CPU visibility — prevents the reference from being published before the constructor finishes; prevents instruction reordering |
| Which Singleton implementation is best in Java and why? | Enum Singleton — JVM guarantees one instance per enum constant; immune to reflection and serialization attacks |
| Why is Singleton not the same as a global variable? | Singleton can be mocked, subclassed, and controlled via interface injection; global variables cannot be substituted in tests |
| When should you reach for Singleton? | Logger, config manager, DB connection pool, in-memory cache — any resource that must be shared and have exactly one copy |
