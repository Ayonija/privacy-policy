**Week 20 | Phase 3 — Mock Interviews | Month 5**

## Focus
Two structurally challenging hard tree problems — finding the maximum sum BST subtree using post-order validation (LC 1373) and reconstructing a binary tree from a depth-encoded preorder string (LC 1028) — then pivot the system design to Uber's dynamic pricing engine.

---

## DSA (1.5 hours)

### Pattern: Post-Order BST Validation with Sum Tracking
- For LC 1373, traverse in post-order; for each node return a tuple: (isBST, minVal, maxVal, subtreeSum). A subtree is a valid BST if: (1) both children are valid BSTs, (2) left's max < node.val < right's min. If valid, update the global max with subtreeSum. Return (-inf, +inf, sum=0, false) for invalid BSTs to propagate failure upward.
- Trigger condition: "Find the maximum sum subtree that satisfies the BST property — post-order is mandatory since validity depends on children."
- Time complexity: O(n) | Space complexity: O(h)

### Pattern: Stack-Based Tree Reconstruction from Depth Encoding
- For LC 1028, parse the preorder string to extract (depth, value) pairs. Use a stack that holds the path from root to current node. To attach a new node at depth d: pop from the stack until stack size equals d (stack[-1] is the parent). The new node becomes the right child if the parent already has a left child, otherwise the left child.
- Trigger condition: "Reconstruct a binary tree from a preorder traversal string that encodes depth with dashes."
- Time complexity: O(n) | Space complexity: O(n)

### Problems

| # | Problem | LC # | Difficulty | Pattern | Key Insight |
|---|---------|------|------------|---------|-------------|
| 1 | Maximum Sum BST in Binary Tree | 1373 | Hard | Post-Order BST Validation + DP | Return (isBST, min, max, sum) from each node; valid if both children valid and BST property holds; update global max |
| 2 | Recover a Tree From Preorder Traversal | 1028 | Hard | Stack-Based Reconstruction | Stack holds current path from root; depth indicates level; pop until stack has `depth` entries, then attach as left/right child |
| 3 | House Robber III | 337 | Medium (Revision) | Post-Order Tree DP (Rob vs Skip) | REVISION: at each node return `(rob, skip)`; `rob = node.val + skip_left + skip_right`; `skip = max(rob_left, skip_left) + max(rob_right, skip_right)` |

### Code Skeleton

```java
class Solution {

    // LC 1373 — Maximum Sum BST in Binary Tree
    private int maxBSTSum = 0;

    public int maxSumBST(TreeNode root) {
        postOrder(root);
        return maxBSTSum;
    }

    // Returns int[]: {isBST (1/0), minVal, maxVal, subtreeSum}
    private int[] postOrder(TreeNode node) {
        if (node == null) return new int[]{1, Integer.MAX_VALUE, Integer.MIN_VALUE, 0};

        int[] left = postOrder(node.left);
        int[] right = postOrder(node.right);

        // Check BST property: both children valid + ordering constraint
        if (left[0] == 1 && right[0] == 1
                && left[2] < node.val && node.val < right[1]) {
            int sum = left[3] + node.val + right[3];
            maxBSTSum = Math.max(maxBSTSum, sum);
            int minVal = Math.min(left[1], node.val);   // left min, or node if leaf
            int maxVal = Math.max(right[2], node.val);  // right max, or node if leaf
            return new int[]{1, minVal, maxVal, sum};
        }
        // Invalid BST subtree
        return new int[]{0, Integer.MIN_VALUE, Integer.MAX_VALUE, 0};
    }


    // LC 1028 — Recover a Tree From Preorder Traversal
    public TreeNode recoverFromPreorder(String traversal) {
        Deque<TreeNode> stack = new ArrayDeque<>();
        int i = 0;
        int n = traversal.length();

        while (i < n) {
            // Count dashes to determine depth
            int depth = 0;
            while (i < n && traversal.charAt(i) == '-') { depth++; i++; }

            // Parse the node value
            int val = 0;
            while (i < n && Character.isDigit(traversal.charAt(i))) {
                val = val * 10 + (traversal.charAt(i) - '0');
                i++;
            }

            TreeNode node = new TreeNode(val);

            // Pop stack until we reach the parent level
            while (stack.size() > depth) stack.pollLast();

            // Attach as left or right child of top of stack
            if (!stack.isEmpty()) {
                TreeNode parent = stack.peekLast();
                if (parent.left == null) parent.left = node;
                else parent.right = node;
            }

            stack.addLast(node);
        }

        // Root is the first element pushed
        while (stack.size() > 1) stack.pollLast();
        return stack.peekLast();
    }
}

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Dynamic Pricing & Surge

**Core components:**
- **Pricing Service** — computes fare for a ride request: `fare = base_fare + (per_mile_rate × distance) + (per_minute_rate × estimated_time) + surge_multiplier`
- **Surge Pricing Engine** — monitors supply (available drivers) vs demand (open ride requests) per zone (H3 cell) in real time; surge multiplier = f(demand/supply ratio); updated every 1–5 minutes per zone
- **Supply/Demand Aggregator** — consumes Kafka events (ride requests, driver status changes); maintains per-zone counters in Redis; published to Pricing Service
- **ML Inference Service** — more sophisticated models use features: historical demand patterns (time of day, day of week, events, weather), competitor pricing, cancellation rates; serves predictions via a low-latency gRPC endpoint
- **Fare Estimate Cache** — for the rider's estimate screen, cache recent fare computations by (origin_zone, destination_zone, time_bucket) in Redis with short TTL (30s)

**Key trade-offs:**
- **Rule-based surge vs ML model:** Rule-based (demand/supply ratio threshold) is transparent and auditable; ML models capture complex patterns but are harder to explain — Uber uses both (rule-based as floor, ML as ceiling)
- **Zone granularity:** Finer zones give more accurate surge but increase computational load and create discontinuities at borders; Uber uses H3 resolution 7–9 depending on city density
- **Fare transparency vs dynamic adjustment:** Riders prefer stable fare estimates; post-estimate changes require UX disclosure; Uber locks fare for 2 minutes after estimate — locks require storing the locked estimate with expiry

**Interview talking point:** "If asked how Uber prevents a rider from seeing one price and being charged another, answer: Uber implements fare locking — when a rider requests a fare estimate, the Pricing Service generates a unique `fare_lock_id` stored in Redis with a 2-minute TTL and the locked fare. When the trip completes, the Payment Service retrieves the fare by `fare_lock_id`. If the lock has expired (rider waited > 2 min), a new estimate is generated and the rider must re-confirm. This prevents bait-and-switch."

| In LC 1373, why does a null node return `Integer.MAX_VALUE` for min and `Integer.MIN_VALUE` for max? | A null node adds no constraint to the BST check; MAX_VALUE for min means "no lower bound from left child"; MIN_VALUE for max means "no upper bound from right child" — any real node value will satisfy the comparison |
| In LC 1373, what four values does the post-order helper return for each node? | (isBST: 1 or 0, minVal in subtree, maxVal in subtree, subtreeSum) |
| In LC 1028, what does the number of dashes before a node value represent? | The depth of that node in the tree (0 dashes = root, 1 dash = depth 1, etc.) |
| In LC 1028, when do you attach a new node as the right child vs left child of its parent? | If the parent already has a left child → attach as right child; otherwise → attach as left child |
| In Uber's surge pricing, what triggers the surge multiplier to increase? | When demand (open ride requests per zone) exceeds supply (available drivers per zone) by a threshold ratio; multiplier scales with the ratio |
