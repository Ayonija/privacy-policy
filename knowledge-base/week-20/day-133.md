## LLD: SOLID — I, D

### I — Interface Segregation Principle (ISP)
**Smell it fixes:** Fat interfaces force implementing classes to provide methods they don't use — stub methods or `UnsupportedOperationException` everywhere.

**Principle:** A class should not be forced to implement interfaces it does not use. Prefer many small, role-specific interfaces over one large general one.

**Analogy:** A restaurant menu vs. a combined restaurant-hotel-spa menu — a café doesn't need the spa pricing section forced on it.

**Bad code (fat interface):**
```java
interface Printer {
    void print(Document d);
    void scan(Document d);
    void fax(Document d);
    void photocopy(Document d);
}

class BasicInkjetPrinter implements Printer {
    public void print(Document d) { /* real impl */ }
    public void scan(Document d)  { throw new UnsupportedOperationException(); } // forced
    public void fax(Document d)   { throw new UnsupportedOperationException(); } // forced
    public void photocopy(Document d) { throw new UnsupportedOperationException(); }
}
```

**Fixed code (role interfaces):**
```java
interface Printable   { void print(Document d); }
interface Scannable   { void scan(Document d); }
interface Faxable     { void fax(Document d); }

class BasicInkjetPrinter implements Printable {
    public void print(Document d) { /* real impl */ }
    // no unused methods
}

class MultifunctionPrinter implements Printable, Scannable, Faxable {
    public void print(Document d) { /* impl */ }
    public void scan(Document d)  { /* impl */ }
    public void fax(Document d)   { /* impl */ }
}
```

**Interview questions:**
- "What is ISP?" → Don't force classes to implement methods they don't need; split fat interfaces.
- "How do you spot ISP violations?" → `UnsupportedOperationException` in interface implementations; stub/empty method bodies.

### D — Dependency Inversion Principle (DIP)
**Smell it fixes:** High-level business logic is tightly coupled to low-level implementation details — swapping a database or email provider requires changing core business code.

**Principle:**
1. High-level modules should not depend on low-level modules. Both should depend on abstractions.
2. Abstractions should not depend on details. Details should depend on abstractions.

**Analogy (charger socket):** Your laptop charges via a USB-C socket (abstraction). You don't rewire the laptop when you switch from a 45W to a 100W charger — both chargers implement the USB-C contract. The laptop (high-level) depends on the socket standard (abstraction), not on any specific charger (concretion).

**Bad code:**
```java
class OrderService {
    private MySQLOrderRepository repo = new MySQLOrderRepository(); // concrete dep
    // If we switch to PostgreSQL, OrderService must change — wrong layer coupling
    void placeOrder(Order o) { repo.save(o); }
}
```

**Fixed code:**
```java
interface OrderRepository { void save(Order o); }

class MySQLOrderRepository implements OrderRepository {
    public void save(Order o) { /* MySQL impl */ }
}

class PostgreSQLOrderRepository implements OrderRepository {
    public void save(Order o) { /* PG impl */ }
}

class OrderService {
    private final OrderRepository repo; // depends on abstraction

    OrderService(OrderRepository repo) { this.repo = repo; } // injected

    void placeOrder(Order o) { repo.save(o); } // never changes when impl swaps
}
```

**DIP in practice:** This is the foundation of Dependency Injection (DI) frameworks (Spring, Guice). DIP is the principle; DI is the pattern; DI containers are the mechanism.

**Interview questions:**
- "What is DIP?" → High-level modules depend on abstractions, not concrete classes.
- "How is DIP related to Dependency Injection?" → DI is the pattern that enforces DIP; the dependency is injected (constructor/setter), not instantiated inside.

## STAR: "Describe a time you applied Dependency Inversion"

**Situation:** "Our notification service hardcoded `new SMTPEmailClient()` directly inside `NotificationService`. When we needed to add SMS support and write unit tests, we couldn't mock the email client — tests required a live SMTP server."

**Task:** "Refactor the notification layer to support multiple channels and be testable without live infrastructure."

**Action:**
1. *Defined the abstraction:* I introduced a `NotificationChannel` interface with a single `send(recipient, message)` method — the contract any channel must fulfill.
2. *Moved concrete clients behind the interface:* `EmailChannel` and `SMSChannel` each implemented `NotificationChannel`; neither the interface nor `NotificationService` knew about the other's details.
3. *Injected via constructor:* `NotificationService` accepted a `List<NotificationChannel>` in its constructor. The caller (or DI container) wired the concretions.
4. *Enabled testing:* Unit tests injected a `MockChannel` that recorded calls — no SMTP server needed. Test time dropped from 45 seconds to under 1 second.

**Result:** "Adding WhatsApp notifications two sprints later took 90 minutes — implement `WhatsAppChannel`, register it. Zero changes to `NotificationService`. Test coverage went from 12% to 78% on the notification module."

## Flashcards

| Q | A |
|---|---|
| What is ISP and what smell does it fix? | No class should be forced to implement unused interface methods; fixes fat interfaces with stub/exception implementations |
| How do you fix a fat interface? | Split into role-specific interfaces; each class implements only the roles it genuinely supports |
| What are DIP's two rules? | High-level modules depend on abstractions; abstractions don't depend on details — details depend on abstractions |
| Charger socket analogy for DIP | Laptop (high-level) depends on USB-C socket (abstraction); 45W or 100W charger (detail) implements the socket — laptop never rewired |
| How is DIP different from Dependency Injection? | DIP is the principle (depend on abstractions); DI is the design pattern that enforces it (inject the abstraction via constructor/setter) |
| What is the surest sign you need DIP? | A high-level class calling `new ConcreteClass()` internally, making it untestable without the real dependency |
