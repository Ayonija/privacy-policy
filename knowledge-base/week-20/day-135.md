## LLD: Factory Method Pattern

### What it Does
Delegates object creation to a factory method instead of calling `new ConcreteClass()` directly. The caller never knows which concrete class it's getting.

### Analogy: Chai Tapri
You walk up to the tapri and say "ek chai." You don't specify how the chai is made, which tea leaves are used, or what vessel is used. The tapri (factory) decides. You just drink the result.

### What it Solves
- Caller code is decoupled from concrete types — adding a new type doesn't require editing callers.
- Replaces brittle `if-else` chains in creation logic with polymorphism.

### Code Example 1 — Shape Factory:
```java
interface Shape {
    void draw();
}

class Circle implements Shape {
    public void draw() { System.out.println("Drawing Circle"); }
}

class Square implements Shape {
    public void draw() { System.out.println("Drawing Square"); }
}

class ShapeFactory {
    public static Shape create(String type) {
        switch (type) {
            case "CIRCLE": return new Circle();
            case "SQUARE": return new Square();
            default: throw new IllegalArgumentException("Unknown shape: " + type);
        }
    }
}

// Caller
Shape s = ShapeFactory.create("CIRCLE");
s.draw(); // caller never wrote `new Circle()`
```

### Code Example 2 — Notification Factory:
```java
interface Notification {
    void send(String recipient, String message);
}

class EmailNotification  implements Notification { /* SMTP send */ }
class SMSNotification    implements Notification { /* Twilio send */ }
class PushNotification   implements Notification { /* FCM send */ }

class NotificationFactory {
    public static Notification create(String channel) {
        switch (channel) {
            case "EMAIL": return new EmailNotification();
            case "SMS":   return new SMSNotification();
            case "PUSH":  return new PushNotification();
            default: throw new IllegalArgumentException(channel);
        }
    }
}
```

---

### Production Tip: Registration Map Over if-else Chain
A switch-case factory must be edited for every new type — OCP violation. Replace it with a registration map:

```java
class NotificationFactory {
    private static final Map<String, Supplier<Notification>> REGISTRY = new HashMap<>();

    static {
        REGISTRY.put("EMAIL", EmailNotification::new);
        REGISTRY.put("SMS",   SMSNotification::new);
        REGISTRY.put("PUSH",  PushNotification::new);
    }

    public static void register(String type, Supplier<Notification> creator) {
        REGISTRY.put(type, creator); // new types plug in without touching this class
    }

    public static Notification create(String type) {
        Supplier<Notification> creator = REGISTRY.get(type);
        if (creator == null) throw new IllegalArgumentException(type);
        return creator.get();
    }
}
```
Adding `WhatsAppNotification` = one `register()` call at startup. Zero edits to factory class.

## LLD: Abstract Factory Pattern

### What it Does
A factory of factories — creates families of related objects that must be used together, without specifying their concrete classes.

### Analogy: Furniture Set
A furniture store sells complete room sets: a Victorian set (Victorian chair + Victorian table + Victorian sofa) or a Modern set (Modern chair + Modern table + Modern sofa). You pick a style, and the factory produces all matching pieces. You never end up with a Victorian chair at a Modern table.

### What it Solves
Ensures product family consistency — objects from the same factory are guaranteed to be compatible. Adding a new family means adding a new concrete factory.

### Code Example — UI Widget Factory:
```java
// Abstract products
interface Button  { void render(); }
interface Checkbox { void render(); }

// Windows family
class WindowsButton  implements Button  { public void render() { /* Win style */ } }
class WindowsCheckbox implements Checkbox { public void render() { /* Win style */ } }

// Mac family
class MacButton   implements Button  { public void render() { /* Mac style */ } }
class MacCheckbox implements Checkbox { public void render() { /* Mac style */ } }

// Abstract factory
interface UIFactory {
    Button   createButton();
    Checkbox createCheckbox();
}

// Concrete factories — one per family
class WindowsUIFactory implements UIFactory {
    public Button   createButton()   { return new WindowsButton(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}

class MacUIFactory implements UIFactory {
    public Button   createButton()   { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// Application
class Application {
    private Button button;
    private Checkbox checkbox;

    Application(UIFactory factory) {
        button   = factory.createButton();
        checkbox = factory.createCheckbox();
        // Never mixes Windows button with Mac checkbox
    }
}
```

---

### Factory Method vs. Abstract Factory

| | Factory Method | Abstract Factory |
|---|---|---|
| **Creates** | One type of product | A family of related products |
| **How many products** | One | Multiple (must be compatible) |
| **Extension** | Add new subclass + update factory | Add new concrete factory class |
| **Analogy** | Chai tapri gives you one chai | Furniture set gives you matching chair + table + sofa |
| **Machine coding trigger** | "Create the right type of X" | "Create a set of compatible X, Y, Z" |

---

### Machine Coding Triggers

**Factory Method:** "The system should create the right type of vehicle / notification / payment processor based on a type string."

**Abstract Factory:** "The system needs to support multiple UI themes / database families / cloud providers where all components within a family must be compatible."

---

### Don't Confuse: Factory vs. Builder

| | Factory | Builder |
|---|---|---|
| **Problem** | Which type to create | How to construct one complex object |
| **Caller's interaction** | One call, pass a type hint | Multiple calls setting parts, then `.build()` |
| **Output** | Different types of same interface | One fully configured instance |
| **Analogy** | Chai tapri (one request, right type) | Subway sandwich (bread → fillings → sauce → build) |

## STAR: "Design a notification system that supports email, SMS, and push"

**Situation:** "Our platform sent order confirmations via email only. Product asked for SMS and push in the same sprint. The existing `sendEmail()` was hardcoded throughout the checkout flow — 14 call sites."

**Task:** "Redesign the notification layer to support three channels with zero changes to the checkout flow, and make it extensible for future channels."

**Action:**
1. *Defined the contract:* Introduced `NotificationChannel` interface with `send(userId, message, metadata)` — one contract all channels implement.
2. *Factory with registration map:* Built `NotificationFactory` backed by a `Map<String, Supplier<NotificationChannel>>`. Email, SMS, and Push registered at app startup. Checkout code now calls `factory.create(channel).send(...)` — no `new EmailClient()` anywhere in business logic.
3. *Multi-channel dispatch:* Added `NotificationService.broadcast(channels, userId, message)` — iterates the list and calls factory per channel. Order confirmation now dispatches email + push in parallel (ExecutorService), SMS sequentially (rate-limited carrier).
4. *OCP guaranteed:* When WhatsApp was requested two sprints later, I implemented `WhatsAppChannel` and called `factory.register("WHATSAPP", WhatsAppChannel::new)` in the startup config. Zero changes to checkout, factory class, or `NotificationService`.

**Result:** "Time to add a new notification channel went from 3-day refactor to 2-hour implementation. Notification failures became channel-specific — an SMS API outage no longer blocked email delivery. Test coverage on the notification module reached 91% using mock channels."

## Flashcards

| Q | A |
|---|---|
| What is the Factory Method pattern? | Delegates object creation to a factory; caller never calls `new ConcreteClass()`; decouples caller from concrete types |
| What is the production-grade upgrade to a switch-case factory? | Registration map: `Map<String, Supplier<T>>`; new types register themselves without editing the factory |
| What is the Abstract Factory pattern? | Creates families of related products that must be used together; ensures compatibility within a product family |
| Factory Method vs. Abstract Factory — one-line difference | Factory Method creates one type of product; Abstract Factory creates a family of compatible products |
| Furniture set — which pattern and why? | Abstract Factory — chair + table + sofa must match the style; one factory guarantees consistent family |
| Factory vs. Builder — how the caller interacts | Factory: one call, pass a type → get object; Builder: chain multiple calls setting parts → call `.build()` |
