## DSA (1.5 hours)

### Pattern: Topological Sort from Pairwise Character Constraints
- For LC 269, compare adjacent words character by character — the first difference gives an ordering constraint: `A[i] → B[i]` (A's char comes before B's). Build a directed graph of characters and run topological sort (Kahn's BFS or DFS cycle detection). If a longer word precedes a shorter prefix (e.g., "abc" before "ab"), the input is invalid → return "".
- Trigger condition: "Derive a character ordering from a sorted list of words where the sort key is unknown — each adjacent pair gives one constraint edge."
- Time complexity: O(N·L + U + E) — N words, L max length, U unique chars, E edges | Space: O(U + E)

### Pattern: DFS + Memoization on Grid (Longest Path)
- For LC 329, the answer is the longest path where each step moves to an adjacent cell with a strictly larger value. DFS from each cell with memoization: if `memo[r][c]` is set, return it immediately. Recursively try all 4 directions where `grid[nr][nc] > grid[r][c]`. No visited array needed — strictly increasing values guarantee no cycles.
- Trigger condition: "Longest increasing path in a grid — DFS with memo avoids re-computing from cells already explored; strictly increasing values eliminate cycle risk."
- Time complexity: O(mn) — each cell computed once | Space: O(mn) for memo

### Code Skeleton

```java
class Solution {

    // LC 269 — Alien Dictionary
    public String alienOrder(String[] words) {
        // Build adjacency list and in-degree map for all unique chars
        Map<Character, Set<Character>> adj = new HashMap<>();
        Map<Character, Integer> inDegree = new HashMap<>();

        for (String w : words) {
            for (char c : w.toCharArray()) {
                adj.putIfAbsent(c, new HashSet<>());
                inDegree.putIfAbsent(c, 0);
            }
        }

        // Extract ordering constraints from adjacent word pairs
        for (int i = 0; i < words.length - 1; i++) {
            String w1 = words[i], w2 = words[i + 1];
            int minLen = Math.min(w1.length(), w2.length());
            // Invalid: longer word is a prefix of shorter next word
            if (w1.length() > w2.length() && w1.startsWith(w2)) return "";
            for (int j = 0; j < minLen; j++) {
                if (w1.charAt(j) != w2.charAt(j)) {
                    char from = w1.charAt(j), to = w2.charAt(j);
                    if (!adj.get(from).contains(to)) {
                        adj.get(from).add(to);
                        inDegree.put(to, inDegree.get(to) + 1);
                    }
                    break;  // only first differing char gives a constraint
                }
            }
        }

        // Kahn's BFS topological sort
        Deque<Character> queue = new ArrayDeque<>();
        for (char c : inDegree.keySet()) {
            if (inDegree.get(c) == 0) queue.offer(c);
        }

        StringBuilder result = new StringBuilder();
        while (!queue.isEmpty()) {
            char c = queue.poll();
            result.append(c);
            for (char neighbor : adj.get(c)) {
                inDegree.put(neighbor, inDegree.get(neighbor) - 1);
                if (inDegree.get(neighbor) == 0) queue.offer(neighbor);
            }
        }

        // If not all chars in result, there is a cycle
        return result.length() == inDegree.size() ? result.toString() : "";
    }


    // LC 329 — Longest Increasing Path in a Matrix
    private static final int[][] DIRS = {{0,1},{0,-1},{1,0},{-1,0}};
    private int[][] memo;

    public int longestIncreasingPath(int[][] matrix) {
        int m = matrix.length, n = matrix[0].length;
        memo = new int[m][n];
        int result = 0;
        for (int r = 0; r < m; r++) {
            for (int c = 0; c < n; c++) {
                result = Math.max(result, dfs(matrix, r, c, m, n));
            }
        }
        return result;
    }

    private int dfs(int[][] matrix, int r, int c, int m, int n) {
        if (memo[r][c] != 0) return memo[r][c];
        int best = 1;  // path of length 1 (just this cell)
        for (int[] d : DIRS) {
            int nr = r + d[0], nc = c + d[1];
            if (nr >= 0 && nr < m && nc >= 0 && nc < n && matrix[nr][nc] > matrix[r][c]) {
                best = Math.max(best, 1 + dfs(matrix, nr, nc, m, n));
            }
        }
        return memo[r][c] = best;
    }
}
```

## System Design (1 hour)

### Topic: Uber / Lyft — Trip Service & State Machine

**Core components:**
- **Trip state machine:** `REQUESTED → DRIVER_ACCEPTED → DRIVER_ARRIVED → IN_PROGRESS → COMPLETED | CANCELLED`
- **Trip Service** — manages state transitions; each transition is a DB write + Kafka event; state is stored in PostgreSQL with optimistic locking (version column)
- **Idempotency:** driver app retries on network failure — each state transition is keyed by `(trip_id, from_state, to_state)`; duplicate transitions are no-ops if already applied
- **Event sourcing option:** instead of mutable state, store each transition as an immutable event; current state is derived by replaying events; enables full audit trail
- **Notifications trigger:** each state transition publishes to Kafka `trip-events`; Notification Service consumes and pushes to rider and driver apps (APNs/FCM)
- **Cancellation handling:** rider or driver can cancel at each state; cancellation fee logic depends on state (no fee if REQUESTED, fee if DRIVER_ARRIVED or IN_PROGRESS)

**Key trade-offs:**
- **Mutable state vs event sourcing:** Mutable is simpler (single row update); event sourcing provides full history and is easier to audit — Uber uses mutable state with a separate audit log table
- **Optimistic vs pessimistic locking:** Optimistic (version column increment) allows high concurrency; pessimistic (SELECT FOR UPDATE) serializes — optimistic works for trip state since concurrent state transitions on the same trip are rare
- **Synchronous vs async notification on state change:** synchronous risks adding latency to the trip API; async via Kafka adds a small delay (~1-2s) but keeps API latency low — Uber uses async

**Interview talking point:** "If asked how Uber prevents a driver from accepting two rides simultaneously, answer: The driver's status in Redis is atomically set from AVAILABLE to MATCHED using `SET driver:{id}:status MATCHED NX PX 30000` (atomic set-if-not-exists with 30s TTL). When the driver's Trip Service processes an acceptance, it first checks this Redis lock. If another accept already acquired it, the request is rejected with a 409 Conflict. The DB trip record is updated only after the Redis lock is confirmed."

## Flashcards

| Q | A |
|---|---|
| In LC 269, why does `"abc"` appearing before `"ab"` in the sorted list make the ordering invalid? | In a sorted order, a word that is a proper prefix must come before the longer word — seeing it after means the input violates the sort invariant |
| In LC 269, why break after finding the first differing character in an adjacent word pair? | Only the first difference gives a definitive ordering constraint; subsequent characters are irrelevant since their relative order is unknown |
| In LC 329, why is no visited array needed to prevent cycles? | Strictly increasing values guarantee you can never revisit a cell — each step increases the grid value, so revisiting would require going to a smaller value |
| In LC 329, what does `memo[r][c]` represent? | The length of the longest increasing path starting at cell (r, c) |
| In Uber's Trip Service, how does optimistic locking prevent two drivers from accepting the same trip? | Each trip row has a version counter; accepting sets status to MATCHED where version = current_version; concurrent accept would fail (stale version) and return 409 |
