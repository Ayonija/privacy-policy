## DSA (1.5 hours)

### Pattern: Tree Global State Update via Post-Order (Revisit)
- For the LC 124 warm-up, the global variable `maxSum` is updated at every node with `leftGain + node.val + rightGain`, but only `node.val + max(leftGain, rightGain)` is returned to the parent. This asymmetry — "update globally with both branches, return only one branch" — is the defining signature of this pattern.
- Trigger condition: "Path through node that cannot continue upward with both branches — update global max locally, propagate single branch upward."

### Pattern: DFS Memoization on Grid with Strict Monotone Constraint (Revisit)
- For LC 329, strictly increasing values are the key insight that eliminates the need for a visited set — you can never cycle on a strictly increasing path. Memoization converts exponential DFS to O(mn).
- Trigger condition: "Longest path in a grid/DAG where cycles are impossible — DFS + memo, no visited array needed."

    public int diameterOfBinaryTree(TreeNode root) {

## System Design (1 hour)

### Topic: Uber / Lyft — Driver App Backend

**Core components:**
- **Driver App** — mobile app (iOS/Android) communicating with backend via WebSocket (persistent) for real-time events + REST for state mutations
- **Driver Gateway** — WebSocket server fleet handling persistent driver connections; each server handles ~10K connections; horizontally scalable; uses sticky sessions (consistent hash on driver_id)
- **Location Publisher** — Driver App sends GPS ping every 4s → Driver Gateway → publishes to Kafka `driver-location`; consumed by Location Worker → Redis Geo update
- **Status Management** — driver AVAILABLE/OFFLINE/IN_TRIP status stored in Redis; updated via REST calls (`POST /driver/status`); change events published to Kafka for matching and analytics
- **Geofencing** — driver app receives polygon boundaries for surge zones, restricted zones (airports requiring special license), and pickup/dropoff zones; computed server-side, pushed to driver app on status change or zone update
- **Batch location optimization** — driver app can buffer 3-5 location pings and send as a batch to reduce radio wake-ups (battery optimization on mobile); server processes batch in order

**Key trade-offs:**
- **WebSocket vs MQTT:** MQTT is purpose-built for IoT/mobile telemetry (very low overhead, QoS levels, offline message buffering); WebSocket is more general but higher overhead — Uber uses their own protocol over TCP/TLS (similar to MQTT) for driver app; rider app uses WebSocket
- **Sticky sessions for WebSocket:** consistent hash routes each driver_id to the same gateway server; avoids state sync across servers; risk: gateway server crash routes driver to new server with no buffered messages — mitigated by Kafka replay
- **Battery vs freshness:** 4-second GPS polling drains battery; increasing interval to 8s halves battery impact but degrades matching accuracy; Uber uses adaptive polling (4s when in a trip or recently requested, 8s when idle)

**Interview talking point:** "If asked how the driver app handles intermittent network connectivity (tunnels, dead zones), answer: The driver app buffers GPS pings locally (in-memory queue, max 60 entries = 4 minutes). On reconnect, it sends the buffered batch in order. The Location Service processes the batch sequentially, updating the Redis Geo position for each ping. The final ping's position is the current location. Trip state changes (arrive, start, complete) are retried with exponential backoff until the server acknowledges with a 200 response."
