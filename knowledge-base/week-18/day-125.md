## DSA (1.5 hours)

### Pattern: Two-Heap Sliding Window (Running Median)
- Maintain a max-heap `lo` (lower half) and min-heap `hi` (upper half) such that `|lo| - |hi| ≤ 1` and `lo.peek() ≤ hi.peek()`. The median is `lo.peek()` (odd size) or `(lo.peek() + hi.peek()) / 2` (even). On every window slide: add new element to appropriate heap, rebalance; lazy-delete the exiting element using a hash map tracking invalid entries.
- Trigger condition: "Median (or other order statistic) of all elements in a sliding window of fixed size k."
- Time complexity: O(n log k) | Space complexity: O(k)

### Pattern: Forward-Backward Two Pointer (Minimum Window Subsequence)
- Phase 1 (forward): scan `s` left to right to find the first index where all characters of `t` appear in order — record `end`. Phase 2 (backward): from `end`, scan right to left to find the leftmost `start` from which `t` is a subsequence ending at `end`. Record this window. Move forward and repeat from `start + 1`.
- Trigger condition: "Shortest substring of S in which T appears as a subsequence (order matters, not just frequency)."
- Time complexity: O(n * m) worst case | Space complexity: O(1)

## System Design (1 hour)

### Topic: Twitter / X — Search: Trending Topics & Full-Text Search

**Core components:**
- Full-text tweet search: Elasticsearch cluster ingests tweet events from Kafka; indexes `content`, `created_at`, `user_id`, `hashtags`; supports `bool` queries with `match` on content and `range` on timestamp
- Trending topics: computed by a streaming aggregation service (Apache Flink or Spark Streaming) consuming the tweet Kafka stream; counts hashtag/phrase frequency in a sliding 24-hour window; stores top-N trends per geographic region in Redis
- Typeahead/autocomplete: a Trie or Elasticsearch `prefix` query on recently popular terms; results cached aggressively (low update frequency)
- Search result ranking: score = relevance (TF-IDF) × recency factor × author influence (follower count log-scaled); promoted tweets injected as separate layer
- Index freshness: tweets indexed within ~5–10s of creation via near-real-time Elasticsearch indexing

**Key trade-offs:**
- Elasticsearch vs purpose-built search: ES provides out-of-the-box full-text, ranking, and geo-filtering but is expensive at Twitter's scale — Twitter historically built custom Earlybird search engine (inverted index in RAM for recent tweets)
- Trending window: a fixed 24h window misses sudden spikes; use an exponential moving average or a multi-resolution sliding window (last 1h, 6h, 24h) to detect rapid trend emergence
- Consistency between Kafka ingest and ES index: use idempotent writes with tweet_id as the document ID; at-least-once delivery is safe

**Interview talking point:** "If asked how Twitter computes trending topics in real time, answer: A streaming job (e.g., Flink) consumes the Kafka tweet stream, maintains a time-decayed frequency count of hashtags and n-grams in a sliding window, and writes the top-N results per region to Redis every 30–60 seconds. The client fetches trends from Redis, giving sub-100ms latency for trend queries without touching the raw tweet stream."

### Interview Tips

- For the two-heap sliding window, state the balancing invariant before coding: "`len(lo) == len(hi)` (even k) or `len(lo) == len(hi) + 1` (odd k), and `max(lo) ≤ min(hi)`."
- Distinguish LC 727 from LC 76 immediately: "Order matters here — t must appear as a subsequence, not just with the right frequency."
- For Twitter trending: Flink → Redis write every 30-60 seconds — mention the write frequency to show system design granularity.

### STAR Interview Framework

> **Two-Heap Sliding Window with Lazy Deletion:** brute-force O(n² log k) → O(n log k) time, O(k) space

**S:** "Given array of n numbers and window size k. Compute median for each of n-k+1 windows. Naive O(n × k log k) re-sort per window fails at n=10^5, k=10^4."
**T:** "Need O(n log k) by maintaining two heaps with lazy deletion — stale entries marked and purged before reading median."
**A (60%):**
1. *Classify:* "Two-heap: max-heap `lo` (lower half), min-heap `hi` (upper half); lazy-delete via stale map."
2. *Init:* "`lo=[], hi=[], stale=defaultdict(int)`, balance so len(lo) == len(hi)+1 or len(hi)."
3. *Loop/Step:* "Add new element to lo or hi based on value; rebalance. When element exits: mark in stale map. Before reading median: pop stale tops from both heaps."
4. *Termination:* "After lazy_clean: median = lo top (odd k) or average of both tops (even k)."
5. *Gotcha:* "Rebalance AFTER adding new element, then check stale BEFORE reading — doing these out of order gives wrong heap sizes and wrong medians."
**R:** "O(n log k) time, O(k) space. Handles n=10^5, k=10^4 in under a second."

**Alternatives & why not:**
| Alternative | Use when | Why not here |
|------------|----------|-------------|
| Re-sort window O(n × k log k) | k ≤ 100 | Too slow at large k |
| Order-statistic tree (BST) | Language supports it | Python has no built-in; heap+lazy is idiomatic |

|---|---|
| How does lazy deletion work in the two-heap sliding window median? | When an element exits the window, mark it in a stale map; before reading the heap top, pop all stale entries — only then is the top guaranteed to be valid |
| What is the balancing invariant for the two heaps in Sliding Window Median? | `len(lo) == len(hi)` (even window) or `len(lo) == len(hi) + 1` (odd window); and `max(lo) ≤ min(hi)` |
| In LC 727, what does the backward pass accomplish that the forward pass cannot? | The forward pass finds any valid window ending; the backward pass finds the minimum-length window ending at that same position by starting t from its last character |
| How is LC 727 different from LC 76 (Minimum Window Substring)? | LC 76 requires all characters of t to appear (any order, any frequency); LC 727 requires t as a subsequence (relative order must be preserved) |
