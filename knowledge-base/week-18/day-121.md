## DSA (1.5 hours)

### Pattern: Variable Sliding Window + Frequency Map (Minimum Window)
- Expand `right` to grow the window; shrink `left` once the window satisfies the condition. Track character frequencies with a hash map. Use a `have` counter (distinct chars meeting required frequency) vs `required` (total distinct chars needed in window). Only update `have` when `window[c]` exactly reaches `need[c]` — this avoids over-counting.
- Trigger condition: "Find the smallest subarray/substring containing all required elements or satisfying a character-count constraint."
- Time complexity: O(n + m) | Space complexity: O(k) where k = unique chars in t

### Pattern: Monotonic Deque Sliding Window (Window Maximum)
- Maintain a deque of indices in strictly decreasing order of their corresponding values. Before appending index `i`, pop all back indices whose values are ≤ `nums[i]` — they can never be the maximum while `nums[i]` is in the window. Pop front when it falls outside the window `[i - k + 1, i]`.
- Trigger condition: "Return the maximum (or minimum) in every contiguous subarray of size k."
- Time complexity: O(n) | Space complexity: O(k)

from collections import Counter

def minWindow(s: str, t: str) -> str:
    need = Counter(t)
    window = {}
    have, required = 0, len(need)

## System Design (1 hour)

### Topic: Twitter / X — System Overview & Scale Estimation

**Core components:**
- 400M DAU; ~500M tweets written per day; ~1B+ read requests per day (strongly read-heavy)
- Core services: Tweet Service (write path), Timeline Service (read path), User Graph Service (follows), Search Service, Notification Service, Media Service
- Primary data stores: PostgreSQL (users, follows, tweet metadata), Redis (precomputed timeline caches, hot counters), Cassandra (high-write event streams), S3 (media objects)
- Fan-out model: **hybrid** — push fan-out (write to follower Redis caches on tweet creation) for regular users; pull fan-out (compute at read time) for celebrities (>~50k followers)
- Scale challenge: a celebrity with 10M followers creates 10M Redis writes per tweet — pull model prevents this

**Key trade-offs:**
- Push fan-out = fast reads (pre-computed), expensive writes (fan-out storms for celebs)
- Pull fan-out = cheap writes, slow reads (must assemble timeline on demand)
- Hybrid uses push for standard users and merges celebrity tweets at read time in the Timeline Service

**Interview talking point:** "If asked how Twitter handles a celebrity tweeting to 10M followers, answer: Twitter uses a hybrid fan-out — regular users get push delivery (tweet written to each follower's Redis timeline cache immediately), but for high-follower accounts the tweet is stored once and injected into the requesting user's timeline at read time by the Timeline Service, avoiding 10M cache writes per tweet."

### Interview Tips

- Classify before coding: "Minimum window with all required chars → variable sliding window + frequency map."
- State the loop invariant explicitly: "When `have == required`, the window contains all characters of t at their required frequencies."
- Mention the monotonic deque pattern for LC 239 as a distinct technique from frequency maps.

### STAR Interview Framework

> **Variable Sliding Window + Frequency Map:** brute-force O(n²) → sliding window O(n + m) time, O(k) space

**S:** "Given strings s (length n) and t (length m). Naive O(n²) substring check with frequency comparison fails at n=10^5."
**T:** "Need O(n + m) by using a `have` counter that tracks satisfied character requirements."
**A (60%):**
1. *Classify:* "Variable sliding window: expand right; shrink left only when `have == required`."
2. *Init:* "`need = Counter(t); window = {}; have = 0; required = len(need)`."
3. *Loop/Step:* "Expand: increment window[c]; if window[c] == need[c], increment have. Shrink while have == required: record window, remove s[left], decrement have if needed."
4. *Termination:* "Minimum recorded window is the answer."
5. *Gotcha:* "`have` increments only at the exact threshold — extra copies of c beyond need[c] don't trigger it. Missing this causes have to overshoot and the window never shrinks."
**R:** "O(n + m) time, O(k) space where k = unique chars in t. Handles n=10^5 in microseconds."

**Alternatives & why not:**
| Alternative | Use when | Why not here |
|------------|----------|-------------|
| Brute force O(n²) substring scan | n ≤ 1000, m ≤ 50 | Quadratic — fails at n=10^5 |
| Regex matching | Unordered char set matching | No frequency awareness; far slower |

## Flashcards

| Q | A |
|---|---|
| How does the `have` counter avoid over-counting in Minimum Window Substring? | `have` only increments when `window[c]` exactly reaches `need[c]`; extra occurrences beyond `need[c]` don't trigger an increment |
| What invariant does the monotonic deque maintain in Sliding Window Maximum? | Indices stored in decreasing order of their values; deque front always holds the index of the current window's maximum element |
| When do you pop from the back of the monotonic deque? | When `nums[new_i] >= nums[dq[-1]]`; the back element can never be the window max while `nums[new_i]` is present |
| What is the time complexity of Minimum Window Substring and why is it O(n + m)? | Right pointer traverses `s` once (O(n)); left pointer also moves at most n times (O(n)); building `need` from `t` is O(m) — total O(n + m) |
| What causes the `left` pointer to stop moving in Minimum Window Substring? | When `have < required` again after removing `s[left]` from the window; window is no longer valid so we expand right |
