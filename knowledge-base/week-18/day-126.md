## DSA (1.5 hours)

### Pattern: Sliding Window with Deferred Effect Tracking (Bit Flips)
- For LC 995, flipping a window of length k centered at index i affects all bits i through i+k-1. Instead of actually flipping (O(n·k)), track cumulative flips with a difference array or a running flip count. At each position, check the current bit XOR the current flip count's parity to determine effective value. If it would be 0 (needs a flip), perform a "virtual flip" and increment the flip counter; decrement when the flip's window expires.
- Trigger condition: "Apply operations to contiguous windows with delayed expiry; avoid re-processing each window element."
- Time complexity: O(n) | Space complexity: O(n) for difference array, or O(k) for a deque

### Pattern: Contribution Count (Character Uniqueness in Substrings)
- For LC 828, instead of iterating over all O(n²) substrings, count how many substrings in which each character is unique. For character `s[i]`, find `prev[i]` (previous occurrence of same char) and `next[i]` (next occurrence). The number of substrings where `s[i]` is the only occurrence = `(i - prev[i]) * (next[i] - i)`. Sum over all indices.
- Trigger condition: "Aggregate property (uniqueness, frequency) over all substrings — direct O(n²) iteration is too slow."
- Time complexity: O(n) | Space complexity: O(1) per character (26 letters)

## System Design (1 hour)

### Topic: Twitter / X — Notification Service

**Core components:**
- Notification types: push (mobile APNs/FCM), email, in-app; each has different latency SLAs (push: <5s, email: <60s, in-app: eventual)
- Event ingestion: notification events published to a dedicated Kafka topic (`notification-events`); consumers fan out to per-channel workers (push worker, email worker, in-app worker)
- Notification Service workers: pull from Kafka, batch events per user, check user preferences (do-not-disturb, notification settings), deduplicate (user may get 100 likes in 5 min → one batched notification), then call APNs/FCM gateway
- User preference store: Redis hash `user:{id}:notif_prefs` for fast lookup; backed by DB for durability
- Deduplication and batching: sliding time window (e.g., 5 min) per (user, notification_type) pair; if X events arrive in the window, send one summary notification ("5 people liked your tweet")
- At-least-once delivery: Kafka consumer commits offset only after successful push; failed pushes are retried with exponential backoff; dead-letter queue for permanently failed deliveries

**Key trade-offs:**
- Push vs pull for in-app notifications: pull (user polls on app open) is simple but stale; push via WebSocket or Server-Sent Events gives real-time updates at higher server cost
- Deduplication granularity: per-user per-type batching reduces notification fatigue but delays individual notifications; configurable per notification type
- APNs/FCM reliability: external dependencies with their own SLAs; always assume possible failure and retry with backoff; token expiry must be handled (device re-registration)

**Interview talking point:** "If asked how Twitter prevents notification spam when a tweet goes viral, answer: The Notification Service uses a time-window deduplication strategy — events for the same (user, notification_type) pair are aggregated in a 5-minute window. If more than a threshold number arrive, a single batched notification is sent ('100+ people liked your tweet'). This is implemented as a sliding-window counter in Redis with TTL expiry on the counter key."

### STAR Interview Framework

> **Deferred Effect Tracking via Difference Array:** brute-force O(nk) → O(n) time, O(n) space

**S:** "Given binary array and flip window k. Apply minimum flips to make all elements 1. Naive: flip k elements per operation = O(nk) total."
**T:** "Need O(n) by tracking flip effects with a difference array instead of applying each flip physically."
**A (60%):**
1. *Classify:* "Sliding window with deferred effect: flip_diff[i+k] ^= 1 marks where each flip's effect expires."
2. *Init:* "`flip_diff = [0]*(n+1); flip_count = 0; result = 0`."
3. *Loop/Step:* "XOR flip_count with flip_diff[i] to apply expired flips. Compute effective bit = nums[i] ^ flip_count. If effective==0: if i+k>n return -1; result++; flip_count^=1; flip_diff[i+k]^=1."
4. *Termination:* "Return result after processing all n positions."
5. *Gotcha:* "Check i+k>n BEFORE incrementing result — if the required window extends past the array, the answer is -1."
**R:** "O(n) time, O(n) space. Handles n=10^5 instantly vs O(nk) which can be O(n²) when k=n/2."

**Alternatives & why not:**
| Alternative | Use when | Why not here |
|------------|----------|-------------|
| Brute force: flip k elements directly | k is small constant | O(nk) = O(n²) when k is large |
| Deque tracking flip windows | Prefer O(k) space over O(n) | Both O(n); difference array is simpler to implement |

| In LC 995, why use a difference array instead of actually flipping each element? | Flipping updates O(k) elements per operation = O(nk) total; difference array marks only the start and end of each flip's effect, reducing to O(n) with a running XOR |
