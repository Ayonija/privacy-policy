**Problem 1 — Revision (10 min): LC 337 House Robber III**

Expected solution (write from memory):

```python
def rob(root):
    def dp(node):
        if not node: return (0, 0)
        lr, ls = dp(node.left)
        rr, rs = dp(node.right)
        rob_this = node.val + ls + rs
        skip_this = max(lr, ls) + max(rr, rs)
        return (rob_this, skip_this)
    return max(dp(root))
```

**Self-grade criteria:**
- [ ] Returns tuple (rob, skip) — not a single value
- [ ] rob_this uses `ls` and `rs` (skip children) — not `lr` and `rr`
- [ ] skip_this takes `max(lr,ls)` per child — independent choices
- [ ] Base case returns (0,0) for None
- [ ] Final answer is `max(dp(root))` not `dp(root)[0]`

**Problem 2 — Hard (25 min): LC 698 Partition to K Equal Sum Subsets**

Expected solution:

```python
def canPartitionKSubsets(nums, k):
    total = sum(nums)
    if total % k != 0: return False
    target = total // k
    nums.sort(reverse=True)
    if nums[0] > target: return False
    n = len(nums)
    dp = [-1] * (1 << n)
    dp[0] = 0
    for mask in range(1 << n):
        if dp[mask] == -1: continue
        for i in range(n):
            if mask & (1 << i): continue
            new_mask = mask | (1 << i)
            new_sum = dp[mask] + nums[i]
            if new_sum <= target:
                dp[new_mask] = new_sum % target
    return dp[(1 << n) - 1] == 0
```

**Self-grade criteria:**
- [ ] Early exits: total % k != 0, nums[0] > target
- [ ] dp[mask] = running partial sum (not count of complete subsets)
- [ ] new_sum % target resets to 0 when a subset completes
- [ ] Condition is `new_sum <= target` (not `< target`)
- [ ] Final check: dp[full] == 0

**Problem 3 — Hard (25 min): LC 837 New 21 Game**

Expected solution:

```python
def new21Game(n, k, maxPts):
    if k == 0 or n >= k + maxPts - 1: return 1.0
    dp = [0.0] * (n + 1)
    dp[0] = 1.0
    window_sum = 1.0
    result = 0.0
    for i in range(1, n + 1):
        dp[i] = window_sum / maxPts
        if i < k:
            window_sum += dp[i]
        else:
            result += dp[i]
        if i >= maxPts:
            window_sum -= dp[i - maxPts]
    return result
```

**Self-grade criteria:**
- [ ] Edge case: k==0 returns 1.0; n >= k+maxPts-1 returns 1.0
- [ ] dp[0] = 1.0 (starting state)
- [ ] window_sum starts at 1.0 (= dp[0])
- [ ] Add dp[i] to window when i < k (player continues)
- [ ] Subtract dp[i-maxPts] when i >= maxPts (window slide)
- [ ] Accumulate result only for i >= k

## System Design (30 min)

### Full Mock: "Design an in-memory cache for a distributed system handling 10M QPS"

**Interviewer prompt:** Design an in-memory cache layer for a product that serves 10M requests per second. The dataset is 200GB. 5% of keys receive 80% of traffic. SLA: p99 < 5ms. What is your design?

---

**Step 1: Clarify requirements (2 min)**
- Read-heavy vs. write-heavy? → Read-heavy (90% reads, 10% writes)
- Strong consistency or eventual? → Eventual acceptable for product catalog; strong for user balances
- Single datacenter or global? → Single region for now
- Item TTL? → Varies: product catalog 5 min; user session 30 min

**Step 2: Capacity estimation (2 min)**
- 200GB dataset → single Redis node (256GB max) barely fits
- 10M QPS × ~100 bytes average payload → 1GB/s throughput → exceeds single node (~100K-500K QPS per Redis node)
- Decision: Redis Cluster with sharding required

**Step 3: Cluster design (8 min)**
```
10M QPS / 300K QPS per node = ~34 nodes → use 8 primaries (+ 8 replicas) with room to scale
200GB / 8 = 25GB per shard — fits in a single node (128GB nodes)

Hash slots: 16384 slots / 8 primaries = ~2048 slots per primary
Client library: lettuce (Java) or redis-py-cluster (Python) for slot routing
```

**Step 4: Eviction policy (5 min)**
- Base policy: allkeys-lru (temporal locality for product catalog)
- Hot key problem (5% keys → 80% traffic): ~10 keys receive ~100K-500K QPS each
  - Solution: local in-process cache (e.g., Caffeine in Java) per app server
  - TTL: 1 second (acceptable staleness for product info)
  - Each app server absorbs ~10K QPS/hot key locally; 100 app servers → 1M QPS removed from Redis

**Step 5: Cache invalidation (5 min)**
- Product catalog: write-around + TTL (5 min). Writes go to DB directly; cache expires naturally
- User balances: write-through (must be consistent)
- Event-driven for high-frequency updates: Debezium CDC → Kafka → cache delete consumer

**Step 6: Consistent hashing for sharding (5 min)**
- Redis Cluster uses hash slots (explicit); for custom cache, consistent hashing ring with 150 virtual nodes
- On node add: only 1/n fraction of keys remapped
- Monitoring: cluster_stats_messages_sent, hit_rate per shard, p99 latency

**Step 7: Failure handling (3 min)**
- Each primary has 1 replica (async replication)
- Automatic failover in ~15s if primary unreachable
- Risk: async replication → writes since last sync are lost during failover
- Mitigation: client-side retry for writes + idempotent writes where possible

**Full mock answer:**
*"I'd deploy Redis Cluster with 8 primaries and 8 replicas. 200GB / 8 shards = 25GB per shard, fits in 128GB nodes. 10M QPS / 8 shards = 1.25M QPS per shard. For the hot key problem — 5% of keys driving 80% of traffic — I'd add local Caffeine caches on each app server with 1-second TTL. For eviction, allkeys-lru by default, switching to allkeys-lfu if I observe high miss rates on newly popular items. Cache invalidation: write-around + 5-minute TTL for catalog, write-through for balances, CDC via Debezium for anything requiring near-real-time consistency."*

| Write the rerooting DP formula for Sum of Distances in Tree (LC 834). | ans[child] = ans[parent] - count[child] + (n - count[child]). count[child] subtree nodes get 1 closer; n-count[child] nodes get 1 farther. Two DFS passes: first computes count[] and ans[0]; second propagates to all nodes. |
| In Bitmask DP, write the subset enumeration idiom and state its complexity. | `sub = mask; while sub > 0: process(sub); sub = (sub-1) & mask`. Each element is either: not in mask, in mask but not sub, or in both → O(3^n) total across all masks. |
| In TSP bitmask DP (LC 943), what is the transition and what is precomputed? | Precompute overlap[i][j] = longest suffix of words[i] matching prefix of words[j]. Transition: dp[mask\|(1<<nxt)][nxt] = min(dp[mask][last] + len(words[nxt]) - overlap[last][nxt]) over all last in mask. |
| In digit DP, what does the `tight` flag control and what triggers it becoming False? | tight=True: prefix exactly matches limit — current digit constrained to ≤ limit[pos]. Becomes False when we choose a digit strictly less than limit[pos] — all subsequent digits are free (0..9). Once False, it stays False. |
| In game theory DP, when do you use score-based dp (int) vs. win/lose dp (bool)? | Score-based: when the value collected matters (e.g., who scores more). bool: when only winning/losing matters and moves don't accumulate score (e.g., stone removal games). Rule: if the problem asks "who wins" without score tracking → bool; if asks "by how much" or "what is Player 1's maximum score" → int. |
| In XOR Trie, why is the greedy "take opposite bit" strategy globally optimal? | MSB carries the most weight (2^31 vs. lower bits). Setting the highest achievable XOR bit first gives the maximum possible contribution before any lower bit is decided. No future bit decision can compensate for losing a higher bit. Hence greedy MSB→LSB is globally optimal. |
| What is the sliding window optimization in New 21 Game (LC 837) and what is its invariant? | window_sum = Σ dp[j] for j ∈ [max(0, i-maxPts), i-1]. Maintained in O(1): add dp[i] when i<k (this value can transition to future scores); subtract dp[i-maxPts] when i≥maxPts (this value falls out of reach). dp[i] = window_sum / maxPts. |
| In consistent hashing, what is the formula for expected keys remapped when adding one server to an n-server ring? | K/n where K = total keys, n = current number of servers. Only the new server's successor node is affected. On average, 1/(n+1) fraction of the successor's keys migrate. With virtual nodes, this fraction is distributed evenly across all successor virtual nodes. |
| In LFU Cache, write the _update(key) method logic in pseudocode. | freq = key_freq[key]; key_freq[key] = freq+1; remove key from freq_keys[freq]; if freq_keys[freq] is empty and freq == min_freq: min_freq += 1; add key to freq_keys[freq+1]. |
| What are the two passes in rerooting DP and what does each compute? | Pass 1 (dfs1, post-order rooted at 0): compute count[i] = subtree size at node i; compute ans[0] = sum of all distances from root 0. Pass 2 (dfs2, top-down from 0): use rerooting formula to compute ans[i] for all i from ans[parent]. Total O(n). |
