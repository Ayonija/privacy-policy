## DSA (1.5 hours)

### Pattern: Heap-Guided Two Pointer on Multiple Sorted Lists
- For LC 632, maintain one pointer per sorted list. Use a min-heap to find the global minimum efficiently. Simultaneously track the global maximum as pointers advance. The range `[min_val, max_val]` is valid when every list has at least one element in it. Advance the pointer of the list containing the minimum to try to shrink the range.
- Trigger condition: "Find a range/window that covers at least one element from each of k sorted lists."
- Time complexity: O(n * k * log k) | Space complexity: O(k)

### Pattern: Binary Search on Answer + Two Pointer Verification
- For LC 719, binary search on the answer (the distance value). For a given candidate distance `D`, count how many pairs `(i, j)` satisfy `nums[j] - nums[i] ≤ D` using a sliding window (two pointers on the sorted array). If count ≥ k, the answer is ≤ D; else > D.
- Trigger condition: "K-th smallest/largest in a collection of pairwise values — direct enumeration is O(n²); binary search on the value space + efficient counting reduces to O(n log n)."
- Time complexity: O(n log n + n log W) where W = max - min | Space complexity: O(1)

```python
# LC 632 — Smallest Range Covering Elements from K Lists
import heapq

def smallestRange(nums: list) -> list:
    # Initialize heap with (value, list_index, element_index)
    heap = []
    global_max = float('-inf')
    for i, lst in enumerate(nums):
        heapq.heappush(heap, (lst[0], i, 0))
        global_max = max(global_max, lst[0])

    best = [heap[0][0], global_max]

    while True:
        min_val, list_idx, elem_idx = heapq.heappop(heap)
        # TODO: update best range if global_max - min_val is smaller
        # TODO: if list_idx has no more elements, break (range must include all lists)
        next_elem_idx = elem_idx + 1
        if next_elem_idx == len(nums[list_idx]):
            break
        next_val = nums[list_idx][next_elem_idx]
        global_max = max(global_max, next_val)
        heapq.heappush(heap, (next_val, list_idx, next_elem_idx))

    return best


# LC 719 — Find K-th Smallest Pair Distance
def smallestDistancePair(nums: list, k: int) -> int:
    nums.sort()
    lo, hi = 0, nums[-1] - nums[0]

    def count_pairs_le(mid: int) -> int:
        # sliding window: count pairs (i, j) where nums[j] - nums[i] <= mid
        count = left = 0
        for right in range(len(nums)):
            while nums[right] - nums[left] > mid:
                left += 1
            count += right - left
        return count

    while lo < hi:
        mid = (lo + hi) // 2
        if count_pairs_le(mid) >= k:
            hi = mid
        else:
            lo = mid + 1
    return lo
```

## System Design (1 hour)

### Topic: Twitter / X — User Graph & Follow Relationships

**Core components:**
- Follow relationship: directed graph — `(follower_id, followee_id, created_at)`; stored in a dedicated Follows table (sharded by `follower_id` for "who do I follow" queries; secondary index or separate table for "who follows me" for fan-out)
- Graph storage options: relational (PostgreSQL) works for most users; dedicated graph DB (Neo4j) or adjacency list in Cassandra for deep graph traversal (mutual friends, suggestions)
- Follow table schema: `follower_id BIGINT, followee_id BIGINT, created_at TIMESTAMP` with composite PK and bidirectional index
- Friend/follow suggestions: computed offline (Spark batch jobs); result cached in Redis `user:{id}:suggestions` list; refreshed periodically
- Follower count: stored as a separate counter (Redis `INCR`/`DECR` or denormalized in users table) — never computed by `COUNT(*)` on follow table at request time

**Key trade-offs:**
- Sharding by `follower_id` is fast for "get my following list" (fan-out feed) but slow for "get all followers of user X" (used for fan-out-on-write); requires a second shard key or table
- Denormalizing follower count trades write amplification (update count on every follow/unfollow) for fast read of profile pages (no aggregation needed)
- For celebrity follow counts (100M+ followers), use Redis HyperLogLog or an approximate counter — exact count is expensive to maintain transactionally

**Interview talking point:** "If asked how to efficiently retrieve 'all followers of celebrity X' for fan-out purposes, answer: Maintain two separate lookup tables — follows_by_follower (sharded by follower_id) for feed assembly, and follows_by_followee (sharded by followee_id) for fan-out writes to followers. This denormalization doubles storage for follows but eliminates expensive cross-shard scans at write time."

## Flashcards

| Q | A |
|---|---|
| In LC 632, why do you advance the pointer of the list containing the minimum element? | To potentially shrink the current range; the minimum determines the range's lower bound, so replacing it with the next element in its list might raise the lower bound |
| What stops the loop in LC 632 (Smallest Range)? | When the list that contains the current minimum has no more elements — every remaining range must include that list, so no smaller valid range exists |
| What is the answer space for binary search in LC 719? | `[0, nums[-1] - nums[0]]` after sorting; the smallest possible distance is 0 (duplicates), the largest is max - min |
| In the `count_pairs_le(D)` sliding window for LC 719, why does `right - left` count correctly? | For each `right`, all indices from `left` to `right - 1` form a valid pair with `right`; there are `right - left` such pairs |
| What query pattern is avoided by denormalizing follower counts in Twitter's user table? | `SELECT COUNT(*) FROM follows WHERE followee_id = X` — a full index scan at request time; replaced by a pre-maintained counter |
