## DSA (1.5 hours)

### Pattern: Exactly-K via At-Most-K Subtraction
- Counting subarrays with **exactly** K distinct elements is equivalent to `atMost(k) - atMost(k - 1)`. The `atMost(k)` function uses a standard variable sliding window: expand right, shrink left when distinct count exceeds k. Each valid right pointer contributes `right - left + 1` subarrays.
- Trigger condition: "Count subarrays/substrings with exactly K distinct elements/occurrences."
- Time complexity: O(n) | Space complexity: O(k)

### Pattern: Fixed-Length Window with Multi-Word Matching
- For LC 30, maintain a window of exactly `len(words) * word_len` characters. Slide one `word_len` step at a time. Use a frequency map of `words` and check each word in the current window. Run this from every offset in `[0, word_len)` to cover all alignments.
- Trigger condition: "Fixed-size window matching a permutation of a word list."
- Time complexity: O(n * k) where k = word length | Space complexity: O(num_words)

# LC 992 — Subarrays with K Different Integers
from collections import defaultdict

def subarraysWithKDistinct(nums: list, k: int) -> int:
    def atMost(k: int) -> int:
        count = defaultdict(int)
        left = 0
        result = 0
        for right in range(len(nums)):
            count[nums[right]] += 1
            while len(count) > k:
                # TODO: shrink from left; remove from count if freq hits 0
                left += 1
            result += right - left + 1
        return result

    return atMost(k) - atMost(k - 1)


# LC 30 — Substring with Concatenation of All Words
from collections import Counter

def findSubstring(s: str, words: list) -> list:
    if not s or not words:
        return []
    word_len = len(words[0])
    num_words = len(words)
    window_size = word_len * num_words
    word_count = Counter(words)
    result = []

    for offset in range(word_len):  # all possible starting alignments
        left = offset
        seen = defaultdict(int)
        matched = 0
        for right in range(offset, len(s) - word_len + 1, word_len):
            word = s[right:right + word_len]
            if word in word_count:
                seen[word] += 1
                if seen[word] == word_count[word]:
                    matched += 1

    return result

## System Design (1 hour)

### Topic: Twitter / X — Tweet Write Path & Storage Schema

**Core components:**
- Tweet write path: client → API gateway → Tweet Service → message queue (Kafka) → async fan-out workers → follower timeline caches
- Tweet table schema: `tweet_id (BIGINT PK, Snowflake ID), user_id (BIGINT FK), content (VARCHAR 280), media_ids (ARRAY), created_at (TIMESTAMP), reply_to_tweet_id, retweet_of_tweet_id, like_count, retweet_count`
- Snowflake IDs: 64-bit IDs = timestamp (41 bits) + datacenter (5) + machine (5) + sequence (12); time-sortable, globally unique without coordination
- Async fan-out: after tweet is persisted, Kafka event triggers fan-out service to write tweet_id into each follower's Redis timeline list (LPUSH + LTRIM to cap at ~800 entries)
- Sharding: tweets table sharded by `user_id`; timeline cache sharded by `follower_user_id`

**Key trade-offs:**
- Synchronous fan-out blocks the write API until all followers are updated — unacceptable at scale; async fan-out via Kafka decouples write latency from follower count
- Storing full tweet content vs storing only tweet_id in timeline cache: store only tweet_id + author_id and hydrate tweet content on read (cheaper fan-out, extra read-time lookup)
- `created_at` index on tweets table enables efficient range scans for user timelines but adds write overhead

**Interview talking point:** "If asked what happens when a user tweets, answer: The tweet is persisted to the tweets table with a Snowflake ID (guarantees time-ordering without central coordination). A Kafka event is then published; fan-out workers consume it asynchronously and push the tweet_id into each follower's Redis timeline list. This decouples API write latency from fan-out cost — the user sees the tweet immediately even while fan-out completes in the background."

## Flashcards

| Q | A |
|---|---|
| What is the "exactly K" sliding window trick? | `count(exactly K) = atMost(K) - atMost(K - 1)`; implement a helper that counts subarrays with at most K distinct elements |
| In the `atMost(k)` helper, why does `right - left + 1` count subarrays correctly? | Every subarray ending at `right` and starting anywhere from `left` to `right` satisfies the at-most-k constraint; there are `right - left + 1` such subarrays |
| Why does LC 30 require iterating over `word_len` different starting offsets? | Words align only at multiples of `word_len` from a given start; without iterating offsets `0..word_len-1`, you miss concatenations not aligned to index 0 |
| What is a Snowflake ID and why does Twitter use it for tweet IDs? | 64-bit ID encoding timestamp + datacenter + machine + sequence; provides globally unique, time-sortable IDs without central coordination or locking |
| Why does Twitter use async fan-out via Kafka instead of synchronous fan-out? | Synchronous fan-out would make write API latency proportional to follower count (10M followers = 10M writes before returning); async decouples API response from fan-out completion |
