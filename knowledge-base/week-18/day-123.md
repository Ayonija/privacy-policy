## DSA (1.5 hours)

### Pattern: Two Pointer with Running Max (Trapping Rain Water)
- Water trapped at index `i` = `min(max_left[i], max_right[i]) - height[i]`. Precompute max heights from both sides, or use two-pointer approach: maintain `left_max` and `right_max`; move the pointer with the smaller max inward (that side's contribution is bounded by the smaller max). No precomputation array needed — O(1) space.
- Trigger condition: "Amount of water/area trapped or bounded between elements; depends on minimum of two boundaries."
- Time complexity: O(n) | Space complexity: O(1) with two pointers

### Pattern: Binary Search on Answer Space (Median of Two Sorted Arrays)
- Partition both arrays such that all elements on the left side ≤ all elements on the right side, and left half size == right half size (or +1 for odd total). Binary search on the partition point of the smaller array. Check: `A[partition_A - 1] ≤ B[partition_B]` and `B[partition_B - 1] ≤ A[partition_A]`. Adjust binary search bounds accordingly.
- Trigger condition: "Find kth element across two sorted arrays in O(log(min(m, n))) without merging."
- Time complexity: O(log(min(m, n))) | Space complexity: O(1)

def findMedianSortedArrays(nums1: list, nums2: list) -> float:
    # Ensure nums1 is the smaller array
    if len(nums1) > len(nums2):
        nums1, nums2 = nums2, nums1
    m, n = len(nums1), len(nums2)
    half = (m + n) // 2
    lo, hi = 0, m

    while lo <= hi:
        partition_a = (lo + hi) // 2
        partition_b = half - partition_a

        left_a  = nums1[partition_a - 1] if partition_a > 0 else float('-inf')
        right_a = nums1[partition_a]     if partition_a < m else float('inf')

## System Design (1 hour)

### Topic: Twitter / X — Timeline Feed Generation (Fan-out Strategies)

**Core components:**
- Timeline Service assembles a user's home feed from: (1) precomputed Redis timeline cache, (2) tweets from followed celebrities fetched at read time, (3) promoted/ad content injected
- Redis timeline cache: sorted set or list per user storing the 800 most recent tweet IDs; `ZRANGEBYSCORE` or `LRANGE` retrieves them in O(log n + k)
- Fan-out-on-write worker: consumes Kafka tweet events; for each follower of the author (if follower count < threshold), appends tweet_id to follower's Redis timeline; caps list at 800 with `LTRIM`
- Fan-out-on-read path: when user requests feed, Timeline Service queries: `SELECT tweet_id FROM tweets WHERE user_id IN (celebrity_ids I follow) ORDER BY created_at DESC LIMIT 20` — then merges with cached timeline
- Merge step: k-way merge of celebrity tweets + cached timeline by `created_at` (like merge-k-sorted-lists)

**Key trade-offs:**
- Cache size cap (800 tweets): inactive users ("cold start") miss old tweets when they return; handled by falling back to DB query if cache is stale
- Consistency: fan-out is async, so a follower's timeline may lag by seconds after a tweet is posted — acceptable for a social feed (eventual consistency)
- Timeline assembly latency target: ≤ 100ms p99; requires celebrity tweet lists to be pre-fetched or cached separately, not computed on-demand

**Interview talking point:** "If asked how Twitter assembles a home timeline, answer: The Timeline Service performs a k-way merge — it pulls the user's precomputed Redis timeline list (most recent 800 tweet IDs from normal follows) and merges it with a real-time DB query for celebrity tweets the user follows. Celebrity tweets are merged at read time to avoid the fan-out write storm. The result is sorted by timestamp and the top N tweets are returned."

## Flashcards

| Q | A |
|---|---|
| In the two-pointer solution for Trapping Rain Water, why is it safe to process the side with the smaller running max? | Water at that side is bounded by its own max (which is smaller than the other side's max); we know the min boundary without checking the other side |
| What is the partition invariant in Median of Two Sorted Arrays? | `left_a ≤ right_b` AND `left_b ≤ right_a`; all elements in left halves ≤ all elements in right halves, and total left size = ⌊(m+n)/2⌋ |
| How do you handle the case where partition_a = 0 in LC 4? | `left_a = -infinity` (no left partition element exists in nums1); similarly `right_a = +infinity` when `partition_a = m` |
| Why binary search on the smaller array in LC 4? | Ensures O(log(min(m, n))) time; partition_b is derived from partition_a so searching on the smaller array bounds all valid partitions |
| In Trapping Rain Water (two-pointer), when do you update `left_max` vs add water? | Update `left_max` when `height[left] >= left_max`; add `left_max - height[left]` to water when `height[left] < left_max` |
